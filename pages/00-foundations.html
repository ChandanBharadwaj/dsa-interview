Foundations: How to Think About DSA

Big-O intuition, the Java collections you already use, and the pattern-recognition mindset that makes every later topic click.

Phase 0 · Week 1 Read this first — everything builds on it

Before you start

Why this topic exists. Interviews don't test whether you can write Java — you've done that for 10 years. They test whether you can predict the cost of code before running it, and whether you can recognize which known pattern a new problem secretly is. This page gives you the vocabulary (Big-O) and the mental map (data structure ↔ Java class ↔ cost) that every other page relies on.

Prerequisites

None — this is the starting point. You only need everyday Java: loops, arrays, HashMap, List.

Interview relevance: stating time and space complexity unprompted — and comparing approaches by cost — is expected in nearly every interview.

How this runbook builds incrementally

Each phase depends only on the ones before it: Foundations → array patterns (the highest-frequency interview questions) → sorting & binary search → linked structures → recursion & trees → graphs → dynamic programming → advanced topics → a capstone showing how real systems (Google Maps, Netflix recommendations, Git) compose all of it. Don't skip ahead: graph BFS is easy if tree BFS is instinct; DP is learnable if recursion is instinct.

Core concepts

1. Big-O: counting work, not seconds

Big-O answers one question: as the input gets bigger, how fast does the work grow? Constants and hardware don't matter; the shape of growth does.

ClassNameIf n = 1,000,000Typical source
O(1)constant1 stepHashMap get/put, array index, stack push/pop
O(log n)logarithmic~20 stepsbinary search, balanced-tree (TreeMap) ops, heap push/pop
O(n)linear1,000,000 stepsone pass over the input; building a HashMap
O(n log n)linearithmic~20,000,000 stepsgood sorting (merge/quick/Tim), n heap operations
O(n²)quadratic10¹² steps — too slownested loops over the same input
O(2ⁿ)exponentialastronomicaltrying every subset; unmemoized recursion
Interview rule of thumb. Around 10⁸ simple operations per second is a safe mental budget. If n ≤ 10⁵ and your idea is O(n²) (10¹⁰ ops), it will be "correct but too slow" — the interviewer is waiting for you to notice and reach for a better tool. Build the habit: check n, budget the operations.

2. Reading complexity straight off the code

Counting work — four shapes you'll see everywhere

// O(n): one pass, constant work per element
int max = Integer.MIN_VALUE;
for (int x : arr) {            // runs n times
    max = Math.max(max, x);    // O(1) work each time
}

// O(n^2): for EACH element, scan all elements again
for (int i = 0; i < n; i++)
    for (int j = i + 1; j < n; j++)
        check(arr[i], arr[j]);         // runs ~n*n/2 times

// O(log n): the problem HALVES every iteration
while (lo <= hi) {
    int mid = lo + (hi - lo) / 2;      // half the range remains
    ...
}

// O(n log n): n elements, each paying an O(log n) price
PriorityQueue<Integer> pq = new PriorityQueue<>();
for (int x : arr) pq.offer(x);         // n pushes * O(log n) each
Reading recursive costs off the recursion tree. When code calls itself, draw the tree of calls and multiply levels × work per level. T(n) = 2T(n/2) + O(n) — two half-size children, linear work to combine — has log n levels each doing n total work: O(n log n) (merge sort). T(n) = T(n/2) + O(1) is one child, constant work: O(log n) (binary search). T(n) = 2T(n−1) + O(1) doubles the call count per level for n levels: O(2ⁿ) (naive Fibonacci). The general rule: if per-level work grows or shrinks geometrically, the biggest level dominates; if it's constant per level, multiply by the level count. The formalized version is the master theorem — page 10 applies it to real recursions.

3. Space complexity & the classic trade

Space complexity counts extra memory beyond the input. The single most common interview move is trading space for time: spend O(n) memory on a HashMap/HashSet to turn an O(n²) "search inside a loop" into O(n).

The trade in action: Two Sum (LeetCode 1), the most famous interview question

// Brute force: O(n^2) time, O(1) space — check every pair
public int[] twoSumBrute(int[] nums, int target) {
    for (int i = 0; i < nums.length; i++)
        for (int j = i + 1; j < nums.length; j++)
            if (nums[i] + nums[j] == target) return new int[]{i, j};
    return new int[]{};
}

// HashMap: O(n) time, O(n) space — one pass
public int[] twoSum(int[] nums, int target) {
    // value -> index of every number we've already walked past
    Map<Integer, Integer> seen = new HashMap<>();
    for (int i = 0; i < nums.length; i++) {
        int need = target - nums[i];        // the partner we're hoping exists
        if (seen.containsKey(need))         // O(1) lookup instead of an O(n) rescan
            return new int[]{seen.get(need), i};
        seen.put(nums[i], i);               // remember current number for the future
    }
    return new int[]{};
}

4. Amortized cost (why ArrayList.add is "O(1)")

ArrayList occasionally doubles its backing array (an O(n) copy), but doubling happens so rarely that the average cost per add stays constant — that's amortized O(1). Same idea behind HashMap resizing and StringBuilder growth. If an interviewer asks "worst case?", the honest answer is: a single add can be O(n), but any sequence of n adds is O(n) total.

Senior interviews sometimes push one level deeper: prove the amortized bound. There are three standard ways, and each one is a two-sentence argument for the doubling array:

  • Aggregate method — total the whole sequence. n appends trigger copies of size 1, 2, 4, …, up to n; the geometric series sums to less than 2n, so n appends cost under 3n work total — O(1) each on average. One inequality, done.
  • Accounting (banker's) method — overcharge cheap operations and bank the difference. Charge every append 3 credits: 1 pays for its own write, 1 is saved to move itself at the next resize, 1 is saved to move one element from the older half that already spent its savings. Every copy at resize time is pre-paid, so no operation is ever in debt.
  • Potential method — the same idea as a formula. Define Φ = 2·size − capacity; each cheap append raises Φ by 2, and a resize's O(n) copy is exactly cancelled by Φ collapsing back toward zero. Amortized cost = actual cost + ΔΦ stays constant. This is the accounting method with the bank balance made explicit — and the tool that generalizes to harder structures (the two-stack queue on page 09, splay trees).

One honest detail worth volunteering: the doubling story is the textbook version, but the JDK's ArrayList actually grows by 1.5× (old + (old >> 1) in current JDKs). The proof survives — any fixed factor > 1 gives a geometric series and amortized O(1); growing by a constant amount instead (say +10) does not, degrading n appends to O(n²). Why 1.5 and not 2? A gentler factor wastes less memory (at most ~50% slack instead of ~100%) and plays nicer with allocators: with growth below the golden ratio, the sum of previously freed blocks can eventually be reused for the next allocation, which a strict doubler never manages. HashMap does double — but for a different reason: it needs power-of-two table sizes so bucket indexing can be a bit-mask (page 01).

5. Worst, average, amortized, expected — four different promises

"It's O(1)" hides which kind of O(1). Interviewers at the senior level listen for the qualifier, because each one is a different contract:

GuaranteeWhat it promisesJava example
Worst caseevery single operation, no exceptionsTreeMap.get O(log n) — the strongest promise here
Amortizedany sequence averages out; single ops may spikeArrayList.add, ArrayDeque push, StringBuilder.append
Expectedaverage over randomness (hash spread / coin flips), for any inputHashMap.get O(1) expected, O(log n) worst per bucket since Java 8
Average caseaverage over an assumed input distribution — weakest, distrust itquicksort O(n log n) "on random inputs" (page 06)

The practical difference is latency versus throughput. Amortized O(1) is a throughput promise: a million appends cost a million-ish units of work — but one unlucky append eats an O(n) resize pause. Batch jobs don't care; a p99-latency-sensitive request path does, which is why real systems that can't tolerate spikes either pre-size their collections (new ArrayList<>(expected), new HashMap<>(expected * 4 / 3 + 1)) or resize incrementally — Redis rehashes its hash tables a few buckets per operation precisely to smear that pause out. Saying "amortized O(1), and I'd pre-size it on a hot path" is a senior-level sentence.

6. Lower bounds: knowing when to stop optimizing

Big-O bounds an algorithm from above; Big-Ω bounds the problem from below — "no correct algorithm can beat this". Three arguments cover nearly every interview mention:

  • Ω(n) — you must read the input. Adversary argument: if an algorithm skips even one element, an adversary sets that element to whatever flips the answer, so the algorithm is wrong on some input. This is why "find the max" or "sum the array" can never beat linear — and why an O(n) solution to such a problem is not "slow", it's optimal.
  • Ω(n log n) — comparison sorting. A sorting algorithm that only compares elements is a binary decision tree; it must distinguish all n! input orderings, so the tree needs n! leaves and hence height at least log₂(n!) ≈ n log₂ n. Any comparison sort is Ω(n log n) in the worst case — the full proof, and why counting/radix sorts legally "cheat" past it, lives on page 06.
  • Conjectured barriers. Some bounds aren't proven, just battle-tested: no strongly subquadratic algorithm for 3SUM is known, and dozens of geometry problems are "3SUM-hard" — as fast as 3SUM or no faster. In an interview, O(n²) for 3Sum (page 02) plus one sentence about the barrier is a complete, confident answer.
Why this matters live. Matching your upper bound to the problem's lower bound is the professional way to stop: "this is O(n), the problem is Ω(n) because every element must be read, so we're done." It converts "I can't think of anything faster" into a proof.

Your Java collections are already data structures

You've used these for a decade — here's what they are underneath, which is exactly what interviews test:

Habit to build now. In interviews say the structure, not just the class: "I'll use a hash map for O(1) lookups", "a min-heap to always grab the smallest". It shows you chose the structure, not the Java default.

The table above is the Big-O story; the memory story is what separates candidates who've profiled real JVMs. Every Java object carries a header (~12–16 bytes in current JDKs with compressed references), so a boxed Integer spends roughly 16 bytes to hold 4 bytes of payload — an ArrayList<Integer> costs about 4–5× the memory of an int[], with each value hiding behind a pointer. A LinkedList node is worse still: header + prev + next + element references means ~24 bytes of overhead per element, scattered wherever the allocator put them.

That scattering is the real killer. CPUs fetch memory in ~64-byte cache lines, so scanning an int[] gets 16 values per fetch and the hardware prefetcher streams the next line before you ask; following node.next pointers lands somewhere unpredictable, and every hop risks a cache miss costing ~100× a cached read. This is why ArrayList beats LinkedList in practice even for workloads where the Big-O table says they tie, why ArrayDeque (a circular array) is the JDK's own recommended stack/queue, and why "O(1) insert in the middle!" is a trap answer — finding the middle of a linked list already cost O(n) pointer chases. One sentence of this — "arrays win on cache locality; Big-O ignores the memory hierarchy" — reads as a decade of experience. The full hardware argument continues on page 08.

Where this shows up in real systems

Latency budgets at scale

A feed request touching 10⁹ rows can't afford O(n) scans. Databases keep B-tree and hash indexes (O(log n) / O(1) lookup) precisely because of the growth curves above. Big-O is the language of every performance review you'll ever read.

Java itself

HashMap converts long collision buckets into red-black trees (Java 8+) to cap the worst case at O(log n); Arrays.sort picks dual-pivot quicksort for primitives but TimSort for objects (stability). The JDK is applied DSA — page 06 digs in.

Recommendation systems

Netflix-style pipelines are a chain of exactly these tools: hashing for similarity lookups, heaps for top-K ranking, graphs for "users like you". The capstone (page 25) maps each stage back to a pattern you'll learn here.

Caches everywhere

Redis, CDN edges, and your service's in-memory cache are hash tables with eviction policies — LRU is a HashMap + doubly linked list (page 24). The trade "O(n) memory to make reads O(1)" runs the modern internet.

Practice: read the complexity

No LeetCode yet — first make cost-reading automatic. State time and space for each, then check.

warm-up 1. A single loop summing an array of n ints.
warm-up 2. Nested loops printing every pair (i, j) with i < j.
warm-up 3. A loop that does n = n / 2 until n is 0.
think 4. Sorting an array, then binary-searching it 50 times.
think 5. Building a HashSet from array A, then checking every element of array B against it.
think 6. String concatenation with += inside a loop of n iterations.
Answers with reasoning
  1. O(n) time, O(1) space — n iterations, constant work, one accumulator.
  2. O(n²) time — about n²/2 pairs; Big-O drops the ½.
  3. O(log n) time — halving until 0 takes log₂n steps. Halving ⇒ log. Always.
  4. O(n log n) — sort costs O(n log n); 50 searches cost 50·log n, a constant multiple of log n. The sort dominates.
  5. O(|A| + |B|) time, O(|A|) space — this is the space-for-time trade you'll use constantly.
  6. O(n²) time — each += copies the whole string built so far (1+2+…+n chars). Fix: StringBuilder, which is amortized O(1) per append ⇒ O(n) total. A very common Java-specific interview probe.

Pitfalls

  • Don't say "O(2n)" or "O(n/2)". Constants are dropped: both are O(n). Nested-but-shrinking loops (pairs) are still O(n²).
  • Two different inputs get two variables: processing lists of sizes n and m is O(n + m) (or O(n·m) if nested) — never collapse them into one n.
  • HashMap is O(1) average. Saying "average, with a treeified O(log n) worst case in Java 8+" earns senior-level credit.
  • Recursion isn't free space. The call stack counts: recursing n deep is O(n) space even with no allocations. (Big theme from page 10 on.)
  • Finish every solution with its costs: "time O(…), space O(…) because …". It becomes a reflex within a week of practice.
  • Integer overflow: int tops out at ~2.1·10⁹ — sums and products need long. Cheap point lost constantly.

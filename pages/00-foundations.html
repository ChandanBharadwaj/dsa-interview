Foundations: How to Think About DSA

Big-O intuition, the Java collections you already use, and the pattern-recognition mindset that makes every later topic click.

Phase 0 · Week 1 Read this first — everything builds on it

Before you start

Why this topic exists. Interviews don't test whether you can write Java — you've done that for 10 years. They test whether you can predict the cost of code before running it, and whether you can recognize which known pattern a new problem secretly is. This page gives you the vocabulary (Big-O) and the mental map (data structure ↔ Java class ↔ cost) that every other page relies on.

Prerequisites

None — this is the starting point. You only need everyday Java: loops, arrays, HashMap, List.

Interview relevance: stating time and space complexity unprompted — and comparing approaches by cost — is expected in nearly every interview.

How this runbook builds incrementally

Each phase depends only on the ones before it: Foundations → array patterns (the highest-frequency interview questions) → sorting & binary search → linked structures → recursion & trees → graphs → dynamic programming → advanced topics → a capstone showing how real systems (Google Maps, Netflix recommendations, Git) compose all of it. Don't skip ahead: graph BFS is easy if tree BFS is instinct; DP is learnable if recursion is instinct.

Core concepts

1. Big-O: counting work, not seconds

Big-O answers one question: as the input gets bigger, how fast does the work grow? Constants and hardware don't matter; the shape of growth does.

ClassNameIf n = 1,000,000Typical source
O(1)constant1 stepHashMap get/put, array index, stack push/pop
O(log n)logarithmic~20 stepsbinary search, balanced-tree (TreeMap) ops, heap push/pop
O(n)linear1,000,000 stepsone pass over the input; building a HashMap
O(n log n)linearithmic~20,000,000 stepsgood sorting (merge/quick/Tim), n heap operations
O(n²)quadratic10¹² steps — too slownested loops over the same input
O(2ⁿ)exponentialastronomicaltrying every subset; unmemoized recursion
Interview rule of thumb. Around 10⁸ simple operations per second is a safe mental budget. If n ≤ 10⁵ and your idea is O(n²) (10¹⁰ ops), it will be "correct but too slow" — the interviewer is waiting for you to notice and reach for a better tool. Build the habit: check n, budget the operations.

2. Reading complexity straight off the code

Counting work — four shapes you'll see everywhere

// O(n): one pass, constant work per element
int max = Integer.MIN_VALUE;
for (int x : arr) {            // runs n times
    max = Math.max(max, x);    // O(1) work each time
}

// O(n^2): for EACH element, scan all elements again
for (int i = 0; i < n; i++)
    for (int j = i + 1; j < n; j++)
        check(arr[i], arr[j]);         // runs ~n*n/2 times

// O(log n): the problem HALVES every iteration
while (lo <= hi) {
    int mid = lo + (hi - lo) / 2;      // half the range remains
    ...
}

// O(n log n): n elements, each paying an O(log n) price
PriorityQueue<Integer> pq = new PriorityQueue<>();
for (int x : arr) pq.offer(x);         // n pushes * O(log n) each

3. Space complexity & the classic trade

Space complexity counts extra memory beyond the input. The single most common interview move is trading space for time: spend O(n) memory on a HashMap/HashSet to turn an O(n²) "search inside a loop" into O(n).

The trade in action: Two Sum (LeetCode 1), the most famous interview question

// Brute force: O(n^2) time, O(1) space — check every pair
public int[] twoSumBrute(int[] nums, int target) {
    for (int i = 0; i < nums.length; i++)
        for (int j = i + 1; j < nums.length; j++)
            if (nums[i] + nums[j] == target) return new int[]{i, j};
    return new int[]{};
}

// HashMap: O(n) time, O(n) space — one pass
public int[] twoSum(int[] nums, int target) {
    // value -> index of every number we've already walked past
    Map<Integer, Integer> seen = new HashMap<>();
    for (int i = 0; i < nums.length; i++) {
        int need = target - nums[i];        // the partner we're hoping exists
        if (seen.containsKey(need))         // O(1) lookup instead of an O(n) rescan
            return new int[]{seen.get(need), i};
        seen.put(nums[i], i);               // remember current number for the future
    }
    return new int[]{};
}

4. Amortized cost (why ArrayList.add is "O(1)")

ArrayList occasionally doubles its backing array (an O(n) copy), but doubling happens so rarely that the average cost per add stays constant — that's amortized O(1). Same idea behind HashMap resizing and StringBuilder growth. If an interviewer asks "worst case?", the honest answer is: a single add can be O(n), but any sequence of n adds is O(n) total.

Where this shows up in real systems

Latency budgets at scale

A feed request touching 10⁹ rows can't afford O(n) scans. Databases keep B-tree and hash indexes (O(log n) / O(1) lookup) precisely because of the growth curves above. Big-O is the language of every performance review you'll ever read.

Java itself

HashMap converts long collision buckets into red-black trees (Java 8+) to cap the worst case at O(log n); Arrays.sort picks dual-pivot quicksort for primitives but TimSort for objects (stability). The JDK is applied DSA — page 06 digs in.

Recommendation systems

Netflix-style pipelines are a chain of exactly these tools: hashing for similarity lookups, heaps for top-K ranking, graphs for "users like you". The capstone (page 25) maps each stage back to a pattern you'll learn here.

Caches everywhere

Redis, CDN edges, and your service's in-memory cache are hash tables with eviction policies — LRU is a HashMap + doubly linked list (page 24). The trade "O(n) memory to make reads O(1)" runs the modern internet.

Practice: read the complexity

No LeetCode yet — first make cost-reading automatic. State time and space for each, then check.

warm-up 1. A single loop summing an array of n ints.
warm-up 2. Nested loops printing every pair (i, j) with i < j.
warm-up 3. A loop that does n = n / 2 until n is 0.
think 4. Sorting an array, then binary-searching it 50 times.
think 5. Building a HashSet from array A, then checking every element of array B against it.
think 6. String concatenation with += inside a loop of n iterations.
Answers with reasoning
  1. O(n) time, O(1) space — n iterations, constant work, one accumulator.
  2. O(n²) time — about n²/2 pairs; Big-O drops the ½.
  3. O(log n) time — halving until 0 takes log₂n steps. Halving ⇒ log. Always.
  4. O(n log n) — sort costs O(n log n); 50 searches cost 50·log n, a constant multiple of log n. The sort dominates.
  5. O(|A| + |B|) time, O(|A|) space — this is the space-for-time trade you'll use constantly.
  6. O(n²) time — each += copies the whole string built so far (1+2+…+n chars). Fix: StringBuilder, which is amortized O(1) per append ⇒ O(n) total. A very common Java-specific interview probe.

Pitfalls

  • Don't say "O(2n)" or "O(n/2)". Constants are dropped: both are O(n). Nested-but-shrinking loops (pairs) are still O(n²).
  • Two different inputs get two variables: processing lists of sizes n and m is O(n + m) (or O(n·m) if nested) — never collapse them into one n.
  • HashMap is O(1) average. Saying "average, with a treeified O(log n) worst case in Java 8+" earns senior-level credit.
  • Recursion isn't free space. The call stack counts: recursing n deep is O(n) space even with no allocations. (Big theme from page 10 on.)
  • Finish every solution with its costs: "time O(…), space O(…) because …". It becomes a reflex within a week of practice.
  • Integer overflow: int tops out at ~2.1·10⁹ — sums and products need long. Cheap point lost constantly.

Arrays & Hashing

Frequency counting, complement lookups, canonical keys, seen-sets — the five moves behind the most-asked interview questions, and what your trusty HashMap actually does underneath.

Phase 1 · Weeks 2–3 #1 most-asked interview category

Before you start

Why this topic exists. Almost every array question secretly asks the same thing: "you're about to write a nested loop — can you avoid it?" The answer, most of the time, is a hash table: spend O(n) memory to remember what you've seen so any "does X exist / how many X" question becomes O(1) instead of a rescan. Master five small patterns here and you can solve the single largest slice of the interview question pool.

Prerequisites

00 · Foundations — Big-O reflexes, the space-for-time trade, and the Java-collections-to-structure map.

What the interviewer is actually testing

  • Do you spot the O(n²) → O(n) upgrade? Writing brute force is fine — staying there is the fail.
  • Can you choose the right key? Group Anagrams is entirely about inventing a canonical key; Two Sum is about keying on the complement.
  • Do you know your tools honestly? "HashMap is O(1) average, and Java treeifies long buckets to O(log n)" is a senior answer; "HashMap is O(1)" is a junior one.
  • Edge-case discipline: duplicates, the empty array, an element pairing with itself.

How this builds on the previous topic

Foundations gave you the cost table and the space-for-time idea in the abstract. This page turns it into muscle memory: every pattern here is that one trade applied with a different choice of key and value. The next two pages (Two Pointers, Sliding Window) then show when you can get the same speedup without paying the O(n) memory.

Core concepts

1. The array cost model, in one breath

An array is a contiguous block of memory. Index math (base + i × size) makes arr[i] O(1); contiguity makes iteration cache-friendly and fast. But searching an unsorted array is O(n), and inserting/removing in the middle is O(n) because everything after the slot must shift. Interview array problems are rarely about arrays — they're about avoiding those O(n) scans.

2. What a hash table actually does

A hash table is just an array with a trick for picking the index: run the key through a hash function, take it modulo the array length, and that's the bucket where the entry lives. One arithmetic step replaces an O(n) search.

Two different keys can land in the same bucket — a collision. Java's HashMap handles this by chaining: each bucket holds a little linked list of entries, and a lookup walks the chain comparing keys with equals(). As long as the hash function spreads keys evenly and the table resizes when it fills up (Java doubles at 75% load), chains stay short and operations stay O(1) on average.

Java internals worth quoting. Since Java 8, when a single bucket's chain exceeds 8 entries (and the table has ≥ 64 buckets), HashMap treeifies it into a red-black tree, capping the worst-case lookup at O(log n) even under hostile keys. This also explains the golden rule you've lived by for years: equals() and hashCode() must agree, or your key hashes into one bucket and gets compared in another.

3. The five patterns of this page

  • Frequency counting — map value → count. One pass builds it; questions like Valid Anagram and Top K Frequent fall out.
  • Complement lookup — for each element, ask "have I seen the partner I need?" (Two Sum family).
  • Group by canonical key — design a key that is identical for all "equivalent" items (sorted chars, or a 26-count signature) and let the map group them.
  • Seen-set — a HashSet answers "have I encountered this before?" (Contains Duplicate) or "does neighbor x+1 exist?" (Longest Consecutive).
  • Cyclic sort — special case: values known to be in 1..n can be swapped to index value−1 in place, finding missing/duplicate numbers with O(1) extra space (First Missing Positive).
The reflex to train. The moment you write for (i) for (j) over the same array, pause and ask: "what would I need to remember from the first loop to delete the second one?" That memory — value→index, value→count, or just a set — is your hash table.

Pattern templates (Java)

Template 1 — frequency counting (map or int[26])

// General values: HashMap counter
Map<Integer, Integer> freq = new HashMap<>();
for (int x : nums)
    freq.merge(x, 1, Integer::sum);   // count[x]++ in one call

// Lowercase letters only: an array IS a hash table with a perfect hash
int[] count = new int[26];
for (char c : s.toCharArray())
    count[c - 'a']++;                 // 'a'->0 ... 'z'->25, no collisions ever

Template 2 — complement lookup (one pass, check-then-insert)

Map<Integer, Integer> seen = new HashMap<>();   // value -> index
for (int i = 0; i < nums.length; i++) {
    int need = target - nums[i];       // what would complete a pair?
    if (seen.containsKey(need)) { /* found: (seen.get(need), i) */ }
    seen.put(nums[i], i);              // insert AFTER checking: no self-pairing
}

Template 3 — group by canonical key

Map<String, List<String>> groups = new HashMap<>();
for (String s : words) {
    String key = canonical(s);         // sorted chars, count signature, etc.
    groups.computeIfAbsent(key, k -> new ArrayList<>()).add(s);
}

Template 4 — seen-set

Set<Integer> seen = new HashSet<>();
for (int x : nums)
    if (!seen.add(x)) { /* add() returns false => x is a duplicate */ }

Template 5 — cyclic sort (values 1..n, O(1) extra space)

// Put every value v at index v-1 by swapping until settled
for (int i = 0; i < nums.length; i++) {
    while (nums[i] >= 1 && nums[i] <= nums.length
           && nums[nums[i] - 1] != nums[i]) {   // target slot not settled yet
        int t = nums[nums[i] - 1];               // swap nums[i] into its home
        nums[nums[i] - 1] = nums[i];
        nums[i] = t;
    }
}
// Second pass: first index i with nums[i] != i+1 exposes a missing/duplicate

Decision cues:

  • "Have I seen…?", "count of…", "find a pair summing to…" → HashMap/HashSet.
  • "Group these strings/items by equivalence" → canonical key + map.
  • "Only lowercase letters" in the constraints → int[26] — say it out loud, it's free credit.
  • "O(1) extra space" + values in 1..n → cyclic sort (or bit tricks, page 22).
  • Sorted input, or follow-up "can you do it without extra memory?" → the two-pointer page is the intended answer.

Where this shows up in real systems

Database hash indexes & hash joins

Postgres hash indexes and every serious engine's hash join are Template 2 at terabyte scale: build a hash table on the smaller table's join key, then probe it once per row of the larger table — O(n+m) instead of the O(n·m) nested-loop join.

Dedup pipelines

Log ingestion (Splunk, Datadog) and ETL jobs drop duplicate events with exactly the seen-set pattern, keyed on an event hash. At scale the exact set becomes a Bloom filter — a lossy, bit-packed cousin you'll meet on page 24.

Redis

Redis is, at heart, a giant networked hash table — O(1) GET/SET — plus hash-field objects (HSET) implemented as… another hash table, with incremental rehashing so a resize never blocks the event loop. Your cache layer is this page in production.

Count-Min Sketch in analytics

Trending topics and heavy-hitter detection (e.g. in stream processors like Flink) use Count-Min Sketch: several small counter arrays indexed by different hash functions — frequency counting when the key space is too huge to store exactly.

★ Worked solution — Two Sum (LeetCode 1, easy)

Problem in one line: return the indices of the two numbers in nums that add up to target (exactly one answer exists).

Idea: walking the array, each element only needs one question answered: "has my partner target − nums[i] already gone by?" A map from value→index answers that in O(1). Checking before inserting the current element guarantees we never pair an element with itself.

public static int[] twoSum(int[] nums, int target) {
    // value -> index of every number we've already walked past
    Map<Integer, Integer> seen = new HashMap<>();
    for (int i = 0; i < nums.length; i++) {
        // The partner nums[i] needs to reach the target
        int need = target - nums[i];
        // O(1) average lookup — this replaces an O(n) rescan of the array
        if (seen.containsKey(need)) {
            // Partner was seen earlier, so its index comes first
            return new int[]{seen.get(need), i};
        }
        // Add AFTER the check — prevents pairing an element with itself
        // (e.g. target 6 must not match a single 3 twice)
        seen.put(nums[i], i);
    }
    // The problem guarantees exactly one answer, so we never get here
    return new int[]{-1, -1};
}

Complexity: time O(n) — one pass with O(1) work per element; space O(n) — the map can hold every element. Runnable mirror: code/01-arrays-hashing/TwoSum.java.

★ Worked solution — Group Anagrams (LeetCode 49, medium)

Problem in one line: given an array of strings, group the anagrams together.

Idea: invent a key that all anagrams share and nothing else does — their letter-count signature. Serialize the 26 counts to a string, use it as a map key, and the map does the grouping for you. Counting is O(L) per word, beating the O(L log L) sorted-characters key (both are accepted answers; mention both in an interview).

public static List<List<String>> groupAnagrams(String[] strs) {
    // canonical key -> all words that share it (i.e. one anagram group)
    Map<String, List<String>> groups = new HashMap<>();
    for (String s : strs) {
        // Canonical key = letter counts. Anagrams have identical counts,
        // and building this is O(L) vs O(L log L) for a sorted-chars key.
        int[] count = new int[26];
        for (char c : s.toCharArray()) count[c - 'a']++;
        // Serialize counts into a string key like "1#0#0#...".
        // The '#' separator matters: without it, counts (1,12) and (11,2)
        // could collide as "112".
        StringBuilder sb = new StringBuilder();
        for (int n : count) sb.append(n).append('#');
        String key = sb.toString();
        // computeIfAbsent creates the group list on first sight, then appends
        groups.computeIfAbsent(key, k -> new ArrayList<>()).add(s);
    }
    // The map's values ARE the answer — one list per anagram group
    return new ArrayList<>(groups.values());
}

Complexity: time O(n·L) for n words of average length L; space O(n·L). Runnable mirror: code/01-arrays-hashing/GroupAnagrams.java.

★ Worked solution — Longest Consecutive Sequence (LeetCode 128, medium)

Problem in one line: in an unsorted array, find the length of the longest run of consecutive integers (e.g. 1,2,3,4) — in O(n), so sorting is off the table.

Idea: put everything in a HashSet, then only begin counting a run at its true start — an element whose predecessor n−1 is absent. Every other element gets skipped by the guard, so despite the nested-looking while loop, each number is walked over at most once.

public static int longestConsecutive(int[] nums) {
    // Dump everything in a set: O(1) membership tests, duplicates collapse
    Set<Integer> set = new HashSet<>();
    for (int n : nums) set.add(n);

    int best = 0;
    for (int n : set) {
        // KEY INSIGHT: only count from the START of a run.
        // If n-1 exists, n is mid-run — skip it, the run's start will
        // count it later. This is what keeps the total work O(n).
        if (set.contains(n - 1)) continue;
        // n starts a run: walk forward while consecutive neighbors exist
        int len = 1;
        while (set.contains(n + len)) len++;
        best = Math.max(best, len);
    }
    return best;
}

Complexity: time O(n) — each element is touched at most twice (once by the outer loop, once inside exactly one run's walk); space O(n) for the set. Runnable mirror: code/01-arrays-hashing/LongestConsecutiveSequence.java.

Pitfalls & interview tips

  • Insert after checking in complement-lookup problems, or an element can "find" itself (Two Sum with target 6 and a single 3).
  • Integer boxing is not ==-safe: map.get(a) == map.get(b) compares references outside the −128..127 cache. Use .equals() or unbox to int. Classic Java-specific trap.
  • Never use a mutable object as a key (or an int[], whose hashCode is identity-based) — serialize count arrays to a String, as in Group Anagrams.
  • Say "average" when you say HashMap is O(1), and mention Java 8+ treeification to O(log n) — it's the difference between using the tool and knowing it.
  • Read the constraints for a small alphabet: "lowercase English letters" is an invitation to swap the HashMap for an int[26] and say why.
  • When asked for O(1) space on a 1..n array, the intended answer is cyclic sort (or index-sign tricks) — hashing is explicitly the thing being banned.

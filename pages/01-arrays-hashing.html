Arrays & Hashing

Frequency counting, complement lookups, canonical keys, seen-sets — the five moves behind the most-asked interview questions, and what your trusty HashMap actually does underneath.

Phase 1 · Weeks 2–3 #1 most-asked interview category

Before you start

Why this topic exists. Almost every array question secretly asks the same thing: "you're about to write a nested loop — can you avoid it?" The answer, most of the time, is a hash table: spend O(n) memory to remember what you've seen so any "does X exist / how many X" question becomes O(1) instead of a rescan. Master five small patterns here and you can solve the single largest slice of the interview question pool.

Prerequisites

00 · Foundations — Big-O reflexes, the space-for-time trade, and the Java-collections-to-structure map.

Interview relevance: the single most-asked category — the core skill tested is spotting the O(n²) → O(n) upgrade and choosing the right map key.

How this builds on the previous topic

Foundations gave you the cost table and the space-for-time idea in the abstract. This page turns it into muscle memory: every pattern here is that one trade applied with a different choice of key and value. The next two pages (Two Pointers, Sliding Window) then show when you can get the same speedup without paying the O(n) memory.

Core concepts

1. The array cost model, in one breath

An array is a contiguous block of memory. Index math (base + i × size) makes arr[i] O(1); contiguity makes iteration cache-friendly and fast. But searching an unsorted array is O(n), and inserting/removing in the middle is O(n) because everything after the slot must shift. Interview array problems are rarely about arrays — they're about avoiding those O(n) scans.

2. What a hash table actually does

A hash table is just an array with a trick for picking the index: run the key through a hash function, take it modulo the array length, and that's the bucket where the entry lives. One arithmetic step replaces an O(n) search.

Two different keys can land in the same bucket — a collision. Java's HashMap handles this by chaining: each bucket holds a little linked list of entries, and a lookup walks the chain comparing keys with equals(). As long as the hash function spreads keys evenly and the table resizes when it fills up (Java doubles at 75% load), chains stay short and operations stay O(1) on average.

Java internals worth quoting. Since Java 8, when a single bucket's chain exceeds 8 entries (and the table has ≥ 64 buckets), HashMap treeifies it into a red-black tree, capping the worst-case lookup at O(log n) even under hostile keys. This also explains the golden rule you've lived by for years: equals() and hashCode() must agree, or your key hashes into one bucket and gets compared in another.

3. The five patterns of this page

  • Frequency counting — map value → count. One pass builds it; questions like Valid Anagram and Top K Frequent fall out.
  • Complement lookup — for each element, ask "have I seen the partner I need?" (Two Sum family).
  • Group by canonical key — design a key that is identical for all "equivalent" items (sorted chars, or a 26-count signature) and let the map group them.
  • Seen-set — a HashSet answers "have I encountered this before?" (Contains Duplicate) or "does neighbor x+1 exist?" (Longest Consecutive).
  • Cyclic sort — special case: values known to be in 1..n can be swapped to index value−1 in place, finding missing/duplicate numbers with O(1) extra space (First Missing Positive).
The reflex to train. The moment you write for (i) for (j) over the same array, pause and ask: "what would I need to remember from the first loop to delete the second one?" That memory — value→index, value→count, or just a set — is your hash table.

Pattern templates (Java)

Template 1 — frequency counting (map or int[26])

// General values: HashMap counter
Map<Integer, Integer> freq = new HashMap<>();
for (int x : nums)
    freq.merge(x, 1, Integer::sum);   // count[x]++ in one call

// Lowercase letters only: an array IS a hash table with a perfect hash
int[] count = new int[26];
for (char c : s.toCharArray())
    count[c - 'a']++;                 // 'a'->0 ... 'z'->25, no collisions ever

Template 2 — complement lookup (one pass, check-then-insert)

Map<Integer, Integer> seen = new HashMap<>();   // value -> index
for (int i = 0; i < nums.length; i++) {
    int need = target - nums[i];       // what would complete a pair?
    if (seen.containsKey(need)) { /* found: (seen.get(need), i) */ }
    seen.put(nums[i], i);              // insert AFTER checking: no self-pairing
}

Template 3 — group by canonical key

Map<String, List<String>> groups = new HashMap<>();
for (String s : words) {
    String key = canonical(s);         // sorted chars, count signature, etc.
    groups.computeIfAbsent(key, k -> new ArrayList<>()).add(s);
}

Template 4 — seen-set

Set<Integer> seen = new HashSet<>();
for (int x : nums)
    if (!seen.add(x)) { /* add() returns false => x is a duplicate */ }

Template 5 — cyclic sort (values 1..n, O(1) extra space)

// Put every value v at index v-1 by swapping until settled
for (int i = 0; i < nums.length; i++) {
    while (nums[i] >= 1 && nums[i] <= nums.length
           && nums[nums[i] - 1] != nums[i]) {   // target slot not settled yet
        int t = nums[nums[i] - 1];               // swap nums[i] into its home
        nums[nums[i] - 1] = nums[i];
        nums[i] = t;
    }
}
// Second pass: first index i with nums[i] != i+1 exposes a missing/duplicate

Decision cues:

  • "Have I seen…?", "count of…", "find a pair summing to…" → HashMap/HashSet.
  • "Group these strings/items by equivalence" → canonical key + map.
  • "Only lowercase letters" in the constraints → int[26] — say it out loud, it's free credit.
  • "O(1) extra space" + values in 1..n → cyclic sort (or bit tricks, page 22).
  • Sorted input, or follow-up "can you do it without extra memory?" → the two-pointer page is the intended answer.

Where this shows up in real systems

Database hash indexes & hash joins

Postgres hash indexes and every serious engine's hash join are Template 2 at terabyte scale: build a hash table on the smaller table's join key, then probe it once per row of the larger table — O(n+m) instead of the O(n·m) nested-loop join.

Dedup pipelines

Log ingestion (Splunk, Datadog) and ETL jobs drop duplicate events with exactly the seen-set pattern, keyed on an event hash. At scale the exact set becomes a Bloom filter — a lossy, bit-packed cousin you'll meet on page 24.

Redis

Redis is, at heart, a giant networked hash table — O(1) GET/SET — plus hash-field objects (HSET) implemented as… another hash table, with incremental rehashing so a resize never blocks the event loop. Your cache layer is this page in production.

Count-Min Sketch in analytics

Trending topics and heavy-hitter detection (e.g. in stream processors like Flink) use Count-Min Sketch: several small counter arrays indexed by different hash functions — frequency counting when the key space is too huge to store exactly.

easy Contains Duplicate — LeetCode 217

The problem. You're given an integer array nums. Return true if any value appears at least twice, and false if every element is distinct.

Example. [1,2,3,1] → true (the 1 repeats). [1,2,3,4] → false. [1,1,1,3,3,4,3,2,4,2] → true.

Variation 1 — brute force: compare every pair · O(n²) time, O(1) space

public boolean containsDuplicateBrute(int[] nums) {
    for (int i = 0; i < nums.length; i++)
        for (int j = i + 1; j < nums.length; j++)   // every pair (i, j)
            if (nums[i] == nums[j]) return true;
    return false;
}

Correct, but 10⁵ elements ⇒ ~10¹⁰ comparisons. The nested loop exists only to answer "have I seen this value before?" — which is exactly what faster structures answer directly.

Variation 2 — sort first: duplicates become neighbors · O(n log n) time, O(1) extra

public boolean containsDuplicateSort(int[] nums) {
    Arrays.sort(nums);                            // duplicates now sit side by side
    for (int i = 1; i < nums.length; i++)
        if (nums[i] == nums[i - 1]) return true;  // just compare neighbors
    return false;
}

Better, and uses no extra memory — but it mutates the input and still pays for a full sort.

Variation 3 — seen-set (optimal) · O(n) time, O(n) space

public boolean containsDuplicate(int[] nums) {
    Set<Integer> seen = new HashSet<>();
    for (int x : nums)
        if (!seen.add(x))      // add() returns false if x was already present
            return true;       // -> x is the duplicate, stop immediately
    return false;
}

Takeaway. The three variations are the standard ladder for almost every problem on this page: pairs (n²) → sort (n log n) → hash (n). Time improved at the cost of O(n) memory.

easy Valid Anagram — LeetCode 242

The problem. Given two strings s and t, return true if t is an anagram of s — i.e. it uses exactly the same letters with exactly the same counts, just possibly in a different order.

Example. s="anagram", t="nagaram" → true. s="rat", t="car" → false (c and t differ). Different lengths → always false.

Variation 1 — sort both strings · O(n log n) time

public boolean isAnagramSort(String s, String t) {
    if (s.length() != t.length()) return false;   // cheap early exit
    char[] a = s.toCharArray(), b = t.toCharArray();
    Arrays.sort(a);                               // anagrams sort to the
    Arrays.sort(b);                               // exact same char sequence
    return Arrays.equals(a, b);
}

Variation 2 — one count array (optimal) · O(n) time, O(1) space

public boolean isAnagram(String s, String t) {
    if (s.length() != t.length()) return false;
    int[] count = new int[26];                    // 26 lowercase letters
    for (int i = 0; i < s.length(); i++) {
        count[s.charAt(i) - 'a']++;               // +1 for every letter of s
        count[t.charAt(i) - 'a']--;               // -1 for every letter of t
    }
    for (int c : count)
        if (c != 0) return false;                 // any imbalance -> not an anagram
    return true;
}

Takeaway. The +1/−1 trick folds two frequency maps into one array and one pass. The int[26] works because the constraint says lowercase letters; for full Unicode you'd fall back to a HashMap<Character,Integer> — same idea, bigger key space.

easy Two Sum — LeetCode 1

The problem. Given an array nums and an integer target, return the indices of the two different elements that add up to target. Exactly one answer exists, and you may not use the same element twice.

Example. nums=[2,7,11,15], target=9 → [0,1] because 2 + 7 = 9. nums=[3,2,4], target=6 → [1,2] (2+4 — note NOT [0,0]: the single 3 can't be used twice).

Variation 1 — brute force: try every pair · O(n²) time, O(1) space

public int[] twoSumBrute(int[] nums, int target) {
    for (int i = 0; i < nums.length; i++)
        for (int j = i + 1; j < nums.length; j++)
            if (nums[i] + nums[j] == target) return new int[]{i, j};
    return new int[]{};
}

Variation 2 — one-pass HashMap (optimal) · O(n) time, O(n) space

public int[] twoSum(int[] nums, int target) {
    // value -> index of every number we've already walked past
    Map<Integer, Integer> seen = new HashMap<>();
    for (int i = 0; i < nums.length; i++) {
        int need = target - nums[i];        // the partner that would complete a pair
        if (seen.containsKey(need))         // O(1) lookup replaces an O(n) rescan
            return new int[]{seen.get(need), i};
        seen.put(nums[i], i);               // insert AFTER checking -> no self-pairing
    }
    return new int[]{};                     // unreachable: an answer is guaranteed
}

Takeaway. Rather than searching forward for a partner, each element checks backward in O(1): "did my partner already pass by?" The check-before-insert order is what prevents an element pairing with itself. If the array were sorted, two pointers would solve it with O(1) memory — that's Two Sum II on the next page. Mirror: code/01-arrays-hashing/TwoSum.java.

easy Ransom Note — LeetCode 383

The problem. Can ransomNote be built from the letters of magazine, each magazine letter usable at most once? Both are lowercase.

Example. note "a", magazine "b" → false. note "aa", magazine "ab" → false (only one 'a' in stock). note "aa", magazine "aab" → true.

Count, then spend · O(n+m) time, O(1) space (26 counters)

public static boolean canConstruct(String ransomNote, String magazine) {
    int[] counts = new int[26];                 // lowercase letters only
    for (char c : magazine.toCharArray())
        counts[c - 'a']++;                      // stock the shelf
    for (char c : ransomNote.toCharArray())
        if (--counts[c - 'a'] < 0)              // spend; negative = shortage
            return false;
    return true;
}

Takeaway. The purest form of frequency counting: one array as inventory, decrement to consume. Valid Anagram (above) is this same problem with the extra demand that the inventory come out exactly empty. A fixed alphabet means an int[26] beats a HashMap — say that out loud in an interview, it's a free point. Mirror: code/01-arrays-hashing/RansomNote.java.

easy Majority Element — LeetCode 169

The problem. An element appears more than ⌊n/2⌋ times (guaranteed to exist). Find it. Follow-up: O(1) space.

Example. [3,2,3] → 3. [2,2,1,1,1,2,2] → 2.

Variation 1 — frequency map · O(n) time, O(n) space

public static int majorityByCounting(int[] nums) {
    Map<Integer, Integer> freq = new HashMap<>();
    for (int x : nums) {
        int c = freq.merge(x, 1, Integer::sum); // increment and read back
        if (c > nums.length / 2) return x;      // strictly more than half
    }
    throw new IllegalArgumentException("no majority element");
}

Variation 2 — Boyer-Moore voting (optimal) · O(n) time, O(1) space

public static int majorityElement(int[] nums) {
    int candidate = 0, count = 0;
    for (int x : nums) {
        if (count == 0) candidate = x;          // previous run cancelled out
        count += (x == candidate) ? 1 : -1;     // vote for or against
    }
    return candidate;                           // the true majority survives
}

Takeaway. Boyer-Moore is a pairing argument: every time the count hits zero, the discarded block contained the candidate and non-candidates in equal number — so it threw away at most as many majority copies as others. With more than n/2 copies, the majority can't be fully cancelled. Note it only works because a majority is guaranteed; without that guarantee you'd verify the candidate with one more pass. Mirror: code/01-arrays-hashing/MajorityElement.java.

easy Isomorphic Strings — LeetCode 205

The problem. Can s be transformed into t by a consistent letter substitution? Every occurrence of a letter must map to the same letter, and no two letters may map to the same letter.

Example. "egg"/"add" → true (e→a, g→d). "foo"/"bar" → false (o would need to be both a and r). "badc"/"baba" → false (d and c would both map to a).

Two maps — the substitution must be a bijection · O(n)

public static boolean isIsomorphic(String s, String t) {
    Map<Character, Character> fwd = new HashMap<>();
    Map<Character, Character> back = new HashMap<>();
    for (int i = 0; i < s.length(); i++) {
        char a = s.charAt(i), b = t.charAt(i);
        Character seenB = fwd.putIfAbsent(a, b);   // a's existing image, if any
        Character seenA = back.putIfAbsent(b, a);  // b's existing preimage
        if (seenB != null && seenB != b) return false;  // a mapped elsewhere before
        if (seenA != null && seenA != a) return false;  // b already claimed
    }
    return true;
}

Takeaway. The classic trap is checking only one direction — "badc"/"baba" passes a forward-only check. A consistent substitution is a bijection, and a bijection needs both maps (or one map plus a "used images" set). This is exactly how compilers check α-equivalence of variable renamings. Mirror: code/01-arrays-hashing/IsomorphicStrings.java.

easy Word Pattern — LeetCode 290

The problem. Does the space-separated sentence s follow pattern, with a bijection between pattern letters and whole words?

Example. "abba" / "dog cat cat dog" → true. "abba" / "dog cat cat fish" → false. "abba" / "dog dog dog dog" → false (a and b can't both be "dog").

Isomorphic Strings, letters ↔ words · O(n)

public static boolean wordPattern(String pattern, String s) {
    String[] words = s.split(" ");
    if (words.length != pattern.length()) return false;  // lengths must agree first

    Map<Character, String> letterToWord = new HashMap<>();
    Map<String, Character> wordToLetter = new HashMap<>();
    for (int i = 0; i < pattern.length(); i++) {
        char p = pattern.charAt(i);
        String w = words[i];
        String seenW = letterToWord.putIfAbsent(p, w);
        Character seenP = wordToLetter.putIfAbsent(w, p);
        if (seenW != null && !seenW.equals(w)) return false;  // letter reused differently
        if (seenP != null && seenP != p) return false;        // word already claimed
    }
    return true;
}

Takeaway. Same bijection discipline as Isomorphic Strings, one level up (letters ↔ words instead of letters ↔ letters). The two easy-to-miss cases: a length mismatch between pattern and sentence, and the reverse-map check. Comparing with .equals, not ==, matters now that values are Strings. Mirror: code/01-arrays-hashing/WordPattern.java.

easy Happy Number — LeetCode 202

The problem. Repeatedly replace n by the sum of the squares of its digits. If the process reaches 1, n is "happy"; otherwise it loops forever. Decide which.

Example. 19 → 1²+9²=82 → 68 → 100 → 1 → happy. 2 → 4 → 16 → 37 → … enters the cycle 4→16→37→58→89→145→42→20→4 → not happy.

Variation 1 — HashSet cycle detection · O(1)-ish per step

public static boolean isHappySet(int n) {
    Set<Integer> seen = new HashSet<>();
    while (n != 1 && seen.add(n))   // add() returns false on a repeat
        n = next(n);
    return n == 1;                  // stopped at 1, or on a proven cycle
}

private static int next(int n) {
    int sum = 0;
    while (n > 0) {
        int d = n % 10;             // peel the last digit
        sum += d * d;
        n /= 10;
    }
    return sum;
}

Variation 2 — Floyd's tortoise and hare · O(1) space

public static boolean isHappy(int n) {
    int slow = n, fast = next(n);
    while (fast != 1 && slow != fast) {
        slow = next(slow);          // one step
        fast = next(next(fast));    // two steps
    }
    return fast == 1;               // met at 1 = happy; met elsewhere = cycle
}

Takeaway. "Does this iteration ever repeat?" is cycle detection, whether the steps follow pointers or arithmetic. The set version is the honest first answer; Floyd removes the memory by running the same sequence at two speeds — the identical trick as Linked List Cycle on page 08, just with next() instead of .next. Mirror: code/01-arrays-hashing/HappyNumber.java.

easy Contains Duplicate II — LeetCode 219

The problem. Are there two equal elements at most k indices apart?

Example. [1,2,3,1], k=3 → true. [1,0,1,1], k=1 → true. [1,2,3,1,2,3], k=2 → false (the equal pairs are 3 apart).

A sliding set of the last k elements · O(n)

public static boolean containsNearbyDuplicate(int[] nums, int k) {
    Set<Integer> window = new HashSet<>();
    for (int i = 0; i < nums.length; i++) {
        if (i > k)
            window.remove(nums[i - k - 1]);   // evict what fell out of range
        if (!window.add(nums[i]))             // add() is false if already present
            return true;
    }
    return false;
}

Takeaway. Contains Duplicate (above) with a distance constraint — and the constraint turns the global set into a windowed set: membership over only the last k items. It's your first taste of the sliding-window idea formalized on page 03. (An alternative: a value→lastIndex map, comparing index gaps.) Mirror: code/01-arrays-hashing/ContainsDuplicateII.java.

easy Find All Numbers Disappeared in an Array — LeetCode 448

The problem. nums has n entries, each in [1, n]. Return every value in [1, n] that never appears — ideally O(1) extra space, treating the returned list as free.

Example. [4,3,2,7,8,2,3,1] → [5,6]. [1,1] → [2].

The array is its own hash table: mark by negation · O(n) time, O(1) space

public static List<Integer> findDisappearedNumbers(int[] nums) {
    for (int x : nums) {
        int idx = Math.abs(x) - 1;      // abs: x may already carry a mark
        if (nums[idx] > 0)
            nums[idx] = -nums[idx];     // mark "value idx+1 exists"
    }
    List<Integer> missing = new ArrayList<>();
    for (int i = 0; i < nums.length; i++)
        if (nums[i] > 0)                // never marked
            missing.add(i + 1);
    return missing;
}

Takeaway. When values are confined to [1, n], the indices themselves can act as hash buckets and the sign bit as the "present" flag — hashing with zero extra memory. This is the gentle version of the cyclic-sort idea that First Missing Positive (below) pushes to its hard extreme. Mirror: code/01-arrays-hashing/DisappearedNumbers.java.

medium Group Anagrams — LeetCode 49

The problem. Given an array of strings, group the anagrams together and return the groups (any order). E.g. "eat", "tea", "ate" all use letters {a,e,t} once each, so they form one group.

Example. ["eat","tea","tan","ate","nat","bat"] → [["eat","tea","ate"], ["tan","nat"], ["bat"]].

The insight. A map groups values that share a key — so invent a key that all anagrams share and nothing else does (a canonical key). Two standard choices:

Variation 1 — sorted-characters key · O(n · L log L)

public List<List<String>> groupAnagramsSortKey(String[] strs) {
    Map<String, List<String>> groups = new HashMap<>();
    for (String s : strs) {
        char[] chars = s.toCharArray();
        Arrays.sort(chars);                       // "tea" -> "aet", "ate" -> "aet"
        String key = new String(chars);           // all anagrams produce the same key
        groups.computeIfAbsent(key, k -> new ArrayList<>()).add(s);
    }
    return new ArrayList<>(groups.values());
}

Variation 2 — letter-count signature key (faster) · O(n · L)

public List<List<String>> groupAnagrams(String[] strs) {
    // canonical key -> all words that share it (i.e. one anagram group)
    Map<String, List<String>> groups = new HashMap<>();
    for (String s : strs) {
        // Counting is O(L) vs O(L log L) for sorting the characters
        int[] count = new int[26];
        for (char c : s.toCharArray()) count[c - 'a']++;
        // Serialize counts into a key like "1#0#0#...". The '#' separator
        // matters: without it, counts (1,12) and (11,2) could both read "112".
        StringBuilder sb = new StringBuilder();
        for (int n : count) sb.append(n).append('#');
        groups.computeIfAbsent(sb.toString(), k -> new ArrayList<>()).add(s);
    }
    return new ArrayList<>(groups.values());      // the map's values ARE the groups
}

Takeaway. Both are accepted everywhere; the count key wins asymptotically for long words. Never use the raw int[] as the map key — array hashCode() is identity-based, so equal counts would not collide. Mirror: code/01-arrays-hashing/GroupAnagrams.java.

medium Top K Frequent Elements — LeetCode 347

The problem. Given an integer array and a number k, return the k values that occur most often (any order). k is guaranteed valid and the answer is unique.

Example. nums=[1,1,1,2,2,3], k=2 → [1,2] (1 occurs 3×, 2 occurs 2×, 3 occurs 1×). nums=[1], k=1 → [1].

Variation 1 — count, then sort by frequency · O(n log n)

public int[] topKFrequentSort(int[] nums, int k) {
    Map<Integer, Integer> freq = new HashMap<>();
    for (int x : nums) freq.merge(x, 1, Integer::sum);       // value -> count

    // Sort the distinct values by their count, biggest first
    List<Integer> vals = new ArrayList<>(freq.keySet());
    vals.sort((a, b) -> freq.get(b) - freq.get(a));
    int[] res = new int[k];
    for (int i = 0; i < k; i++) res[i] = vals.get(i);
    return res;
}

Variation 2 — bucket by frequency (optimal) · O(n)

public int[] topKFrequent(int[] nums, int k) {
    Map<Integer, Integer> freq = new HashMap<>();
    for (int x : nums) freq.merge(x, 1, Integer::sum);

    // Key idea: a count can never exceed n, so use the COUNT as an array
    // index. bucket[c] = all values that occur exactly c times.
    List<Integer>[] bucket = new List[nums.length + 1];
    for (Map.Entry<Integer, Integer> e : freq.entrySet()) {
        int c = e.getValue();
        if (bucket[c] == null) bucket[c] = new ArrayList<>();
        bucket[c].add(e.getKey());
    }

    // Walk buckets from the highest count down, collecting k values
    int[] res = new int[k];
    int idx = 0;
    for (int c = nums.length; c >= 1 && idx < k; c--)
        if (bucket[c] != null)
            for (int v : bucket[c]) {
                res[idx++] = v;
                if (idx == k) break;
            }
    return res;
}

Takeaway. Sorting compares counts against each other (n log n); bucketing exploits that counts live in the small fixed range 1..n, so they can be used as indexes — the same idea as counting sort (page 06). A third variation — a size-k min-heap, O(n log k), best for streaming data — is worked on page 12.

medium Product of Array Except Self — LeetCode 238

The problem. Return an array answer where answer[i] is the product of every element of nums except nums[i]. You must run in O(n) and division is not allowed (it would break on zeros anyway).

Example. [1,2,3,4] → [24,12,8,6] (e.g. answer[0] = 2·3·4 = 24). [-1,1,0,-3,3] → [0,0,9,0,0].

The insight. The product of "everything except me" = (product of everything to my left) × (product of everything to my right). Both sides can be precomputed in one pass each.

Variation 1 — two extra arrays · O(n) time, O(n) space

public int[] productExceptSelfTwoArrays(int[] nums) {
    int n = nums.length;
    int[] left = new int[n], right = new int[n], ans = new int[n];
    left[0] = 1;                                    // nothing to the left of index 0
    for (int i = 1; i < n; i++)
        left[i] = left[i - 1] * nums[i - 1];        // product of nums[0..i-1]
    right[n - 1] = 1;                               // nothing to the right of the end
    for (int i = n - 2; i >= 0; i--)
        right[i] = right[i + 1] * nums[i + 1];      // product of nums[i+1..n-1]
    for (int i = 0; i < n; i++)
        ans[i] = left[i] * right[i];
    return ans;
}

Variation 2 — reuse the output array (optimal) · O(n) time, O(1) extra space

public int[] productExceptSelf(int[] nums) {
    int n = nums.length;
    int[] ans = new int[n];
    ans[0] = 1;
    for (int i = 1; i < n; i++)                 // pass 1: ans[i] = product of the left side
        ans[i] = ans[i - 1] * nums[i - 1];
    int right = 1;                              // running product of the right side
    for (int i = n - 1; i >= 0; i--) {
        ans[i] *= right;                        // left-product * right-product
        right *= nums[i];                       // extend the right side to include nums[i]
    }
    return ans;                                 // output array doesn't count as extra space
}

Takeaway. This is prefix products — the multiplicative twin of the prefix sums on page 04. Variation 2 folds the right-side array into a single running variable, a space squeeze you'll reuse in DP (page 18).

medium Longest Consecutive Sequence — LeetCode 128

The problem. Given an unsorted array, return the length of the longest run of consecutive integers present in it (the run's members can appear anywhere in the array). Required time: O(n).

Example. [100,4,200,1,3,2] → 4, because 1,2,3,4 are all present. [0,3,7,2,5,8,4,6,0,1] → 9 (0 through 8).

Variation 1 — sort and scan · O(n log n)

public int longestConsecutiveSort(int[] nums) {
    if (nums.length == 0) return 0;
    Arrays.sort(nums);
    int best = 1, len = 1;
    for (int i = 1; i < nums.length; i++) {
        if (nums[i] == nums[i - 1]) continue;       // skip duplicates
        len = (nums[i] == nums[i - 1] + 1) ? len + 1 : 1;  // extend or restart the run
        best = Math.max(best, len);
    }
    return best;
}

Simple, but the problem explicitly demands O(n) — sorting is the thing being ruled out.

Variation 2 — seen-set with run-start guard (optimal) · O(n)

public int longestConsecutive(int[] nums) {
    // Dump everything in a set: O(1) membership tests, duplicates collapse
    Set<Integer> set = new HashSet<>();
    for (int n : nums) set.add(n);

    int best = 0;
    for (int n : set) {
        // KEY INSIGHT: only count from the START of a run.
        // If n-1 exists, n is mid-run — skip it; the run's true start
        // will count it later. This guard keeps total work O(n).
        if (set.contains(n - 1)) continue;
        int len = 1;
        while (set.contains(n + len)) len++;   // walk the run upward
        best = Math.max(best, len);
    }
    return best;
}

Takeaway. The while loop looks nested but isn't quadratic: every element is walked by exactly one run (the one whose start owns it), so the total walking across all runs is n steps. Mirror: code/01-arrays-hashing/LongestConsecutiveSequence.java.

medium Encode and Decode Strings — LeetCode 271

The problem. Design encode(List<String>) → String and decode(String) → List<String> so any list of strings round-trips through a single string — including strings containing any character your format uses.

Example. ["hello","wor#ld","","12#34"] must decode back exactly. A naive comma-join dies on strings containing commas; there is no safe delimiter.

Length-prefix framing · O(total length)

public static String encode(List<String> strs) {
    StringBuilder sb = new StringBuilder();
    for (String s : strs)
        sb.append(s.length()).append('#').append(s);   // "5#hello4#worl…"
    return sb.toString();
}

public static List<String> decode(String data) {
    List<String> out = new ArrayList<>();
    int i = 0;
    while (i < data.length()) {
        int hash = data.indexOf('#', i);            // end of the length field
        int len = Integer.parseInt(data.substring(i, hash));
        out.add(data.substring(hash + 1, hash + 1 + len));  // exactly len chars —
        i = hash + 1 + len;                         // a '#' inside them is just data
    }
    return out;
}

Takeaway. The insight is that no delimiter can be safe when content is arbitrary — but a length is unambiguous: after reading "5#", the next 5 characters are data no matter what they contain. This is exactly how network protocols (HTTP's Content-Length, TCP framing, Redis's RESP) delimit messages. Empty strings ("0#") work for free. Mirror: code/01-arrays-hashing/EncodeDecodeStrings.java.

medium Detect Squares — LeetCode 2013

The problem. Design a class with add(point) (duplicates allowed) and count(q): how many axis-aligned squares of positive area can be formed using q and three previously added points?

Example. add (3,10), (11,2), (3,2); count((11,10)) → 1 — the square with corners (3,10),(11,2),(3,2),(11,10). Add (11,2) again → count((11,10)) → 2, because the duplicated corner participates in a second (identical) square.

Count map + iterate diagonal partners · add O(1), count O(points)

public class DetectSquares {
    // Multiset of points: duplicates multiply the number of squares.
    private final Map<String, Integer> counts = new HashMap<>();
    private final List<int[]> points = new ArrayList<>();

    public void add(int[] point) {
        counts.merge(point[0] + "," + point[1], 1, Integer::sum);
        points.add(point);
    }

    public int count(int[] q) {
        int total = 0;
        // Try every stored point p as the corner DIAGONAL to q. A diagonal
        // pins the square: the other two corners are forced.
        for (int[] p : points) {
            int dx = p[0] - q[0], dy = p[1] - q[1];
            if (dx == 0 || Math.abs(dx) != Math.abs(dy)) continue;  // not diagonal
            // remaining corners: (q.x, p.y) and (p.x, q.y) — multiply their counts
            total += counts.getOrDefault(q[0] + "," + p[1], 0)
                   * counts.getOrDefault(p[0] + "," + q[1], 0);
        }
        return total;
    }
}

Takeaway. The geometric trick: enumerate the diagonal partner, because a diagonal determines the square — then the two forced corners are O(1) hash lookups, and multiplying counts handles duplicates automatically. "Fix the piece that determines everything, look up the rest" is the same shape as Two Sum's complement lookup, upgraded to 2D. Mirror: code/01-arrays-hashing/DetectSquares.java.

hard First Missing Positive — LeetCode 41

The problem. Given an unsorted integer array (may contain zeros, negatives, duplicates), return the smallest positive integer that is not present. Required: O(n) time and O(1) extra space.

Example. [1,2,0] → 3. [3,4,-1,1] → 2. [7,8,9,11,12] → 1.

The insight. The answer must lie in 1..n+1 (n values can cover at most 1..n). So only values in 1..n matter — and each such value v has a natural home: index v−1.

Variation 1 — seen-set · O(n) time but O(n) space (violates the constraint)

public int firstMissingPositiveSet(int[] nums) {
    Set<Integer> set = new HashSet<>();
    for (int x : nums) set.add(x);
    for (int v = 1; v <= nums.length + 1; v++)  // answer is in 1..n+1
        if (!set.contains(v)) return v;
    return -1;                                   // unreachable
}

The natural first idea — write it, then note it breaks the O(1)-space requirement, which is the cue for cyclic sort.

Variation 2 — cyclic sort (optimal) · O(n) time, O(1) space

public int firstMissingPositive(int[] nums) {
    int n = nums.length;
    // Phase 1: swap every value v in 1..n to its home index v-1.
    for (int i = 0; i < n; i++) {
        // Keep swapping until nums[i] is out of range, or already home,
        // or its home already holds a correct copy (guards duplicates).
        while (nums[i] >= 1 && nums[i] <= n
               && nums[nums[i] - 1] != nums[i]) {
            int home = nums[i] - 1;             // where nums[i] belongs
            int t = nums[home];                 // classic swap
            nums[home] = nums[i];
            nums[i] = t;
        }
    }
    // Phase 2: the first index whose value disagrees exposes the answer.
    for (int i = 0; i < n; i++)
        if (nums[i] != i + 1) return i + 1;
    return n + 1;                               // 1..n all present -> answer is n+1
}

Takeaway. The while loop is amortized O(n) overall: every swap puts one value into its final home forever, and there are only n homes. Cyclic sort is the standard escape hatch whenever hashing is banned but values are confined to 1..n.

Pitfalls

  • Insert after checking in complement-lookup problems, or an element can "find" itself (Two Sum with target 6 and a single 3).
  • Integer boxing is not ==-safe: map.get(a) == map.get(b) compares references outside the −128..127 cache. Use .equals() or unbox to int. Classic Java-specific trap.
  • Never use a mutable object as a key (or an int[], whose hashCode is identity-based) — serialize count arrays to a String, as in Group Anagrams.
  • Say "average" when you say HashMap is O(1), and mention Java 8+ treeification to O(log n).
  • Read the constraints for a small alphabet: "lowercase English letters" is a cue to swap the HashMap for an int[26].
  • When asked for O(1) space on a 1..n array, the intended answer is cyclic sort (or index-sign tricks) — hashing is explicitly the thing being banned.

Two Pointers

Two indexes walking one array — converging from the ends, or reading and writing in place. The cheapest O(n²) → O(n) upgrade in the toolbox, and the technique sliding window grows out of.

Phase 1 · Weeks 2–3 Top-5 pattern; feeds sliding window

Before you start

Why this topic exists. Page 01's answer to "avoid the nested loop" was memory: a hash table. Two pointers is the answer when memory is banned or unnecessary — usually because the data is sorted. Order gives every comparison a direction ("too small → move left pointer up"), and each comparison permanently discards part of the search space. You examine n² pairs' worth of possibilities in n steps, with zero extra memory.

Prerequisites

  • 00 · Foundations — Big-O and the ops budget.
  • 01 · Arrays & Hashing — you should know the HashMap solution to Two Sum cold, because this page replaces it under sortedness.

Interview relevance: a top-5 pattern; the skill tested is justifying why each pointer move safely discards candidates.

How this builds on the previous topic

Arrays & Hashing solved "find a pair" by remembering everything in O(n) space. Here, sortedness lets you get the same answer in O(1) space — and when an interviewer asks the follow-up "now without the HashMap?", this page is the expected reply. The next page, Sliding Window, is two pointers moving in the same direction with state maintained between them.

Core concepts

1. Converging pointers: why sortedness lets you discard

Start with l at the first element and r at the last. In a sorted array, nums[r] is the largest partner nums[l] could ever have. So if nums[l] + nums[r] is still too small, no partner anywhere can rescue nums[l] — discard it forever with l++. Symmetrically, a too-big sum retires nums[r]. Every comparison kills one element's candidacy, so the loop ends after at most n steps having implicitly ruled out all ~n²/2 pairs.

That sentence — "each step discards a provably useless part of the search space" — is the same logic binary search uses (page 07); two pointers is its two-ended cousin.

2. Read/write pointers: editing in place

The second flavor has both pointers moving left to right: a fast read pointer scans every element, and a slow write pointer marks where the next "kept" element belongs. Everything left of write is finished output; everything between write and read is discarded. Remove Duplicates and Move Zeroes are this one idea. It's how you compact an array in one pass with O(1) space — and conceptually how compacting garbage collectors slide live objects over dead ones.

3. Fast & slow pointers (preview)

A third variant runs both pointers in the same direction at different speeds — fast moves two steps per slow's one. On linked structures this finds the middle node and detects cycles (Floyd's algorithm). It needs linked-list mechanics first, so the full treatment lives in 08 · Linked Lists; just file it under the same family for now.

Unsorted data? Converging pointers need order to justify the discard. If the input isn't sorted, either sort first (paying O(n log n) — fine when you'd sort anyway, as in 3Sum) or go back to page 01's HashMap.

Pattern templates (Java)

Template 1 — converging pointers on a sorted array

int l = 0, r = arr.length - 1;
while (l < r) {                    // strict <: a pair needs two distinct indexes
    long sum = (long) arr[l] + arr[r];   // long guards against int overflow
    if (sum == target) { /* found the pair */ }
    else if (sum < target) l++;    // arr[l] can't work with ANY partner: discard
    else r--;                      // arr[r] too big for everyone left: discard
}

Template 2 — read/write pointers (in-place filtering)

int write = 0;                          // next slot to fill; [0, write) = kept
for (int read = 0; read < arr.length; read++) {
    if (keep(arr[read])) {              // e.g. != previous, != 0, != val
        arr[write] = arr[read];         // safe: write <= read, never clobbers unread
        write++;
    }
}
// write is the new logical length

Template 3 — palindrome check (converging, comparing not summing)

int l = 0, r = s.length() - 1;
while (l < r) {
    // skip non-alphanumerics from either side (LeetCode 125 flavor)
    if (!Character.isLetterOrDigit(s.charAt(l))) { l++; continue; }
    if (!Character.isLetterOrDigit(s.charAt(r))) { r--; continue; }
    if (Character.toLowerCase(s.charAt(l)) !=
        Character.toLowerCase(s.charAt(r))) return false;  // mirror mismatch
    l++; r--;                            // matched: shrink from both ends
}
return true;

When to use what

TechniqueNeeds sorted?TimeExtra spaceBest for
Two pointers (converging)YesO(n)O(1)pair/triple sums, palindromes, max-area style problems
HashMap (page 01)NoO(n)O(n)unsorted "find the pair", need original indexes
Binary search (page 07)YesO(log n) per queryO(1)find ONE value/boundary, not a pair relation
Sliding window (page 03)NoO(n)O(1)–O(k)contiguous subarray/substring, pointers moving the SAME way

Decision cues:

  • "Sorted array" + "pair/two numbers that…" → converging pointers.
  • Unsorted + need indexes back → HashMap; sorting would scramble the indexes.
  • "In place", "O(1) extra space", "remove/compact" → read/write pointers.
  • "Contiguous subarray/substring with property" → sliding window, next page.
  • k-sum problems (3Sum, 4Sum) → sort once, fix anchors, finish with converging pointers.

Where this shows up in real systems

The merge step everywhere

Merging two sorted runs with one pointer per run is the heart of merge sort and of Java's own Arrays.sort for objects (TimSort merges naturally-occurring runs). Same-direction two pointers, production-grade.

Merge joins in databases

When Postgres or SQL Server picks a merge join, it walks two sorted inputs with exactly this pattern — advance whichever side's key is smaller — joining huge tables in O(n+m) without a hash table's memory bill.

Log stream reconciliation

Comparing two time-ordered streams — say, payment events vs. ledger entries to find mismatches — is two pointers over timestamps: advance the earlier stream, flag anything unmatched. Every reconciliation job you've seen does this.

Deduplication & compaction

Sorted-input dedup (sort -u, keeping the first of each equal run) is the read/write pattern; LSM-tree engines like RocksDB use the same idea during compaction to merge SSTables and drop shadowed keys.

easy Valid Palindrome — LeetCode 125

The problem. A string is a palindrome if, after removing every non-alphanumeric character and lowercasing, it reads the same forwards and backwards. Return whether s is one.

Example. "A man, a plan, a canal: Panama" → true (cleans to "amanaplanacanalpanama"). "race a car" → false ("raceacar" reversed is "racaecar"). " " → true (empty after cleaning).

Variation 1 — clean, reverse, compare · O(n) time, O(n) space

public boolean isPalindromeReverse(String s) {
    StringBuilder clean = new StringBuilder();
    for (char c : s.toCharArray())
        if (Character.isLetterOrDigit(c))
            clean.append(Character.toLowerCase(c));   // keep only a-z0-9, lowercased
    return clean.toString().equals(clean.reverse().toString());
}

Perfectly fine — but it builds two throwaway strings. The follow-up "without extra memory?" leads to:

Variation 2 — converging pointers (optimal) · O(n) time, O(1) space

public boolean isPalindrome(String s) {
    int l = 0, r = s.length() - 1;
    while (l < r) {
        // Skip characters that don't count, from either side
        if (!Character.isLetterOrDigit(s.charAt(l))) { l++; continue; }
        if (!Character.isLetterOrDigit(s.charAt(r))) { r--; continue; }
        // Compare the mirror pair, case-insensitively
        if (Character.toLowerCase(s.charAt(l)) !=
            Character.toLowerCase(s.charAt(r))) return false;
        l++; r--;                        // matched: shrink from both ends
    }
    return true;                         // pointers met -> every pair matched
}

Takeaway. A palindrome question is a mirror question, and mirrors are two pointers converging. The same skeleton returns for "expand around center" on page 14, just running outward instead of inward.

medium Two Sum II (sorted input) — LeetCode 167

The problem. Same as Two Sum, but the array is already sorted ascending and the answer must be the 1-indexed positions, using O(1) extra space (exactly one solution exists).

Example. numbers=[2,7,11,15], target=9 → [1,2]. numbers=[2,3,4], target=6 → [1,3] (2+4).

Variation 1 — binary-search each partner · O(n log n) time, O(1) space

public int[] twoSumBinarySearch(int[] numbers, int target) {
    for (int i = 0; i < numbers.length; i++) {
        int need = target - numbers[i];
        // Search only to the RIGHT of i so we never reuse the same element
        int lo = i + 1, hi = numbers.length - 1;
        while (lo <= hi) {
            int mid = lo + (hi - lo) / 2;
            if (numbers[mid] == need) return new int[]{i + 1, mid + 1};
            if (numbers[mid] < need) lo = mid + 1;
            else hi = mid - 1;
        }
    }
    return new int[]{};
}

Uses sortedness, but pays log n per element. Converging pointers use it better:

Variation 2 — converging pointers (optimal) · O(n) time, O(1) space

public int[] twoSum(int[] numbers, int target) {
    int l = 0, r = numbers.length - 1;   // start at the two extremes
    while (l < r) {
        int sum = numbers[l] + numbers[r];
        if (sum == target)
            return new int[]{l + 1, r + 1};     // 1-indexed positions
        if (sum < target) {
            // numbers[r] is the LARGEST partner available to numbers[l];
            // if even that is too small, numbers[l] can never work — discard it.
            l++;
        } else {
            // Symmetric: numbers[l] is the smallest partner available to
            // numbers[r]; too big means numbers[r] is done — discard it.
            r--;
        }
    }
    return new int[]{-1, -1};            // unreachable: a solution is guaranteed
}

Takeaway. The three Two Sums line up as a decision table: unsorted → HashMap O(n)/O(n) (page 01); sorted → two pointers O(n)/O(1); sorted + many queries for single values → binary search. Mirror: code/02-two-pointers/TwoSumII.java.

easy Remove Duplicates from Sorted Array — LeetCode 26

The problem. Given a sorted array, remove the duplicates in place so each value appears once, keeping relative order. Return the new logical length k; the first k slots must hold the answer. O(1) extra space required.

Example. [1,1,2] → k=2, array becomes [1,2,_]. [0,0,1,1,1,2,2,3,3,4] → k=5, array becomes [0,1,2,3,4,...].

Variation 1 — copy into a set/list · O(n) space (violates the constraint)

public int removeDuplicatesExtra(int[] nums) {
    // LinkedHashSet keeps insertion order while dropping repeats
    Set<Integer> set = new LinkedHashSet<>();
    for (int x : nums) set.add(x);
    int i = 0;
    for (int x : set) nums[i++] = x;     // copy the survivors back
    return set.size();
}

Works, but "in place / O(1) space" is the actual exercise. Enter read/write pointers:

Variation 2 — read/write pointers (optimal) · O(n) time, O(1) space

public int removeDuplicates(int[] nums) {
    int write = 1;                        // nums[0] is always kept
    for (int read = 1; read < nums.length; read++) {
        // Sorted input => duplicates are adjacent. Keep nums[read] only
        // if it differs from the last KEPT value.
        if (nums[read] != nums[write - 1]) {
            nums[write] = nums[read];     // safe: write <= read always
            write++;
        }
    }
    return write;                         // first `write` slots = deduped array
}

Takeaway. The invariant to hold in your head: [0, write) is finished output, [write, read) is discarded junk, [read, n) is unread. The same skeleton solves Move Zeroes (keep = "not zero") and Remove Element (keep = "!= val") — one template, three problems.

medium Container With Most Water — LeetCode 11

The problem. height[i] is a vertical wall at position i. Choose two walls so the container between them holds the most water: area = min(height[l], height[r]) × (r − l). Return that maximum area.

Example. [1,8,6,2,5,4,8,3,7] → 49: walls of height 8 (index 1) and 7 (index 8) give min(8,7) × 7 = 49. [1,1] → 1.

Variation 1 — brute force: every pair of walls · O(n²)

public int maxAreaBrute(int[] height) {
    int best = 0;
    for (int l = 0; l < height.length; l++)
        for (int r = l + 1; r < height.length; r++)
            best = Math.max(best,
                Math.min(height[l], height[r]) * (r - l));
    return best;
}

Variation 2 — converging pointers (optimal) · O(n) time, O(1) space

public int maxArea(int[] height) {
    int l = 0, r = height.length - 1;    // widest possible container first
    int best = 0;
    while (l < r) {
        // Water level is capped by the SHORTER wall; width is the gap
        int area = Math.min(height[l], height[r]) * (r - l);
        best = Math.max(best, area);
        // Move the shorter wall inward. Why it's safe: any container
        // keeping the shorter wall is narrower (width shrinks) AND still
        // capped by that same short wall — it can never beat what we just
        // measured. So nothing is lost by discarding it.
        if (height[l] < height[r]) l++;
        else r--;
    }
    return best;
}

Takeaway. This is the discard argument without sortedness: what justifies moving a pointer isn't order but the min() in the area formula — the shorter wall personally caps every container it could ever be part of. Mirror: code/02-two-pointers/ContainerWithMostWater.java.

medium 3Sum — LeetCode 15

The problem. Return all unique triples [a, b, c] from the array with a + b + c = 0. The same triple must not appear twice, even if the array holds repeated values.

Example. [-1,0,1,2,-1,-4] → [[-1,-1,2], [-1,0,1]] (note only one copy of [-1,0,1] although two −1s exist). [0,1,1] → []. [0,0,0] → [[0,0,0]].

Variation 1 — brute force: every triple + a dedup set · O(n³)

public List<List<Integer>> threeSumBrute(int[] nums) {
    Set<List<Integer>> res = new HashSet<>();      // set kills duplicate triples
    for (int i = 0; i < nums.length; i++)
        for (int j = i + 1; j < nums.length; j++)
            for (int k = j + 1; k < nums.length; k++)
                if (nums[i] + nums[j] + nums[k] == 0) {
                    List<Integer> t = Arrays.asList(nums[i], nums[j], nums[k]);
                    Collections.sort(t);            // canonical order for the set
                    res.add(t);
                }
    return new ArrayList<>(res);
}

O(n³) dies around n = 1000. The upgrade: fix one element, and the remaining task is exactly Two Sum II.

Variation 2 — sort + anchor + converging pointers (optimal) · O(n²)

public List<List<Integer>> threeSum(int[] nums) {
    Arrays.sort(nums);   // O(n log n) — enables two pointers AND duplicate skips
    List<List<Integer>> res = new ArrayList<>();
    for (int i = 0; i < nums.length - 2; i++) {
        // Sorted, so if the smallest of the three is positive,
        // no triple from here on can sum to zero — stop early.
        if (nums[i] > 0) break;
        // Same anchor value as last iteration would rebuild the same
        // triples — skip it (this is what makes the answer unique).
        if (i > 0 && nums[i] == nums[i - 1]) continue;
        // Solve Two Sum II on nums[i+1..end] with target -nums[i]
        int l = i + 1, r = nums.length - 1;
        while (l < r) {
            int sum = nums[i] + nums[l] + nums[r];
            if (sum < 0) l++;              // too small — need a bigger left value
            else if (sum > 0) r--;         // too big — need a smaller right value
            else {
                res.add(Arrays.asList(nums[i], nums[l], nums[r]));
                l++; r--;
                // Skip equal values so the same triple isn't emitted again
                while (l < r && nums[l] == nums[l - 1]) l++;
                while (l < r && nums[r] == nums[r + 1]) r--;
            }
        }
    }
    return res;
}

Takeaway. "k-sum" problems reduce one dimension at a time: fix an anchor, solve (k−1)-sum on the rest, with two pointers as the base case. Sorting costs O(n log n) but is dominated by the O(n²) sweep — and it's what makes duplicate-skipping a two-line affair instead of a hash set. Mirror: code/02-two-pointers/ThreeSum.java.

hard Trapping Rain Water — LeetCode 42

The problem. Bars of heights height[i] form a terrain. After rain, water collects in the dips. Return the total units of trapped water.

Example. [0,1,0,2,1,0,1,3,2,1,2,1] → 6. [4,2,0,3,2,5] → 9.

The insight. Water standing on top of bar i rises to min(tallest bar to my left, tallest bar to my right) — minus the bar's own height. Everything that follows is just computing those two maxima efficiently.

Variation 1 — per-bar scans · O(n²) time, O(1) space

public int trapBrute(int[] height) {
    int total = 0;
    for (int i = 0; i < height.length; i++) {
        int maxL = 0, maxR = 0;
        for (int j = 0; j <= i; j++) maxL = Math.max(maxL, height[j]);   // scan left
        for (int j = i; j < height.length; j++) maxR = Math.max(maxR, height[j]); // scan right
        total += Math.min(maxL, maxR) - height[i];   // water column above bar i
    }
    return total;
}

Variation 2 — precompute prefix maxima · O(n) time, O(n) space

public int trapPrefix(int[] height) {
    int n = height.length;
    if (n == 0) return 0;
    int[] maxL = new int[n], maxR = new int[n];
    maxL[0] = height[0];
    for (int i = 1; i < n; i++)             // tallest bar at or left of i
        maxL[i] = Math.max(maxL[i - 1], height[i]);
    maxR[n - 1] = height[n - 1];
    for (int i = n - 2; i >= 0; i--)        // tallest bar at or right of i
        maxR[i] = Math.max(maxR[i + 1], height[i]);
    int total = 0;
    for (int i = 0; i < n; i++)
        total += Math.min(maxL[i], maxR[i]) - height[i];
    return total;
}

The rescans became two prefix-max arrays (the pattern of page 04). One more squeeze removes the arrays entirely:

Variation 3 — converging pointers (optimal) · O(n) time, O(1) space

public int trap(int[] height) {
    int l = 0, r = height.length - 1;
    int maxL = 0, maxR = 0;              // tallest bar seen so far from each end
    int total = 0;
    while (l < r) {
        if (height[l] < height[r]) {
            // The KEY step: maxR' (the true right max for l) is at least
            // height[r] > height[l], so min(maxL, trueMaxR) = maxL here.
            // The left side's water is fully decided — settle it.
            maxL = Math.max(maxL, height[l]);
            total += maxL - height[l];   // 0 if height[l] is a new max
            l++;
        } else {
            maxR = Math.max(maxR, height[r]);
            total += maxR - height[r];
            r--;
        }
    }
    return total;
}

Takeaway. The full ladder — O(n²) rescan → O(n)-space precompute → O(1)-space two pointers — is worth internalizing as a sequence: each variation removes exactly one redundancy. The two-pointer step works because whichever side is currently shorter has its water level already provably determined.

Pitfalls

  • Loop on l < r, not l <= r for pair problems — l == r would pair an element with itself.
  • Always narrate the discard proof. Moving a pointer without saying why the skipped candidates can't win leaves the correctness argument unstated.
  • Sum overflow: two ints near 2³¹ overflow on +. Cast to long (or note the constraints rule it out).
  • Duplicate skipping in 3Sum: skip the anchor with continue, and skip equal l/r values after recording a hit — mixing these up either drops or duplicates triples.
  • Read/write invariant: write ≤ read at all times, so you never overwrite unread data. If you can state that invariant, in-place problems stop being scary.
  • Sorting destroys indexes. If the answer must be original indexes (Two Sum I), sort-then-two-pointers needs an index-carrying copy — usually the HashMap is simpler.

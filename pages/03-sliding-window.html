Sliding Window

Best/longest/shortest contiguous subarray or substring? Keep a window that slides forward, updating its state in O(1) per move instead of recomputing — the pattern behind a huge share of string questions.

Phase 1 · Weeks 2–3 Asked constantly for strings & subarrays

Before you start

Why this topic exists. "Longest substring such that…", "shortest subarray with…", "max sum of k consecutive…" — the brute force checks O(n²) ranges, each in O(n): cubic. The fix is to notice that neighboring ranges differ by one element. Maintain a window plus a little bookkeeping (a sum, a count array), pay O(1) per slide, and the whole family collapses to O(n). It's incremental computation — the same instinct as caching a computed value instead of recomputing it per request.

Prerequisites

  • 01 · Arrays & Hashing — frequency counting with maps and int[26] is the window's state.
  • 02 · Two Pointers — a window is two pointers moving the same direction; the discard reasoning carries over.

What the interviewer is actually testing

  • Do you recognize "contiguous"? That single word is the trigger; subsequences (non-contiguous) are a different topic (DP, page 18).
  • Can you state the window invariant? "My window always satisfies P; I expand until P breaks, shrink until it holds" — say it explicitly.
  • O(1) updates: adding/removing one element must not trigger a recount of the window.
  • The amortized argument: why a loop-in-a-loop is still O(n) — each pointer only moves forward.

How this builds on the previous topic

Two pointers converged from opposite ends and needed sorted data. Slide both pointers the same way and sortedness stops mattering — what matters instead is that the window's property is easy to maintain incrementally. The new ingredient over page 02 is state carried between moves (a running sum, a count array from page 01), so the window "knows" whether it's valid without re-scanning itself.

Core concepts

1. Fixed-size window: recompute in O(1), not O(k)

For "max sum of k consecutive elements", the windows at i and i+1 share k−1 elements. So don't re-add k numbers — subtract the element leaving, add the element entering: sum += arr[i] − arr[i−k]. Total cost O(n) instead of O(n·k). Any recomputation dearer than O(1) per slide means you're holding the pattern wrong.

2. Variable-size window: the grow/shrink invariant

When the window size is what you're optimizing, run two same-direction pointers with a contract:

  • right expands the window one element per iteration — greedily take more.
  • The moment the window violates the constraint (a repeated char, sum too big), advance left just far enough to restore it.
  • Measure the window whenever it's valid (longest) or whenever it's just barely valid (shortest).

The nested while looks quadratic but isn't: left and right each move only forward, at most n steps apiece — amortized O(n). That argument, said out loud, is worth real points.

3. Window state: what the window "knows" about itself

The window must answer "am I valid?" in O(1). Pick the lightest structure that can:

  • A number — running sum/product (max average subarray).
  • int[26]/int[128] counts — anagram/permutation checks, distinct-char limits.
  • HashMap + a counter — Minimum Window Substring's required trick: one integer summarizes "how many characters am I still missing?", making validity a required == 0 check instead of a 52-entry comparison.

4. Monotonic deque: sliding window maximum

"Max of each k-window" breaks the plain pattern — when the max leaves the window, what's the new max? The fix is a deque of indexes whose values stay in decreasing order: push a new element after popping all smaller ones from the back (they can never be a future max while a bigger, younger element lives); pop from the front when the front index slides out of range. The front is always the current max, and every index enters/leaves the deque once ⇒ O(n). Full monotonic-structure treatment in 09 · Stacks & Queues.

When the window logic breaks: negative numbers. "Shrink while sum > target" assumes growing the window can only grow the sum — monotonicity. Negative values kill that assumption (a longer window may sum to less), so "subarray sum equals k with negatives" is not a sliding-window problem: it belongs to prefix sums + HashMap, next page. Recognizing this boundary is a favorite follow-up question.

Pattern templates (Java)

Template 1 — fixed-size window (size k)

long sum = 0;
for (int i = 0; i < k; i++) sum += arr[i];   // seed the first window
long best = sum;
for (int i = k; i < arr.length; i++) {
    sum += arr[i] - arr[i - k];   // one enters, one leaves: O(1), not O(k)
    best = Math.max(best, sum);
}

Template 2 — variable window (longest while valid)

int left = 0;
for (int right = 0; right < n; right++) {
    add(arr[right]);                       // extend window by one element
    while (windowInvalid()) {              // constraint broken?
        remove(arr[left]);                 // shrink from the left...
        left++;                            // ...until it holds again
    }
    best = Math.max(best, right - left + 1);   // window valid here
}
// Amortized O(n): left and right each advance at most n times total

Template 3 — variable window (shortest that satisfies)

int left = 0;
for (int right = 0; right < n; right++) {
    add(arr[right]);
    while (windowSatisfies()) {            // valid — try to make it tighter
        best = Math.min(best, right - left + 1);   // record BEFORE shrinking
        remove(arr[left]);
        left++;
    }
}

Decision cues:

  • "Longest/shortest substring/subarray with condition" → variable window.
  • "Exactly k consecutive" / "every window of size k" → fixed window.
  • "Contains a permutation/anagram of t" → fixed window (size |t|) + count compare.
  • "Subarray sum equals k" with negatives allowed → prefix sum + HashMap (page 04).
  • "Max/min of each window" → monotonic deque (here + page 09).

Where this shows up in real systems

Rate limiters

"Max 100 requests per 60 seconds" is a sliding window over a timestamp stream. The sliding-window-log limiter keeps a deque of timestamps, evicting those older than now−60s — literally Template 2 with time as the constraint. Redis and Envoy both ship this and the cheaper sliding-window-counter approximation.

Streaming analytics

Rolling averages, "p95 latency over the last N samples", Prometheus rate() over a range: monitoring pipelines maintain window aggregates incrementally, add-one-evict-one, because recomputing per data point would melt the collector.

TCP flow control

TCP's receive and congestion windows are literal sliding windows over the byte stream: the sender may have only the window's worth of unacknowledged bytes in flight; ACKs slide the left edge forward while the congestion algorithm grows and shrinks the right.

Fraud velocity checks

"More than 3 cards tried from one IP in 10 minutes", "5 logins across 4 countries in an hour" — risk engines keep per-key sliding windows with counts (page 01's maps as window state) and flag when a window's stats cross a threshold.

Curated problems (easy → hard)

easy Best Time to Buy and Sell Stock A degenerate window: track min price so far; best profit = price − minSoFar at each step.

★ Worked solution — Longest Substring Without Repeating Characters (LeetCode 3, medium)

Problem in one line: length of the longest substring of s containing no repeated character.

Idea: keep a window that is always duplicate-free. Remember each character's last position; when the incoming char was last seen inside the window, snap left to just past that old occurrence — one jump instead of a step-by-step shrink. Measure after every extension.

public static int lengthOfLongestSubstring(String s) {
    // last[c] = most recent index where character c appeared; -1 = never.
    // An int[128] covers ASCII and beats a HashMap on constant factors.
    int[] last = new int[128];
    Arrays.fill(last, -1);

    int best = 0;
    int left = 0;                          // window start
    for (int right = 0; right < s.length(); right++) {
        char c = s.charAt(right);
        // If c was last seen INSIDE the current window, the invariant
        // would break — jump left just past that old occurrence.
        // (If last[c] < left, the old copy is already outside: ignore it.)
        if (last[c] >= left) {
            left = last[c] + 1;
        }
        last[c] = right;                   // record newest position of c
        // Window [left..right] is now duplicate-free — measure it
        best = Math.max(best, right - left + 1);
    }
    return best;
}

Complexity: time O(n) — one pass, left only jumps forward; space O(1) — a fixed 128-slot array. Runnable mirror: code/03-sliding-window/LongestSubstringNoRepeat.java.

★ Worked solution — Permutation in String (LeetCode 567, medium)

Problem in one line: does s2 contain any permutation of s1 as a contiguous substring?

Idea: a permutation of s1 is just some window of length |s1| whose letter counts equal s1's. So slide a fixed-size window across s2, updating counts by the one char entering and the one leaving, and compare the two 26-slot arrays — an O(26) = O(1) check per slide.

public static boolean checkInclusion(String s1, String s2) {
    int k = s1.length();
    if (k > s2.length()) return false;     // can't fit — fail fast

    // need = target letter counts; window = counts of current k-slice
    int[] need = new int[26];
    int[] window = new int[26];
    for (char c : s1.toCharArray()) need[c - 'a']++;

    for (int i = 0; i < s2.length(); i++) {
        // Slide right edge in: count the new character
        window[s2.charAt(i) - 'a']++;
        // Once the window exceeds size k, evict the char falling off
        // the left edge — O(1) update instead of recounting k chars
        if (i >= k) {
            window[s2.charAt(i - k) - 'a']--;
        }
        // Window is full-size from i == k-1 onward; compare counts.
        // Arrays.equals over 26 ints is O(26) = O(1).
        if (i >= k - 1 && Arrays.equals(window, need)) {
            return true;
        }
    }
    return false;
}

Complexity: time O(n·26) = O(n) for n = |s2|; space O(1) — two fixed arrays. Runnable mirror: code/03-sliding-window/PermutationInString.java.

★ Worked solution — Minimum Window Substring (LeetCode 76, hard)

Problem in one line: smallest window of s containing every character of t (with multiplicity).

Idea: the shortest-valid template. Expand right until the window covers t, then shrink left while it still does, recording each valid size. The elegant part is the required counter: need[c] goes negative for surplus copies, and required only counts genuinely missing characters — so "is the window valid?" is the O(1) test required == 0 rather than a full count comparison.

public static String minWindow(String s, String t) {
    if (t.length() > s.length()) return "";

    // need[c] > 0  => window still owes that many copies of c.
    // need[c] < 0  => window holds surplus copies of c.
    int[] need = new int[128];
    for (char c : t.toCharArray()) need[c]++;
    int required = t.length();      // total characters still missing

    int bestLen = Integer.MAX_VALUE, bestStart = 0;
    int left = 0;
    for (int right = 0; right < s.length(); right++) {
        char in = s.charAt(right);
        // Taking `in`: it only fills a real gap if we still owed one
        if (need[in] > 0) required--;
        need[in]--;                 // may go negative = surplus copy

        // Window covers all of t — shrink from the left while it still does
        while (required == 0) {
            // Record the window if it's the smallest so far
            if (right - left + 1 < bestLen) {
                bestLen = right - left + 1;
                bestStart = left;
            }
            // Give the leftmost char back before moving left forward
            char out = s.charAt(left);
            need[out]++;
            // If that char is now owed again, the window just broke —
            // exit the shrink loop and go back to expanding
            if (need[out] > 0) required++;
            left++;
        }
    }
    return bestLen == Integer.MAX_VALUE
            ? ""
            : s.substring(bestStart, bestStart + bestLen);
}

Complexity: time O(|s| + |t|) — each pointer sweeps forward once; space O(1) — a fixed 128-slot array. Runnable mirror: code/03-sliding-window/MinimumWindowSubstring.java.

Pitfalls & interview tips

  • Window length is right − left + 1 with inclusive pointers. Off-by-one here is the #1 sliding-window bug — write it once, reuse it everywhere.
  • Order of operations: add the new element first, then fix validity, then measure. For shortest-window problems, record the answer before shrinking past validity.
  • "Substring" vs "subsequence": windows only handle contiguous ranges. If characters may be skipped, you're in DP territory (page 18) — clarify with the interviewer before coding.
  • Negatives kill grow/shrink monotonicity for sum constraints — switch to prefix sum + HashMap (page 04) and say why.
  • Prefer int[128]/int[26] over HashMap<Character,Integer> when the alphabet is fixed: no boxing, no getOrDefault noise, faster — and mention the assumption ("ASCII input") as you make it.
  • Justify the nested loop: "left never moves backward, so both pointers advance at most n times — amortized O(n)". Interviewers explicitly listen for this sentence.

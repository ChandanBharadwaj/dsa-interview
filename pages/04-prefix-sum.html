Prefix Sums: Precompute Once, Answer Instantly

Turn repeated O(n) range scans into O(1) lookups, count subarrays that sliding windows can't touch, and apply thousands of range updates in one pass.

Phase 1 · Weeks 2–3 Quiet workhorse — rarely the headline, unlocks many mediums

Before you start

Why this topic exists. You've been doing prefix sums for years without the name: a bank balance is the prefix sum of transactions; a "requests served so far" counter is the prefix sum of per-second request counts. The interview version is the same idea made deliberate: spend one O(n) pass building cumulative totals, and every range question afterwards becomes a subtraction. It's the simplest example of the precompute-then-query mindset that databases, image processing, and monitoring systems are built on — and it quietly powers a surprising number of medium-level interview problems that look like they need something fancier.

Prerequisites

  • 01 · Arrays & Hashing — the HashMap complement-lookup trick returns here in disguise.
  • 03 · Sliding Window — you need to know the window invariant to appreciate exactly when it breaks.

What the interviewer is actually testing

  • Do you notice "many range queries" as a signal? Recomputing a sum inside a loop is the O(n²) trap they're watching for.
  • Can you handle negative numbers correctly? Reaching for a sliding window on "subarray sum equals k" with negatives present is the classic wrong turn — they want to see you recognize why it fails.
  • Off-by-one discipline. The prefix[i+1] - prefix[i] indexing is a tiny fence-post minefield; clean handling reads as experience.
  • Can you invert the idea? Difference arrays (prefix sums run backwards) show you understand the concept, not just the recipe.

How this builds on the previous topic

Sliding window (page 03) answers questions about one contiguous range that you grow and shrink as you scan. Prefix sums generalize that: they let you answer questions about any range, in any order, as many times as you like — and they keep working when negative numbers destroy the grow/shrink logic a window depends on. Where a window maintains one running total, a prefix array remembers all of them.

Core concepts

1. The core move: cumulative totals turn ranges into subtractions

Define prefix[i] = sum of the first i elements (so prefix[0] = 0, "the sum of nothing"). Then the sum of any slice nums[i..j] is prefix[j+1] - prefix[i]: take everything up to j, subtract everything strictly before i, and the shared head cancels out.

The array is one slot longer than the input on purpose. That extra leading zero is not decoration — it means a range starting at index 0 needs no special case, which removes the single most common bug in this pattern.

The mental model. A prefix array is a savings-account statement: each entry is "balance after transaction i". To know how much you gained between March and July you don't replay four months of transactions — you subtract two balances. That subtraction is the whole trick.

2. Prefix + HashMap: counting subarrays that sum to k

The follow-up interviewers love: how many subarrays sum to exactly k? Rearranging the range formula: a subarray ending at index j sums to k exactly when some earlier prefix equals running − k. So walk left to right, keep the running sum, and ask a HashMap "how many times have I already seen running − k?" — the same complement-lookup shape as Two Sum from page 01, applied to prefix sums instead of elements.

Why not a sliding window? A window relies on a monotone invariant: growing the window can only increase the sum, shrinking can only decrease it — that's what tells you which pointer to move. Negative numbers break both directions: extending the window might shrink the sum, so "sum too big → shrink" no longer reasons correctly, and the window will silently skip valid answers. The moment a subarray-sum problem allows negatives (or asks to count all answers, not find one), drop the window and reach for prefix + HashMap.

3. Difference arrays: prefix sums in reverse

Now invert the idea. Suppose you must apply many range updates — "add 3 to every element from index 2 to 7" — and only read the final result at the end. Updating every cell per request is O(n) each. Instead keep a difference array: record +3 at the start and −3 just past the end, in O(1) per update. One final prefix-sum pass "integrates" the markers back into actual values. Bookings, seat counts, and interval-overlap problems fall to this in a few lines.

4. Two dimensions, same trick (brief)

A 2D prefix table P[r][c] = sum of the rectangle from (0,0) to (r−1,c−1) makes any rectangle sum O(1) by inclusion–exclusion: sum = P[r2+1][c2+1] − P[r1][c2+1] − P[r2+1][c1] + P[r1][c1] (subtract the strip above and the strip to the left; add back the corner you subtracted twice). Computer vision calls this an integral image. Worth knowing exists; full 2D drills live with the matrix patterns on page 05 and beyond.

Pattern templates (Java)

Template 1 — build once, query ranges in O(1)

// Build: prefix has ONE EXTRA slot; prefix[0] = 0 means "sum of nothing".
long[] prefix = new long[nums.length + 1];        // long: n large values can overflow int
for (int i = 0; i < nums.length; i++) {
    prefix[i + 1] = prefix[i] + nums[i];          // each entry extends the running total
}

// Query: sum of nums[i..j] inclusive — two lookups, no rescan.
long rangeSum = prefix[j + 1] - prefix[i];        // total up to j, minus everything before i

Template 2 — prefix + HashMap: count subarrays with sum k (negatives welcome)

Map<Long, Integer> seen = new HashMap<>();  // prefix value -> times seen so far
seen.put(0L, 1);                    // the empty prefix: lets subarrays starting at 0 count
long running = 0;
int count = 0;
for (int x : nums) {
    running += x;                                   // prefix sum ending HERE
    count += seen.getOrDefault(running - k, 0);     // each earlier match = one valid start
    seen.merge(running, 1, Integer::sum);           // record AFTER counting (length >= 1)
}

Template 3 — difference array: many range updates, one final read

int[] diff = new int[n + 1];        // +1 slot so "end marker" never falls off the array
// apply "add v to every index in [l, r]" in O(1):
diff[l]     += v;                   // the effect starts at l ...
diff[r + 1] -= v;                   // ... and is cancelled just PAST r

// integrate once at the end: value[i] = diff[0] + diff[1] + ... + diff[i]
int[] value = new int[n];
int running = 0;
for (int i = 0; i < n; i++) {
    running += diff[i];             // prefix sum of the markers = actual value
    value[i] = running;
}

Decision cues:

  • The problem says "range sum query", "sum between indices", or a loop keeps re-summing slices → prefix array.
  • It says "subarray sum equals / divisible by / at most k" and values can be negative (or it asks for a count) → prefix + HashMap.
  • It says "apply these bookings/updates, then report the result" → difference array.
  • It mixes updates and queries throughout → prefix arrays go stale in O(n); that's the cue for a Fenwick tree (page 24), not this page.

Where this shows up in real systems

OLAP & analytics dashboards

"Revenue month-to-date" and "cumulative signups" charts are prefix sums over daily rollups. Warehouses (BigQuery, Snowflake) expose it directly as the window function SUM(x) OVER (ORDER BY day) — and "revenue between any two dates" is exactly the two-balance subtraction from this page.

Integral images in computer vision

The Viola–Jones face detector (the classic real-time face-box algorithm in cameras) evaluates thousands of rectangular brightness features per frame. Each is O(1) thanks to a 2D prefix table computed once per image — the textbook industrial win for inclusion–exclusion.

Calendar & booking systems

"How many meetings overlap at 10:15?" or "can this flight take more standby passengers?" — booking systems drop +1 at each start and −1 at each end, then prefix-sum along the timeline to get concurrent load at every instant. Car Pooling (worked below) is precisely this.

Monitoring counters

Prometheus counters are ever-growing cumulative totals — prefix sums by design. When Grafana graphs a rate, it subtracts two cumulative samples over a time range: counter(t2) − counter(t1). Storing the running total instead of raw deltas makes every range query cheap and loss-tolerant.

Worked solution — Range Sum Query, Immutable (LC 303)

Problem in one line: given a fixed array, answer many sumRange(left, right) queries fast.

Idea: the constructor pays O(n) once to build the cumulative array with a leading zero; after that every query is prefix[right+1] − prefix[left]. This is the pattern in its purest form — if you internalize the +1 offset here, you'll never fight it again. (LeetCode names the class NumArray.)

RangeSumQueryImmutable.java

public class RangeSumQueryImmutable {

    // prefix[i] = sum of the first i elements: nums[0] + ... + nums[i-1].
    // One extra slot (prefix[0] = 0) means no special case for ranges starting at 0.
    private final long[] prefix;

    public RangeSumQueryImmutable(int[] nums) {
        prefix = new long[nums.length + 1];          // prefix[0] stays 0: "sum of nothing"
        for (int i = 0; i < nums.length; i++) {
            prefix[i + 1] = prefix[i] + nums[i];     // extend the running total by one element
        }
    }

    // Sum of nums[left..right] inclusive, in O(1):
    // everything up to right, minus everything strictly before left.
    public int sumRange(int left, int right) {
        return (int) (prefix[right + 1] - prefix[left]);
    }
}

Complexity: O(n) construction, O(1) per query, O(n) extra space.

Worked solution — Subarray Sum Equals K (LC 560)

Problem in one line: count how many contiguous subarrays sum to exactly k; values may be negative.

Idea: a subarray (i..j] sums to k exactly when prefix[j] − prefix[i] = k, i.e. some earlier prefix equals running − k. So scan once, and let a HashMap remember how many times each prefix value has occurred — every earlier occurrence of running − k is a distinct valid start index. Seeding the map with {0 → 1} represents the empty prefix, so subarrays starting at index 0 are counted too.

SubarraySumEqualsK.java

public static int subarraySum(int[] nums, int k) {
    // How many times have we seen each prefix sum so far?
    // Seed: the empty prefix (sum 0) has been seen once — it lets a
    // subarray that starts at index 0 be counted.
    Map<Long, Integer> seen = new HashMap<>();
    seen.put(0L, 1);

    long running = 0;                  // prefix sum ending at the current index
    int count = 0;
    for (int x : nums) {
        running += x;                  // extend the prefix by one element
        // A subarray ending here sums to k  <=>  some earlier prefix equals running - k.
        // Each earlier occurrence is a distinct start index, so add them all at once.
        count += seen.getOrDefault(running - k, 0);
        // Only NOW record the current prefix — recording first would let the
        // zero-length subarray match itself when k == 0.
        seen.merge(running, 1, Integer::sum);
    }
    return count;
}

Complexity: O(n) time, O(n) space for the map. Compare: the brute force checks all O(n²) ranges; the sliding window is simply incorrect here once negatives appear.

Worked solution — Car Pooling (LC 1094, difference array)

Problem in one line: given trips [passengers, from, to] along a one-way road and a seat capacity, decide whether the car ever overflows.

Idea: we don't need per-trip simulation — we need the passenger count at every kilometer. Record each trip as two markers (+passengers at pickup, −passengers at drop-off), then one prefix-sum sweep down the road reconstructs the onboard count everywhere. If it ever exceeds capacity, fail. Note the drop-off index is exclusive: a passenger leaving at km 5 frees their seat for someone boarding at km 5.

CarPooling.java

public static boolean carPooling(int[][] trips, int capacity) {
    // diff[p] = net change in passengers at kilometer p.
    // Positions are bounded (0..1000 in the constraints), so a plain
    // array beats a TreeMap of events here.
    int[] diff = new int[1001];
    for (int[] t : trips) {
        int passengers = t[0], from = t[1], to = t[2];
        diff[from] += passengers;   // they board at 'from'
        diff[to]   -= passengers;   // and are gone by 'to' (drop-off is exclusive)
    }
    // Prefix-sum the markers: onboard = passengers in the car at position p.
    int onboard = 0;
    for (int p = 0; p < diff.length; p++) {
        onboard += diff[p];
        if (onboard > capacity) return false;   // overfull at this kilometer
    }
    return true;
}

Complexity: O(t + R) time for t trips over road length R, O(R) space. This "+1 at start, −1 past end, integrate once" shape is the standard tool for every interval-overlap counting problem — it returns with Meeting Rooms II on page 21.

Pitfalls & interview tips

  • Size the prefix array n+1, not n. The leading zero eliminates the "range starts at 0" special case. If you find yourself writing if (left == 0), you built it one slot short.
  • Overflow: 10⁵ elements near ±10⁹ each push prefix totals past int range. Build prefixes in long by default — it costs nothing and interviewers notice.
  • Seed the map with (0, 1). Forgetting the empty prefix silently drops every subarray that starts at index 0 — the most common wrong-answer on LC 560.
  • Count before you record. In the HashMap pattern, look up running − k before inserting running; inserting first lets a zero-length subarray match itself when k == 0.
  • Difference arrays end exclusive. The cancel marker goes at r + 1 (one past the last updated index). Size the diff array n + 1 so that marker always has a home.
  • Say the limitation out loud: "prefix arrays assume the data is immutable — if updates interleave with queries I'd move to a Fenwick tree for O(log n) both ways." One sentence, and you've flagged the page-24 upgrade path like a senior engineer.

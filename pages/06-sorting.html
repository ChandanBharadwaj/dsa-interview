Sorting

Merge sort, quicksort, and the non-comparison sorts — plus what Arrays.sort actually runs, and why "is it stable?" is a senior-level question.

Phase 2 · Week 4 Concept questions + the building block under half of all patterns

Before you start

Why this topic exists. You will rarely be asked to hand-write a full sort in an interview — but you will constantly be asked about sorting: "can you do better than O(n²)?", "is your sort stable?", "why did you sort at all?". And dozens of patterns (two pointers, intervals, greedy, binary search) begin with the sentence "first, sort the input". Understanding how the classic sorts work — and their exact costs — is the price of admission for everything after this page.

Interview relevance: mostly concept questions (tradeoffs, stability, comparators) plus quickselect; sorting is also step 1 of many other patterns.

How this builds on the previous topic

Phase 1 gave you patterns that mostly assume unsorted input. Sorting is the great pre-processor: spend O(n log n) once and suddenly two pointers, binary search (next page), and interval reasoning all become available. The recurring interview decision is exactly: "HashMap in O(n), or sort first and get a simpler O(n log n) solution?"

Core concepts

1. Merge sort — divide, conquer, merge

Split the array in half, sort each half (recursively), then merge the two sorted halves with a two-pointer walk. The recursion is log n levels deep and every level touches all n elements once ⇒ O(n log n) always — best case, worst case, adversarial case. The price: O(n) extra space for the merge buffer. Merge sort is also stable: equal elements keep their original relative order.

2. Quicksort — partition around a pivot

Pick a pivot, then partition: rearrange so everything smaller sits left of the pivot, everything bigger sits right. The pivot lands in its final sorted position; recurse on the two sides. Average O(n log n), in place (O(log n) stack only) — but a consistently bad pivot (already-sorted input with naive "pick last") degrades to O(n²). Real implementations randomize or median-of-three the pivot. Quicksort is not stable.

3. Heapsort, and the O(n log n) floor

Heapsort builds a max-heap (O(n)) then extracts the max n times (O(log n) each): O(n log n), in place, not stable, but with poor cache behavior — so it's rarely the default. It matters as a concept (see 12 · Heaps) and as quicksort's safety net (see Introsort below). And a fact worth quoting: no comparison sort can beat O(n log n) — sorting n items must distinguish n! orderings, and each comparison splits the possibilities in half, so you need at least log₂(n!) ≈ n log n comparisons. To go faster you must stop comparing…

That halving argument deserves its full form, because "prove the lower bound" is a genuine senior-loop question. Model any comparison sort as a decision tree: each internal node is one "is a[i] < a[j]?" comparison, and each leaf is a finished output ordering. To sort correctly, the tree needs at least n! leaves — every distinct input permutation must reach its own leaf, or two different inputs would get the same (wrong for one of them) answer. A binary tree of height h has at most 2ʰ leaves, so the worst-case comparison count is h ≥ log₂(n!), and Stirling's approximation turns that into log₂(n!) = n log₂ n − n log₂ e + O(log n) ≈ n log₂ n − 1.44n. Every comparison sort — including ones nobody has invented yet — pays that on some input. Merge sort's ~n log₂ n comparisons sit within a linear term of the floor, which is why the search for asymptotically faster comparison sorts is over, permanently.

Why radix sort isn't a contradiction. The Ω(n log n) bound binds only algorithms whose sole access to keys is pairwise comparison — one bit of information per question. Counting and radix sorts never ask "is a < b?"; they index arrays with the key's digits, extracting log₂ k bits per operation. Different model, different lower bound. That sentence is exactly the answer to the follow-up "you just said O(n log n) is optimal — so how is radix sort O(n)?"

4. Counting, bucket & radix — sorting without comparisons

Counting sort — keys in a small range ⇒ O(n + k)

// Sort an array whose values are all in 0..k (here k = 100), stably and in O(n + k).
int[] count = new int[101];
for (int x : arr) count[x]++;          // tally each value — O(n)
int idx = 0;
for (int v = 0; v <= 100; v++)         // walk values in order — O(k)
    while (count[v]-- > 0) arr[idx++] = v;   // write out v, count[v] times

Bucket sort spreads uniformly-distributed values into buckets and sorts each small bucket. Radix sort counting-sorts numbers digit by digit (least significant first — stability of each pass makes it work): O(d·(n+b)) for d digits. The interview signal: whenever the problem bounds the values ("ages 0–120", "letters a–z"), say "counting sort gives O(n) here."

5. What Java actually runs — and stability

  • Arrays.sort(int[]) → dual-pivot quicksort: fastest in practice for primitives, not stable — but for primitives equal values are indistinguishable, so stability is irrelevant.
  • Arrays.sort(T[]) and Collections.sort → TimSort: a merge sort that detects already-sorted "runs" in real data; stable, O(n) best case on nearly-sorted input, O(n log n) worst.
  • C++'s std::sort uses introsort: quicksort that switches to heapsort if recursion gets too deep — killing the O(n²) worst case. Good trivia when asked "how do libraries avoid bad pivots?"
Why stability matters in real code. Sort employees by salary, then by department: if the second sort is stable, each department's employees stay salary-ordered — you've built "sort by department, then salary" out of two passes. Unstable sorts destroy that. This is exactly why Java forces stable TimSort for objects.

How TimSort actually works is worth two minutes, because "tell me about TimSort" is a real senior question. It scans for runs — maximal already-ascending stretches (strictly descending ones are reversed in place, so reverse-sorted input is also O(n)). Runs shorter than minrun (16–32, computed so that n/minrun is close to a power of two, making the final merges balanced) get extended with binary insertion sort. Each run is pushed onto a stack, and merges fire whenever the top run lengths break the stack invariants — roughly, each run must be longer than the next two combined, so lengths grow like Fibonacci numbers, keeping the stack O(log n) deep. During a merge, if one run supplies seven elements in a row, TimSort switches to galloping: exponential search (page 07) finds how far the streak extends, then one bulk arraycopy moves it — which is how merging two barely-interleaved runs costs far less than element-by-element O(n).

The formal-verification anecdote. In 2015, researchers building a machine-checked proof of TimSort discovered the run-stack invariant wasn't fully re-established after certain merges — adversarial inputs could overflow the run stack with an ArrayIndexOutOfBoundsException. The bug had shipped in Python and Java for over a decade of planetary-scale use. The JDK's fix enlarged the stack; Python repaired the invariant. Quotable moral: a decade of testing missed what a proof attempt surfaced in weeks — and "my sort is obviously correct" is exactly the confidence stability bugs hide behind.

Dual-pivot quicksort carries its own asterisk: it is still a quicksort, and crafted adversarial inputs can drive it quadratic. That's why, in current JDKs, Arrays.sort(int[]) tracks recursion depth and falls back to heapsort when partitioning goes pathological — the same introsort insurance C++ has always used — and routes nearly-sorted primitive arrays to a run-merging strategy instead of partitioning at all. The one-line summary: the JDK sorts primitives with a quicksort wearing a heapsort parachute, and objects with a merge sort that exploits whatever order already exists.

6. Quickselect — the K-th element without sorting

Quicksort's partition places the pivot at its final index. If you only need the K-th smallest element, recurse into just the side containing K: n + n/2 + n/4 + … ⇒ average O(n), versus O(n log n) for a full sort. This powers "K-th largest element" (worked below) and is the one-shot alternative to heaps (page 12 compares them).

The honest caveat — quickselect's O(n) is average; hostile input or hostile luck makes it O(n²) — has a classical fix worth knowing by name: median of medians (BFPRT, 1973). Split the array into groups of 5, take each group's median (constant work per group), recursively select the median of those n/5 medians, and use that as the pivot. The guarantee: half the group-medians are ≤ the chosen pivot, and each such median has two more group members below it — so at least ~3n/10 elements are ≤ the pivot, and symmetrically ~3n/10 are ≥ it. Neither recursion side ever exceeds ~7n/10. The recurrence T(n) ≤ T(n/5) + T(7n/10) + O(n) solves to worst-case O(n) because 1/5 + 7/10 < 1: the level costs form a shrinking geometric series (the recursion-tree reading from page 00).

What to actually say in an interview. "Randomized quickselect is expected O(n); if we need a worst-case guarantee, median-of-medians gives deterministic O(n), but its constants are large enough that real libraries use introselect — quickselect that only falls back to the deterministic pivot when recursion misbehaves; that's C++'s nth_element." Naming that trade-off is the senior answer; nobody expects BFPRT reproduced at a whiteboard. Expected-O(n) with your own random pivot is acceptable whenever the adversary can't see your random bits — which is every interview and almost all of production.

7. The comparator contract, and sorting beyond RAM

Handing Arrays.sort a comparator asserts a total order: antisymmetry (sgn(c(a,b)) == -sgn(c(b,a))), transitivity (c(a,b) ≤ 0 and c(b,c) ≤ 0 ⇒ c(a,c) ≤ 0), and consistency — the same pair must compare the same way every time it's asked. Break any of these and a quicksort would just quietly mis-sort; TimSort instead notices: its merges rely on runs being genuinely sorted, and when a comparison contradicts that assumption it throws the exception quoted in the Pitfalls below — "Comparison method violates its general contract". Typical culprits: the subtraction-overflow comparator (also in Pitfalls), comparing doubles with < so NaN answers inconsistently (use Double.compare), and keys mutated concurrently mid-sort. Because the throw depends on which runs happen to merge, it surfaces "randomly, only in production, only on some data" — a classic war story worth telling before it happens to you.

And when data outgrows memory, sorting doesn't stop — it becomes external merge sort, the pattern behind the LSM-compaction card below: read RAM-sized chunks, sort each in memory (that part is this page), write them out as sorted runs, then k-way merge the runs using a min-heap holding each run's current head — O(N log k) comparisons and strictly sequential disk I/O (page 12 builds that heap; the capstone shows it running forever inside database compaction). Note the symmetry with TimSort: both are bets on presortedness. TimSort's true cost is O(n + n log r) for r runs — one run costs O(n), n runs cost O(n log n) — so "how sorted is this input already?" is a question worth asking out loud before choosing any sort at all.

Pattern templates (Java)

Merge sort — the template (also the shape of "count inversions" problems)

public static void mergeSort(int[] a, int lo, int hi) {   // sorts a[lo..hi]
    if (lo >= hi) return;                 // base case: 0 or 1 element is sorted
    int mid = lo + (hi - lo) / 2;         // overflow-safe midpoint
    mergeSort(a, lo, mid);                // sort left half
    mergeSort(a, mid + 1, hi);            // sort right half
    merge(a, lo, mid, hi);                // two-pointer merge of the sorted halves
}

private static void merge(int[] a, int lo, int mid, int hi) {
    int[] tmp = new int[hi - lo + 1];     // the O(n) extra space
    int i = lo, j = mid + 1, t = 0;
    while (i <= mid && j <= hi)
        // <= keeps equal elements' left-first order ⇒ stability
        tmp[t++] = a[i] <= a[j] ? a[i++] : a[j++];
    while (i <= mid) tmp[t++] = a[i++];   // drain leftovers (only one loop runs)
    while (j <= hi)  tmp[t++] = a[j++];
    System.arraycopy(tmp, 0, a, lo, tmp.length);
}

Comparators — the Java you'll write in every sorting-adjacent problem

int[][] intervals = ...;
// sort 2D arrays by first column (start time) — intervals, meetings, events
Arrays.sort(intervals, (x, y) -> Integer.compare(x[0], y[0]));

// multi-key: by length ascending, then alphabetically — Comparator composition
List<String> words = ...;
words.sort(Comparator.comparingInt(String::length)
                     .thenComparing(Comparator.naturalOrder()));

// descending: reverseOrder needs boxed types (Integer[], not int[])
Arrays.sort(boxed, Collections.reverseOrder());
Comparator bug to avoid. Never write (x, y) -> x[0] - y[0] — the subtraction overflows for values like Integer.MIN_VALUE and silently mis-sorts. Always Integer.compare(x[0], y[0]).

Decision cues

  • Problem bounds the values to a small range → counting sort O(n), mention it even if you then use a library sort.
  • "K-th largest / smallest", one-shot → quickselect O(n); streaming/repeated → heap (12).
  • Only 3 distinct values (0/1/2) → Dutch national flag one-pass partition (worked below).
  • Custom order ("largest concatenation", "closest to origin") → library sort + comparator; your job is only the comparator.
  • "Nearly sorted" input → say TimSort is O(n) here; or an insertion-sort/heap argument for k-out-of-place variants.

Where this shows up in real systems

TimSort runs the world's sorting

Invented for Python, adopted by Java (Collections.sort), Android, and V8's Array.prototype.sort. Its bet — real data arrives in partially sorted runs — pays off so often that a 2002 heuristic became the planetary default.

External sort & LSM-tree compaction

When data exceeds RAM (database spills, MapReduce shuffle), systems sort chunks in memory, write sorted "runs" to disk, then K-way merge them — merge sort at terabyte scale. RocksDB/Cassandra compaction is this merge running forever in the background (see capstone).

Radix sort in graphics & networking

GPUs radix-sort millions of primitives per frame for rendering order; routers and flow analyzers radix-bucket packets by IP prefix — comparison-free sorting where keys are fixed-width bits.

Top-K everywhere

"Trending now" and leaderboard features don't fully sort a billion items; they quickselect or heap-select the top 100 — the exact interview distinction between O(n log n), O(n log k) and O(n).

easy Merge Sorted Array — LeetCode 88

The problem. nums1 (length m+n, last n slots empty) and nums2 (length n) are both sorted. Merge nums2 into nums1 so nums1 ends up sorted — in place, without extra arrays.

Example. nums1=[1,2,3,0,0,0], m=3, nums2=[2,5,6], n=3 → nums1=[1,2,2,3,5,6].

Variation 1 — copy, then sort · O((m+n) log(m+n)), ignores the structure

public void mergeSortEverything(int[] nums1, int m, int[] nums2, int n) {
    System.arraycopy(nums2, 0, nums1, m, n);   // append nums2 into the free tail
    Arrays.sort(nums1);                        // let the library do the rest
}

Works, but throws away the fact that both inputs are already sorted — a merge should be O(m+n).

Variation 2 — merge backwards (optimal) · O(m+n) time, O(1) space

public void merge(int[] nums1, int m, int[] nums2, int n) {
    int i = m - 1;          // last real element of nums1
    int j = n - 1;          // last element of nums2
    int w = m + n - 1;      // last slot of nums1 — write position

    // Merge from the BACK: the free space is at the end, so writing
    // backwards never overwrites an unread nums1 element.
    while (j >= 0) {                       // nums2 exhausted => rest is already in place
        if (i >= 0 && nums1[i] > nums2[j])
            nums1[w--] = nums1[i--];       // nums1's tail element is bigger — place it
        else
            nums1[w--] = nums2[j--];       // nums2's tail element is bigger (or nums1 done)
    }
}

Takeaway. Forward merging would clobber unread nums1 values; walking from the back turns the free tail into a safe write zone. "Merge from the side where the free space is" is a trick that recurs in in-place array problems. Mirror: code/06-sorting/MergeSortedArray.java.

easy How Many Numbers Are Smaller Than the Current Number — LeetCode 1365

The problem. For each element, count how many other elements are strictly smaller. Values are guaranteed to be in 0..100.

Example. [8,1,2,2,3] → [4,0,1,1,3] (four numbers are smaller than 8; nothing is smaller than 1; one number is smaller than each 2; …).

Variation 1 — compare every pair · O(n²)

public int[] smallerNumbersBrute(int[] nums) {
    int[] res = new int[nums.length];
    for (int i = 0; i < nums.length; i++)
        for (int j = 0; j < nums.length; j++)
            if (nums[j] < nums[i]) res[i]++;
    return res;
}

Variation 2 — counting sort + prefix counts (optimal) · O(n + k)

public int[] smallerNumbersThanCurrent(int[] nums) {
    // Values are 0..100 — the constraint that unlocks counting sort.
    int[] count = new int[101];
    for (int x : nums) count[x]++;            // tally each value

    // prefix[v] = how many elements are <= v. After this loop,
    // count[v-1] answers "how many are STRICTLY smaller than v".
    for (int v = 1; v <= 100; v++)
        count[v] += count[v - 1];

    int[] res = new int[nums.length];
    for (int i = 0; i < nums.length; i++)
        res[i] = nums[i] == 0 ? 0 : count[nums[i] - 1];
    return res;
}

Takeaway. A bounded value range ("0 ≤ nums[i] ≤ 100") is the standing invitation to replace comparisons with counting — O(n + k) beats any comparison sort. The prefix pass on the counts is page 04's prefix sum applied to a histogram. Mirror: code/06-sorting/SmallerNumbersThanCurrent.java.

easy Sort Array By Parity — LeetCode 905

The problem. Rearrange so all even numbers come before all odd numbers (any order within each group), ideally in place.

Example. [3,1,2,4] → [2,4,3,1] (any even-first arrangement accepted).

Two-way partition — quicksort's inner move · O(n) time, O(1) space

public static int[] sortArrayByParity(int[] nums) {
    int write = 0;                          // invariant: [0, write) is all even
    for (int read = 0; read < nums.length; read++)
        if (nums[read] % 2 == 0) {
            int t = nums[read];             // swap the even back to the boundary
            nums[read] = nums[write];
            nums[write++] = t;
        }
    return nums;
}

Takeaway. This is quicksort's partition step, with "is even" playing the pivot test — worth seeing stripped bare, because partition is the piece of quicksort people fumble at the whiteboard. Sort Colors (above) is the three-way version of the same move. Mirror: code/06-sorting/SortArrayByParity.java.

easy Relative Sort Array — LeetCode 1122

The problem. Sort arr1 so its elements follow the relative order given by arr2; elements not in arr2 go last, ascending. Values are 0..1000.

Example. arr1 [2,3,1,3,2,4,6,7,9,2,19], arr2 [2,1,4,3,9,6] → [2,2,2,1,4,3,3,9,6,7,19].

Counting sort with a custom emission order · O(n + m + 1000)

public static int[] relativeSortArray(int[] arr1, int[] arr2) {
    int[] count = new int[1001];        // bounded values -> counting sort
    for (int x : arr1) count[x]++;

    int[] out = new int[arr1.length];
    int w = 0;
    for (int v : arr2)                  // arr2's order first, with multiplicity
        while (count[v]-- > 0)
            out[w++] = v;
    for (int v = 0; v <= 1000; v++)     // leftovers ascending — free, because
        while (count[v]-- > 0)          // we scan values in increasing order
            out[w++] = v;
    return out;
}

Takeaway. Counting sort's superpower isn't just speed — it lets you emit in any order you can enumerate: here arr2's order first, then ascending. (The comparator alternative — rank map + custom sort — is O(n log n) and boxes every int.) Whenever values are small and dense, think buckets before comparisons. Mirror: code/06-sorting/RelativeSortArray.java.

medium Sort Colors (Dutch National Flag) — LeetCode 75

The problem. An array contains only 0s, 1s and 2s (colors). Sort it in place — ideally in a single pass without counting.

Example. [2,0,2,1,1,0] → [0,0,1,1,2,2].

Variation 1 — counting sort, two passes · O(n) time, O(1) space

public void sortColorsCounting(int[] nums) {
    int[] count = new int[3];
    for (int x : nums) count[x]++;            // pass 1: tally 0s, 1s, 2s
    int i = 0;
    for (int v = 0; v <= 2; v++)              // pass 2: rewrite in order
        while (count[v]-- > 0) nums[i++] = v;
}

Perfectly fine — but the follow-up asks for one pass, which is the Dutch National Flag partition:

Variation 2 — three-zone partition, one pass (optimal) · O(n) time, O(1) space

public void sortColors(int[] nums) {
    int lo = 0;                    // next slot for a 0 (zone boundary)
    int hi = nums.length - 1;      // next slot for a 2 (zone boundary)
    int i = 0;                     // current element under inspection
    while (i <= hi) {              // hi side is UNSORTED territory — stop when we meet it
        if (nums[i] == 0) {
            swap(nums, i, lo);     // send 0 to the left zone
            lo++; i++;             // the swapped-in value came from the 1-zone — safe to advance
        } else if (nums[i] == 2) {
            swap(nums, i, hi);     // send 2 to the right zone
            hi--;                  // do NOT advance i: the swapped-in value is UNSEEN — recheck it
        } else {
            i++;                   // 1 is already in the middle zone
        }
    }
}
private void swap(int[] a, int i, int j) { int t = a[i]; a[i] = a[j]; a[j] = t; }

Takeaway. The one-pass version is quicksort's partition generalized to three zones. The classic bug: advancing i after the hi-swap — the value that arrived from the right end is unexamined and must be rechecked. Mirror: code/06-sorting/SortColors.java.

medium Kth Largest Element in an Array — LeetCode 215

The problem. Return the k-th largest element (by sorted position, not distinct values) of an unsorted array.

Example. nums=[3,2,1,5,6,4], k=2 → 5. nums=[3,2,3,1,2,4,5,5,6], k=4 → 4.

Variation 1 — sort and index · O(n log n)

public int findKthLargestSort(int[] nums, int k) {
    Arrays.sort(nums);                        // full sort — does more work than needed
    return nums[nums.length - k];             // k-th largest sits k from the end
}

Variation 2 — size-k min-heap · O(n log k), best for streams

public int findKthLargestHeap(int[] nums, int k) {
    // Min-heap of the k largest seen so far; its ROOT is the k-th largest.
    PriorityQueue<Integer> heap = new PriorityQueue<>();
    for (int x : nums) {
        heap.offer(x);
        if (heap.size() > k) heap.poll();     // evict the smallest of the k+1
    }
    return heap.peek();
}

Variation 3 — quickselect (optimal one-shot) · average O(n)

private static final Random RAND = new Random();

public int findKthLargest(int[] nums, int k) {
    int target = nums.length - k;              // index the answer occupies in sorted order
    int lo = 0, hi = nums.length - 1;
    while (true) {
        int p = partition(nums, lo, hi);       // pivot lands at its FINAL sorted index p
        if (p == target) return nums[p];       // pivot is exactly the K-th largest — done
        if (p < target) lo = p + 1;            // answer lies right of p — discard left side
        else hi = p - 1;                       // answer lies left of p — discard right side
    }
}

// Lomuto partition with a random pivot: returns the pivot's final index
private int partition(int[] a, int lo, int hi) {
    int r = lo + RAND.nextInt(hi - lo + 1);    // random pivot defeats sorted-input worst case
    swap(a, r, hi);                            // park pivot at the end
    int pivot = a[hi];
    int i = lo;                                // a[lo..i-1] = the "< pivot" zone
    for (int j = lo; j < hi; j++)
        if (a[j] < pivot) swap(a, i++, j);     // grow the smaller-zone by one
    swap(a, i, hi);                            // drop pivot just after the smaller-zone
    return i;                                  // pivot's final resting index
}
private void swap(int[] a, int i, int j) { int t = a[i]; a[i] = a[j]; a[j] = t; }

Takeaway. Three tools, three costs: full sort O(n log n) when you need total order anyway; heap O(n log k) when data streams in or k is tiny (page 12); quickselect average O(n) for a one-shot query on an array you may mutate. Recursing into only one side is what collapses n log n to n: n + n/2 + n/4 + … = 2n. Mirror: code/06-sorting/KthLargest.java.

medium Largest Number — LeetCode 179

The problem. Arrange non-negative integers so their concatenation forms the largest possible number; return it as a string (it can exceed long).

Example. [10,2] → "210". [3,30,34,5,9] → "9534330" (note 3 before 30: "330" beats "303").

The insight. Comparing values doesn't work (3 must outrank 30). What matters is which concatenation order is bigger: put a before b exactly when a+b > b+a as strings. That comparator is a valid total order, so sorting with it maximizes the whole result.

Custom comparator sort · O(n log n · L)

public String largestNumber(int[] nums) {
    // work with strings — concatenation order is what we're optimizing
    String[] s = new String[nums.length];
    for (int i = 0; i < nums.length; i++) s[i] = String.valueOf(nums[i]);

    // order pair (a, b) by which concatenation is bigger: "b+a vs a+b" descending
    Arrays.sort(s, (a, b) -> (b + a).compareTo(a + b));

    if (s[0].equals("0")) return "0";     // all zeros collapse to a single "0"

    StringBuilder sb = new StringBuilder(); // O(n) build — never += in a loop (page 00!)
    for (String part : s) sb.append(part);
    return sb.toString();
}

Takeaway. Sometimes the entire problem is the comparator. Why the order is consistent (transitive) is the exchange argument of page 21: if a+b ≥ b+a pairwise, any arrangement can be bubble-swapped into the sorted one without ever shrinking the result. The all-zeros edge case ("00" → "0") is the classic missed test. Mirror: code/06-sorting/LargestNumber.java.

medium Sort List — LeetCode 148

The problem. Sort a singly linked list in O(n log n) time. (Constant space is the follow-up; recursion's O(log n) stack is generally accepted.)

Example. 4→2→1→3 → 1→2→3→4. -1→5→3→4→0 → -1→0→3→4→5.

Why merge sort here. Quicksort wants O(1) random access for partitioning — lists don't have it. Merge sort only ever needs "walk forward" and "splice nodes": find the middle (fast/slow pointers, page 08), sort both halves, merge by relinking.

Merge sort on a linked list · O(n log n) time, O(log n) stack

public ListNode sortList(ListNode head) {
    if (head == null || head.next == null) return head;   // 0/1 nodes: sorted

    // 1) Split in half: slow ends at the middle (fast moves 2x).
    ListNode slow = head, fast = head.next;
    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
    }
    ListNode second = slow.next;
    slow.next = null;                     // cut the list in two

    // 2) Recursively sort each half.
    ListNode a = sortList(head);
    ListNode b = sortList(second);

    // 3) Merge two sorted lists by relinking nodes (no allocation).
    ListNode dummy = new ListNode(0), tail = dummy;
    while (a != null && b != null) {
        if (a.val <= b.val) { tail.next = a; a = a.next; }
        else                { tail.next = b; b = b.next; }
        tail = tail.next;
    }
    tail.next = (a != null) ? a : b;      // append the leftover run
    return dummy.next;
}

Takeaway. Starting fast at head.next (not head) makes the split land left-of-center on even lengths, preventing infinite recursion on two-node lists — a detail worth understanding rather than memorizing. Merging by relinking is why merge sort's usual O(n) buffer disappears on lists. Mirror: code/06-sorting/SortList.java.

medium Sort an Array — LeetCode 912

The problem. Sort the array in O(n log n) without built-in sort — the "implement a real sorting algorithm, correctly" question. LeetCode's tests include the two classic quicksort killers: already-sorted input and long runs of equal values.

Example. [5,2,3,1] → [1,2,3,5].

Variation 1 — merge sort · stable, guaranteed O(n log n), O(n) extra

private static void mergeSort(int[] a, int[] tmp, int lo, int hi) {
    if (lo >= hi) return;
    int mid = lo + (hi - lo) / 2;
    mergeSort(a, tmp, lo, mid);         // sort left half
    mergeSort(a, tmp, mid + 1, hi);     // sort right half
    int i = lo, j = mid + 1, w = lo;
    while (i <= mid && j <= hi)         // merge: smaller head wins
        tmp[w++] = a[i] <= a[j] ? a[i++] : a[j++];
    while (i <= mid) tmp[w++] = a[i++];
    while (j <= hi) tmp[w++] = a[j++];
    System.arraycopy(tmp, lo, a, lo, hi - lo + 1);
}

Variation 2 — quicksort, random pivot + three-way partition · in place

private static void quickSort(int[] a, int lo, int hi) {
    if (lo >= hi) return;
    // Random pivot defuses the sorted-input killer; the three-way (Dutch
    // flag) partition defuses the all-equal killer — equal values settle
    // in the middle band and are never recursed into again.
    int pivot = a[lo + RAND.nextInt(hi - lo + 1)];
    int lt = lo, gt = hi, i = lo;
    while (i <= gt) {
        if (a[i] < pivot) swap(a, lt++, i++);
        else if (a[i] > pivot) swap(a, i, gt--);
        else i++;                       // equal: stays in the middle band
    }
    quickSort(a, lo, lt - 1);           // strictly-less zone
    quickSort(a, gt + 1, hi);           // strictly-greater zone
}

Variation 3 — heap sort · O(n log n) worst case, O(1) extra, not stable

public static void heapSort(int[] a) {
    int n = a.length;
    for (int i = n / 2 - 1; i >= 0; i--)   // heapify: bottom-up sift-downs, O(n)
        siftDown(a, i, n);
    for (int end = n - 1; end > 0; end--) {
        swap(a, 0, end);                   // current max to its final slot
        siftDown(a, 0, end);               // restore the heap on the prefix
    }
}

Takeaway. One problem, all three engines from this page — and the reason naive quicksort fails here is the lesson: random pivots handle adversarial order, three-way partitioning handles adversarial equality; you need both. All three variants are cross-checked against Arrays.sort on hundreds of random arrays in the mirror: code/06-sorting/SortAnArray.java.

medium H-Index — LeetCode 274

The problem. A researcher's h-index is the largest h such that h of their papers have at least h citations each. Compute it from the citation counts.

Example. [3,0,6,1,5] → 3 (three papers with ≥ 3 citations). [1,3,1] → 1.

Variation 1 — sort ascending, scan · O(n log n)

public static int hIndexSort(int[] citations) {
    Arrays.sort(citations);                  // ascending
    int n = citations.length;
    for (int i = 0; i < n; i++) {
        int papersAtLeast = n - i;           // papers with ≥ citations[i] cites
        if (citations[i] >= papersAtLeast)   // first index that qualifies:
            return papersAtLeast;            // that count IS the h-index
    }
    return 0;
}

Variation 2 — counting buckets, clamped at n · O(n)

public static int hIndex(int[] citations) {
    int n = citations.length;
    int[] buckets = new int[n + 1];
    for (int c : citations)
        buckets[Math.min(c, n)]++;           // h ≤ n, so 1000 citations act like n
    int papers = 0;
    for (int h = n; h >= 0; h--) {           // walk h downward,
        papers += buckets[h];                // accumulating papers with ≥ h
        if (papers >= h) return h;           // first h that enough papers support
    }
    return 0;
}

Takeaway. The clamp min(c, n) is the insight that unlocks counting sort here: the h-index can't exceed the paper count, so citation counts above n are interchangeable — suddenly the value range is bounded and O(n) drops out. "Can I clamp or bucket the values?" is the standard question to ask before paying for a comparison sort. Mirror: code/06-sorting/HIndex.java.

hard Count of Smaller Numbers After Self — LeetCode 315

The problem. For every element, count how many elements to its right are strictly smaller. (Each such pair is an "inversion" anchored at the left element.)

Example. [5,2,6,1] → [2,1,1,0] (right of 5: 2 and 1 are smaller; right of 2: just 1; right of 6: just 1; right of 1: nothing).

Variation 1 — scan right per element · O(n²)

public List<Integer> countSmallerBrute(int[] nums) {
    List<Integer> res = new ArrayList<>();
    for (int i = 0; i < nums.length; i++) {
        int c = 0;
        for (int j = i + 1; j < nums.length; j++)
            if (nums[j] < nums[i]) c++;
        res.add(c);
    }
    return res;
}

Variation 2 — count during merge sort (optimal) · O(n log n)

public List<Integer> countSmaller(int[] nums) {
    int n = nums.length;
    int[] counts = new int[n];
    int[] idx = new int[n];               // we sort INDEXES, not values,
    for (int i = 0; i < n; i++) idx[i] = i;   // so each count lands at its owner
    mergeCount(nums, idx, new int[n], counts, 0, n - 1);
    List<Integer> res = new ArrayList<>(n);
    for (int c : counts) res.add(c);
    return res;
}

private void mergeCount(int[] a, int[] idx, int[] tmp, int[] counts, int lo, int hi) {
    if (lo >= hi) return;
    int mid = lo + (hi - lo) / 2;
    mergeCount(a, idx, tmp, counts, lo, mid);        // sort left half (by value)
    mergeCount(a, idx, tmp, counts, mid + 1, hi);    // sort right half

    // Merge, counting cross-half inversions. Both halves are sorted, so
    // when a left element is placed, every right element already placed
    // (rightPlaced of them) is a smaller element that was ORIGINALLY to
    // its right — exactly what we're counting. One merge counts a whole
    // batch of pairs at once; that's the n^2 -> n log n collapse.
    int i = lo, j = mid + 1, t = lo, rightPlaced = 0;
    while (i <= mid || j <= hi) {
        if (j > hi || (i <= mid && a[idx[i]] <= a[idx[j]])) {
            counts[idx[i]] += rightPlaced;   // credit the left element
            tmp[t++] = idx[i++];
        } else {
            rightPlaced++;                   // a right (smaller) element goes first
            tmp[t++] = idx[j++];
        }
    }
    System.arraycopy(tmp, lo, idx, lo, hi - lo + 1);
}

Takeaway. Merge sort isn't only a sorter — its merge step visits pairs across the two halves in sorted order, which lets it count ordered relationships in bulk. Sorting indexes instead of values is the standard device for routing per-element results back to their owners. (A Fenwick tree solves this too — page 24.) Mirror: code/06-sorting/CountSmallerAfterSelf.java.

hard Maximum Gap — LeetCode 164

The problem. The maximum difference between successive elements of the sorted array — computed in O(n) time, without actually sorting.

Example. [3,6,9,1] → 3 (sorted: 1,3,6,9 — gaps 2,3,3).

Pigeonhole buckets: the answer lives between buckets · O(n)

public static int maximumGap(int[] nums) {
    int n = nums.length;
    if (n < 2) return 0;
    int lo = Integer.MAX_VALUE, hi = Integer.MIN_VALUE;
    for (int x : nums) {
        lo = Math.min(lo, x);
        hi = Math.max(hi, x);
    }
    if (lo == hi) return 0;                       // all equal

    // Pigeonhole: n numbers over range R force some sorted gap ≥ R/(n−1).
    // Buckets of that size (or smaller) can never CONTAIN the max gap —
    // so only each bucket's min and max matter.
    int bucketSize = Math.max(1, (hi - lo) / (n - 1));
    int bucketCount = (hi - lo) / bucketSize + 1;
    int[] mins = new int[bucketCount], maxs = new int[bucketCount];
    Arrays.fill(mins, Integer.MAX_VALUE);
    Arrays.fill(maxs, Integer.MIN_VALUE);
    for (int x : nums) {
        int b = (x - lo) / bucketSize;            // which bucket x lands in
        mins[b] = Math.min(mins[b], x);
        maxs[b] = Math.max(maxs[b], x);
    }

    int best = 0, prevMax = maxs[0];              // bucket 0 surely holds `lo`
    for (int b = 1; b < bucketCount; b++) {
        if (mins[b] == Integer.MAX_VALUE) continue;   // empty bucket — skip it
        best = Math.max(best, mins[b] - prevMax); // gap across the boundary
        prevMax = maxs[b];
    }
    return best;
}

Takeaway. The rare problem where bucket sort's idea beats every comparison sort provably: pigeonhole guarantees the max gap spans a bucket boundary, so each bucket only needs its min/max — not its contents, not its order. When a problem asks for a property of the sorted order but not the order itself, look for an argument that skips the sort. Mirror (cross-checked vs sorting on random arrays): code/06-sorting/MaximumGap.java.

Pitfalls

  • "Just sort it" isn't free. Before sorting, say what it costs (O(n log n)) and check a hash-based O(n) alternative isn't cleaner. Sorting mutates — copy first if indexes matter later.
  • Subtraction comparators overflow. a - b breaks on extreme values; use Integer.compare / Long.compare. Also never return inconsistent comparators — TimSort throws IllegalArgumentException: Comparison method violates its general contract.
  • Primitives can't take comparators. To sort int[] descending, either sort ascending and read backwards, or box to Integer[] (O(n) extra).
  • Quicksort's worst case is a fair question. Answer: O(n²) on adversarial pivots; fixed by random pivot / median-of-three / introsort's heapsort fallback.
  • Stability only matters for objects. If asked "is your sort stable?" for an int[], point out equal ints are interchangeable — that's why the JDK dares use unstable dual-pivot quicksort there.
  • Merge from the back when merging into an array that has free space at the end (Merge Sorted Array) — forward merging overwrites unread elements.

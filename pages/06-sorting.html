Sorting

Merge sort, quicksort, and the non-comparison sorts — plus what Arrays.sort actually runs, and why "is it stable?" is a senior-level question.

Phase 2 · Week 4 Concept questions + the building block under half of all patterns

Before you start

Why this topic exists. You will rarely be asked to hand-write a full sort in an interview — but you will constantly be asked about sorting: "can you do better than O(n²)?", "is your sort stable?", "why did you sort at all?". And dozens of patterns (two pointers, intervals, greedy, binary search) begin with the sentence "first, sort the input". Understanding how the classic sorts work — and their exact costs — is the price of admission for everything after this page.

Prerequisites

  • 00 · Foundations — Big-O, and why O(n log n) beats O(n²)
  • 01 · Arrays & Hashing — array manipulation fluency

What the interviewer is actually testing

  • Do you know the O(n log n) standard sorts and their tradeoffs (space, stability, worst case) — not just their names?
  • Do you spot when a non-comparison sort (counting/bucket) gives O(n) because the key range is small?
  • Can you write a custom comparator correctly in Java, and reason about stability?
  • Do you know quickselect — getting the K-th element in average O(n) without fully sorting?

How this builds on the previous topic

Phase 1 gave you patterns that mostly assume unsorted input. Sorting is the great pre-processor: spend O(n log n) once and suddenly two pointers, binary search (next page), and interval reasoning all become available. The recurring interview decision is exactly: "HashMap in O(n), or sort first and get a simpler O(n log n) solution?"

Core concepts

1. Merge sort — divide, conquer, merge

Split the array in half, sort each half (recursively), then merge the two sorted halves with a two-pointer walk. The recursion is log n levels deep and every level touches all n elements once ⇒ O(n log n) always — best case, worst case, adversarial case. The price: O(n) extra space for the merge buffer. Merge sort is also stable: equal elements keep their original relative order.

2. Quicksort — partition around a pivot

Pick a pivot, then partition: rearrange so everything smaller sits left of the pivot, everything bigger sits right. The pivot lands in its final sorted position; recurse on the two sides. Average O(n log n), in place (O(log n) stack only) — but a consistently bad pivot (already-sorted input with naive "pick last") degrades to O(n²). Real implementations randomize or median-of-three the pivot. Quicksort is not stable.

3. Heapsort, and the O(n log n) floor

Heapsort builds a max-heap (O(n)) then extracts the max n times (O(log n) each): O(n log n), in place, not stable, but with poor cache behavior — so it's rarely the default. It matters as a concept (see 12 · Heaps) and as quicksort's safety net (see Introsort below). And a fact worth quoting: no comparison sort can beat O(n log n) — sorting n items must distinguish n! orderings, and each comparison splits the possibilities in half, so you need at least log₂(n!) ≈ n log n comparisons. To go faster you must stop comparing…

4. Counting, bucket & radix — sorting without comparisons

Counting sort — keys in a small range ⇒ O(n + k)

// Sort an array whose values are all in 0..k (here k = 100), stably and in O(n + k).
int[] count = new int[101];
for (int x : arr) count[x]++;          // tally each value — O(n)
int idx = 0;
for (int v = 0; v <= 100; v++)         // walk values in order — O(k)
    while (count[v]-- > 0) arr[idx++] = v;   // write out v, count[v] times

Bucket sort spreads uniformly-distributed values into buckets and sorts each small bucket. Radix sort counting-sorts numbers digit by digit (least significant first — stability of each pass makes it work): O(d·(n+b)) for d digits. The interview signal: whenever the problem bounds the values ("ages 0–120", "letters a–z"), say "counting sort gives O(n) here."

5. What Java actually runs — and stability

  • Arrays.sort(int[]) → dual-pivot quicksort: fastest in practice for primitives, not stable — but for primitives equal values are indistinguishable, so stability is irrelevant.
  • Arrays.sort(T[]) and Collections.sort → TimSort: a merge sort that detects already-sorted "runs" in real data; stable, O(n) best case on nearly-sorted input, O(n log n) worst.
  • C++'s std::sort uses introsort: quicksort that switches to heapsort if recursion gets too deep — killing the O(n²) worst case. Good trivia when asked "how do libraries avoid bad pivots?"
Why stability matters in real code. Sort employees by salary, then by department: if the second sort is stable, each department's employees stay salary-ordered — you've built "sort by department, then salary" out of two passes. Unstable sorts destroy that. This is exactly why Java forces stable TimSort for objects.

6. Quickselect — the K-th element without sorting

Quicksort's partition places the pivot at its final index. If you only need the K-th smallest element, recurse into just the side containing K: n + n/2 + n/4 + … ⇒ average O(n), versus O(n log n) for a full sort. This powers "K-th largest element" (worked below) and is the one-shot alternative to heaps (page 12 compares them).

Pattern templates (Java)

Merge sort — the template (also the shape of "count inversions" problems)

public static void mergeSort(int[] a, int lo, int hi) {   // sorts a[lo..hi]
    if (lo >= hi) return;                 // base case: 0 or 1 element is sorted
    int mid = lo + (hi - lo) / 2;         // overflow-safe midpoint
    mergeSort(a, lo, mid);                // sort left half
    mergeSort(a, mid + 1, hi);            // sort right half
    merge(a, lo, mid, hi);                // two-pointer merge of the sorted halves
}

private static void merge(int[] a, int lo, int mid, int hi) {
    int[] tmp = new int[hi - lo + 1];     // the O(n) extra space
    int i = lo, j = mid + 1, t = 0;
    while (i <= mid && j <= hi)
        // <= keeps equal elements' left-first order ⇒ stability
        tmp[t++] = a[i] <= a[j] ? a[i++] : a[j++];
    while (i <= mid) tmp[t++] = a[i++];   // drain leftovers (only one loop runs)
    while (j <= hi)  tmp[t++] = a[j++];
    System.arraycopy(tmp, 0, a, lo, tmp.length);
}

Comparators — the Java you'll write in every sorting-adjacent problem

int[][] intervals = ...;
// sort 2D arrays by first column (start time) — intervals, meetings, events
Arrays.sort(intervals, (x, y) -> Integer.compare(x[0], y[0]));

// multi-key: by length ascending, then alphabetically — Comparator composition
List<String> words = ...;
words.sort(Comparator.comparingInt(String::length)
                     .thenComparing(Comparator.naturalOrder()));

// descending: reverseOrder needs boxed types (Integer[], not int[])
Arrays.sort(boxed, Collections.reverseOrder());
Comparator bug to avoid. Never write (x, y) -> x[0] - y[0] — the subtraction overflows for values like Integer.MIN_VALUE and silently mis-sorts. Always Integer.compare(x[0], y[0]). Interviewers notice.

Decision cues

  • Problem bounds the values to a small range → counting sort O(n), mention it even if you then use a library sort.
  • "K-th largest / smallest", one-shot → quickselect O(n); streaming/repeated → heap (12).
  • Only 3 distinct values (0/1/2) → Dutch national flag one-pass partition (worked below).
  • Custom order ("largest concatenation", "closest to origin") → library sort + comparator; your job is only the comparator.
  • "Nearly sorted" input → say TimSort is O(n) here; or an insertion-sort/heap argument for k-out-of-place variants.

Where this shows up in real systems

TimSort runs the world's sorting

Invented for Python, adopted by Java (Collections.sort), Android, and V8's Array.prototype.sort. Its bet — real data arrives in partially sorted runs — pays off so often that a 2002 heuristic became the planetary default.

External sort & LSM-tree compaction

When data exceeds RAM (database spills, MapReduce shuffle), systems sort chunks in memory, write sorted "runs" to disk, then K-way merge them — merge sort at terabyte scale. RocksDB/Cassandra compaction is this merge running forever in the background (see capstone).

Radix sort in graphics & networking

GPUs radix-sort millions of primitives per frame for rendering order; routers and flow analyzers radix-bucket packets by IP prefix — comparison-free sorting where keys are fixed-width bits.

Top-K everywhere

"Trending now" and leaderboard features don't fully sort a billion items; they quickselect or heap-select the top 100 — the exact interview distinction between O(n log n), O(n log k) and O(n).

Worked solution: Sort Colors (Dutch National Flag) — Medium

Problem. Array of 0s, 1s, 2s — sort in place, one pass, no counting allowed.

Idea. Maintain three zones via two boundaries: everything left of lo is 0, everything right of hi is 2, between lo and i is 1, and i..hi is unexplored. Swap each element into its zone. This is quicksort's partition generalized to three parts.

public void sortColors(int[] nums) {
    int lo = 0;                    // next slot for a 0 (zone boundary)
    int hi = nums.length - 1;      // next slot for a 2 (zone boundary)
    int i = 0;                     // current element under inspection
    while (i <= hi) {              // hi side is UNSORTED territory — stop when we meet it
        if (nums[i] == 0) {
            swap(nums, i, lo);     // send 0 to the left zone
            lo++; i++;             // the swapped-in value came from the 1-zone — safe to advance
        } else if (nums[i] == 2) {
            swap(nums, i, hi);     // send 2 to the right zone
            hi--;                  // do NOT advance i: the swapped-in value is UNSEEN — recheck it
        } else {
            i++;                   // 1 is already in the middle zone
        }
    }
}
private void swap(int[] a, int i, int j) { int t = a[i]; a[i] = a[j]; a[j] = t; }

Complexity. Time O(n) — each step either advances i or shrinks hi. Space O(1). The "don't advance i after the hi-swap" line is the classic bug — say why out loud.

Worked solution: Kth Largest Element (quickselect) — Medium

Problem. Return the K-th largest element (not distinct) of an unsorted array.

Idea. K-th largest = element at index n - k in sorted order. Partition places the pivot at its final index; recurse only into the side containing n - k. Randomized pivot makes the adversarial O(n²) practically impossible.

private static final Random RAND = new Random();

public int findKthLargest(int[] nums, int k) {
    int target = nums.length - k;              // index the answer occupies in sorted order
    int lo = 0, hi = nums.length - 1;
    while (true) {
        int p = partition(nums, lo, hi);       // pivot lands at its FINAL sorted index p
        if (p == target) return nums[p];       // pivot is exactly the K-th largest — done
        if (p < target) lo = p + 1;            // answer lies right of p — discard left side
        else hi = p - 1;                       // answer lies left of p — discard right side
    }
}

// Lomuto partition with a random pivot: returns the pivot's final index
private int partition(int[] a, int lo, int hi) {
    int r = lo + RAND.nextInt(hi - lo + 1);    // random pivot defeats sorted-input worst case
    swap(a, r, hi);                            // park pivot at the end
    int pivot = a[hi];
    int i = lo;                                // a[lo..i-1] = the "< pivot" zone
    for (int j = lo; j < hi; j++)
        if (a[j] < pivot) swap(a, i++, j);     // grow the smaller-zone by one
    swap(a, i, hi);                            // drop pivot just after the smaller-zone
    return i;                                  // pivot's final resting index
}
private void swap(int[] a, int i, int j) { int t = a[i]; a[i] = a[j]; a[j] = t; }

Complexity. Average O(n) time (n + n/2 + n/4 + … = 2n), O(1) space (iterative). Compare: full sort O(n log n); size-k min-heap O(n log k) — better for streams (page 12).

Worked solution: Largest Number (custom comparator) — Medium

Problem. Arrange non-negative ints so their concatenation is the largest number: [10, 2] → "210".

Idea. For any two strings a, b: put a first exactly when a+b > b+a as strings ("9"+"34"="934" beats "34"+"9"="349"). This comparator is a valid total order, so sorting with it maximizes the whole concatenation — a taste of the greedy exchange argument from page 21.

public String largestNumber(int[] nums) {
    // work with strings — concatenation order is what we're optimizing
    String[] s = new String[nums.length];
    for (int i = 0; i < nums.length; i++) s[i] = String.valueOf(nums[i]);

    // order pair (a, b) by which concatenation is bigger: "b+a vs a+b" descending
    Arrays.sort(s, (a, b) -> (b + a).compareTo(a + b));

    if (s[0].equals("0")) return "0";     // all zeros collapse to a single "0"

    StringBuilder sb = new StringBuilder(); // O(n) build — never += in a loop (page 00!)
    for (String part : s) sb.append(part);
    return sb.toString();
}

Complexity. O(n log n · L) time where L is average digit length (each comparison costs O(L)); O(n·L) space for the string copies.

Pitfalls & interview tips

  • "Just sort it" isn't free. Before sorting, say what it costs (O(n log n)) and check a hash-based O(n) alternative isn't cleaner. Sorting mutates — copy first if indexes matter later.
  • Subtraction comparators overflow. a - b breaks on extreme values; use Integer.compare / Long.compare. Also never return inconsistent comparators — TimSort throws IllegalArgumentException: Comparison method violates its general contract.
  • Primitives can't take comparators. To sort int[] descending, either sort ascending and read backwards, or box to Integer[] (O(n) extra). Know both before the interview.
  • Quicksort's worst case is a fair question. Answer: O(n²) on adversarial pivots; fixed by random pivot / median-of-three / introsort's heapsort fallback.
  • Stability only matters for objects. If asked "is your sort stable?" for an int[], point out equal ints are interchangeable — that's why the JDK dares use unstable dual-pivot quicksort there.
  • Merge from the back when merging into an array that has free space at the end (Merge Sorted Array) — forward merging overwrites unread elements.

Binary Search: Halving Your Way to O(log n)

One invariant-driven template, its boundary variants, and the "search on the answer" trick that quietly solves hard problems.

Phase 2 · Week 4 Top-5 pattern — the "search on answer" trick unlocks hards

Before you start

Why this topic exists. Binary search is the purest expression of the interview's favorite move: use structure to discard work. If the data (or the answer space) is ordered, one comparison eliminates half of everything remaining — a million candidates collapse in ~20 steps. The catch: it's famously easy to get subtly wrong. Jon Bentley found that 90% of professional programmers couldn't write a bug-free binary search, and even the JDK's own version shipped with an overflow bug for nine years. This page gives you one template and one mental model that make it hard to get wrong.

Prerequisites

  • 00 · Foundations — why halving means O(log n).
  • 06 · Sorting — binary search's price of admission is sorted data (or a monotonic answer).

What the interviewer is actually testing

  • Can you avoid the classic bugs? Infinite loops, off-by-one boundaries, and the (lo+hi)/2 overflow are exactly what they watch for.
  • Do you know the boundary variants? "Find the first occurrence" is a different template than "find any occurrence" — most candidates fumble it.
  • Can you spot hidden binary search? Nothing in "minimum eating speed" says "sorted array". Recognizing a monotonic answer space is the senior-level signal.
  • Can you state the invariant? Saying "everything left of lo is known-false, everything from hi on is known-true" proves you're reasoning, not pattern-matching.

How this builds on the previous topic

Page 06 established what sortedness costs to obtain — O(n log n). This page is the payoff: once order exists, membership queries drop from O(n) to O(log n), which is why you sort at all. And the deeper idea carries forward: binary search doesn't actually need a sorted array, it needs a monotonic predicate — a yes/no question whose answers look like F F F F T T T. That reframing powers half the problems below.

Core concepts

1. The classic search, done safely

Keep a range [lo..hi] that is guaranteed to contain the target if it exists. Probe the middle; one comparison tells you which half to throw away. Repeat until the range is empty or you hit the target.

The classic template — memorize this exact shape

// Returns the index of target, or -1. Works on any sorted int[].
public static int binarySearch(int[] nums, int target) {
    int lo = 0, hi = nums.length - 1;      // INVARIANT: if target exists, its index is in [lo..hi]
    while (lo <= hi) {                     // <= because [lo..lo] is still a valid 1-element range
        int mid = lo + (hi - lo) / 2;      // overflow-safe midpoint — see the warning below
        if (nums[mid] == target) return mid;
        if (nums[mid] < target) lo = mid + 1;  // mid is too small: mid and everything left of it is out
        else                    hi = mid - 1;  // mid is too big: mid and everything right of it is out
    }
    return -1;                             // range emptied — target isn't there
}
The overflow pitfall. The "obvious" midpoint mid = (lo + hi) / 2 is a real bug: for arrays near a billion elements, lo + hi can exceed Integer.MAX_VALUE and wrap negative. This exact bug sat in the JDK's Arrays.binarySearch from 1997 until 2006. Always write mid = lo + (hi - lo) / 2 — the subtraction can't overflow because hi ≥ lo. Mentioning this unprompted is an easy senior point.
No infinite loops, guaranteed. The loop always terminates because every branch strictly shrinks the range: lo = mid + 1 or hi = mid - 1 — never bare lo = mid. If you ever write lo = mid in a lo <= hi loop, a 2-element range can compute mid == lo and spin forever. When a variant genuinely needs to keep mid in play, switch to the boundary template below, whose lo < hi guard plus hi = mid is safe.

2. The mental model that unifies everything: a predicate over F...FT...T

Here's the reframe that turns binary search from a memorized ritual into a tool you can adapt. Forget arrays for a second. Imagine any yes/no question P(x) over an ordered range of candidates, with one property: once it becomes true, it stays true: 

candidates:  x0   x1   x2   x3   x4   x5   x6   x7
P(x):         F    F    F    T    T    T    T    T
                             ↑
                             the boundary — binary search finds THIS

Binary search is really a boundary finder for any such monotonic predicate. Every variant is just a different choice of predicate:

  • First position of target → P(i) = "nums[i] ≥ target".
  • Last position of target → one left of the first i where "nums[i] > target".
  • First Bad Version → P(v) = "isBadVersion(v)" (no array anywhere!).
  • Koko's eating speed → P(s) = "at speed s, she finishes in time" — slow speeds fail, fast speeds work.

The boundary template — "first index where the predicate is true"

// Returns the FIRST index where p(i) is true, or n if it's never true.
// Requires: p is monotonic (false...false, true...true).
public static int firstTrue(int n, java.util.function.IntPredicate p) {
    int lo = 0, hi = n;                    // hi is EXCLUSIVE — "n" encodes "never true"
    while (lo < hi) {                      // INVARIANT: answer is always inside [lo..hi]
        int mid = lo + (hi - lo) / 2;
        if (p.test(mid)) hi = mid;         // mid is true: it might BE the first true — keep it
        else             lo = mid + 1;     // mid is false: first true is strictly right of mid
    }
    return lo;                             // lo == hi: the range closed onto the boundary
}

Notice the two deliberate differences from the classic template: the loop guard is lo < hi (we stop when the range is a single point — the answer), and the true-branch keeps mid via hi = mid rather than discarding it. This pair is safe together because mid is always strictly less than hi, so the range still shrinks every iteration.

3. Rotated sorted arrays: order survives, mostly

Take a sorted array and rotate it: [4,5,6,7,0,1,2]. Global order is broken, but a crucial local fact survives: cut it anywhere, and at least one of the two halves is perfectly sorted (the rotation point can only be in one of them). A sorted half is checkable in O(1) — compare its endpoints — so you can still decide which half to discard, and O(log n) survives. The worked solution below shows the exact decision table.

4. Search on the ANSWER: when the input isn't sorted at all

This is the trick that turns mediums and hards into templates. Many optimization problems have the shape "find the minimum capacity / speed / size that satisfies a constraint". The inputs are unsorted — but the candidate answers are monotonic: if capacity 10 works, capacity 11 certainly works. That's exactly a F F F T T T predicate, so binary-search the answer range and write a canDo(x) checker: 

// The "search on answer" recipe:
// 1. Identify the answer range [lo..hi]  (e.g. speed 1 .. max pile size)
// 2. Write boolean canDo(x) — usually a simple O(n) greedy check
// 3. Confirm monotonicity: canDo(x) true ⇒ canDo(x+1) true
// 4. firstTrue over [lo..hi]  →  total cost O(n log(range))
How to spot it. Trigger phrases: "minimum speed so that…", "least capacity to ship within D days", "smallest divisor such that…", "minimize the maximum…". If checking a given answer is easy but finding the best answer looks hard, you're almost certainly meant to binary-search the answer space.

Pattern templates (Java)

Two templates cover every problem on this page. Know when each applies:

Template A — exact match (classic): find target's index or -1

int lo = 0, hi = n - 1;                    // inclusive range
while (lo <= hi) {                         // 1-element range still gets probed
    int mid = lo + (hi - lo) / 2;
    if (nums[mid] == target) return mid;   // can exit early
    if (nums[mid] < target) lo = mid + 1;  // both branches EXCLUDE mid
    else                    hi = mid - 1;
}
return -1;

Template B — boundary (first true): for duplicates, insert position, answer-space search

int lo = 0, hi = n;                        // hi EXCLUSIVE; n means "no true exists"
while (lo < hi) {                          // stop at a single point — no early exit
    int mid = lo + (hi - lo) / 2;
    if (predicate(mid)) hi = mid;          // true: keep mid as a candidate answer
    else                lo = mid + 1;      // false: discard mid and all before it
}
return lo;                                 // the boundary (first true index)
Which template? Ask: "do I need any match, or a boundary?" Unique elements + "does it exist" → Template A. Duplicates, first/last occurrence, insert position, min-that-works → Template B. When in doubt, Template B answers everything Template A does (search firstTrue(nums[i] ≥ target), then check the value) — many strong candidates use only B.

Decision cues:

  • Problem says "sorted array" → binary search is almost certainly the intended tool.
  • Problem asks for O(log n) explicitly → it's binary search (possibly twice, or on a rotated array).
  • "Find first/last/leftmost/rightmost…" → Template B boundary search.
  • "Minimum X such that condition holds" with an easy checker → search on the answer space.
  • Unsorted input, exact-match lookup, no order requirement → hash, not search (page 01).

Where this shows up in real systems

git bisect

Finding the commit that broke the build is exactly "first bad version": commits are ordered, and once a bug is introduced it stays introduced — a monotonic predicate. git bisect binary-searches history, so 4,096 commits take 12 test runs, not 4,096. Arguably the most-used debugging algorithm in industry.

Database indexes: B-tree node search

Every lookup in PostgreSQL or MySQL's InnoDB descends a B-tree; inside each node, the engine binary-searches a sorted array of keys to pick the child page. Same story inside SSTable index blocks in Cassandra/RocksDB. Your WHERE id = ? runs binary search several times per query.

Feature-flag & release bisection

When a metric regresses after a rollout, platforms like Chrome's bisect-builds tooling binary-search across build numbers to isolate the culprit build, and experiment platforms bisect config changes. "Once broken, stays broken" monotonicity is what makes it valid.

Autoscaling & capacity search

"Smallest instance count that keeps p99 under 200ms" is search-on-answer in production: capacity is monotonic (more replicas never raises latency), the checker is a load test. Kubernetes VPA-style recommenders and load-test harnesses walk this boundary just like Koko's bananas.

★ Worked: Find First and Last Position of Element in Sorted Array (LC 34)

Problem in one line: in a sorted array with duplicates, return the first and last index of target, or [-1,-1] — in O(log n).

Idea: Template A can land on any duplicate, so it can't answer this. Instead find two boundaries with Template B: the first index where nums[i] ≥ target is the left edge; the first index where nums[i] ≥ target+1 is one past the right edge. Two boundary searches, no scanning — duplicates never hurt because we never walk along equal values.

public static int[] searchRange(int[] nums, int target) {
    // First index whose value is >= target  (first position of target, if present)
    int first = lowerBound(nums, target);
    // If we ran off the end, or the value there isn't target, target is absent
    if (first == nums.length || nums[first] != target) return new int[]{-1, -1};
    // First index whose value is >= target+1 sits one past the LAST target
    int last = lowerBound(nums, target + 1) - 1;
    return new int[]{first, last};
}

// Classic "first index where predicate is true" template.
// Predicate here: nums[i] >= target. It's false,false,...,true,true — monotonic.
static int lowerBound(int[] nums, int target) {
    int lo = 0, hi = nums.length;          // hi is EXCLUSIVE: answer may be nums.length ("never true")
    while (lo < hi) {                      // stop when the range is empty: lo == hi is the answer
        int mid = lo + (hi - lo) / 2;      // overflow-safe midpoint (never write (lo+hi)/2)
        if (nums[mid] >= target) {
            hi = mid;                      // mid satisfies the predicate: it COULD be the first true — keep it in range
        } else {
            lo = mid + 1;                  // mid is false: the first true must be strictly to the right
        }
    }
    return lo;                             // lo == hi == first index where nums[i] >= target
}

Complexity: time O(log n) — two boundary searches; space O(1). Runnable version: code/07-binary-search/SearchRange.java.

★ Worked: Search in Rotated Sorted Array (LC 33)

Problem in one line: a sorted, distinct array was rotated at an unknown pivot ([4,5,6,7,0,1,2]); find target's index in O(log n).

Idea: at any mid, the rotation point lies in only one half — so the other half is perfectly sorted, and nums[lo] ≤ nums[mid] tells you which (using ≤ handles the 1-element half). A sorted half's endpoints tell you in O(1) whether the target is inside it: if yes, search it; if no, the target must be in the messy half. Either way, half the range is discarded per step.

public static int search(int[] nums, int target) {
    int lo = 0, hi = nums.length - 1;      // inclusive range this time — we're hunting an exact index
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;      // overflow-safe midpoint
        if (nums[mid] == target) return mid;

        if (nums[lo] <= nums[mid]) {
            // Left half [lo..mid] is sorted (no rotation point inside it)
            if (nums[lo] <= target && target < nums[mid]) {
                hi = mid - 1;              // target sits inside the sorted left half — search there
            } else {
                lo = mid + 1;              // otherwise it can only be in the (possibly rotated) right half
            }
        } else {
            // Right half [mid..hi] is sorted
            if (nums[mid] < target && target <= nums[hi]) {
                lo = mid + 1;              // target sits inside the sorted right half
            } else {
                hi = mid - 1;              // otherwise it's in the rotated left half
            }
        }
    }
    return -1;                             // range emptied without a hit — not present
}

Complexity: time O(log n), space O(1). (With duplicates — LC 81 — the nums[lo] == nums[mid] case becomes ambiguous and worst case degrades to O(n).) Runnable version: code/07-binary-search/SearchRotated.java.

★ Worked: Koko Eating Bananas (LC 875)

Problem in one line: given banana piles and h hours (one pile per hour max, eating k bananas/hour), find the minimum speed k that finishes everything in time.

Idea: nothing here is sorted — but the answers are monotonic: if speed 4 works, every speed above 4 works. So the speeds form F F F T T T and we binary-search for the first true. The checker is a one-pass simulation: hours for a pile of p at speed s is ceil(p/s). Answer range: 1 to the largest pile (faster than that changes nothing).

public static int minEatingSpeed(int[] piles, int h) {
    int lo = 1, hi = 0;                    // candidate speeds: 1 .. max(pile)
    for (int p : piles) hi = Math.max(hi, p);  // eating faster than the biggest pile is pointless
    while (lo < hi) {                      // "first true" template: shrink to a single candidate
        int mid = lo + (hi - lo) / 2;      // overflow-safe midpoint speed to test
        if (canFinish(piles, mid, h)) {
            hi = mid;                      // mid works — the minimum is mid or something slower
        } else {
            lo = mid + 1;                  // mid too slow — minimum must be faster
        }
    }
    return lo;                             // lo == hi == slowest speed that still finishes in h hours
}

// The predicate: can Koko finish every pile within h hours at this speed?
static boolean canFinish(int[] piles, int speed, int h) {
    long hours = 0;                        // long: up to 10^4 piles * 10^9 bananas — int would overflow
    for (int p : piles) {
        hours += (p + (long) speed - 1) / speed;   // ceil(p / speed) without floating point
    }
    return hours <= h;                     // monotonic: a faster speed never needs MORE hours
}

Complexity: time O(n · log(max pile)) — each of ~30 probes runs an O(n) check; space O(1). Runnable version: code/07-binary-search/KokoBananas.java.

Pitfalls & interview tips

  • Never write (lo + hi) / 2. Use lo + (hi - lo) / 2 and say why (int overflow — the nine-year JDK bug). One sentence, instant credibility.
  • Match the guard to the moves. Template A: lo <= hi with mid ± 1 moves. Template B: lo < hi with hi = mid. Mixing them (e.g. lo <= hi with hi = mid) is the #1 infinite-loop recipe.
  • Test the tiny cases by hand: empty array, one element, two elements, target absent, target at each end. Nearly every binary-search bug shows up on n ≤ 2 — trace before declaring done.
  • Duplicates change the problem. "Find target" is Template A; "find first/last target" needs Template B. Say which variant you're using and why.
  • Know the JDK quirk: Arrays.binarySearch returns -(insertionPoint) - 1 for misses and an arbitrary matching index for duplicates — fine in application code, but interviewers want the hand-rolled version.
  • Say the invariant out loud. "Everything before lo is known too small; everything after hi is known too big." If every branch of your code preserves that sentence, the code is correct — and the interviewer hears you prove it, not guess it.

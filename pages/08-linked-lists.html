Linked Lists: Pointer Fluency Under Pressure

Dummy heads, the three-pointer reversal, and fast/slow tricks — the screener classics that test whether you can rewire references without dropping the list.

Phase 3 · Week 5 Screener classic — a pointer-fluency test more than a data-structure test

Before you start

Why this topic exists. Nobody is going to ask you to build a production system on LinkedList — cache locality killed that idea (details below). Interviewers use linked lists for a different reason: they are the cheapest way to test whether you can manipulate references precisely. One wrong assignment order and the rest of the list is garbage-collected out from under you. The same reference discipline is exactly what trees (page 11) and the LRU cache (page 24) demand — this page is where you build it.

Prerequisites

  • 00 · Foundations — the ArrayList vs LinkedList cost table.
  • 02 · Two Pointers — fast/slow was introduced there; here it gets its full treatment.

Interview relevance: a screener classic — pointer fluency with dummy heads and fast/slow pairs is the skill under test.

How this builds on the previous topic

Binary search (page 07) exploited O(1) random access — jump to any index instantly. Linked lists are the opposite trade: no random access at all (getting element i costs O(i), which is why you can't binary-search a linked list), but O(1) splicing anywhere you already hold a reference. Every technique on this page is about navigating cleverly under that constraint — and it's your first step into linked structures, where trees and graphs are "linked lists with more pointers".

Core concepts

1. The node model: Java references are pointers with seatbelts

A linked list is just objects pointing at each other. In Java there's no pointer arithmetic, but a ListNode next field is exactly a pointer: a value that names another object (or null). 

// The whole data structure is two fields:
static class ListNode {
    int val;
    ListNode next;      // reference to the following node, or null at the tail
    ListNode(int val) { this.val = val; }
}
// head ──▶ [1|•] ──▶ [2|•] ──▶ [3|∅]
// Losing `head` (or any node's only incoming reference) loses everything after it — the GC takes it.
The one rule of pointer surgery. Before overwriting a reference you still need, save it. Nearly every linked-list bug is an assignment executed one line too early. When stuck, narrate each line: "who points where now?" — and draw it.

It's worth knowing what a node actually costs, because it explains this page's central irony (spelled out under "When to use what"). In current JDKs, that little ListNode is roughly 24 bytes with compressed references — a 12-byte object header, 4 bytes of val, 4 bytes for next, padded to an 8-byte boundary — and up to 32 bytes without compression. Call it 24–32 bytes of heap for 4 bytes of payload; java.util.LinkedList is heavier still, since its doubly linked node adds prev plus a reference to a boxed Integer (~16 more bytes). Worse than the size is the layout: nodes land wherever the allocator put them, so every .next hop is a dependent load the CPU cannot begin until the previous one resolves — the hardware prefetcher, which happily streams a contiguous array 16 ints per 64-byte cache line, gets nothing to predict, and each hop risks a ~100 ns cache miss. That's the mechanical reason ArrayDeque (page 09) beats LinkedList at its own queue game, and why the linked list's one honest production victory is the LRU cache's doubly linked list (page 24) — the case where a HashMap hands you the exact node and the pointer-chasing walk never happens at all.

2. The dummy head: delete your special cases

Operations at the head ("what if the node I remove is the head?") normally need special-case code. The fix is one line: allocate a fake node in front, work exclusively with .next pointers of predecessors, and return dummy.next at the end. The head is now just another node with a predecessor. You'll see it carry both worked solutions below. 

ListNode dummy = new ListNode(0);   // value never read — it exists only to be a predecessor
dummy.next = head;
// ... operate via predecessor.next, never via `head` directly ...
return dummy.next;                  // the true head, even if the original was removed

3. Reversal: the three-pointer dance

Reverse a list in place by walking it once, flipping each next arrow backwards. You need three references — the reversed prefix (prev), the node being flipped (curr), and a saved handle on the unvisited remainder (next) so the flip doesn't orphan it. Watch the invariant: at every step, prev heads a fully reversed sublist and curr heads the untouched rest.

4. Fast & slow: two speeds, three superpowers

Run two pointers down the list, one advancing 1 node per step, the other 2. This single trick yields:

  • Middle of the list: when fast hits the end, slow has covered half the distance — it's standing on the middle. One pass, no length count.
  • Cycle detection (Floyd): if there's a cycle, fast can never "finish" — both pointers eventually orbit the cycle, and they must meet.
  • Nth from the end: a variant — give one pointer an n-node head start and move both at the same speed (worked solution below).
Why Floyd's meeting is guaranteed — the actual math. Once both pointers are inside the cycle (length C), look at the gap from fast to slow: each step, fast closes it by exactly 1 (moves 2, slow moves 1). A gap that shrinks by 1 each step must reach 0 — they collide; fast can never "jump over" slow. Bonus fact for the follow-up (LC 142): if the meeting happens after slow walks distance a + b (a = head→cycle-start, b = into the cycle), fast walked 2(a+b), and the difference a+b is a whole number of laps — which rearranges to: head-to-cycle-start equals meeting-point-to-cycle-start (mod C). So restart one pointer at the head, advance both at speed 1, and they meet exactly at the cycle's entrance. 
// Floyd's cycle detection — O(1) space (vs. the obvious HashSet's O(n))
public static boolean hasCycle(ListNode head) {
    ListNode slow = head, fast = head;
    while (fast != null && fast.next != null) {  // fast does the null-checking for both
        slow = slow.next;                        // +1
        fast = fast.next.next;                   // +2 — closes the gap by 1 per step inside a cycle
        if (slow == fast) return true;           // reference equality: same NODE, not equal values
    }
    return false;                                // fast found a null — no cycle possible
}

Because page 02 and the cycle problems below all lean on it, here is the full Floyd proof, slowly — this is the canonical treatment in the runbook. Name three distances: a = head to the cycle's entrance, b = entrance to the meeting point (measured forward along the cycle), C = the cycle's length.

  • They meet, and quickly. Slow reaches the entrance after a steps; fast is already somewhere in the cycle. From then on the gap from fast to slow shrinks by exactly 1 per step (the tip above), so they collide within at most C further steps. Total: slow walks at most a + C ≤ n steps — the detection is O(n) time, and slow never completes even one full lap before being caught.
  • The meeting point is special. At the collision, slow has walked a + b. Fast, at double speed, has walked exactly 2(a + b). Fast's surplus distance, 2(a + b) − (a + b) = a + b, was spent entirely going around in circles — so it must be a whole number of laps: a + b = kC for some integer k ≥ 1.
  • Head and meeting point are equidistant from the entrance. Rearrange: a = kC − b. Now start one pointer at the head and leave the other at the meeting point, both moving at speed 1. After a steps the first stands at the entrance by definition. The second has moved a = kC − b forward from a point b past the entrance — position b + (kC − b) = kC along the cycle, which is 0 mod C: also the entrance. They arrive together, so the first place they can ever be equal is the entrance — which is exactly the phase-2 loop of Linked List Cycle II below.
Why interviewers care about the derivation. Reciting phase 2 without the k is where candidates get caught: fast may lap the cycle several times before the meeting (whenever a > C), so the surplus is kC, not C. The proof never needed k's value — only that the surplus is some multiple of the cycle length. If you can reproduce the three bullets above unprompted, the entire fast/slow family is yours.

5. The rest of the standard kit

  • Merge two sorted lists — dummy head + always-take-the-smaller. Worked below; it's also the heart of Merge k Sorted Lists (page 12) and merge sort on lists.
  • Reorder List (L0→Ln→L1→Ln-1…) — a three-trick combo: fast/slow finds the middle, reverse the second half, then interleave the two halves. If you know this decomposition, the "hard-looking" problem is three easies.
  • Deep copy with random pointers (LC 138) — either a HashMap<Node,Node> from originals to clones (O(n) space, obvious), or the elegant O(1)-space interleave: weave each clone in right after its original (A→A'→B→B'), set clone.random = orig.random.next, then unweave.
Merge k lists — two roads to the same O(N log k). With N total nodes across k lists, the min-heap of current heads (page 12) pays O(log k) per node output. The divide-and-conquer alternative merges lists pairwise like a tournament bracket: ⌈log k⌉ rounds, each touching all N nodes with the two-list merge above — O(N log k) again, O(1) extra space where the heap costs O(k). The unifying picture is a tournament tree: every output node wins ⌈log k⌉ head-to-head comparisons on its way out, and the two methods just schedule those comparisons differently (heap = lazily per element, D&C = eagerly per round). Naive fold-left merging is the trap answer: early lists get re-walked in every later merge, O(N·k).

6. Exotic variants worth recognizing

None of these are "implement it live" material — but each is a one-sentence answer to a follow-up that separates seniors from script-followers.

  • Sentinel circular doubly linked list — the production workhorse. One permanent dummy node whose next/prev close the ring; an empty list is the sentinel pointing at itself. Every insert and unlink becomes the same four assignments with zero null checks — no head case, no tail case, no empty case. This is exactly how LinkedHashMap threads its entries and how the LRU cache on page 24 should be written; the Linux kernel's ubiquitous list_head is the same design in C.
  • Unrolled linked lists — each node carries a small array of up to B elements instead of one. Pointer overhead drops ~B-fold and each hop delivers a full cache line of payload — a direct fix for the memory math in §1. Text-editor rope/gap structures and some deque implementations use this layout.
  • XOR lists — trivia with a lesson: store prev ⊕ next in a single field and recover either neighbor given the other, halving pointer memory. Impossible in Java, and saying why is the point — the GC must be able to see real references, and an XOR of two addresses isn't one.
  • Skip lists — a sorted linked list with randomized "express lanes" stacked on top, giving O(log n) expected search/insert without rotations. Java ships one as ConcurrentSkipListMap; the expected-height argument lives on page 24.

Pattern templates (Java)

Template A — dummy head + predecessor walking (insert/delete/merge)

ListNode dummy = new ListNode(0);
dummy.next = head;
ListNode pred = dummy;                     // pred = last node of the already-processed prefix
while (pred.next != null) {
    if (shouldRemove(pred.next)) {
        pred.next = pred.next.next;        // bypass — do NOT advance pred (next node is unvetted)
    } else {
        pred = pred.next;                  // keep — advance
    }
}
return dummy.next;                         // correct even if the original head was removed

Template B — in-place reversal (the three-pointer loop)

ListNode prev = null, curr = head;
while (curr != null) {
    ListNode next = curr.next;             // 1. SAVE the remainder first
    curr.next = prev;                      // 2. flip the arrow
    prev = curr;                           // 3. reversed prefix grows
    curr = next;                           // 4. advance into the saved remainder
}
// prev is the new head — memorize this 4-line body until it's automatic

Template C — fast/slow (middle, cycle, nth-from-end)

ListNode slow = head, fast = head;
while (fast != null && fast.next != null) {   // guard BOTH hops fast will take
    slow = slow.next;
    fast = fast.next.next;
}
// even length: slow = second middle; start fast at head.next for the first middle

The dirty secret: LinkedList almost never wins in practice. Its O(1) splicing only pays if you're already standing at the splice point; getting there is O(n) of pointer-chasing, where every hop is a potential CPU cache miss because nodes are scattered across the heap. ArrayList's O(n) "shift" is a contiguous System.arraycopy the hardware races through. Even Joshua Bloch (who wrote LinkedList) jokes that nobody should use it. For queue/deque needs, use ArrayDeque (page 09). So why is this page here at all? Because interviews use lists to test pointer manipulation — and because intrusive linked lists genuinely power real systems where you already hold the node: see below.

Decision cues:

  • Problem hands you a ListNode → it's a pointer-manipulation exercise; think dummy head, reversal, fast/slow.
  • "Detect a cycle" / "find where the cycle begins" → Floyd's fast/slow.
  • "Middle", "nth from end", "palindrome list" → fast/slow (+ reversal for palindrome).
  • "In O(1) space" on a list problem → in-place reversal or interleaving is intended; a HashMap is the O(n)-space fallback worth mentioning first.
  • Real Java code needing a sequence → ArrayList; needing both-ends ops → ArrayDeque; needing O(1) removal of a node you hold a handle to → hand-rolled DLL (LRU, page 24).

Where this shows up in real systems

LRU caches: HashMap + doubly linked list

The #1 design interview question (page 24) and the eviction backbone of real caches: a DLL keeps entries in recency order — move-to-front and evict-from-tail are O(1) because the map hands you the exact node, no walking. This is the case where a linked list beats an array: removal from the middle when you already hold the node. Linux's page LRU lists work the same way.

Allocator free lists

Memory allocators (glibc malloc's bins, kernel slab allocators, JVM free-region tracking) chain free blocks into linked lists — the pointer is stored inside the free block itself, so bookkeeping is free and coalescing/splitting blocks is O(1) splicing. Intrusive lists like these are everywhere in systems code.

HashMap collision chaining

Java's own HashMap resolves collisions by chaining entries into a singly linked list per bucket (treeifying long chains, page 01). Every get on a collided key walks a linked list. LinkedHashMap additionally threads a DLL through all entries for iteration order — which is exactly why it trivializes LRU.

Undo chains & MRU playlists

Editor undo/redo systems keep operations in a doubly linked chain — O(1) append, O(1) walk back and forward, cheap truncation when you undo-then-edit. Media players' "recently played" and browser-tab MRU ordering are small LRU/MRU lists: move-to-front on every access.

easy Reverse Linked List — LeetCode 206

The problem. Reverse a singly linked list in place and return the new head.

Example. 1→2→3→4→5 → 5→4→3→2→1. Empty list → empty list.

Variation 1 — iterative three-pointer flip (the default) · O(n) time, O(1) space

// Iterative: walk the list once, flipping each .next backwards.
public static ListNode reverseList(ListNode head) {
    ListNode prev = null;                  // everything BEFORE curr, already reversed (starts empty)
    ListNode curr = head;                  // the node whose arrow we're about to flip
    while (curr != null) {
        ListNode next = curr.next;         // 1. save the rest of the list BEFORE we cut the link
        curr.next = prev;                  // 2. flip: curr now points backwards
        prev = curr;                       // 3. the reversed prefix grows by one node
        curr = next;                       // 4. step into the saved remainder
    }
    return prev;                           // curr fell off the end; prev is the new head
}

Variation 2 — recursive · O(n) time, O(n) call-stack space

// Recursive: trust reverseRecursive(rest) to hand back the reversed tail,
// then hook the old head onto the end of it.
public static ListNode reverseRecursive(ListNode head) {
    if (head == null || head.next == null) return head;   // base: empty or single node is already reversed
    ListNode newHead = reverseRecursive(head.next);        // leap of faith: rest is now reversed; newHead = old tail
    head.next.next = head;                 // my old successor is now the reversed tail's last node — point it back at me
    head.next = null;                      // I'm the new tail; sever my stale forward link
    return newHead;                        // the new head never changes on the way back up
}

Takeaway. The iterative flip is the single most reused fragment in list problems (Reorder List and k-Group below both embed it). The recursive form costs O(n) stack — worth knowing, and worth knowing why you'd still default to the loop. Mirror: code/08-linked-lists/ReverseLinkedList.java.

easy Merge Two Sorted Lists — LeetCode 21

The problem. Splice two sorted lists into one sorted list, reusing the existing nodes (no new allocations besides an optional dummy).

Example. 1→2→4 + 1→3→4 → 1→1→2→3→4→4. Both empty → empty.

Dummy head + take-the-smaller · O(n+m) time, O(1) space

public static ListNode mergeTwoLists(ListNode l1, ListNode l2) {
    ListNode dummy = new ListNode(0);      // fake node BEFORE the real head — kills all "is this the first node?" special cases
    ListNode tail = dummy;                 // tail always points at the last node of the merged result
    while (l1 != null && l2 != null) {     // while both lists still have candidates
        if (l1.val <= l2.val) {            // <= keeps the merge STABLE (ties keep l1's order)
            tail.next = l1;                // splice the smaller node onto the result
            l1 = l1.next;                  // advance only the list we consumed from
        } else {
            tail.next = l2;
            l2 = l2.next;
        }
        tail = tail.next;                  // result grew by one — move tail forward
    }
    tail.next = (l1 != null) ? l1 : l2;    // one list is empty; the other is sorted — attach it whole (no loop needed)
    return dummy.next;                     // the real head lives just after the dummy
}

Takeaway. This is merge sort's merge step (page 06) on nodes instead of array slots — attaching the leftover list whole is the O(1) bonus arrays don't get. Merging k lists at once upgrades this with a heap (page 12). Mirror: code/08-linked-lists/MergeTwoLists.java.

easy Linked List Cycle — LeetCode 141

The problem. Determine whether a linked list contains a cycle (some node's next points back to an earlier node), using O(1) memory.

Example. 3→2→0→−4 with −4's next pointing back to 2 → true. 1→2 with no back-link → false.

Variation 1 — seen-set of nodes · O(n) time, O(n) space

public static boolean hasCycleSet(ListNode head) {
    Set<ListNode> seen = new HashSet<>();   // identity of NODES, not values
    for (ListNode cur = head; cur != null; cur = cur.next)
        if (!seen.add(cur)) return true;      // revisited the same object -> cycle
    return false;
}

Variation 2 — Floyd's fast & slow pointers (optimal) · O(n) time, O(1) space

public static boolean hasCycle(ListNode head) {
    ListNode slow = head, fast = head;
    while (fast != null && fast.next != null) {   // fast does all the null-checking
        slow = slow.next;                          // tortoise: 1 step
        fast = fast.next.next;                     // hare: 2 steps
        if (slow == fast) return true;             // same OBJECT -> they're stuck in a loop
    }
    return false;                                  // fast found the end -> no cycle
}

Takeaway. Why they must meet: once both are inside the cycle, the gap between them shrinks by exactly 1 each step (fast gains one node per tick), so it reaches 0 in at most cycle-length steps — no jumping over. Compare nodes with == (same object), never equals (duplicate values would lie). Mirror: code/08-linked-lists/LinkedListCycle.java.

easy Middle of the Linked List — LeetCode 876

The problem. Return the middle node of a list; if the length is even, return the second of the two middles.

Example. 1→2→3→4→5 → node 3. 1→2→3→4→5→6 → node 4.

Variation 1 — count, then walk half · two passes

public static ListNode middleNodeTwoPass(ListNode head) {
    int n = 0;
    for (ListNode c = head; c != null; c = c.next) n++;   // pass 1: length
    ListNode mid = head;
    for (int i = 0; i < n / 2; i++) mid = mid.next;       // pass 2: walk n/2
    return mid;
}

Variation 2 — fast & slow (one pass) · O(n) time, O(1) space

public static ListNode middleNode(ListNode head) {
    ListNode slow = head, fast = head;
    while (fast != null && fast.next != null) {
        slow = slow.next;          // 1 step
        fast = fast.next.next;     // 2 steps — when fast finishes, slow is halfway
    }
    return slow;                   // odd length: exact middle; even: second middle
}

Takeaway. The speed ratio IS the fraction: fast at 2× leaves slow at 1/2. Where slow lands on even lengths depends on the starting offset — fast = head gives the second middle; fast = head.next gives the first (which is what Sort List's split on page 06 needs). Trace a 2-node list to see the difference. Mirror: code/08-linked-lists/MiddleOfList.java.

easy Remove Duplicates from Sorted List — LeetCode 83

The problem. Delete duplicates from a sorted linked list so each value appears once.

Example. 1→1→2 becomes 1→2. 1→1→2→3→3 becomes 1→2→3.

Compare with the successor, splice out equals · O(n)

public static ListNode deleteDuplicates(ListNode head) {
    ListNode curr = head;
    while (curr != null && curr.next != null) {
        if (curr.next.val == curr.val)
            curr.next = curr.next.next;   // splice out; DON'T advance —
        else                              // the new next might be equal too
            curr = curr.next;
    }
    return head;
}

Takeaway. Sortedness makes duplicates adjacent, so one local comparison suffices — the list cousin of page 02's Remove Duplicates from Sorted Array. The classic bug is advancing after a splice and skipping a triple (1→1→1); the else-only advance handles runs of any length. No dummy node needed because the head never gets deleted — contrast LC 82, where heads can die and the dummy becomes mandatory. Mirror: code/08-linked-lists/RemoveDuplicatesSortedList.java.

easy Palindrome Linked List — LeetCode 234

The problem. Is the list a palindrome? Follow-up demands O(n) time and O(1) space — so no copying into an array.

Example. 1→2→2→1 → true. 1→2 → false.

Middle + reverse second half + lockstep compare · O(n) time, O(1) space

public static boolean isPalindrome(ListNode head) {
    // 1. find the middle (fast/slow — slow lands at the second half's start)
    ListNode slow = head, fast = head;
    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
    }
    // 2. reverse from slow onward (the 3-pointer loop from Reverse List)
    ListNode prev = null, curr = slow;
    while (curr != null) {
        ListNode next = curr.next;
        curr.next = prev;
        prev = curr;
        curr = next;
    }
    // 3. march inward from both ends
    ListNode a = head, b = prev;              // b: head of the reversed tail
    while (b != null) {                       // second half is never longer
        if (a.val != b.val) return false;
        a = a.next;
        b = b.next;
    }
    return true;
}

Takeaway. A composition test: middle-finding, reversal, and dual-pointer comparison — three techniques from this page chained into one answer. Interviewers often ask you to restore the list afterwards (reverse the second half back); mentioning that unprompted reads as production maturity. Mirror: code/08-linked-lists/PalindromeLinkedList.java.

easy Intersection of Two Linked Lists — LeetCode 160

The problem. Two lists may merge into a shared tail. Return the node where they intersect (by reference, not value), or null. O(1) memory expected.

Example. A = 4→1→8→4→5, B = 5→6→1→8→4→5 (underlined shared) → the node 8.

The pointer-switch trick · O(m + n) time, O(1) space

public static ListNode getIntersectionNode(ListNode headA, ListNode headB) {
    // Walk A's list then B's; walk B's list then A's. Both pointers travel
    // exactly lenA + lenB steps, so they arrive at the shared tail ALIGNED —
    // or reach null together if the lists never meet.
    ListNode a = headA, b = headB;
    while (a != b) {
        a = (a == null) ? headB : a.next;   // exhausted A? restart on B
        b = (b == null) ? headA : b.next;   // exhausted B? restart on A
    }
    return a;                                // intersection node, or null
}

Takeaway. The switch equalizes the length difference without ever measuring it — each pointer covers a+c+b and b+c+a (c = shared tail), which are equal. The pedestrian alternative (compute both lengths, advance the longer by the difference) is equally valid to say first. Note the null-vs-null exit makes disjoint lists terminate naturally — no cycle risk. Mirror: code/08-linked-lists/IntersectionTwoLists.java.

medium Remove Nth Node From End — LeetCode 19

The problem. Delete the n-th node counting from the end of the list, in one pass, and return the head.

Example. 1→2→3→4→5, n=2 → 1→2→3→5. 1→2, n=2 → 2 (the head itself is deleted).

Variation 1 — count length, then walk to victim − 1 · two passes

public static ListNode removeNthTwoPass(ListNode head, int n) {
    int len = 0;
    for (ListNode c = head; c != null; c = c.next) len++;
    ListNode dummy = new ListNode(0);
    dummy.next = head;
    ListNode prev = dummy;
    for (int i = 0; i < len - n; i++) prev = prev.next;  // stop at the predecessor
    prev.next = prev.next.next;                          // bypass the victim
    return dummy.next;
}

Variation 2 — fixed-gap two pointers (one pass) · O(n) time, O(1) space

public static ListNode removeNthFromEnd(ListNode head, int n) {
    ListNode dummy = new ListNode(0);      // dummy head: removing the FIRST node becomes a normal case
    dummy.next = head;
    ListNode fast = dummy, slow = dummy;   // both start at the dummy, gap = 0
    for (int i = 0; i < n + 1; i++) {      // advance fast n+1 steps → gap of n+1 nodes
        fast = fast.next;                  // (n+1, not n, so slow lands BEFORE the node to delete)
    }
    while (fast != null) {                 // walk both until fast runs off the end
        fast = fast.next;
        slow = slow.next;                  // gap stays n+1 the whole way
    }
    slow.next = slow.next.next;            // slow is the predecessor — bypass the victim (GC reclaims it)
    return dummy.next;                     // head may have changed (n == list length) — dummy.next is always right
}

Takeaway. "Count from the end of something you can only walk forward" → maintain a fixed gap. The dummy head converts "delete the head" from a special case into the normal path — reach for it whenever the head might change. Mirror: code/08-linked-lists/RemoveNthFromEnd.java.

medium Add Two Numbers — LeetCode 2

The problem. Two non-negative integers are stored as linked lists in reverse order (ones digit first). Add them and return the sum as a list in the same format.

Example. 2→4→3 (=342) + 5→6→4 (=465) → 7→0→8 (=807). 9→9 + 1 → 0→0→1 (99+1=100 — note the extra final digit).

Grade-school addition with a carry · O(max(n,m))

public static ListNode addTwoNumbers(ListNode l1, ListNode l2) {
    ListNode dummy = new ListNode(0), tail = dummy;
    int carry = 0;
    // Keep going while EITHER list has digits OR a carry is pending —
    // the carry alone can mint one final node (99 + 1 = 100).
    while (l1 != null || l2 != null || carry != 0) {
        int sum = carry;
        if (l1 != null) { sum += l1.val; l1 = l1.next; }   // consume a digit if present
        if (l2 != null) { sum += l2.val; l2 = l2.next; }
        tail.next = new ListNode(sum % 10);   // this position's digit
        carry = sum / 10;                     // 0 or 1, into the next position
        tail = tail.next;
    }
    return dummy.next;
}

Takeaway. Reverse order is a gift — you start at the ones digit, exactly like manual addition. The three-way loop condition (either list or carry) is the whole difficulty; missing the carry case drops the last digit of 99+1. This is also BigInteger addition in miniature. Mirror: code/08-linked-lists/AddTwoNumbers.java.

medium Reorder List — LeetCode 143

The problem. Reorder L0→L1→…→Ln into L0→Ln→L1→Ln−1→L2→… (outside-in interleaving), in place.

Example. 1→2→3→4 → 1→4→2→3. 1→2→3→4→5 → 1→5→2→4→3.

Three known tricks chained · O(n) time, O(1) space

public static void reorderList(ListNode head) {
    if (head == null || head.next == null) return;

    // Step 1: find the middle (fast/slow) — slow ends at the split point.
    ListNode slow = head, fast = head;
    while (fast.next != null && fast.next.next != null) {
        slow = slow.next;
        fast = fast.next.next;
    }
    ListNode second = slow.next;
    slow.next = null;                      // cut: first half ends at slow

    // Step 2: reverse the second half (the three-pointer flip).
    ListNode prev = null;
    while (second != null) {
        ListNode next = second.next;
        second.next = prev;
        prev = second;
        second = next;
    }
    second = prev;                         // head of the reversed back half

    // Step 3: interleave — alternate one node from each half.
    ListNode first = head;
    while (second != null) {               // second is the shorter (or equal) half
        ListNode n1 = first.next, n2 = second.next;   // save both continuations
        first.next = second;               // L0 -> Ln
        second.next = n1;                  // Ln -> L1
        first = n1;                        // advance in the front half
        second = n2;                       // advance in the back half
    }
}

Takeaway. Nothing new is invented here — middle (fast/slow) + reverse (three-pointer) + merge-style interleave, staple-gunned together. Composite list problems almost always decompose into the primitives; practicing the decomposition matters more than memorizing this exact code. Mirror: code/08-linked-lists/ReorderList.java.

medium Copy List with Random Pointer — LeetCode 138

The problem. Each node has next plus a random pointer to any node (or null). Deep-copy the list: brand-new nodes whose next/random wiring mirrors the original.

Example. For list 7→13→11→10→1 with randoms [null, 0, 4, 2, 0], the copy must reproduce the same shape with entirely new nodes.

Variation 1 — HashMap original → clone · O(n) time, O(n) space

public static Node copyRandomList(Node head) {
    // Pass 1: clone every node; map original -> clone
    Map<Node, Node> map = new HashMap<>();
    for (Node cur = head; cur != null; cur = cur.next)
        map.put(cur, new Node(cur.val));
    // Pass 2: wire the clones using the map as a lookup
    for (Node cur = head; cur != null; cur = cur.next) {
        map.get(cur).next   = map.get(cur.next);     // map.get(null) is null — handy
        map.get(cur).random = map.get(cur.random);
    }
    return map.get(head);
}

Variation 2 — interleave clones A→A'→B→B' · O(n) time, O(1) extra space

public static Node copyRandomListInterleaved(Node head) {
    if (head == null) return null;
    // Pass 1: insert each clone right AFTER its original: A -> A' -> B -> B' ...
    for (Node cur = head; cur != null; cur = cur.next.next) {
        Node clone = new Node(cur.val);
        clone.next = cur.next;
        cur.next = clone;
    }
    // Pass 2: set randoms — the clone of X.random is X.random.next (its neighbor!)
    for (Node cur = head; cur != null; cur = cur.next.next)
        if (cur.random != null)
            cur.next.random = cur.random.next;
    // Pass 3: unzip the two lists, restoring the original intact
    Node newHead = head.next;
    for (Node cur = head; cur != null; cur = cur.next) {
        Node clone = cur.next;
        cur.next = clone.next;                       // restore original's next
        clone.next = (clone.next != null) ? clone.next.next : null;
    }
    return newHead;
}

Takeaway. The map version is Clone Graph's technique (page 15) on a list. The interleaving replaces the map with position: parking each clone beside its original makes "find the clone of X" a one-step walk. Mirror: code/08-linked-lists/CopyListRandomPointer.java.

medium Linked List Cycle II — LeetCode 142

The problem. If the list has a cycle, return the node where the cycle begins (not just whether one exists); null otherwise. O(1) memory.

Example. 3→2→0→−4 with −4 linking back to 2 → the node holding 2.

Floyd phase 2 — reset one pointer to head · O(n) time, O(1) space

public static ListNode detectCycle(ListNode head) {
    // Phase 1: standard fast/slow until they meet inside the cycle.
    ListNode slow = head, fast = head;
    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
        if (slow == fast) {
            // Phase 2: restart one pointer at head; walk BOTH at speed 1.
            // They meet exactly at the cycle's entrance.
            ListNode p = head;
            while (p != slow) {
                p = p.next;
                slow = slow.next;
            }
            return p;
        }
    }
    return null;                           // fast hit the end -> acyclic
}

Takeaway. The why, in three lines: let the head-to-entrance distance be a, and the meeting point sit b into the cycle (length c). Fast having gone twice slow's distance forces a + b ≡ 0 (mod c) — so walking a more steps from the meeting point lands exactly on the entrance, which is also a steps from the head. Hence two speed-1 pointers, one from each place, collide there. Mirror: code/08-linked-lists/LinkedListCycle.java.

medium Swap Nodes in Pairs — LeetCode 24

The problem. Swap every two adjacent nodes (rewire the nodes themselves, not the values).

Example. 1→2→3→4 becomes 2→1→4→3. Odd leftover stays: 1→2→3 becomes 2→1→3.

Dummy head + three-arrow rewire per pair · O(n)

public static ListNode swapPairs(ListNode head) {
    ListNode dummy = new ListNode(0);
    dummy.next = head;
    ListNode prev = dummy;
    while (prev.next != null && prev.next.next != null) {
        ListNode first = prev.next;
        ListNode second = first.next;
        first.next = second.next;   // 1. first jumps over second
        second.next = first;        // 2. second points back at first
        prev.next = second;         // 3. the pair's new head hooks in
        prev = first;               // first is now the pair's tail
    }
    return dummy.next;
}

Takeaway. Reverse Nodes in k-Group (below) with k=2 — small enough to hand- rewire, which makes it the ideal rehearsal for the arrow-order discipline: draw the three arrows, number them, then code. The dummy head removes the "new head" special case exactly as in the merge problems. Mirror: code/08-linked-lists/SwapNodesInPairs.java.

medium Partition List — LeetCode 86

The problem. Rearrange so all nodes with value < x come before nodes ≥ x, preserving relative order within each group.

Example. 1→4→3→2→5→2, x=3 → 1→2→2→4→3→5.

Two builder lists, then stitch · O(n)

public static ListNode partition(ListNode head, int x) {
    ListNode smallHead = new ListNode(0), small = smallHead;
    ListNode bigHead = new ListNode(0), big = bigHead;
    for (ListNode n = head; n != null; n = n.next) {
        if (n.val < x) {
            small.next = n;         // append to the smalls
            small = n;
        } else {
            big.next = n;           // append to the bigs
            big = n;
        }
    }
    big.next = null;                // CRITICAL: sever the old tail link
    small.next = bigHead.next;      // stitch smalls -> bigs
    return smallHead.next;
}

Takeaway. Quicksort's partition again (compare page 06's Sort Colors and Sort By Parity) — but on a list, stability comes free because appending preserves encounter order, where the array version has to give it up or pay for it. The one bug that bites everyone: forgetting big.next = null, leaving a cycle through the old links. Mirror: code/08-linked-lists/PartitionList.java.

medium Rotate List — LeetCode 61

The problem. Rotate the list right by k places; k can exceed the length.

Example. 1→2→3→4→5, k=2 → 4→5→1→2→3. 0→1→2, k=4 → 2→0→1.

Close into a ring, cut at n − k%n · O(n)

public static ListNode rotateRight(ListNode head, int k) {
    if (head == null || head.next == null) return head;

    int n = 1;
    ListNode tail = head;
    while (tail.next != null) {     // one pass finds length AND tail
        tail = tail.next;
        n++;
    }
    k %= n;                          // k may far exceed the length
    if (k == 0) return head;

    tail.next = head;                // 1. close the ring
    ListNode newTail = head;
    for (int i = 0; i < n - k - 1; i++)   // 2. walk to the cut point
        newTail = newTail.next;
    ListNode newHead = newTail.next; // 3. cut the ring open
    newTail.next = null;
    return newHead;
}

Takeaway. Rotation is a relabeling, not a movement — closing the ring makes that literal, and the only real work is choosing where to cut (index n − k%n). The k %= n normalization is the same missed edge case as array rotation (page 23). Also compare: arrays rotate via three reversals because they lack a ring; lists get the ring for one pointer write. Mirror: code/08-linked-lists/RotateList.java.

hard Reverse Nodes in k-Group — LeetCode 25

The problem. Reverse the list k nodes at a time: the first k nodes get reversed, then the next k, and so on. A final group with fewer than k nodes stays in original order.

Example. 1→2→3→4→5, k=2 → 2→1→4→3→5. 1→2→3→4→5, k=3 → 3→2→1→4→5.

Reverse per block + careful stitching · O(n) time, O(1) space

public static ListNode reverseKGroup(ListNode head, int k) {
    ListNode dummy = new ListNode(0);
    dummy.next = head;
    ListNode groupPrev = dummy;            // node just BEFORE the current group

    while (true) {
        // 1) Find the k-th node of this group; if fewer than k remain, done.
        ListNode kth = groupPrev;
        for (int i = 0; i < k && kth != null; i++) kth = kth.next;
        if (kth == null) break;
        ListNode groupNext = kth.next;     // first node AFTER the group

        // 2) Reverse the group in place. Standard flip, but prev starts at
        //    groupNext so the group's old head ends up pointing PAST the group.
        ListNode prev = groupNext;
        ListNode curr = groupPrev.next;    // group's first node (will become its last)
        while (curr != groupNext) {
            ListNode next = curr.next;
            curr.next = prev;
            prev = curr;
            curr = next;
        }

        // 3) Stitch: groupPrev must now point at the NEW group head (the old kth).
        ListNode oldGroupHead = groupPrev.next;   // now the group's tail
        groupPrev.next = kth;
        groupPrev = oldGroupHead;          // advance: tail of this group precedes the next
    }
    return dummy.next;
}

Takeaway. Everything is the three-pointer flip plus bookkeeping: know where the group starts (groupPrev.next), ends (kth), and what follows (groupNext) before touching any links. Starting prev at groupNext is the elegant move — the reversal lands pre-stitched on one side. This problem is the pointer-fluency final exam; if it flows, lists are done. Mirror: code/08-linked-lists/ReverseKGroup.java.

Pitfalls

  • Save before you sever. The canonical bug is curr.next = prev before saving curr.next — the rest of the list is instantly unreachable. Write the save line first, always.
  • Reach for the dummy head by default whenever the head might change (delete, merge, partition). It converts three edge cases into zero and takes one line.
  • Guard fast/slow correctly: while (fast != null && fast.next != null) — in that order, or you NPE on even-length lists. And know that where slow stops differs by one between odd and even lengths — trace a 2-node list to check.
  • Compare nodes with ==, not equals. Cycle detection asks "is this the same object?" — value equality would false-positive on duplicate values.
  • Recursion depth is real: a recursive solution on a 10⁵-node list can blow the JVM stack (~10⁴ frames by default). Prefer the iterative version for long lists.
  • After surgery, check for accidental cycles: in reorder/reverse-in-parts problems, forgetting to null-terminate the new tail makes printing loop forever.

Linked Lists: Pointer Fluency Under Pressure

Dummy heads, the three-pointer reversal, and fast/slow tricks — the screener classics that test whether you can rewire references without dropping the list.

Phase 3 · Week 5 Screener classic — a pointer-fluency test more than a data-structure test

Before you start

Why this topic exists. Nobody is going to ask you to build a production system on LinkedList — cache locality killed that idea (details below). Interviewers use linked lists for a different reason: they are the cheapest way to test whether you can manipulate references precisely. One wrong assignment order and the rest of the list is garbage-collected out from under you. The same reference discipline is exactly what trees (page 11) and the LRU cache (page 24) demand — this page is where you build it.

Prerequisites

  • 00 · Foundations — the ArrayList vs LinkedList cost table.
  • 02 · Two Pointers — fast/slow was introduced there; here it gets its full treatment.

What the interviewer is actually testing

  • Reference precision: can you reorder a.next = b assignments so nothing is lost? (Save before you overwrite.)
  • Edge-case instinct: empty list, single node, operating on the head — do you handle them by design (dummy node) or by patching?
  • The standard tricks on recall: reversal, fast/slow middle, cycle detection, dummy-head merging. These are assumed knowledge in later rounds.
  • Can you draw it? Strong candidates sketch boxes and arrows before coding. Pointer bugs are visible on paper before they're visible in code.

How this builds on the previous topic

Binary search (page 07) exploited O(1) random access — jump to any index instantly. Linked lists are the opposite trade: no random access at all (getting element i costs O(i), which is why you can't binary-search a linked list), but O(1) splicing anywhere you already hold a reference. Every technique on this page is about navigating cleverly under that constraint — and it's your first step into linked structures, where trees and graphs are "linked lists with more pointers".

Core concepts

1. The node model: Java references are pointers with seatbelts

A linked list is just objects pointing at each other. In Java there's no pointer arithmetic, but a ListNode next field is exactly a pointer: a value that names another object (or null). 

// The whole data structure is two fields:
static class ListNode {
    int val;
    ListNode next;      // reference to the following node, or null at the tail
    ListNode(int val) { this.val = val; }
}
// head ──▶ [1|•] ──▶ [2|•] ──▶ [3|∅]
// Losing `head` (or any node's only incoming reference) loses everything after it — the GC takes it.
The one rule of pointer surgery. Before overwriting a reference you still need, save it. Nearly every linked-list bug is an assignment executed one line too early. When stuck, narrate each line: "who points where now?" — and draw it.

2. The dummy head: delete your special cases

Operations at the head ("what if the node I remove is the head?") normally need special-case code. The fix is one line: allocate a fake node in front, work exclusively with .next pointers of predecessors, and return dummy.next at the end. The head is now just another node with a predecessor. You'll see it carry both worked solutions below. 

ListNode dummy = new ListNode(0);   // value never read — it exists only to be a predecessor
dummy.next = head;
// ... operate via predecessor.next, never via `head` directly ...
return dummy.next;                  // the true head, even if the original was removed

3. Reversal: the three-pointer dance

Reverse a list in place by walking it once, flipping each next arrow backwards. You need three references — the reversed prefix (prev), the node being flipped (curr), and a saved handle on the unvisited remainder (next) so the flip doesn't orphan it. Watch the invariant: at every step, prev heads a fully reversed sublist and curr heads the untouched rest.

4. Fast & slow: two speeds, three superpowers

Run two pointers down the list, one advancing 1 node per step, the other 2. This single trick yields:

  • Middle of the list: when fast hits the end, slow has covered half the distance — it's standing on the middle. One pass, no length count.
  • Cycle detection (Floyd): if there's a cycle, fast can never "finish" — both pointers eventually orbit the cycle, and they must meet.
  • Nth from the end: a variant — give one pointer an n-node head start and move both at the same speed (worked solution below).
Why Floyd's meeting is guaranteed — the actual math. Once both pointers are inside the cycle (length C), look at the gap from fast to slow: each step, fast closes it by exactly 1 (moves 2, slow moves 1). A gap that shrinks by 1 each step must reach 0 — they collide; fast can never "jump over" slow. Bonus fact for the follow-up (LC 142): if the meeting happens after slow walks distance a + b (a = head→cycle-start, b = into the cycle), fast walked 2(a+b), and the difference a+b is a whole number of laps — which rearranges to: head-to-cycle-start equals meeting-point-to-cycle-start (mod C). So restart one pointer at the head, advance both at speed 1, and they meet exactly at the cycle's entrance. 
// Floyd's cycle detection — O(1) space (vs. the obvious HashSet's O(n))
public static boolean hasCycle(ListNode head) {
    ListNode slow = head, fast = head;
    while (fast != null && fast.next != null) {  // fast does the null-checking for both
        slow = slow.next;                        // +1
        fast = fast.next.next;                   // +2 — closes the gap by 1 per step inside a cycle
        if (slow == fast) return true;           // reference equality: same NODE, not equal values
    }
    return false;                                // fast found a null — no cycle possible
}

5. The rest of the standard kit

  • Merge two sorted lists — dummy head + always-take-the-smaller. Worked below; it's also the heart of Merge k Sorted Lists (page 12) and merge sort on lists.
  • Reorder List (L0→Ln→L1→Ln-1…) — a three-trick combo: fast/slow finds the middle, reverse the second half, then interleave the two halves. If you know this decomposition, the "hard-looking" problem is three easies.
  • Deep copy with random pointers (LC 138) — either a HashMap<Node,Node> from originals to clones (O(n) space, obvious), or the elegant O(1)-space interleave: weave each clone in right after its original (A→A'→B→B'), set clone.random = orig.random.next, then unweave.

Pattern templates (Java)

Template A — dummy head + predecessor walking (insert/delete/merge)

ListNode dummy = new ListNode(0);
dummy.next = head;
ListNode pred = dummy;                     // pred = last node of the already-processed prefix
while (pred.next != null) {
    if (shouldRemove(pred.next)) {
        pred.next = pred.next.next;        // bypass — do NOT advance pred (next node is unvetted)
    } else {
        pred = pred.next;                  // keep — advance
    }
}
return dummy.next;                         // correct even if the original head was removed

Template B — in-place reversal (the three-pointer loop)

ListNode prev = null, curr = head;
while (curr != null) {
    ListNode next = curr.next;             // 1. SAVE the remainder first
    curr.next = prev;                      // 2. flip the arrow
    prev = curr;                           // 3. reversed prefix grows
    curr = next;                           // 4. advance into the saved remainder
}
// prev is the new head — memorize this 4-line body until it's automatic

Template C — fast/slow (middle, cycle, nth-from-end)

ListNode slow = head, fast = head;
while (fast != null && fast.next != null) {   // guard BOTH hops fast will take
    slow = slow.next;
    fast = fast.next.next;
}
// even length: slow = second middle; start fast at head.next for the first middle

The dirty secret: LinkedList almost never wins in practice. Its O(1) splicing only pays if you're already standing at the splice point; getting there is O(n) of pointer-chasing, where every hop is a potential CPU cache miss because nodes are scattered across the heap. ArrayList's O(n) "shift" is a contiguous System.arraycopy the hardware races through. Even Joshua Bloch (who wrote LinkedList) jokes that nobody should use it. For queue/deque needs, use ArrayDeque (page 09). So why is this page here at all? Because interviews use lists to test pointer manipulation — and because intrusive linked lists genuinely power real systems where you already hold the node: see below.

Decision cues:

  • Problem hands you a ListNode → it's a pointer-manipulation exercise; think dummy head, reversal, fast/slow.
  • "Detect a cycle" / "find where the cycle begins" → Floyd's fast/slow.
  • "Middle", "nth from end", "palindrome list" → fast/slow (+ reversal for palindrome).
  • "In O(1) space" on a list problem → in-place reversal or interleaving is intended; a HashMap is the O(n)-space fallback worth mentioning first.
  • Real Java code needing a sequence → ArrayList; needing both-ends ops → ArrayDeque; needing O(1) removal of a node you hold a handle to → hand-rolled DLL (LRU, page 24).

Where this shows up in real systems

LRU caches: HashMap + doubly linked list

The #1 design interview question (page 24) and the eviction backbone of real caches: a DLL keeps entries in recency order — move-to-front and evict-from-tail are O(1) because the map hands you the exact node, no walking. This is the case where a linked list beats an array: removal from the middle when you already hold the node. Linux's page LRU lists work the same way.

Allocator free lists

Memory allocators (glibc malloc's bins, kernel slab allocators, JVM free-region tracking) chain free blocks into linked lists — the pointer is stored inside the free block itself, so bookkeeping is free and coalescing/splitting blocks is O(1) splicing. Intrusive lists like these are everywhere in systems code.

HashMap collision chaining

Java's own HashMap resolves collisions by chaining entries into a singly linked list per bucket (treeifying long chains, page 01). Every get on a collided key walks a linked list. LinkedHashMap additionally threads a DLL through all entries for iteration order — which is exactly why it trivializes LRU.

Undo chains & MRU playlists

Editor undo/redo systems keep operations in a doubly linked chain — O(1) append, O(1) walk back and forward, cheap truncation when you undo-then-edit. Media players' "recently played" and browser-tab MRU ordering are small LRU/MRU lists: move-to-front on every access.

★ Worked: Reverse Linked List (LC 206)

Problem in one line: reverse a singly linked list in place; return the new head.

Idea: walk once with three references. prev always heads the already-reversed prefix, curr the untouched suffix. Each iteration: save curr.next (or the flip strands the rest of the list), point curr backwards, then shift both references one node right. The recursive version is the same surgery driven by the call stack — worth knowing because it previews the "trust the recursion" leap of faith trees demand (page 11).

// Iterative: walk the list once, flipping each .next backwards.
public static ListNode reverseList(ListNode head) {
    ListNode prev = null;                  // everything BEFORE curr, already reversed (starts empty)
    ListNode curr = head;                  // the node whose arrow we're about to flip
    while (curr != null) {
        ListNode next = curr.next;         // 1. save the rest of the list BEFORE we cut the link
        curr.next = prev;                  // 2. flip: curr now points backwards
        prev = curr;                       // 3. the reversed prefix grows by one node
        curr = next;                       // 4. step into the saved remainder
    }
    return prev;                           // curr fell off the end; prev is the new head
}

// Recursive: trust reverseList(rest) to hand back the reversed tail,
// then hook the old head onto the end of it.
public static ListNode reverseRecursive(ListNode head) {
    if (head == null || head.next == null) return head;   // base: empty or single node is already reversed
    ListNode newHead = reverseRecursive(head.next);        // leap of faith: rest is now reversed; newHead = old tail
    head.next.next = head;                 // my old successor is now the reversed tail's last node — point it back at me
    head.next = null;                      // I'm the new tail; sever my stale forward link
    return newHead;                        // the new head never changes on the way back up
}

Complexity: iterative — time O(n), space O(1). Recursive — time O(n), space O(n) for the call stack (say this unprompted!). Runnable version: code/08-linked-lists/ReverseLinkedList.java.

★ Worked: Merge Two Sorted Lists (LC 21)

Problem in one line: splice two sorted lists into one sorted list, reusing the existing nodes.

Idea: exactly the merge step of merge sort (page 06). Keep a tail on the result; repeatedly attach the smaller of the two front nodes and advance that list. The dummy head removes the "which list provides the first node?" special case, and when one list runs dry the other is attached whole — it's already sorted, no loop required.

public static ListNode mergeTwoLists(ListNode l1, ListNode l2) {
    ListNode dummy = new ListNode(0);      // fake node BEFORE the real head — kills all "is this the first node?" special cases
    ListNode tail = dummy;                 // tail always points at the last node of the merged result
    while (l1 != null && l2 != null) {     // while both lists still have candidates
        if (l1.val <= l2.val) {            // <= keeps the merge STABLE (ties keep l1's order)
            tail.next = l1;                // splice the smaller node onto the result
            l1 = l1.next;                  // advance only the list we consumed from
        } else {
            tail.next = l2;
            l2 = l2.next;
        }
        tail = tail.next;                  // result grew by one — move tail forward
    }
    tail.next = (l1 != null) ? l1 : l2;    // one list is empty; the other is sorted — attach it whole (no loop needed)
    return dummy.next;                     // the real head lives just after the dummy
}

Complexity: time O(n + m), space O(1) — nodes are relinked, not copied. Runnable version: code/08-linked-lists/MergeTwoLists.java.

★ Worked: Remove Nth Node From End of List (LC 19)

Problem in one line: delete the nth node counting from the end, in one pass.

Idea: you can't count from the end of a singly linked list — but you can maintain a fixed gap. Send fast ahead by n+1 nodes, then advance both pointers together; when fast falls off the end, slow is exactly n+1 from the end — i.e. the predecessor of the victim, which is who you need for the bypass. Starting both at a dummy makes "delete the head" (n = length) work with zero extra code.

public static ListNode removeNthFromEnd(ListNode head, int n) {
    ListNode dummy = new ListNode(0);      // dummy head: removing the FIRST node becomes a normal case
    dummy.next = head;
    ListNode fast = dummy, slow = dummy;   // both start at the dummy, gap = 0
    for (int i = 0; i < n + 1; i++) {      // advance fast n+1 steps → gap of n+1 nodes
        fast = fast.next;                  // (n+1, not n, so slow lands BEFORE the node to delete)
    }
    while (fast != null) {                 // walk both until fast runs off the end
        fast = fast.next;
        slow = slow.next;                  // gap stays n+1 the whole way
    }
    slow.next = slow.next.next;            // slow is the predecessor — bypass the victim (GC reclaims it)
    return dummy.next;                     // head may have changed (n == list length) — dummy.next is always right
}

Complexity: time O(n) — one pass; space O(1). Runnable version: code/08-linked-lists/RemoveNthFromEnd.java.

Pitfalls & interview tips

  • Save before you sever. The canonical bug is curr.next = prev before saving curr.next — the rest of the list is instantly unreachable. Write the save line first, always.
  • Reach for the dummy head by default whenever the head might change (delete, merge, partition). It converts three edge cases into zero and takes one line. Interviewers notice candidates who deploy it unprompted.
  • Guard fast/slow correctly: while (fast != null && fast.next != null) — in that order, or you NPE on even-length lists. And know that where slow stops differs by one between odd and even lengths; trace a 2-node list before claiming done.
  • Compare nodes with ==, not equals. Cycle detection asks "is this the same object?" — value equality would false-positive on duplicate values.
  • Recursion depth is real: a recursive solution on a 10⁵-node list can blow the JVM stack (~10⁴ frames by default). Offer the iterative version and mention this as the reason — easy senior credit.
  • After surgery, check for accidental cycles: in reorder/reverse-in-parts problems, forgetting to null-terminate the new tail makes printing loop forever. State "and finally the tail's next becomes null" out loud.

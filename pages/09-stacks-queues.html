Stacks & Queues: Order of Processing as a Tool

LIFO for nesting and "most recent", FIFO for arrival order — and the monotonic stack, a one-trick structure that mass-produces medium-problem solutions.

Phase 3 · Week 5 Monotonic stack = medium-question factory

Before you start

Why this topic exists. Stacks and queues aren't really containers — they're policies about what to process next. LIFO says "the most recent unfinished thing first" (that's nesting, matching, undo, the call stack itself). FIFO says "in arrival order" (that's fairness, buffering, BFS). Interviews lean on them because picking the right processing order is the solution to a huge family of problems — and because the monotonic stack, a small twist on LIFO, single-handedly powers a shelf of classic mediums.

Prerequisites

  • 00 · Foundations — the ArrayDeque row of the collections table.
  • 08 · Linked Lists — deques are the linked structure idea with both ends exposed.

Interview relevance: the monotonic stack alone generates a whole family of medium questions; stacks also underpin parsing and undo problems.

How this builds on the previous topic

Page 08 was about wiring linked nodes by hand. Here you consume the polished version: ArrayDeque gives O(1) at both ends without any pointer surgery (ironically via a circular array, not links — cache locality wins again). Conceptually this page is a bridge: the stack you use here becomes the call stack in recursion (page 10), the queue becomes BFS's frontier (pages 11/15), and the monotonic deque closes a loose end from sliding window (page 03).

Core concepts

1. One class to rule them both: ArrayDeque

In interview Java, every stack and every queue is an ArrayDeque — a resizable circular array with O(1) amortized push/pop/offer/poll at both ends. 

Deque<Integer> stack = new ArrayDeque<>();
stack.push(x);        // add at head          LIFO: last in...
int top = stack.pop();    // remove from head  ...first out
int peek = stack.peek();  // look, don't remove

Deque<Integer> queue = new ArrayDeque<>();
queue.offer(x);       // add at tail          FIFO: arrival order
int front = queue.poll(); // remove from head
// Same class, opposite end pairs. A deque exposes all four operations at once.
Never java.util.Stack. It's a 1995 legacy class that extends Vector: every method is synchronized (pointless locking cost in single-threaded code), and being a Vector it exposes get(i)/insertElementAt — so it isn't even a real stack abstraction. The Javadoc itself tells you to use ArrayDeque. Saying this in one sentence when you declare your stack is free senior signal. (Same story for LinkedList-as-queue: it works, but ArrayDeque is the idiomatic, faster choice.)

Knowing how ArrayDeque pulls off O(1) at both ends is a favorite internals question. It's a circular array: head and tail indices chase each other around an Object[], and neither end ever shifts elements. The classic design — the one to describe in an interview — keeps the capacity a power of two, so wrapping around the array's edge is a bitmask instead of a modulo or a branch:

The index math (why capacity is a power of two)

// Conceptually, with elements.length always a power of two:
head = (head - 1) & (elements.length - 1);   // addFirst/push: step left, wrap free
tail = (tail + 1) & (elements.length - 1);   // addLast/offer: step right, wrap free
// Two's complement makes it work at the seam: -1 & 15 == 15.
// Full (head meets tail) -> allocate double, copy the ring out in TWO
// System.arraycopy calls, because the live elements may straddle the array's end.

Three consequences worth saying out loud. Amortized O(1): growth doubles the array, so the occasional O(n) copy is prepaid by the n cheap inserts before it — the same aggregate argument as ArrayList on page 00. No nulls, by design: empty slots in the ring are null, and poll() returns null to mean "empty" — a stored null would be indistinguishable from a vacancy, so offer(null) throws NPE (pitfall below). No shrinking: the ring only ever grows; a deque that once held 10⁶ elements keeps that array. In current JDKs the exact sizing details vary, but the circular-array-with-two-cursors design — and everything it implies — has been stable since the class appeared in Java 6.

2. Matching & nesting: the native LIFO problem

Anything with nested open/close structure — brackets, HTML tags, JSON braces, expression parentheses — obeys one law: the most recently opened thing must close first. That law is a stack. Push openers; each closer must match the top. Valid Parentheses (worked below) is the distilled version, and it generalizes to real parsers: push context on open, pop and verify on close.

3. Min Stack: pay memory to remember history

"Design a stack that also returns its minimum in O(1)." You can't keep a single min variable — popping the minimum leaves you asking "what was the min before that?" The fix: a second stack that records, for every element, the min at the time it was pushed. Push and pop move both stacks in lock-step, so the auxiliary top is always the current answer, and history restores itself for free. 

class MinStack {
    private final Deque<Integer> stack = new ArrayDeque<>();
    private final Deque<Integer> mins  = new ArrayDeque<>();  // mins.peek() = min of everything below

    public void push(int x) {
        stack.push(x);
        // push the min-so-far alongside; if mins is empty, x starts the record
        mins.push(mins.isEmpty() ? x : Math.min(x, mins.peek()));
    }
    public void pop()    { stack.pop(); mins.pop(); }  // popping rewinds the min history automatically
    public int  top()    { return stack.peek(); }
    public int  getMin() { return mins.peek(); }       // O(1), always current
}
The design follow-up: Max Stack with popMax() (LC 716). Extend the idea to "also remove the maximum, wherever it sits" and the snapshot trick dies — snapshots answer reads about history, but deleting from the middle invalidates every snapshot above it. The senior answer pairs two structures: a doubly linked list holding the stack's actual order (O(1) removal of a node you hold — page 08's LRU move) plus a TreeMap<Integer, List<Node>> indexing nodes by value. push/pop/ popMax all land in O(log n): the TreeMap finds the largest value (lastKey()), the list unlinks its node in O(1). The lesson generalizes: a plain stack answers "what's on top?"; the moment a question demands arbitrary deletion by rank, you need an ordered index bolted on.

4. Expression evaluation: stacks as tiny interpreters

Reverse Polish Notation ("3 4 + 2 *") is what you get when you let a stack do the parsing: numbers push, an operator pops two, computes, pushes the result. No parentheses, no precedence rules — the ordering encodes them. This isn't a toy: the JVM is a stack machine, and javac compiles (3 + 4) * 2 into exactly the RPN push/pop sequence. Worked solution below. The harder cousin — LC 224 Basic Calculator, with parentheses and unary minus — keeps a sign stack: walk the string with a running result and current sign; on ( push the current result and sign and start fresh; on ) pop them back and fold the inner result in. Same push-context/pop-context move as bracket matching.

5. The monotonic stack — the medium-question factory

Problem shape: "for each element, find the next element to the right (or left) that is greater (or smaller)." Brute force rescans → O(n²). The monotonic stack answers all n queries in one pass: keep a stack of elements still waiting for their answer, arranged so values strictly decrease from bottom to top. When a new element arrives that is bigger than the top, it is the answer for the top — pop and record, repeat while it keeps beating tops, then push the newcomer to wait its own turn.

Why O(n) despite the nested while-loop: count work per element, not per iteration. Each index is pushed exactly once and popped at most once — so all the while-loop iterations across the entire run total at most n pops. Two operations per element, O(n) overall. This amortized argument is half the points; say it unprompted.

Variants you now get almost for free: Next Greater Element (the pure form), Daily Temperatures (worked below), and Largest Rectangle in Histogram — an increasing stack where each pop learns both its left and right smaller neighbors, giving the widest rectangle each bar can anchor. And the monotonic deque from Sliding Window Maximum (page 03) is this same idea with expiry: greater-elements pop from the back, out-of-window elements leave from the front.

The tip above proves O(n) by the aggregate method (count all pops at once); the accounting method (page 00) phrases it per element, which is often easier to say under pressure: hand every arriving element two credits — one pays for its push now, one rides along as prepaid fare for its single future pop. Every iteration of the inner while-loop spends a credit that was already paid, so total work ≤ 2n no matter how the pops cluster. The same two-credit argument certifies the monotonic deque on page 03 verbatim.

And the histogram's width formula deserves its one-line proof, because it's where hand-waving gets caught. When bar j pops (incoming bar i is the first strictly shorter bar to its right), the new stack top is the nearest bar to j's left that survived every pop j survived — i.e. the nearest bar shorter-or-equal on the left. Everything strictly between those two boundaries must be ≥ heights[j], because anything shorter would have popped j earlier. So the maximal rectangle of height heights[j] spans exactly width = i − stack.peek() − 1.

The ≥-vs-> subtlety, resolved. With equal heights [5, 5, 5] and a strict pop condition (h < top, as in the worked LC 84 below), equal bars pile up unpopped — and when a shorter bar finally flushes them, the earlier equal bars compute undercounted widths (their left boundary is a same-height twin, not a shorter bar). The algorithm survives anyway: the last equal bar's pop sees the full span, so the maximum is still right. That asymmetry is fine for "largest rectangle" but fatal for problems that need every element's exact span — for a strict "next greater", equal elements must chain (pop on >= instead), and for "count of subarrays where a[i] is the max" you must break ties one strict / one non-strict or you'll double-count. Decide the comparison from the problem, not from muscle memory.

6. Queue via two stacks: two reversals make a forward

A classic warm-up with a real lesson in amortized analysis: push into an in stack; to poll, if the out stack is empty, dump all of in into out (reversing order), then pop out. Each element makes exactly one trip: pushed on in, moved once, popped from out — so every operation is amortized O(1) even though a single poll can be O(n).

That "one trip each" sentence is the aggregate method. This structure is also the cleanest place to run a potential-function proof once, because interviewers who ask "prove the amortized bound" want this shape (page 00 defines the machinery). Choose Φ = 2 · |in| — twice the size of the in-stack, a "stored energy" that is 0 at the start and never negative. Amortized cost = actual cost + ΔΦ:

  • push: actual 1, Φ rises by 2 → amortized 3. The push overpays, banking energy for the element's future transfer.
  • poll, out non-empty: actual 1, ΔΦ = 0 → amortized 1.
  • poll with a dump of k elements: actual k pops + k pushes + 1 pop = 2k + 1, but Φ drops by 2k → amortized 1. The expensive operation is paid for before it happens.

Every operation is amortized ≤ 3 = O(1), and because Φ ≥ 0 always, total actual cost ≤ total amortized cost — the bank never overdrafts. Same theorem as the aggregate argument, but this version generalizes to structures where "each element moves once" isn't obvious.

7. The JVM stack: where recursion actually lives

Every recursive solution in this runbook secretly uses today's data structure — the thread's call stack. It is small, fixed, and it fails loudly. Knowing its numbers is what turns "I'd use DFS" into a deployable answer. (Pages 10, 15, and 18 all point back here.)

Each Java thread owns a fixed-size, contiguous stack, allocated at thread creation — by default roughly 512 KB–1 MB depending on platform, tunable per thread or via -Xss. Every method call pushes a frame: the method's local variables, its operand stack (the JVM is a stack machine — §4's RPN evaluator in hardware), and the return address. A frame costs on the order of a hundred bytes, more with many locals. Divide and you get the number to memorize: ~10,000 frames of headroom by default, give or take a small factor. When it runs out the JVM throws StackOverflowError — not an exception to catch, a structural fact about your algorithm.

Now read a constraint like n ≤ 10⁵ with that number in mind. A recursion whose depth tracks n — a degenerate BST (page 11), DFS down a path-shaped graph (page 15), a memoized DP recursing to depth n (page 18) — dies at ten times the default budget, and no amount of algorithmic cleverness fixes an exhausted stack. The heap has no such limit: an explicit ArrayDeque holding ten million entries is routine. Hence the standard conversion — push what you would have passed as arguments:

Recursion → explicit stack (the 10⁵-depth insurance policy)

// Before: void dfs(Node u) { ...; for (child) dfs(child); }   // depth-limited
// After: your own frames, on the HEAP — depth limited only by memory.
Deque<Node> stack = new ArrayDeque<>();
stack.push(root);
while (!stack.isEmpty()) {
    Node u = stack.pop();          // "entering the call"
    process(u);
    for (Node child : u.children)  // pushed = "recursive calls scheduled"
        stack.push(child);
}
// Needs post-processing on the way "back up"? Push a (node, state) pair and
// revisit — the two-phase visit trick, spelled out on page 11's iterative traversals.
Facts worth quoting. Java performs no tail-call optimization — even a perfectly tail-recursive method consumes a frame per call (the why lives on page 10). Thread stack size is fixed at thread start; a common competitive-programming escape hatch is running the solver in a new Thread(null, runnable, "main", 1 << 26) with a 64 MB stack. In an interview, the winning sentence is simpler: "recursion depth is O(n) here and n is 10⁵, so I'd either convert to an explicit stack or note the stack-size caveat."

Pattern templates (Java)

Template A — matching / nesting (push context, pop & verify)

Deque<Character> stack = new ArrayDeque<>();
for (char c : s.toCharArray()) {
    if (isOpener(c))       stack.push(c);            // remember the newest unclosed context
    else {                                           // c is a closer
        if (stack.isEmpty()) return false;           // nothing to close
        if (!matches(stack.pop(), c)) return false;  // must close the MOST RECENT opener
    }
}
return stack.isEmpty();                              // no unclosed leftovers

Template B — monotonic stack (next greater to the right)

// stack holds INDICES whose "next greater" is still unknown; values decrease bottom→top
Deque<Integer> stack = new ArrayDeque<>();
for (int i = 0; i < n; i++) {
    while (!stack.isEmpty() && a[i] > a[stack.peek()]) {
        int j = stack.pop();          // a[i] is the first element right of j that beats it
        answer[j] = /* use i and j */; // e.g. i - j (distance) or a[i] (the value)
    }
    stack.push(i);                     // i now waits for its own next-greater
}
// still on the stack at the end ⇒ no next-greater exists (answer 0 / -1)
// Flip the comparison to < for next-SMALLER; scan right-to-left for "previous" variants.

Template C — queue via two stacks (amortized O(1))

class MyQueue {
    private final Deque<Integer> in  = new ArrayDeque<>();   // arrivals pile up here
    private final Deque<Integer> out = new ArrayDeque<>();   // reversed once, served in FIFO order

    public void push(int x) { in.push(x); }
    public int  pop()  { shift(); return out.pop(); }
    public int  peek() { shift(); return out.peek(); }
    public boolean empty() { return in.isEmpty() && out.isEmpty(); }

    private void shift() {                 // only when out runs dry — each element moves ONCE ever
        if (out.isEmpty())
            while (!in.isEmpty()) out.push(in.pop());   // stack reversal restores arrival order
    }
}

Decision cues:

  • "Valid/balanced brackets", "nested", "decode string", "simplify path" → stack (push context, pop on close).
  • "Next greater/smaller element", "days until warmer", "stock span", "histogram rectangle" → monotonic stack.
  • "Process in arrival order", "first come first served", level-by-level anything → queue.
  • "Max/min of each sliding window" → monotonic deque (page 03).
  • "Always take the smallest/largest available next" → not recency or arrival: heap (page 12).
  • "Most recent" + "also evict oldest by capacity" → deque or LRU structure (page 24).

Where this shows up in real systems

The call stack & your stack traces

Every thread runs on a literal stack: a method call pushes a frame (locals + return address), a return pops it. A Java stack trace is that stack printed top-down; StackOverflowError is its capacity limit — why deep recursion dies (page 10). The JVM's bytecode engine is additionally an operand stack machine: your RPN solution is a miniature of it.

Undo/redo & browser history

Editors keep two stacks: undo pops the most recent operation and pushes it onto redo; a fresh edit clears redo. Browser back/forward is the same pair of stacks over visited pages. "Most recent action reversed first" is inherently LIFO — no other policy makes sense.

Parsers, compilers & JSON validators

Every JSON parser, XML validator, and compiler front-end runs bracket-matching at industrial scale: push on {/(/open-tag, pop and verify on close. Shunting-yard converts infix to RPN with an operator stack; recursive-descent parsers use the call stack itself as the context stack.

Message queues & backpressure buffers

Kafka partitions and SQS queues are giant durable FIFOs: producers append at the tail, consumers advance from the head, preserving per-partition arrival order. In-process, bounded queues between stages (thread-pool work queues, Netty/Reactor buffers) absorb bursts and create backpressure when full — FIFO fairness plus flow control.

easy Valid Parentheses — LeetCode 20

The problem. A string contains only ()[]{}. Decide whether it's valid: every bracket closed by the matching type, in the right (properly nested) order.

Example. "()[]{}" → true. "(]" → false. "([)]" → false (interleaved). "{[]}" → true. ")(" → false.

Stack of unclosed openers · O(n) time, O(n) space

public static boolean isValid(String s) {
    Deque<Character> stack = new ArrayDeque<>();  // ArrayDeque, never java.util.Stack
    for (char c : s.toCharArray()) {
        if (c == '(' || c == '[' || c == '{') {
            stack.push(c);                 // opener: remember it — it's now the most recent debt to repay
        } else {
            if (stack.isEmpty()) return false;      // a closer with nothing open — e.g. ")(" — invalid
            char open = stack.pop();                // the most recent unclosed opener MUST match this closer
            if (c == ')' && open != '(') return false;   // mismatched pair like "(]"
            if (c == ']' && open != '[') return false;
            if (c == '}' && open != '{') return false;
        }
    }
    return stack.isEmpty();                // leftovers like "((" mean unclosed openers — invalid
}

Takeaway. Nesting = LIFO by definition: the most recently opened bracket must close first. Three distinct failure modes (mismatch, closer-with-nothing-open, unclosed leftovers), one invariant: the stack holds exactly the currently-unclosed openers, newest on top. Mirror: code/09-stacks-queues/ValidParentheses.java.

medium Min Stack — LeetCode 155

The problem. Design a stack supporting push, pop, top, and getMin — all in O(1). The catch: after any sequence of pops, getMin must reflect the minimum of what remains.

Example. push(−2), push(0), push(−3) → getMin = −3; pop() → getMin = −2.

Variation 1 — scan for min on demand · getMin becomes O(n)

// Keep one stack; on getMin(), walk every element. Correct but O(n) per
// query — the exercise is precisely to avoid this rescan.

Variation 2 — parallel min-stack (optimal) · all ops O(1)

public class MinStack {
    // Two stacks moving in lockstep:
    //   stack    = the actual values
    //   minStack = the minimum AS OF each push
    // Popping both together rewinds the minimum's history automatically.
    private final Deque<Integer> stack = new ArrayDeque<>();
    private final Deque<Integer> minStack = new ArrayDeque<>();

    public void push(int val) {
        stack.push(val);
        // The min after this push: the smaller of val and the previous min
        int min = minStack.isEmpty() ? val : Math.min(val, minStack.peek());
        minStack.push(min);
    }

    public void pop() {
        stack.pop();
        minStack.pop();          // discard that era's minimum along with the value
    }

    public int top() { return stack.peek(); }

    public int getMin() { return minStack.peek(); }   // min of everything still in the stack
}

Takeaway. The insight is that a stack's history only ever rewinds — so recording "the min as of this moment" beside each value lets pops restore old minimums for free. This per-state-snapshot idea also powers max-stacks and versioned registers. Mirror: code/09-stacks-queues/MinStack.java.

easy Implement Queue using Stacks — LeetCode 232

The problem. Build a FIFO queue (push, pop, peek, empty) using only stack operations.

Example. push(1), push(2) → peek() = 1 (front!), pop() = 1, empty() = false.

Two stacks: in + out, lazy transfer · amortized O(1) per op

public class QueueUsingStacks {
    // in  : newly pushed elements (newest on top)
    // out : reversed elements ready to pop in FIFO order (oldest on top)
    private final Deque<Integer> in = new ArrayDeque<>();
    private final Deque<Integer> out = new ArrayDeque<>();

    public void push(int x) { in.push(x); }

    public int pop() {
        shift();
        return out.pop();
    }

    public int peek() {
        shift();
        return out.peek();
    }

    public boolean empty() { return in.isEmpty() && out.isEmpty(); }

    // Only when out runs dry, dump in -> out. Two LIFO reversals = FIFO.
    // Each element makes this trip AT MOST ONCE in its lifetime, which is
    // why every operation is O(1) amortized despite the occasional O(n) dump.
    private void shift() {
        if (out.isEmpty())
            while (!in.isEmpty()) out.push(in.pop());
    }
}

Takeaway. Two reversals cancel: LIFO twice = FIFO. The lazy transfer (only when out is empty) is what makes it amortized O(1) — an eager per-push transfer would be O(n) per operation. The amortized argument ("each element moves once, ever") is the same one behind monotonic stacks. Mirror: code/09-stacks-queues/QueueUsingStacks.java.

easy Next Greater Element I — LeetCode 496

The problem. nums1 is a subset of nums2 (all values distinct). For each element of nums1, find the first greater element to its right in nums2 (−1 if none).

Example. nums1 [4,1,2], nums2 [1,3,4,2] → [-1,3,-1].

Monotonic decreasing stack + answer map · O(n)

public static int[] nextGreaterElement(int[] nums1, int[] nums2) {
    Map<Integer, Integer> nextGreater = new HashMap<>();
    Deque<Integer> stack = new ArrayDeque<>();     // values, decreasing
    for (int x : nums2) {
        // x arriving resolves EVERY smaller value still waiting:
        while (!stack.isEmpty() && stack.peek() < x)
            nextGreater.put(stack.pop(), x);
        stack.push(x);                             // x now waits for ITS greater
    }
    int[] out = new int[nums1.length];             // still on the stack -> -1
    for (int i = 0; i < nums1.length; i++)
        out[i] = nextGreater.getOrDefault(nums1[i], -1);
    return out;
}

Takeaway. The monotonic stack's hello-world: each element is pushed once and popped once, so "next greater for everyone" costs O(n) total — not O(n²). Daily Temperatures (below) is the same loop storing indices; the circular variant (LC 503) just runs the array twice. If you can narrate why the stack stays decreasing, you own the pattern. Mirror: code/09-stacks-queues/NextGreaterElementI.java.

easy Implement Stack using Queues — LeetCode 225

The problem. Build a LIFO stack using only queue operations (offer, poll, peek, isEmpty).

Example. push(1), push(2), top() → 2, pop() → 2, empty() → false.

One queue, rotate after each push · push O(n), pop/top O(1)

private final Queue<Integer> q = new ArrayDeque<>();

public void push(int x) {
    q.offer(x);
    for (int i = 0; i < q.size() - 1; i++)   // rotate the older elements
        q.offer(q.poll());                   // around behind x
}

public int pop() { return q.poll(); }        // head is always the newest
public int top() { return q.peek(); }
public boolean empty() { return q.isEmpty(); }

Takeaway. The inverse of Implement Queue using Stacks (above), and the asymmetry is the lesson: two stacks give queues amortized O(1), but queues can't return the favor — one adapter direction is cheap, the other pays O(n) somewhere. Choosing where to pay (push vs pop) is the interview follow-up; rotating on push keeps all reads O(1). Mirror: code/09-stacks-queues/StackUsingQueues.java.

medium Evaluate Reverse Polish Notation — LeetCode 150

The problem. Evaluate an arithmetic expression in postfix (RPN) form: operators come after their operands. Operators are + − * /; division truncates toward zero.

Example. ["2","1","+","3","*"] = (2+1)×3 → 9. ["4","13","5","/","+"] = 4 + 13/5 → 6.

Operand stack · O(n)

public static int evalRPN(String[] tokens) {
    Deque<Integer> stack = new ArrayDeque<>();
    for (String t : tokens) {
        switch (t) {
            case "+": case "-": case "*": case "/": {
                int b = stack.pop();       // top of stack = RIGHT operand (pushed most recently)
                int a = stack.pop();       // next = LEFT operand — order matters for - and /
                int r = switch (t) {
                    case "+" -> a + b;
                    case "-" -> a - b;     // a - b, NOT b - a: "3 4 -" means 3 - 4
                    case "*" -> a * b;
                    default  -> a / b;     // Java int division truncates toward zero, as LC 150 specifies
                };
                stack.push(r);             // the result becomes an operand for later operators
                break;
            }
            default:
                stack.push(Integer.parseInt(t));   // a number: park it until an operator claims it
        }
    }
    return stack.pop();                    // a valid RPN expression leaves exactly one value
}

Takeaway. RPN needs no parentheses and no precedence rules — the order is the evaluation plan, which is why compilers and old HP calculators loved it. The only trap is operand order on pop (top = right operand). Infix calculators (LC 224/227) are this plus a second stack or a sign convention. Mirror: code/09-stacks-queues/EvalRPN.java.

medium Daily Temperatures — LeetCode 739

The problem. For each day's temperature, output how many days you must wait for a strictly warmer temperature (0 if it never comes).

Example. [73,74,75,71,69,72,76,73] → [1,1,4,2,1,1,0,0].

Variation 1 — scan forward per day · O(n²)

public static int[] dailyTemperaturesBrute(int[] temps) {
    int n = temps.length;
    int[] answer = new int[n];
    for (int i = 0; i < n; i++)
        for (int j = i + 1; j < n; j++)
            if (temps[j] > temps[i]) { answer[i] = j - i; break; }
    return answer;
}

Variation 2 — monotonic decreasing stack (optimal) · O(n)

public static int[] dailyTemperatures(int[] temps) {
    int n = temps.length;
    int[] answer = new int[n];             // defaults to 0 = "no warmer day ever comes"
    Deque<Integer> stack = new ArrayDeque<>();  // indices; temps strictly decrease bottom→top
    for (int i = 0; i < n; i++) {
        // Today resolves every colder day still waiting on the stack
        while (!stack.isEmpty() && temps[i] > temps[stack.peek()]) {
            int j = stack.pop();           // day j finally sees a warmer day: today
            answer[j] = i - j;             // the wait was exactly the index distance
        }
        stack.push(i);                     // today now waits for ITS next warmer day
    }
    return answer;                         // indices left on the stack keep answer 0 — correct by default
}

Takeaway. The stack invariant: indices of days still waiting, temperatures decreasing toward the top (anything warmer would already have resolved what's above it). Each index is pushed once and popped at most once ⇒ O(n) despite the nested while — the same amortized argument as the sliding-window deque (page 03). "Next Greater Element" is this exact code returning values instead of distances. Mirror: code/09-stacks-queues/DailyTemperatures.java.

medium Car Fleet — LeetCode 853

The problem. Cars at positions position[i] drive toward a target at speeds speed[i]; a faster car that catches a slower one ahead slows to its speed and forms a fleet (fleets can't pass each other). How many fleets arrive at the target?

Example. target=12, position=[10,8,0,5,3], speed=[2,4,1,1,3] → 3: cars at 10 & 8 merge (both arrive by t=1), 5 & 3 merge, 0 travels alone.

Sort by position, compare arrival times with a stack · O(n log n)

public static int carFleet(int target, int[] position, int[] speed) {
    int n = position.length;
    // Pair up and sort by starting position, CLOSEST to target first —
    // a car can only ever catch fleets that started ahead of it.
    Integer[] idx = new Integer[n];
    for (int i = 0; i < n; i++) idx[i] = i;
    Arrays.sort(idx, (a, b) -> position[b] - position[a]);

    Deque<Double> fleets = new ArrayDeque<>();   // arrival time of each fleet, front-of-road first
    for (int k : idx) {
        double time = (double) (target - position[k]) / speed[k];  // solo arrival time
        // If this car would arrive AFTER the fleet directly ahead, it never
        // catches it -> it founds a NEW fleet. If it would arrive sooner or
        // equal, it slams into that fleet and dissolves into it (no push).
        if (fleets.isEmpty() || time > fleets.peek())
            fleets.push(time);
    }
    return fleets.size();
}

Takeaway. The physics reduces to one comparison: a car catches the fleet ahead iff its solo arrival time ≤ that fleet's arrival time. Processing front-of-road first means the stack top is always "the fleet directly ahead". Note the stack only ever grows here — a plain counter with a "last fleet time" variable works too; the stack framing just makes the invariant visible. Mirror: code/09-stacks-queues/CarFleet.java.

medium Generate Parentheses — LeetCode 22

The problem. Generate all strings of n pairs of well-formed parentheses.

Example. n=3 → ["((()))","(()())","(())()","()(())","()()()"]. n=1 → ["()"].

Backtracking with the validity invariant · O(4ⁿ/√n) outputs, O(n) depth

public static List<String> generateParenthesis(int n) {
    List<String> res = new ArrayList<>();
    build(res, new StringBuilder(), 0, 0, n);
    return res;
}

// open/close = how many of each we've placed so far.
// Valid prefixes are exactly those where close never exceeds open.
private static void build(List<String> res, StringBuilder sb, int open, int close, int n) {
    if (sb.length() == 2 * n) {            // used all n pairs — a complete valid string
        res.add(sb.toString());
        return;
    }
    if (open < n) {                        // can still open a new pair
        sb.append('(');
        build(res, sb, open + 1, close, n);
        sb.deleteCharAt(sb.length() - 1);  // un-choose (backtrack)
    }
    if (close < open) {                    // may close ONLY if something is open —
        sb.append(')');                    // this prune is Valid Parentheses' invariant
        build(res, sb, open, close + 1, n);
        sb.deleteCharAt(sb.length() - 1);
    }
}

Takeaway. The bridge to page 10: instead of checking validity with a stack after the fact, the recursion never generates an invalid prefix — the close < open guard is the stack invariant enforced at build time. Pruning during generation beats generate-then-filter (2²ⁿ strings) by an enormous margin. Mirror: code/09-stacks-queues/GenerateParentheses.java.

medium Asteroid Collision — LeetCode 735

The problem. Asteroids in a row; sign is direction (+ right, − left), magnitude is size. Equal sizes destroy both; otherwise the smaller explodes. Return the survivors.

Example. [5,10,-5] → [5,10]. [10,2,-5] → [10]. [8,-8] → [].

Stack of survivors; resolve on right-meets-left · O(n)

public static int[] asteroidCollision(int[] asteroids) {
    Deque<Integer> stack = new ArrayDeque<>();
    for (int a : asteroids) {
        boolean alive = true;
        // Only "stack top → right, newcomer ← left" ever collides.
        while (alive && a < 0 && !stack.isEmpty() && stack.peek() > 0) {
            int top = stack.peek();
            if (top < -a)                // top smaller — it explodes,
                stack.pop();             // newcomer keeps plowing left
            else if (top == -a) {        // equal — both explode
                stack.pop();
                alive = false;
            } else                       // top bigger — newcomer dies
                alive = false;
        }
        if (alive) stack.push(a);
    }
    int[] out = new int[stack.size()];   // bottom-to-top = left-to-right
    for (int i = out.length - 1; i >= 0; i--)
        out[i] = stack.pop();
    return out;
}

Takeaway. A monotonic-stack cousin where the "pop condition" is a physics rule instead of a comparison: the stack holds asteroids whose fate isn't settled, and each newcomer resolves as many as it can. The enumeration discipline — exactly one of four sign-pairs collides — is where solutions go wrong; -2,-1 sailing past 1,2 (no collision) is the test to check. Mirror: code/09-stacks-queues/AsteroidCollision.java.

medium Decode String — LeetCode 394

The problem. Decode k[segment] notation, where segment repeats k times and brackets nest arbitrarily.

Example. "3[a]2[bc]" → "aaabcbc". "3[a2[c]]" → "accaccacc".

Two stacks: pending counts + pending prefixes · O(output)

public static String decodeString(String s) {
    Deque<Integer> counts = new ArrayDeque<>();
    Deque<StringBuilder> prefixes = new ArrayDeque<>();
    StringBuilder curr = new StringBuilder();
    int num = 0;
    for (char c : s.toCharArray()) {
        if (c >= '0' && c <= '9') {
            num = num * 10 + (c - '0');       // multi-digit counts ("10[a]")
        } else if (c == '[') {
            counts.push(num);                 // save the outer context
            prefixes.push(curr);
            num = 0;
            curr = new StringBuilder();       // build the inner segment fresh
        } else if (c == ']') {
            StringBuilder inner = curr;
            curr = prefixes.pop();            // restore the outer segment
            int k = counts.pop();
            for (int i = 0; i < k; i++)       // append k copies of the inner
                curr.append(inner);
        } else {
            curr.append(c);
        }
    }
    return curr.toString();
}

Takeaway. Nesting = recursion = a stack; here the "call frame" is the pair (pending count, string built so far), saved at '[' and restored at ']'. This save/restore shape is exactly how interpreters and JSON/XML parsers handle nested structure. Watch multi-digit numbers — "10[a]" breaks the num = c - '0' shortcut. Mirror: code/09-stacks-queues/DecodeString.java.

medium Simplify Path — LeetCode 71

The problem. Canonicalize a Unix path: collapse "//", drop ".", resolve ".." (which cannot go above root). Names like "..." are legal directories.

Example. "/home//foo/" → "/home/foo". "/a/./b/../../c/" → "/c". "/../" → "/".

Split on '/', stack the real names · O(n)

public static String simplifyPath(String path) {
    Deque<String> stack = new ArrayDeque<>();
    for (String seg : path.split("/")) {
        if (seg.isEmpty() || seg.equals("."))
            continue;                        // "//" and "." change nothing
        if (seg.equals("..")) {
            if (!stack.isEmpty())            // ".." above root is a no-op
                stack.pollLast();
        } else {
            stack.offerLast(seg);            // a real directory name
        }
    }
    return "/" + String.join("/", stack);    // bottom-to-top = the path
}

Takeaway. ".." undoing the most recent directory is textbook LIFO — the same reason editors implement undo with a stack. Splitting first turns character-level parsing into clean token handling; the deque-as-stack (offerLast/pollLast) keeps left-to-right output order for the final join. Every OS's realpath() and every browser's URL normalizer runs this loop. Mirror: code/09-stacks-queues/SimplifyPath.java.

medium Basic Calculator II — LeetCode 227

The problem. Evaluate an expression with +, −, *, /, spaces and non-negative integers (no parentheses), honoring precedence, without eval().

Example. "3+2*2" → 7. " 3/2 " → 1. "14-3/2" → 13.

Stack of signed terms; * and / bind immediately · O(n)

public static int calculate(String s) {
    Deque<Integer> terms = new ArrayDeque<>();
    int num = 0;
    char pendingOp = '+';                    // operator waiting to apply to num
    for (int i = 0; i < s.length(); i++) {
        char c = s.charAt(i);
        if (c >= '0' && c <= '9')
            num = num * 10 + (c - '0');
        // On an operator (or the last char), resolve the PENDING op:
        if ((c != ' ' && (c < '0' || c > '9')) || i == s.length() - 1) {
            switch (pendingOp) {
                case '+': terms.push(num); break;          // decide later
                case '-': terms.push(-num); break;         // sign travels along
                case '*': terms.push(terms.pop() * num); break;  // binds NOW
                case '/': terms.push(terms.pop() / num); break;  // binds NOW
            }
            pendingOp = c;
            num = 0;
        }
    }
    int sum = 0;
    for (int t : terms) sum += t;            // only +/- remain — just add up
    return sum;
}

Takeaway. Precedence handled without a parser: high-precedence ops (*, /) collapse into the previous term immediately, low-precedence ones (+, −) defer by pushing signed terms — the stack holds a sum-in-waiting. This one-pass trick generalizes to the full calculator family (LC 224 adds parentheses = save/restore like Decode String). RPN (above) is what you get when the expression arrives pre-flattened. Mirror: code/09-stacks-queues/BasicCalculatorII.java.

hard Largest Rectangle in Histogram — LeetCode 84

The problem. Given bar heights (width 1 each), find the area of the largest rectangle that fits entirely inside the histogram.

Example. [2,1,5,6,2,3] → 10 (the 5 and 6 bars support a 5×2 rectangle). [2,4] → 4.

Variation 1 — expand around each bar · O(n²)

public static int largestRectangleBrute(int[] heights) {
    int best = 0;
    for (int i = 0; i < heights.length; i++) {
        // The rectangle of height heights[i] extends while bars stay >= it
        int l = i, r = i;
        while (l > 0 && heights[l - 1] >= heights[i]) l--;
        while (r < heights.length - 1 && heights[r + 1] >= heights[i]) r++;
        best = Math.max(best, heights[i] * (r - l + 1));
    }
    return best;
}

Variation 2 — increasing monotonic stack (optimal) · O(n)

public static int largestRectangleArea(int[] heights) {
    int n = heights.length;
    Deque<Integer> stack = new ArrayDeque<>();  // indices; heights INCREASE toward the top
    int best = 0;
    // Iterate one past the end with a virtual height-0 bar: it forces
    // every remaining bar to pop and be evaluated (the "sentinel flush").
    for (int i = 0; i <= n; i++) {
        int h = (i == n) ? 0 : heights[i];
        while (!stack.isEmpty() && h < heights[stack.peek()]) {
            // Popping bar j: its rectangle is now fully determined.
            int j = stack.pop();
            int height = heights[j];
            // Right boundary: i (first bar shorter on the right).
            // Left boundary: new stack top (first bar shorter on the left),
            // or the very start if the stack emptied.
            int left = stack.isEmpty() ? -1 : stack.peek();
            int width = i - left - 1;
            best = Math.max(best, height * width);
        }
        stack.push(i);
    }
    return best;
}

Takeaway. Every bar's maximal rectangle is bounded by its nearest shorter neighbor on each side — and the increasing stack delivers both boundaries at the exact moment a bar pops (right = the incoming shorter bar, left = the bar beneath it on the stack). The height-0 sentinel that flushes the stack is the standard finishing move. Maximal Rectangle (LC 85) runs this per matrix row. Mirror: code/09-stacks-queues/LargestRectangle.java.

Pitfalls

  • Declare Deque<T> s = new ArrayDeque<>() — java.util.Stack is legacy (synchronized, extends Vector).
  • Know your API pairs: stack = push/pop/peek, queue = offer/poll/peek. Mixing push (head) with poll (head) on the same deque silently gives you LIFO when you wanted FIFO — a brutal quiet bug.
  • Store indices, not values, in monotonic stacks. Indices give you distances and widths for free; you can always recover the value via a[i], never the reverse.
  • Strict vs non-strict comparison matters: "next greater" (>) vs "next greater-or-equal" (>=) changes how equal values chain. Decide from the problem statement.
  • ArrayDeque rejects null (throws NPE) — it uses null internally as its empty marker. Also, poll/peek return null on empty while pop/remove throw; pick one style and guard accordingly.
  • State the stack invariant before coding: "the stack holds indices of days still awaiting a warmer day, temperatures decreasing". If you can't finish that sentence, you don't have the solution yet.

Recursion & Backtracking

Learn to trust a function that calls itself, read complexity off the recursion tree, and master the choose → explore → un-choose template that generates every combination interviewers ask for.

Phase 4 · Weeks 6–7 Gateway skill — trees, graphs and DP all build on this

Before you start

Why this topic exists. After a decade of Java you write loops on autopilot — and that autopilot actively fights recursion. Interviews use recursion as a filter because it predicts everything that follows: tree traversals are recursion, graph DFS is recursion, dynamic programming is recursion plus a cache. Backtracking is the special case that answers "generate all valid X" — subsets, permutations, board placements — which is an entire category of medium questions. Get the mental model right here and pages 11–20 become variations on a theme.

Prerequisites

  • 00 · Foundations — Big-O, and the fact that the call stack is O(depth) space.
  • 09 · Stacks & Queues — recursion literally runs on a stack; you'll convert between the two.

What the interviewer is actually testing

  • Can you define the base case first? Most broken recursive code fails at "when do we stop", not at the clever part.
  • Do you trust the recursive call? Seniors state "assume f(n-1) returns the right answer" and build on it; juniors try to trace every frame and drown.
  • Can you count the work? Reading O(n·2ⁿ) or O(n·n!) off a recursion tree, unprompted, is a strong signal.
  • Is your state restored? Forgetting the un-choose step (or sharing a mutated list) is the classic backtracking bug — interviewers watch for it.

How this builds on the previous topic

Page 09 made you operate stacks by hand. Recursion is the JVM operating a stack for you: every call pushes a frame (parameters + locals), every return pops one. That's why deep recursion throws StackOverflowError, why recursion depth counts as space complexity, and why any recursive algorithm can be rewritten with an explicit ArrayDeque — you're just managing the frames yourself.

Core concepts

1. Recursion mechanics: base case + leap of faith

A recursive function is a contract: "given a smaller input, I return the right answer." You only ever write two things — the smallest input answered directly (base case), and one step that reduces the problem and trusts the contract for the rest.

The anatomy, on the simplest possible example

// factorial(4) -> 4 * factorial(3) -> ... four stacked frames, then unwinding
long factorial(int n) {
    if (n <= 1) return 1;          // BASE CASE: answered directly, no recursion
    return n * factorial(n - 1);   // LEAP OF FAITH: trust the (n-1) answer, extend it
}
The leap of faith is not laziness — it's induction. You verify the base case and you verify "if the smaller call is correct, my step is correct". That's a proof by induction; mentally tracing 15 nested frames is not. When stuck in an interview, say out loud: "Assume the recursive call works. What do I do with its answer?"

2. The recursion tree: where complexity comes from

Draw every call as a node and its recursive calls as children. Total work = (number of nodes) × (work per node). Three shapes cover almost everything:

ShapeExampleNodesTypical cost
One child, depth nfactorial, linked-list recursionnO(n) time, O(n) stack
Two children per node, depth nsubsets (include/exclude), naive fib~2ⁿO(2ⁿ) — exponential
Branching shrinks per levelpermutations: n, then n−1, …~n!O(n!) — factorial
Preview of DP. When the recursion tree contains the same subproblem many times (fib(3) computed five times), caching answers collapses the tree — that's memoization, page 18. Backtracking problems on this page generate distinct outputs at every leaf, so there is nothing to cache: exponential really is the answer.

3. Converting recursion to iteration

Any recursion can become a loop with an explicit stack — push what you would have passed as parameters, pop instead of calling. You'll use this on page 11 for iterative traversals and whenever depth could exceed the JVM's ~10⁴ default frames. Tail-call-style recursions (one call, last statement) convert even easier: a plain while loop. Java does not optimize tail calls — worth saying in interviews.

4. Backtracking = DFS over the tree of choices

Backtracking answers "enumerate all configurations that satisfy constraints". Frame every such problem as a decision tree: each level decides one thing, each root-to-leaf path is one candidate. Then walk the tree depth-first with the three-beat rhythm — choose (mutate shared state), explore (recurse), un-choose (undo the mutation so siblings start clean). Pruning means noticing a partial path can never succeed and abandoning the whole subtree — it's why real backtracking beats brute force by orders of magnitude.

Pattern templates (Java)

The universal backtracking skeleton — memorize this rhythm

// result: all complete solutions.  path: the partial solution being built.
void backtrack(State state, List<Integer> path, List<List<Integer>> result) {
    if (isComplete(state)) {                 // BASE CASE: path is a full solution
        result.add(new ArrayList<>(path));   // COPY IT — path will be mutated later!
        return;
    }
    for (Choice c : legalChoices(state)) {   // candidates for the NEXT decision
        if (!promising(c, state)) continue;  // PRUNE: this subtree can't succeed
        apply(c, state, path);               // 1. CHOOSE  (mutate shared state)
        backtrack(state, path, result);      // 2. EXPLORE (recurse one level deeper)
        undo(c, state, path);                // 3. UN-CHOOSE (restore, mirror of apply)
    }
}
Two bugs cause 90% of backtracking failures. (1) Adding path itself instead of new ArrayList<>(path) — every "solution" in your result mutates into the same empty list as the recursion unwinds. (2) An un-choose that isn't the exact mirror of the choose (wrong order, or forgetting one of two mutations). Write choose and un-choose as adjacent lines first, then fill in the middle.

Recursion → iteration: the same DFS with an explicit stack

// Each stack entry holds what the recursive frame would have held.
// Useful when depth may exceed the JVM stack, or the interviewer asks for it.
Deque<Frame> stack = new ArrayDeque<>();
stack.push(initialFrame);
while (!stack.isEmpty()) {
    Frame f = stack.pop();               // "entering the call"
    if (isComplete(f)) { record(f); continue; }
    for (Choice c : legalChoices(f))
        stack.push(f.with(c));           // "recursive call" = push child frame
}

  • Problem says "how many ways" or "minimum cost" and n is large → suspect DP; backtracking would recompute overlapping subproblems.
  • Problem says "generate combinations" (order irrelevant) → start-index pattern; "arrangements" (order matters) → used[] pattern.

Where this shows up in real systems

Dependency resolvers (npm, Maven, apt)

Picking versions that satisfy every package's constraints is a search: choose a version, propagate constraints, and on conflict backtrack to the last choice point and try another. Modern resolvers (npm's arborist, Rust's cargo, Dart's PubGrub) are literally backtracking SAT-style solvers with clever pruning.

Regex engines

Java's java.util.regex is a backtracking matcher: alternation tries one branch, and on failure un-chooses and tries the next. It's also why a pathological pattern like (a+)+$ on a long string of a's explodes — "catastrophic backtracking" is the exponential recursion tree from this page hitting production.

SQL query planners

Optimizers explore the tree of join orders and access paths — choose a join, cost the partial plan, prune branches already worse than the best complete plan (branch-and-bound). PostgreSQL switches from exhaustive search to heuristics past ~12 joins precisely because the tree is factorial.

Solvers & filesystem walks

Sudoku/config solvers (choose a value, check constraints, backtrack) are the N-Queens pattern with different constraint sets. And Files.walkFileTree is plain recursion over directories — visit, recurse into children, return — the everyday face of DFS.

Worked solution: Subsets (LeetCode 78)

Problem in one line: given distinct integers, return every possible subset (the power set).

Idea: walk the elements left to right; for each one make two recursive calls — one that excludes it and one that includes it. When you've decided all n elements, the current path is exactly one subset. This is the decision tree from the animation above; the code is a depth-first walk of it.

code/10-recursion-backtracking/Subsets.java

// LeetCode 78. Return ALL subsets (the power set) of a set of distinct ints.
public static List<List<Integer>> subsets(int[] nums) {
    List<List<Integer>> result = new ArrayList<>();
    // path = the choices made so far on the way down one branch
    backtrack(nums, 0, new ArrayList<>(), result);
    return result;
}

private static void backtrack(int[] nums, int i, List<Integer> path,
                              List<List<Integer>> result) {
    // Base case: we have made a decision for EVERY element -> path is one
    // complete subset. Copy it (path itself keeps mutating afterwards!).
    if (i == nums.length) {
        result.add(new ArrayList<>(path));
        return;
    }
    // Branch 1: EXCLUDE nums[i]. No state change, just move on.
    backtrack(nums, i + 1, path, result);

    // Branch 2: INCLUDE nums[i].
    path.add(nums[i]);                  // choose
    backtrack(nums, i + 1, path, result); // explore
    path.remove(path.size() - 1);       // un-choose: restore state for caller
}

Complexity: 2ⁿ leaves, each copied in O(n) → time O(n·2ⁿ); space O(n) for the recursion depth and path (output not counted).

Worked solution: Permutations (LeetCode 46)

Problem in one line: given distinct integers, return every possible ordering of them.

Idea: build the answer slot by slot. Because order matters, the candidates for the next slot are all elements not yet placed — not just "later" ones. A boolean[] used marks what the current branch has consumed; un-choosing clears the mark so sibling branches can use that element.

code/10-recursion-backtracking/Permutations.java

// LeetCode 46. Return ALL orderings of an array of distinct ints.
public static List<List<Integer>> permute(int[] nums) {
    List<List<Integer>> result = new ArrayList<>();
    // used[j] = "nums[j] is already placed somewhere in the current path"
    backtrack(nums, new boolean[nums.length], new ArrayList<>(), result);
    return result;
}

private static void backtrack(int[] nums, boolean[] used, List<Integer> path,
                              List<List<Integer>> result) {
    // Base case: every slot is filled -> path is one complete permutation.
    if (path.size() == nums.length) {
        result.add(new ArrayList<>(path)); // copy — path keeps mutating
        return;
    }
    // Unlike subsets, EVERY still-unused element is a candidate for the
    // next slot (order matters, so we don't restrict to "later" indices).
    for (int j = 0; j < nums.length; j++) {
        if (used[j]) continue;      // prune: already placed in this branch
        used[j] = true;             // choose nums[j] for the current slot
        path.add(nums[j]);
        backtrack(nums, used, path, result);   // explore deeper
        path.remove(path.size() - 1);          // un-choose (mirror order!)
        used[j] = false;
    }
}

Complexity: n! leaves, O(n) copy each → time O(n·n!); space O(n) for recursion depth, path and used.

Worked solution: Combination Sum (LeetCode 39)

Problem in one line: distinct candidates, unlimited reuse of each — return all unique combinations that sum to a target.

Idea: the danger is producing [2,3] and [3,2] as "different" answers. The fix is the start-index pattern: at each level only consider candidates from index start onward, so combinations are always generated in non-decreasing candidate order. Reuse is allowed by recursing with i rather than i+1. Sorting first lets us break the instant a candidate exceeds what's left — real pruning.

code/10-recursion-backtracking/CombinationSum.java

// LeetCode 39. Distinct candidates, each usable UNLIMITED times.
// Return all unique combinations summing to target.
public static List<List<Integer>> combinationSum(int[] candidates, int target) {
    List<List<Integer>> result = new ArrayList<>();
    Arrays.sort(candidates);   // lets us break early once a candidate exceeds what's left
    backtrack(candidates, 0, target, new ArrayList<>(), result);
    return result;
}

private static void backtrack(int[] cand, int start, int remaining,
                              List<Integer> path, List<List<Integer>> result) {
    // Base case: we spent the target exactly -> path is a valid combination.
    if (remaining == 0) {
        result.add(new ArrayList<>(path));
        return;
    }
    // Only consider candidates from `start` onward. This is THE trick that
    // avoids duplicates: [2,3] and [3,2] can never both appear, because once
    // we move past 2 we never pick it again.
    for (int i = start; i < cand.length; i++) {
        if (cand[i] > remaining) break; // prune: sorted, so the rest are bigger too
        path.add(cand[i]);              // choose
        // pass `i`, NOT `i + 1`: same element may be reused (unlimited supply)
        backtrack(cand, i, remaining - cand[i], path, result);
        path.remove(path.size() - 1);   // un-choose
    }
}

Complexity: exponential in the worst case — roughly O(nᵀ/ᵐ) branches where m is the smallest candidate; space O(target/m) recursion depth. Honest answer in interviews: "exponential, bounded by the pruning".

Hint-level walkthrough: N-Queens (LeetCode 51)

Problem in one line: place n queens on an n×n board so none attack another; return all boards.

The insight chain: (1) exactly one queen per row → recursion depth = row number, one decision (the column) per level. (2) A placement conflicts iff its column, its ↘ diagonal (constant r−c), or its ↙ diagonal (constant r+c) is already taken — so keep three HashSet<Integer>s (or boolean arrays) and conflict checks are O(1). (3) Choose = add col, r−c, r+c to the sets and place the queen; un-choose = remove all three. Pruning is savage here: most branches die within a few rows, which is why n=9 (352,696-node full tree in theory) solves instantly. This is the template you'd reuse for Sudoku solvers.

Pitfalls & interview tips

  • result.add(path) without copying — the #1 bug. Every stored reference points at the same list, which unwinds to empty. Always new ArrayList<>(path) at the base case.
  • Un-choose must mirror choose exactly. Two mutations in (add to path, set used) means two mutations out, in reverse order. Write both lines before writing the recursive call.
  • State the complexity from the tree shape: "2ⁿ leaves × O(n) copy = O(n·2ⁿ)". Interviewers accept exponential answers for enumeration problems — they just want them derived, not guessed.
  • Watch the stack. Recursion depth is space: O(n) here, but a 10⁵-deep list/graph recursion overflows the JVM's default stack. Mention the explicit-stack rewrite as the fix.
  • Prune before recursing, not after. if (cand[i] > remaining) break; saves the whole subtree; checking validity only at the leaves is brute force wearing a costume.
  • Say the rhythm out loud — "choose, explore, un-choose" — as you code it. It structures your solution and tells the interviewer you have the pattern, even before the code compiles.

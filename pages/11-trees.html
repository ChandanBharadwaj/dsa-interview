Trees & Binary Search Trees

The recursive data structure: traversals that must become instinct, the top-down vs bottom-up recursion split, and the BST invariant that powers TreeMap and half the databases you've ever queried.

Phase 4 · Weeks 6–7 Top-3 interview area — expect at least one tree question per loop

Before you start

Why this topic exists. Trees are where interviews check that recursion became instinct. A tree is defined recursively — a node plus two smaller trees — so nearly every tree solution is five lines of recursion, and the interviewer watches whether you find those five lines in two minutes or twenty. Trees are also the most-used "advanced" structure in real life: TreeMap, every database index, the DOM, your filesystem, every parsed piece of JSON.

Prerequisites

  • 10 · Recursion & Backtracking — base case + leap of faith; this whole page is that skill applied.
  • 09 · Stacks & Queues — BFS runs on an ArrayDeque queue; iterative DFS on a stack.
  • 07 · Binary Search — the BST is binary search turned into a data structure.

What the interviewer is actually testing

  • Traversal fluency: can you pick pre/in/post/level-order for a task and write it without notes?
  • The top-down vs bottom-up split: do you know when to pass state down (parameters) vs return answers up? This one distinction solves most tree mediums.
  • Invariant thinking: validate-BST is a famous bug trap — do you reason about what all ancestors require, or just the parent?
  • Complexity honesty: O(n) time for visiting every node, O(h) stack space — and h is log n only if the tree is balanced.

How this builds on the previous topic

Page 10 taught you to trust a recursive call on a smaller input. A tree hands you the smaller inputs for free: root.left and root.right are smaller trees. Every backtracking idea transfers — the decision tree you drew for subsets is literally a tree being DFS-walked. What's new here is data living in the nodes, and one killer invariant (the BST ordering) that turns a tree into a search structure.

Core concepts

1. Vocabulary and the recursive definition

A binary tree is either empty, or a node with a value and two binary trees (left, right). That's it — and it's why tree code is short: the definition is already a recursion.

  • Root — the top node. Leaf — a node with no children. Subtree — any node treated as a root.
  • Depth of a node — edges from the root down to it. Height of a tree — edges on the longest downward path. A balanced tree has height ~log₂n; a degenerate one (every node one child) is a linked list with height n.
  • In Java interviews you'll be handed exactly this node — no fields you don't see:

The node every tree question uses

static class TreeNode {
    int val;
    TreeNode left, right;
    TreeNode(int val) { this.val = val; }
}

2. The four traversals (and when each is the right one)

TraversalOrder at each nodeReach for it when…
Pre-order (DFS)node → left → rightcopying / serializing a tree (parent must be written before children)
In-order (DFS)left → node → rightBSTs: emits values in sorted order — kth smallest, validation via previous value
Post-order (DFS)left → right → nodeanything needing children's answers first: height, diameter, delete/free
Level-order (BFS)level by level, via a queueanything about levels: per-level grouping, zigzag, right-side view, nearest leaf

3. The one decision that solves most tree problems: top-down vs bottom-up

Every recursive tree solution passes information in one of two directions. Deciding which is 80% of solving the problem:

  • Top-down (pre-order thinking): ancestors know something the node needs → pass it as a parameter. Example: validate BST — the allowed (min, max) window comes from ancestors above.
  • Bottom-up (post-order thinking): the node needs its children's answers → recurse first, then combine return values. Example: height, diameter, "is balanced".
maxDepth vs diameter is the canonical pair. Both are bottom-up, but diameter shows the pro move: the recursion returns one thing the parent can use (height) while recording a different global answer (best path through each node) on the side. When "the answer might live in any subtree", return the extendable value and track the best seen in a field or array cell.

4. Binary search trees: the ordering invariant

A BST adds one rule: for every node, everything in its left subtree is smaller, and everything in its right subtree is larger. Two consequences you'll use constantly:

  • Search/insert are a walk: compare, go left or right, done in O(h) — binary search as a structure. No rebuilding, unlike a sorted array.
  • In-order traversal emits sorted order. Kth smallest = stop an in-order walk at the kth visit; validation can also be "is my in-order sequence strictly increasing?".
The classic validate-BST bug. Checking only left.val < node.val && right.val > node.val passes trees that are locally fine but globally broken — a 3 hiding in the right subtree of a 5, two levels down. The invariant is about all ancestors, so carry an (exclusive) min/max window down the recursion. This exact trap appears in real interviews weekly.

LCA in a BST is a one-liner of thought: walk from the root; the first node whose value sits between p and q is where they split — that's the lowest common ancestor. (General binary trees need the bottom-up version: "did I find p or q below me?" — both patterns are worth knowing.)

5. Balance: why TreeMap is always O(log n)

Insert sorted data into a naive BST and it degenerates into a linked list — O(n) everything. Self-balancing trees fix this by doing O(1) local rotations on the way up from an insert/delete to keep height O(log n). Java's TreeMap/TreeSet use a red-black tree: nodes carry a color bit, and a handful of color+rotation rules guarantee the longest root-to-leaf path is at most ~2× the shortest. You will never implement one in an interview — but say "TreeMap is a red-black tree, so floorKey/ceilingKey and friends are O(log n) guaranteed" and you've banked senior credit. Databases use the same idea scaled to disk: B-trees are "BSTs with hundreds of keys per node" so each node is one page read (page 24).

6. Serialize / deserialize (the pattern, briefly)

Pre-order with explicit null markers ("4 2 # # 6 # #") captures a tree's exact shape in one pass, and a recursive reader consuming tokens left-to-right rebuilds it — the parent-first property of pre-order is exactly why it works. This is LeetCode 297, and it's also conceptually what every JSON/AST writer does.

Pattern templates (Java)

DFS template — the shape of 80% of tree solutions

ReturnType dfs(TreeNode node /*, top-down state */) {
    if (node == null) return BASE;          // empty tree: the safe default
    // pre-order work here (uses state passed DOWN from ancestors)
    ReturnType left = dfs(node.left);       // trust the recursion (page 10)
    // in-order work here (between the two child calls)
    ReturnType right = dfs(node.right);
    // post-order work here: COMBINE children's answers going UP
    return combine(left, right, node.val);
}

BFS template — queue + the frozen-size trick for levels

Deque<TreeNode> queue = new ArrayDeque<>();
if (root != null) queue.offer(root);
while (!queue.isEmpty()) {
    int levelSize = queue.size();           // freeze: exactly one level's nodes
    for (int i = 0; i < levelSize; i++) {
        TreeNode node = queue.poll();
        // process node — everything in this loop is on the SAME level
        if (node.left != null)  queue.offer(node.left);   // next level
        if (node.right != null) queue.offer(node.right);
    }
    // end of one level — depth counter++, or close the level list here
}

BST search — the invariant doing the work

TreeNode search(TreeNode node, int target) {
    if (node == null || node.val == target) return node;
    return target < node.val
        ? search(node.left, target)     // everything bigger is on the right — skip it
        : search(node.right, target);   // everything smaller is on the left — skip it
}   // O(h): log n balanced, n degenerate

  • Decision cue: only exact lookups → HashMap; order + mutation → TreeMap; order + never mutates → sorted array + binary search.
  • DFS vs BFS on trees: if the question mentions levels, depth-by-depth output, or "closest to root" → BFS. Everything else — heights, paths, subtree properties — is usually shorter as DFS. Same O(n) time either way; DFS costs O(h) stack, BFS O(width) queue (up to n/2 on the bottom level).
  • If the problem says "kth smallest", "range", "successor" on a BST → in-order traversal or the invariant walk, never a full sort.

Where this shows up in real systems

Filesystems and the DOM

Directories, HTML documents, and React component trees are all n-ary trees. Every Files.walkFileTree, CSS selector match, and virtual-DOM diff is a traversal from this page — pre-order to visit parents first, post-order to compute sizes or tear down children first.

TreeMap / TreeSet in the JDK

Red-black trees you've been shipping for years: guaranteed O(log n) put/get, plus the navigation calls (floorKey, ceilingEntry, subMap) that make them the tool for "nearest value" and range problems — and for order books, calendars, and consistent-hash rings (page 24).

Database indexes: B-trees

MySQL InnoDB, Postgres, and SQLite index with B+-trees — the BST invariant generalized to hundreds of keys per node so each hop is one disk page. A 4-level B-tree indexes billions of rows: that's the log in "indexed query". Range scans are exactly in-order traversal.

Compilers, JSON & ASTs

javac parses your code into an abstract syntax tree, then every phase is a traversal: type-checking is bottom-up (children's types first), code generation is post-order. Jackson serializing your DTOs walks a tree pre-order with null markers — LeetCode 297 in production clothes.

Worked solution: Maximum Depth of Binary Tree (LeetCode 104)

Problem in one line: return the number of nodes on the longest root-to-leaf path.

Idea: the recursive definition writes the code for you. The empty tree has depth 0; any other tree is 1 (itself) plus the depth of its deeper subtree. Trust the recursion to handle the subtrees — that's the leap of faith from page 10 in its purest form.

code/11-trees/MaxDepth.java

// LeetCode 104. Depth = number of nodes on the longest root-to-leaf path.
public static int maxDepth(TreeNode root) {
    // Base case: the empty tree has depth 0. This one line also handles
    // leaves cleanly (both their children are null subtrees of depth 0).
    if (root == null) return 0;
    // Leap of faith: trust the recursion to measure each subtree...
    int left = maxDepth(root.left);
    int right = maxDepth(root.right);
    // ...then combine: my depth = the deeper subtree + 1 for myself.
    // "Combine after recursing" makes this a POST-ORDER (bottom-up) pattern.
    return Math.max(left, right) + 1;
}

Complexity: time O(n) — each node visited once; space O(h) recursion stack — O(log n) balanced, O(n) degenerate.

Worked solution: Diameter of Binary Tree (LeetCode 543)

Problem in one line: length in edges of the longest path between any two nodes — the path need not pass through the root.

Idea: at every node, the longest path bending through it is (height of left) + (height of right) + 2 connecting edges. So compute heights bottom-up, and while you're at it, record that bend at every node — the best one anywhere is the diameter. The subtlety: the recursion returns height (what a parent can extend) while the diameter is tracked on the side, because a parent's path can't use both of my subtrees.

code/11-trees/DiameterOfBinaryTree.java

// LeetCode 543. Diameter = length (in EDGES) of the longest path between
// any two nodes. The path does not have to pass through the root.
private static int best; // longest path seen anywhere in the tree

public static int diameterOfBinaryTree(TreeNode root) {
    best = 0;
    height(root);        // we only run the traversal for its side effect
    return best;
}

// Returns the height (edges on the longest downward path) of `node`,
// and as a SIDE EFFECT updates the best diameter seen so far.
private static int height(TreeNode node) {
    if (node == null) return -1; // so a leaf gets height -1 + 1 = 0
    int left = height(node.left);    // bottom-up: children answered first
    int right = height(node.right);
    // The longest path THROUGH this node uses the deepest chain on each
    // side: left height + right height + the 2 edges connecting them.
    best = Math.max(best, left + right + 2);
    // But we RETURN only what the parent can extend: one side + 1 edge.
    // (The parent's path can't bend through both of my subtrees.)
    return Math.max(left, right) + 1;
}

Complexity: time O(n), space O(h). In an interview prefer passing a one-element int[] instead of a static field if asked about thread safety.

Worked solution: Binary Tree Level Order Traversal (LeetCode 102)

Problem in one line: return the values grouped level by level, left to right.

Idea: BFS with a queue naturally visits in level order; the only trick is knowing where one level ends. Freeze queue.size() before the inner loop — at that moment the queue holds exactly the current level, because children enqueued during the loop belong to the next one.

code/11-trees/LevelOrder.java

// LeetCode 102. Return node values grouped level by level, left to right.
public static List<List<Integer>> levelOrder(TreeNode root) {
    List<List<Integer>> result = new ArrayList<>();
    if (root == null) return result;         // empty tree -> no levels
    Deque<TreeNode> queue = new ArrayDeque<>(); // FIFO frontier (never Stack!)
    queue.offer(root);
    while (!queue.isEmpty()) {
        // KEY TRICK: freeze the queue size NOW — that count is exactly one
        // level, because children we enqueue below belong to the NEXT level.
        int levelSize = queue.size();
        List<Integer> level = new ArrayList<>(levelSize);
        for (int i = 0; i < levelSize; i++) {
            TreeNode node = queue.poll();    // take from the front (FIFO)
            level.add(node.val);
            // enqueue children left-to-right so the next level stays ordered
            if (node.left != null) queue.offer(node.left);
            if (node.right != null) queue.offer(node.right);
        }
        result.add(level);                   // one completed level per pass
    }
    return result;
}

Complexity: time O(n); space O(w) where w is the widest level — up to ~n/2 for a full bottom level.

Worked solution: Validate Binary Search Tree (LeetCode 98)

Problem in one line: check whether a binary tree satisfies the BST invariant everywhere.

Idea: top-down recursion carrying the window of legal values. Each node must fall strictly inside the (min, max) imposed by all its ancestors; descending left tightens the ceiling to my value, descending right raises the floor. Using long bounds sidesteps the edge case of node values equal to Integer.MIN_VALUE/MAX_VALUE.

code/11-trees/ValidateBST.java

// LeetCode 98. Is this a valid binary search tree?
public static boolean isValidBST(TreeNode root) {
    // Every node must live inside an OPEN interval (min, max) imposed by
    // ALL of its ancestors — not just its parent. Start unbounded, and use
    // Long so Integer.MIN_VALUE / MAX_VALUE node values still validate.
    return valid(root, Long.MIN_VALUE, Long.MAX_VALUE);
}

private static boolean valid(TreeNode node, long min, long max) {
    if (node == null) return true;           // empty subtree breaks nothing
    // The node itself must respect the bounds inherited from its ancestors.
    // (Checking only node.left.val < node.val is the CLASSIC BUG — a
    // grandchild can violate a grandparent while satisfying its parent.)
    if (node.val <= min || node.val >= max) return false;
    // Going left: values must stay BELOW me -> my value becomes the ceiling.
    // Going right: values must stay ABOVE me -> my value becomes the floor.
    return valid(node.left, min, node.val)
        && valid(node.right, node.val, max);
}

Complexity: time O(n), space O(h). Alternative worth naming: in-order traversal checking the sequence is strictly increasing — same costs, different style.

Pitfalls & interview tips

  • Null is a tree too. Make if (node == null) the first line of every recursion and half the edge cases (empty tree, missing child) vanish. NPEs in tree code almost always mean a missing base case.
  • Validate-BST with parent-only checks is the most famous wrong answer in interviewing. Say the word "ancestors" and carry bounds — and mention the Integer.MIN_VALUE edge that makes long bounds (or nullable Integer) necessary.
  • Don't say O(log n) space by reflex. Recursion stack is O(h); h = n for a skewed tree. "O(log n) if balanced, O(n) worst case" is the answer that sounds senior.
  • Depth in nodes vs height in edges differ by one and problems mix them freely (LC 104 counts nodes, LC 543 counts edges). Restate the definition to the interviewer before coding.
  • Level questions → freeze the queue size. If you're tagging each node with its depth in a wrapper class, you've missed the standard trick.
  • BST questions → say "in-order is sorted" early. Kth smallest, validation, successors, "two nodes swapped" — the sorted-sequence view usually is the solution.

Heaps & Priority Queues

The structure that always knows its smallest element: how a tree hides inside an array, why PriorityQueue is a min-heap, and the size-K inversion trick behind every top-K question.

Phase 4 · Weeks 6–7 Top-K questions are everywhere — from phone screens to system design

Before you start

Why this topic exists. A whole family of problems reduces to one sentence: "repeatedly give me the smallest (or largest) thing I'm currently tracking." Sorting answers it once for static data; a heap answers it forever, while items keep arriving, at O(log n) per operation. That's why "top K", "K-th largest", "merge K streams" and "running median" are all heap questions — and why schedulers, Dijkstra, and ranking pipelines all carry a priority queue at their core.

Prerequisites

  • 11 · Trees & BSTs — a heap is a binary tree (a very disciplined one); parent/child vocabulary carries over.
  • 06 · Sorting — we'll constantly compare "heap it" vs "sort it" vs quickselect.
  • 00 · Foundations — PriorityQueue row of the collections table.

What the interviewer is actually testing

  • The inversion trick: do you know that the k largest elements are kept with a min-heap of size k? Explaining why is the real test.
  • Cost fluency: offer/poll O(log n), peek O(1), building a heap from n items O(n) — and top-K is O(n log k), not O(n log n).
  • Java specifics: PriorityQueue is a min-heap; can you flip it with a Comparator without fumbling? Do you know its iterator is not sorted?
  • Pattern matching: spotting that "merge K sorted anything" or "always process the cheapest next" means heap before you've read the constraints twice.

How this builds on the previous topic

Page 11 gave you trees with an ordering invariant (BST: left < node < right). A heap keeps the tree but swaps in a much weaker invariant — parent ≤ children, nothing about siblings. Weaker means cheaper: no rotations, no node objects, no pointers at all. The tree is so perfectly shaped (complete, filled left to right) that it lives in a plain array with index arithmetic. You give up search (finding an arbitrary element is O(n)); you gain a permanently available minimum. That trade — less order, more speed — is the whole design.

Core concepts

1. The heap property, and a tree with no nodes

A binary min-heap is a complete binary tree where every parent ≤ its children. That single rule puts the global minimum at the root — and says nothing else. Left vs right child order? Unspecified. Where's the second-smallest? Somewhere in the top two levels. Heaps answer exactly one question fast, on purpose.

Because the tree is complete (every level full, last level packed left), it maps onto an array with zero waste and zero pointers — this is the detail interviewers love to probe: 

// The tree lives in the array. For the node at index i:
//   left child   2*i + 1
//   right child  2*i + 2
//   parent       (i - 1) / 2      (integer division)
// Example min-heap:            2
//   array: [2, 5, 3, 8, 7]    / \
//   index:  0  1  2  3  4    5   3
//                           / \
//                          8   7
No node objects means contiguous memory, cache-friendly walks, and allocation-free swaps — one concrete reason heaps outperform pointer-based trees for pure priority work, and why PriorityQueue is backed by an Object[], not a node class.

2. sift-up, sift-down: how order is repaired

  • offer(x): append x at the end (keeps the tree complete), then sift up — swap with the parent while smaller. Path length ≤ tree height → O(log n).
  • poll(): the answer is the root. Fill the hole with the last element (completeness again), then sift down — swap with the smaller child while bigger than either. O(log n).
  • peek(): read the root. O(1).

3. heapify: n items in O(n), not O(n log n)

Building a heap by n offers costs O(n log n). But new PriorityQueue<>(collection) does better: place everything in the array as-is, then sift down from the last parent back to the root — O(n) total. The intuition: half the nodes are leaves (sift distance 0), a quarter sift at most 1, an eighth at most 2… the sum Σ n/2^h · h converges to ~2n. One sentence of "why" here is worth a lot in an interview.

4. Java's PriorityQueue: a MIN-heap (and how to flip it)

PriorityQueue<Integer> min = new PriorityQueue<>();                    // smallest on top
PriorityQueue<Integer> max = new PriorityQueue<>(Comparator.reverseOrder()); // largest on top

// For your own shapes, compare on the priority field:
PriorityQueue<int[]> byCost = new PriorityQueue<>((a, b) -> Integer.compare(a[0], b[0]));

// Negation also works for ints (offer -x, negate on poll) but a Comparator
// says what you MEAN — prefer it. Two more facts worth stating out loud:
//   * pq.iterator() / toString() are NOT in priority order — only poll() is.
//   * remove(Object) is O(n): the heap can't FIND things, only rank them.

5. The pattern gallery

Top-K — the inversion trick, slowly. To keep the K largest items of a stream, use a min-heap capped at size K. Feels backwards, so walk it through: the heap is your top-K club, and its root is the weakest current member. A newcomer only needs to be compared against that weakest member — beat it and it's evicted (poll), lose and the newcomer is discarded. Every element pays O(log k), never O(log n): O(n log k) total, O(k) memory, and it works on infinite streams because you never store more than K items. Symmetrically: K smallest → max-heap of size K. K-th largest in a stream (LC 703) is this exact structure, peeking the root.

Two heaps — running median. Keep the smaller half in a max-heap and the larger half in a min-heap, sizes within one; the median is always a root away. Worked fully below.

K-way merge. The smallest unmerged element across K sorted lists must be one of the K current heads — keep exactly those in a min-heap, poll one, push its successor. Worked fully below; the same shape merges LSM-tree runs and external-sort chunks.

"Process cheapest next". Schedulers popping the highest-priority task, event loops popping the nearest timer, and Dijkstra popping the closest unsettled node (page 17) are all one loop: while (!pq.isEmpty()) process(pq.poll()), occasionally offering new work. Recognize the loop and you've recognized the structure.

Pattern templates (Java)

Top-K skeleton — memorize this five-liner

// K LARGEST elements of anything: min-heap capped at size k.
PriorityQueue<Integer> club = new PriorityQueue<>();   // root = weakest member
for (int x : stream) {
    club.offer(x);                    // candidate joins provisionally, O(log k)
    if (club.size() > k) club.poll(); // evict the weakest of the k+1
}
// club now holds the k largest; club.peek() is the K-TH largest.
// (K smallest: same code with Comparator.reverseOrder().)

Process-cheapest-next skeleton (scheduling, Dijkstra's shape)

PriorityQueue<Task> pq = new PriorityQueue<>(Comparator.comparingLong(t -> t.priority));
pq.addAll(initialTasks);              // heapify: O(n)
while (!pq.isEmpty()) {
    Task t = pq.poll();               // ALWAYS the cheapest remaining, O(log n)
    for (Task next : t.spawns())      // work may create more work...
        pq.offer(next);               // ...which slots in by priority
}

Where this shows up in real systems

OS & cluster schedulers

Linux's CFS keeps runnable tasks ordered by virtual runtime (red-black tree playing the heap role); Kubernetes' scheduler pops the highest-priority pending pod. "Ready work, ordered by priority, popped one at a time" is the process-cheapest-next loop verbatim.

Navigation: Dijkstra's frontier

Route engines keep the frontier of partially-explored roads in a min-heap keyed by distance-so-far — poll the closest, relax its edges, repeat. Page 17 builds this in full; the heap is the reason it's O(E log V) instead of O(V²).

Ranking & trending feeds

"Top 10 trending" over millions of counted items is the size-K min-heap — O(n log 10), constant memory, stream-friendly. Recommendation re-rankers (page 25) do the same: score candidates, keep the K best, never sort the world.

LSM compaction, external sort & timers

RocksDB/Cassandra compaction and external sort merge K sorted runs with a K-way heap merge — exactly LeetCode 23 at terabyte scale. Event loops (Netty, Node) and cron keep pending timers in a min-heap by deadline; Kafka's delayed-operation purgatory does the same for timeouts.

Worked solution: K Closest Points to Origin (LeetCode 973)

Problem in one line: given points on a plane, return the k nearest to (0,0).

Idea: "k closest" = "k smallest by distance", so the inversion trick uses a max-heap: its root is the farthest point currently kept — the only one a new point must beat. Compare on squared distance (monotonic with real distance) to stay in integer land, with long math to dodge overflow.

code/12-heaps/KClosestPoints.java

// LeetCode 973. Return the k points closest to the origin (any order).
public static int[][] kClosest(int[][] points, int k) {
    // THE INVERSION TRICK: to keep the k CLOSEST points, use a MAX-heap
    // ordered by distance. Its root is the WORST point we're keeping —
    // exactly the one to evict when someone better shows up.
    // (Squared distance avoids sqrt AND floating point entirely.)
    PriorityQueue<int[]> worstFirst = new PriorityQueue<>(
            (a, b) -> Long.compare(dist(b), dist(a)));  // reversed = max-heap

    for (int[] p : points) {
        worstFirst.offer(p);              // O(log k)
        if (worstFirst.size() > k) {
            worstFirst.poll();            // evict the farthest of the k+1
        }
    }
    // Whatever survived is the answer. Heap order is NOT sorted order —
    // we just drain it; the problem allows any order.
    int[][] result = new int[k][];
    for (int i = 0; i < k; i++) result[i] = worstFirst.poll();
    return result;
}

// Squared Euclidean distance from the origin, as long to dodge overflow.
private static long dist(int[] p) {
    return (long) p[0] * p[0] + (long) p[1] * p[1];
}

Complexity: time O(n log k), space O(k). Mention quickselect for O(n) average if all points fit in memory and mutation is allowed.

Worked solution: Top K Frequent Elements (LeetCode 347)

Problem in one line: return the k values that occur most often in an array.

Idea: two pages of this runbook composed: a frequency HashMap (page 01) followed by the top-K min-heap over the (value, count) pairs. The heap is capped at k, so we pay log k — not log n — per map entry.

code/12-heaps/TopKFrequent.java

// LeetCode 347. Return the k most frequent elements (heap version).
public static int[] topKFrequent(int[] nums, int k) {
    // Step 1: frequency map — value -> how many times it appears (page 01).
    Map<Integer, Integer> count = new HashMap<>();
    for (int x : nums) count.merge(x, 1, Integer::sum);

    // Step 2: keep a size-k MIN-heap of (value, freq), ordered by freq.
    // Counter-intuitive but key: for the k LARGEST frequencies we use a
    // MIN-heap, because its root is the weakest member of our top-k club —
    // the only one a newcomer ever needs to beat.
    PriorityQueue<int[]> minFreqFirst =
            new PriorityQueue<>((a, b) -> Integer.compare(a[1], b[1]));

    for (Map.Entry<Integer, Integer> e : count.entrySet()) {
        minFreqFirst.offer(new int[]{e.getKey(), e.getValue()}); // O(log k)
        if (minFreqFirst.size() > k) {
            minFreqFirst.poll();  // evict the least frequent of the k+1
        }
    }
    // The heap never grows past k+1: n entries * O(log k) = O(n log k),
    // beating the O(n log n) full sort when k is small.
    int[] result = new int[k];
    for (int i = 0; i < k; i++) result[i] = minFreqFirst.poll()[0];
    return result;
}

Complexity: time O(n log k), space O(n) for the map (heap is only O(k)). Follow-up worth volunteering: bucket-by-frequency achieves O(n).

Worked solution: Merge k Sorted Lists (LeetCode 23)

Problem in one line: merge k already-sorted linked lists into one sorted list.

Idea: at any moment, the globally smallest unmerged node must be the head of one of the k lists — every list's later nodes are ≥ its head. So keep exactly those k heads in a min-heap: poll the winner onto the output tail, then offer that list's next node as its new representative. A dummy head (page 08) removes the empty-result special case.

code/12-heaps/MergeKSortedLists.java

// LeetCode 23. Merge k sorted linked lists into one sorted list.
public static ListNode mergeKLists(ListNode[] lists) {
    // A min-heap holds ONE candidate per list: its current head.
    // The global smallest remaining element is always one of these k heads
    // (each list is sorted!), so poll() hands us the next output node.
    PriorityQueue<ListNode> heap =
            new PriorityQueue<>((a, b) -> Integer.compare(a.val, b.val));
    for (ListNode head : lists) {
        if (head != null) heap.offer(head);   // skip empty lists
    }

    ListNode dummy = new ListNode(0);   // dummy head: no "first node" special case
    ListNode tail = dummy;
    while (!heap.isEmpty()) {
        ListNode smallest = heap.poll();      // global minimum, O(log k)
        tail.next = smallest;                 // append to the merged result
        tail = smallest;
        if (smallest.next != null) {
            heap.offer(smallest.next);        // that list's next head replaces it
        }
    }
    return dummy.next;  // total: n nodes each pushed+popped once -> O(n log k)
}

Complexity: time O(n log k) for n total nodes, space O(k) for the heap (output reuses existing nodes). This is the merge loop inside every LSM-tree compaction.

Worked solution: Find Median from Data Stream (LeetCode 295)

Problem in one line: design a class that accepts numbers one at a time and can report the median of everything seen so far.

Idea: a median splits the data into two halves — so store the halves. A max-heap lo holds the smaller half (root = its largest), a min-heap hi holds the larger half (root = its smallest). Keep sizes within one of each other and the median is read off the roots in O(1). Each insert routes by comparing with lo's root, then one rebalancing move at most restores the size invariant.

code/12-heaps/MedianFinder.java

// LeetCode 295. Support addNum(int) and findMedian() on a growing stream.
//
// TWO-HEAP IDEA: split the numbers into halves around the median.
//   lo = a MAX-heap of the smaller half (its root = biggest small number)
//   hi = a MIN-heap of the larger half  (its root = smallest big number)
// The median only ever involves the two roots — both O(1) peeks.
private final PriorityQueue<Integer> lo =
        new PriorityQueue<>(Comparator.reverseOrder()); // max-heap
private final PriorityQueue<Integer> hi =
        new PriorityQueue<>();                          // min-heap (default)

public void addNum(int num) {
    // Route: anything <= biggest-of-the-small-half belongs in lo.
    if (lo.isEmpty() || num <= lo.peek()) lo.offer(num);
    else hi.offer(num);
    // Rebalance so sizes differ by at most 1 (lo may hold the extra).
    // Moving one root across the boundary keeps both halves valid.
    if (lo.size() > hi.size() + 1) hi.offer(lo.poll());
    else if (hi.size() > lo.size()) lo.offer(hi.poll());
}

public double findMedian() {
    // Odd count: the extra element sits on top of lo.
    if (lo.size() > hi.size()) return lo.peek();
    // Even count: median = average of the two middle values (the roots).
    return (lo.peek() + hi.peek()) / 2.0;
}

Complexity: addNum O(log n), findMedian O(1), space O(n). Follow-up ("sliding-window median") needs arbitrary deletions → swap the heaps for two TreeMap multisets, exactly the trade from the comparison table.

Pitfalls & interview tips

  • Blurting "max-heap!" for k-largest. A max-heap of all n works but is O(n log n) time and O(n) space; the size-k min-heap is the expected answer. Practice saying the club-and-weakest-member explanation until it's fluid.
  • PriorityQueue iteration is not sorted. for (int x : pq) and toString() walk the backing array, not priority order. Only repeated poll() drains in order.
  • Comparator overflow: (a, b) -> a - b breaks on values near Integer.MIN_VALUE. Use Integer.compare / Long.compare — free correctness points.
  • Don't mutate keys inside the heap. Changing an element's priority after offering silently corrupts ordering — there's no decrease-key in Java. The idiom is offer-a-fresh-entry and lazily skip stale ones when polled (Dijkstra, page 17).
  • Heap ≠ search structure. remove(Object) and contains are O(n). The moment a problem needs arbitrary deletion or floor/ceiling, upgrade to TreeMap and say why.
  • Know the numbers cold: peek O(1), offer/poll O(log n), heapify O(n), top-K O(n log k). And "heapsort = heapify + n polls = O(n log n), in place but not stable" ties page 06 in nicely.

Tries: The Prefix Tree

The data structure behind every autocomplete box you've ever typed into — and the one tool that answers "what starts with…?" in time proportional to the prefix, not the dictionary.

Phase 4 · Weeks 6–7 Niche but high-signal — the autocomplete classic

Before you start

Why this topic exists. You already have HashSet for "is this exact word stored?" — O(1), unbeatable. But ask it "how many stored words start with ca?" and it has nothing: hashing scrambles keys precisely so that similar keys land far apart. A trie does the opposite — it stores words by their shared structure, so every prefix question becomes a short walk from the root. Interviewers love tries because they test whether you can pick a structure for the query shape, not just for lookup speed.

Prerequisites

  • 01 · Arrays & Hashing — you should know why HashMap lookups are O(1) and what that trade costs.
  • 11 · Trees & BSTs — a trie is a tree; node-with-children navigation should feel routine.
  • 10 · Recursion & Backtracking — the wildcard search below is a small DFS.

Interview relevance: niche but high-signal — implementing a trie cleanly and knowing when it beats hashing (repeated prefix queries) is the test.

How this builds on the previous topic

Heaps (page 12) organized data by priority; tries organize it by shared structure. Both answer a query that a plain HashMap can't. The tree mechanics are ones you already own from page 11 — a trie is just a tree where the edges carry the information (one letter each) and a path from the root spells a prefix. Nothing new to learn about traversal; everything new is in what the shape lets you ask.

Core concepts

1. Why HashMap-of-words fails prefix queries

A HashSet<String> with a million words answers contains("cat") in O(1). But startsWith("ca") forces a full scan: hash codes of "ca", "cat" and "car" are unrelated by design. Your only HashSet option is to check all million entries — O(n·L).

A trie flips the storage: instead of one bucket per word, it builds one node per prefix. All words starting with "ca" literally hang below the same node, so the prefix query is: walk two edges, done — O(L) where L is the prefix length, no matter whether you stored ten words or ten million.

2. The node: children[26] vs a HashMap of children

A trie node needs exactly two things: where can I go next? and does a word end here? The "where next" part has two classic implementations, and interviewers like hearing you weigh them:

DesignChild lookupMemory per nodeUse when…
Node[26] childrenO(1), a raw array index26 references (~208 bytes) even if only 1 child existslowercase a–z guaranteed (most LeetCode problems)
HashMap<Character,Node>O(1) average, hash overheadproportional to actual childrenUnicode, mixed alphabets, sparse branching

The array wins interviews for a–z inputs: less code, no boxing, cache-friendly. Mention the map variant in one sentence to show you know the array is a special case.

The memory math behind the table's "~208 bytes", concretely: a Node[26] is itself an object — ~16 bytes of array header plus 26 references (104 bytes compressed, 208 uncompressed) — and the node object wrapping it adds ~16 more. Call it ~140–240 bytes per node, where each node represents a single character of some stored prefix; a String stores that character in one or two bytes. In a real dictionary most nodes have exactly one child, so you're paying 26 slots to use 1 — that emptiness, not the asymptotics, is why "why not always a trie?" is answered with memory. Two intermediate designs interviewers reward you for naming:

  • Ternary search tree (TST) — each node holds one character and three pointers: less-than, equal (descend a level), greater-than. It's a BST spliced into every trie level: lookup becomes O(L + log n)-ish instead of a flat O(L), but memory drops to three references per node regardless of alphabet size — the classic Bentley–Sedgewick middle ground between HashMap children and the raw array, and it handles Unicode for free.
  • Bitmap / array-mapped nodes — keep a 26-bit (or 64-bit) mask of which children exist plus a dense array of only those children. The child for letter c lives at index Integer.bitCount(mask & ((1 << c) - 1)) — one popcount instruction, O(1), and a node with two children pays for exactly two references. This is precisely the node layout of the HAMT behind Scala's and Clojure's immutable maps; recognizing the shared trick is real signal.

3. The three operations — all O(L)

insert, search, and startsWith are the same loop: start at the root, follow one child edge per character. Insert creates missing nodes; the two queries fail on missing nodes. The only subtlety is the ending: search additionally requires the final node's isWord flag — otherwise search("ca") would return true just because "cat" passes through. That flag is the single most common trie bug; the animation's rings above are exactly that flag.

4. Counts at nodes → ranked autocomplete

Nothing stops you storing more than a boolean at each node. Keep an int passCount incremented on every insert that walks through, and you can answer "how many words start with this prefix?" in O(L). Keep a small top-K list of popular completions at each node and you've built the heart of a search-box typeahead — production systems precompute exactly this (heaps from page 12 doing the ranking; the capstone, page 25, assembles the full pipeline).

5. Wildcard search: a DFS branch at every dot

If the query may contain '.' meaning "any one letter" (LeetCode 211 below), a plain walk no longer works — at a dot you must try all existing children. That turns the walk into a depth-first search: normal letters follow one branch, dots fan out into up to 26 recursive calls. Worst case for a query of all dots is O(26L), and saying that out loud is worth points.

The honest downside: memory. A trie can hold more node objects than there are characters in your dictionary would suggest — each node is an object with a 26-slot array. A HashSet of the same words is often several times smaller. Tries buy prefix-shaped queries with memory; if you only ever need exact-match lookup, the HashSet wins and you should say so. (Production systems compress tries into radix trees / DAWGs for this reason.)

6. Compressed tries: radix trees and Patricia

That warning's fix deserves its own mechanics. In a real dictionary, long runs of single-child nodes are the norm — "important" spends nine nodes spelling a chain with no branches. Path compression collapses every such chain into one edge labeled with the whole substring.

The result is a radix tree (Patricia trie): edges carry strings, not characters, and every internal node is a genuine decision point with at least two children. That invariant buys a hard bound worth quoting: a radix tree over n words has at most 2n − 1 nodes, no matter how long the words are — node count scales with the word count, not the character count, while the plain trie's scales with total characters. Search is the same O(L) walk; you just compare characters in runs against edge labels (store each label as offsets into the original word — two ints — and even the labels cost nothing extra). Insertion picks up one new case: a new word may diverge mid-edge, splitting that edge at the first mismatching character. Patricia proper is the binary-alphabet special case — one bit per decision — which is exactly the shape of an IP routing table: CIDR routes are bit-prefixes, and the longest-prefix-match walk in the routing card above runs over a compressed binary trie (Linux's LC-trie FIB is this structure, level-compressed further). The same design carries search-engine term dictionaries — Lucene keeps its on-disk terms index as a compressed prefix structure so "all terms starting with X" is one seek — and a DAWG goes one step beyond by also merging identical suffixes, turning the tree into a DAG: "walking", "talking", "stalking" share their tail once. For interviews: implement the plain trie, name the compressed ones and the 2n − 1 bound.

7. Aho–Corasick in one idea

Stream of Characters (worked below) restarts a fresh trie walk at every arriving character — fine when words are short, quadratic-ish when they aren't. Aho–Corasick removes the restart with one addition: give every trie node a failure link pointing to the node for the longest proper suffix of its prefix that is also a prefix of some pattern. If that sentence sounds familiar, it should — it is KMP's prefix function (page 14) generalized from a single pattern string to an entire trie of patterns. Matching then never rewinds the text: on each character, follow the child edge if it exists, otherwise chase failure links until one does (or you're back at the root). Each text character does amortized O(1) work — depth only increases by following a child, and every failure hop strictly decreases it, the same two-credit accounting as page 09's monotonic stack — so matching all patterns simultaneously costs O(text + matches), independent of how many patterns you loaded. The links themselves are built in one BFS from the root (a child's failure = its parent's failure's same-letter child), and word-end flags propagate along failure links so matches ending mid-walk aren't missed. That's the entire engine inside the intrusion-detection and spam-filter systems named in the cards above, matching 10⁵ signatures at wire speed. For interviews: implement the trie, explain "failure links = KMP on a trie" — nobody sane codes Aho–Corasick live.

Alphabets don't have to be letters. A 0/1 trie over 32-bit numbers turns "maximize XOR" into a greedy 32-level walk — worked in full as Maximum XOR of Two Numbers (LC 421) below.
Why you can't just "trie the suffixes". Inserting all n suffixes of one string gives a suffix trie that answers "is p a substring?" in O(|p|) — beautiful, except the suffixes of an n-character string total n(n+1)/2 characters, so the structure is Θ(n²) nodes: a 10⁵-character text explodes into billions of nodes before the first query. Path compression (§6) tames it to O(n) nodes — that's a suffix tree, a radix tree of suffixes — and suffix arrays get the same queries from a plain sorted array of suffix start indexes with no nodes at all. Both live on page 14; this note is the reason they exist.

Pattern templates (Java)

The trie skeleton — memorize this shape

class Trie {
    // The node: 26 outgoing edges + "does a word end here?".
    // Everything else in trie-land is a variation of this.
    static class Node {
        Node[] children = new Node[26];
        boolean isWord;
        int passCount;      // optional: #words whose prefix passes here
    }

    final Node root = new Node();   // root represents the empty prefix

    void insert(String word) {
        Node cur = root;
        for (char ch : word.toCharArray()) {
            int c = ch - 'a';                 // letter -> child slot 0..25
            if (cur.children[c] == null)      // create missing branch
                cur.children[c] = new Node();
            cur = cur.children[c];            // descend
            cur.passCount++;                  // this prefix gained a word
        }
        cur.isWord = true;                    // flag the final node
    }

    // Both queries are the same walk; only the ending differs.
    Node walk(String s) {                     // null if the path breaks
        Node cur = root;
        for (char ch : s.toCharArray()) {
            cur = cur.children[ch - 'a'];
            if (cur == null) return null;
        }
        return cur;
    }
    boolean search(String w)      { Node n = walk(w); return n != null && n.isWord; }
    boolean startsWith(String p)  { return walk(p) != null; }
    int countPrefix(String p)     { Node n = walk(p); return n == null ? 0 : n.passCount; }
}
Say the costs while you code. "Insert and search are O(L) time — L being the word length — and the trie holds O(total characters) nodes worst case, times 26 references each." That one sentence covers time, space, and the memory caveat in a single breath.

Decision cues:

  • Problem says "starts with", "prefix", "autocomplete", "replace by root" → trie.
  • Problem says "is this exact word present" and nothing else → HashSet; a trie is over-engineering.
  • Static dictionary, memory-tight, prefix queries → sorted array + binary search is a legitimate trie substitute (matches are adjacent after sorting) — mentioning it shows range.
  • Problem searches many words in a grid/stream at once → trie of the words, walk it while you scan (Word Search II).
  • Queries contain single-character wildcards → trie + DFS (a HashSet cannot do this at all).

Where this shows up in real systems

Search-box typeahead

Google-style suggestion services keep a trie (usually compressed) over popular queries, with each node caching its top-K completions by frequency so a keystroke is answered from one node, no subtree walk. Page 12's heaps compute those top-K lists offline; page 25 shows the full pipeline.

IP routing: longest-prefix match

A router choosing where to forward a packet matches its destination against thousands of CIDR routes and must pick the longest matching prefix. Routing tables are binary tries over address bits (Linux uses an LC-trie for its FIB) — the "walk and remember the last match" trick is Replace Words, inverted.

Spell-check & T9

Spell-checkers walk a dictionary trie in lock-step with an edit budget — branching like our wildcard DFS — to propose corrections. Old-phone T9 typing mapped digit sequences to trie paths; modern keyboard swipe prediction is the same idea with probabilities on the edges.

Tokenizer vocabularies

LLM tokenizers (BPE/WordPiece) must repeatedly find the longest vocabulary entry matching the upcoming text — a longest-prefix walk over a trie of ~50k tokens. Aho-Corasick, the multi-pattern matcher inside intrusion-detection systems and spam filters, is a trie with failure links (KMP's idea, next page, generalized).

easy Longest Common Prefix — LeetCode 14

The problem. Find the longest string that is a prefix of every string in the array ("" if none).

Example. ["flower","flow","flight"] → "fl". ["dog","racecar","car"] → "".

Variation 1 — vertical scan (the practical answer) · O(total chars)

public static String longestCommonPrefix(String[] strs) {
    // Column by column: compare character i of EVERY string before i+1.
    for (int i = 0; i < strs[0].length(); i++) {
        char c = strs[0].charAt(i);
        for (int j = 1; j < strs.length; j++) {
            // Stop at the first string that ends here or disagrees
            if (i == strs[j].length() || strs[j].charAt(i) != c)
                return strs[0].substring(0, i);
        }
    }
    return strs[0];                      // strs[0] itself prefixes everything
}

Variation 2 — trie view: the prefix is the unbranched spine

// Insert all words into a trie; the common prefix is the path from the
// root that (a) has exactly ONE child at each step and (b) contains no
// word-end before it branches. Overkill for one query — but exactly the
// right structure when prefix queries REPEAT (that's this page's point).

Takeaway. A one-shot question deserves the simple scan; the trie framing explains when a trie earns its memory — repeated prefix queries over a fixed dictionary, like the problems below. Mirror: code/13-tries/LongestCommonPrefix.java.

medium Implement Trie (Prefix Tree) — LeetCode 208

The problem. Build the structure itself: insert(word), search(word) (exact match), and startsWith(prefix).

Example. insert("apple"); search("apple") → true; search("app") → false (no word ends there); startsWith("app") → true; insert("app"); search("app") → true.

The canonical implementation · all ops O(L)

public class Trie {

    // One node per character position. children[c - 'a'] is the branch
    // for letter c; null means "no stored word goes that way".
    private static class Node {
        Node[] children = new Node[26]; // lowercase a-z only
        boolean isWord;                 // does a stored word END here?
    }

    private final Node root = new Node(); // root = the empty prefix ""

    /** Insert a word: walk letter by letter, creating nodes as needed. */
    public void insert(String word) {
        Node cur = root;
        for (int i = 0; i < word.length(); i++) {
            int c = word.charAt(i) - 'a';      // map 'a'..'z' -> 0..25
            if (cur.children[c] == null)       // no branch yet for this letter
                cur.children[c] = new Node();  // create it (the only allocation)
            cur = cur.children[c];             // descend one level
        }
        cur.isWord = true; // mark the node where the word ends
    }

    /** True only if the exact word was inserted (not just a prefix of one). */
    public boolean search(String word) {
        Node end = walk(word);
        return end != null && end.isWord; // must exist AND be a word-end
    }

    /** True if ANY inserted word starts with this prefix. */
    public boolean startsWith(String prefix) {
        return walk(prefix) != null; // reaching a node is enough — no flag check
    }

    // Shared walk: follow the letters of s down from the root.
    // Returns the node where s ends, or null if the path breaks.
    private Node walk(String s) {
        Node cur = root;
        for (int i = 0; i < s.length(); i++) {
            cur = cur.children[s.charAt(i) - 'a'];
            if (cur == null) return null; // path does not exist
        }
        return cur;
    }
}

Takeaway. All three operations are the same root-down walk — factoring it into walk() keeps the code short enough to write flawlessly under time pressure. Costs are O(L) in the query's length, independent of how many words are stored — the property a HashSet of whole words can't give you for prefixes. Mirror: code/13-tries/Trie.java.

medium Replace Words — LeetCode 648

The problem. Given dictionary "roots" and a sentence, replace every word with the shortest root that prefixes it (words with no root stay unchanged).

Example. roots ["cat","bat","rat"], sentence "the cattle was rattled by the battery" → "the cat was rat by the bat".

Trie of roots; stop at the first word-end · O(build + sentence)

public class ReplaceWords {

    private static class Node {
        Node[] children = new Node[26];
        boolean isWord;
    }

    public static String replaceWords(List<String> dictionary, String sentence) {
        // Step 1: build a trie of all roots.
        Node root = new Node();
        for (String w : dictionary) {
            Node cur = root;
            for (int i = 0; i < w.length(); i++) {
                int c = w.charAt(i) - 'a';
                if (cur.children[c] == null) cur.children[c] = new Node();
                cur = cur.children[c];
            }
            cur.isWord = true;
        }

        // Step 2: rewrite each word of the sentence independently.
        StringBuilder out = new StringBuilder();
        for (String word : sentence.split(" ")) {
            if (out.length() > 0) out.append(' ');
            out.append(shortestRoot(root, word));
        }
        return out.toString();
    }

    // Walk the trie along `word`; the FIRST word-end we meet is the
    // shortest root (that's why we check isWord before going deeper).
    private static String shortestRoot(Node root, String word) {
        Node cur = root;
        for (int i = 0; i < word.length(); i++) {
            int c = word.charAt(i) - 'a';
            if (cur.children[c] == null) return word; // no root prefixes this word
            cur = cur.children[c];
            if (cur.isWord) return word.substring(0, i + 1); // shortest root found
        }
        return word; // word ran out before any root ended
    }
}

Takeaway. The naive alternative — test every root against every word — costs O(#roots × sentence). The trie merges all roots into one structure so each sentence word pays a single O(L) walk, and checking isWord before descending guarantees the shortest match. Mirror: code/13-tries/ReplaceWords.java.

medium Design Add and Search Words — LeetCode 211

The problem. Support addWord(word) and search(word) where the query may contain '.', which matches exactly one arbitrary letter.

Example. add "bad", "dad", "mad"; search("pad") → false; search("bad") → true; search(".ad") → true; search("b..") → true.

Trie + wildcard DFS · O(L) without dots, branching with them

public class WordDictionary {

    private static class Node {
        Node[] children = new Node[26];
        boolean isWord;
    }

    private final Node root = new Node();

    /** Standard trie insert — identical to Implement Trie. */
    public void addWord(String word) {
        Node cur = root;
        for (int i = 0; i < word.length(); i++) {
            int c = word.charAt(i) - 'a';
            if (cur.children[c] == null) cur.children[c] = new Node();
            cur = cur.children[c];
        }
        cur.isWord = true;
    }

    /** Search where '.' is a wildcard for exactly one letter. */
    public boolean search(String word) {
        return dfs(word, 0, root);
    }

    // Can word[i..] be matched starting from `node`?
    private boolean dfs(String word, int i, Node node) {
        if (node == null) return false;             // walked off the trie -> fail
        if (i == word.length()) return node.isWord; // all chars consumed: need a word-end HERE
        char ch = word.charAt(i);
        if (ch == '.') {
            // Wildcard: try every existing child. Any success is a match.
            for (Node child : node.children)
                if (dfs(word, i + 1, child)) return true;
            return false; // no branch worked
        }
        // Normal letter: follow exactly one branch (null is caught above).
        return dfs(word, i + 1, node.children[ch - 'a']);
    }
}

Takeaway. A '.' converts the linear walk into a small backtracking search (page 10) over (query position, trie node). The two base cases carry all correctness: off-the-trie fails, fully-consumed succeeds only on a word-end. Mirror: code/13-tries/WordDictionary.java.

medium Map Sum Pairs — LeetCode 677

The problem. Design a map with insert(key, val) (re-inserting a key overwrites its value) and sum(prefix) = sum of values of all keys starting with that prefix.

Example. insert("apple", 3); sum("ap") → 3; insert("app", 2); sum("ap") → 5; insert("apple", 5) (overwrite); sum("ap") → 7.

Trie with per-node running sums + overwrite delta · both ops O(L)

public class MapSum {

    private static class Node {
        Node[] children = new Node[26];
        int subtreeSum;                     // sum of values of ALL keys below here
    }

    private final Node root = new Node();
    private final Map<String, Integer> values = new HashMap<>(); // current value per key

    public void insert(String key, int val) {
        // Overwrite semantics: only the DIFFERENCE propagates down the path.
        int delta = val - values.getOrDefault(key, 0);
        values.put(key, val);
        Node cur = root;
        cur.subtreeSum += delta;
        for (int i = 0; i < key.length(); i++) {
            int c = key.charAt(i) - 'a';
            if (cur.children[c] == null) cur.children[c] = new Node();
            cur = cur.children[c];
            cur.subtreeSum += delta;        // every prefix node absorbs the delta
        }
    }

    public int sum(String prefix) {
        Node cur = root;
        for (int i = 0; i < prefix.length(); i++) {
            cur = cur.children[prefix.charAt(i) - 'a'];
            if (cur == null) return 0;      // no key has this prefix
        }
        return cur.subtreeSum;              // precomputed — no subtree walk needed
    }
}

Takeaway. Storing an aggregate at every node turns "walk the whole subtree per query" into a single O(L) descent — the same precompute-on-the-path idea as prefix sums (page 04). The overwrite delta (val − old) is the detail that keeps the aggregates honest. Autocomplete rankers cache "top k completions" per node the same way. Mirror: code/13-tries/MapSum.java.

medium Search Suggestions System — LeetCode 1268

The problem. Given products and a search word, return — after each typed character — up to three lexicographically smallest products having the current input as a prefix. (This is autocomplete.)

Example. products ["mobile","mouse","moneypot","monitor","mousepad"], word "mouse" → after "m": [mobile, moneypot, monitor]; after "mo": same; after "mou"/"mous"/"mouse": [mouse, mousepad].

Variation 1 — sort + two-pointer window over the sorted list · O(n log n + L·n)

public static List<List<String>> suggestedProducts(String[] products, String searchWord) {
    Arrays.sort(products);                     // lexicographic order once
    List<List<String>> res = new ArrayList<>();
    int lo = 0, hi = products.length - 1;      // window of products still matching
    for (int i = 0; i < searchWord.length(); i++) {
        char c = searchWord.charAt(i);
        // Shrink the window: products must have character c at position i
        while (lo <= hi && (products[lo].length() <= i || products[lo].charAt(i) != c)) lo++;
        while (lo <= hi && (products[hi].length() <= i || products[hi].charAt(i) != c)) hi++;
        // (hi moves DOWN — see mirror; abbreviated here)
        List<String> top3 = new ArrayList<>();
        for (int k = lo; k <= Math.min(hi, lo + 2); k++) top3.add(products[k]);
        res.add(top3);
    }
    return res;
}

Variation 2 — trie with top-3 cached per node · O(build) + O(L) per keystroke

Takeaway. The sorted-window variation is the quickest to write; the trie variation is the one that scales to real autocomplete — precompute per-prefix answers once, serve each keystroke in O(1). medium Longest Word in Dictionary — LeetCode 720

The problem. The longest word buildable one character at a time, with every prefix along the way also in the dictionary. Ties go to the lexicographically smaller word.

Example. ["w","wo","wor","worl","world"] → "world". ["a","banana","app","appl","ap","apply","apple"] → "apple" (beats "apply").

Trie DFS through end-of-word nodes only · O(total letters)

static class Node {
    Node[] next = new Node[26];
    String word;                        // non-null marks end-of-word
}

private static String dfs(Node n, String best) {
    for (int i = 0; i < 26; i++) {      // ascending i = alphabetical order
        Node child = n.next[i];
        if (child == null || child.word == null)
            continue;                   // prefix chain broken -> dead branch
        if (child.word.length() > best.length())
            best = child.word;          // strictly longer wins; ties keep the
        best = dfs(child, best);        // earlier (smaller) find
    }
    return best;
}

Takeaway. "Every prefix must be a word" translates in trie language to "every node on the path is flagged" — so the DFS simply refuses to pass through unflagged nodes, and the constraint enforces itself structurally. Iterating children in index order makes the lexicographic tie-break automatic. When a problem's constraint is about prefixes, restating it as a trie-path property usually collapses the difficulty. Mirror: code/13-tries/LongestWordInDictionary.java.

medium Maximum XOR of Two Numbers in an Array — LeetCode 421

The problem. The maximum value of nums[i] XOR nums[j] over all pairs — in better than O(n²).

Example. [3,10,5,25,2,8] → 28 (5 ^ 25).

Bitwise trie + opposite-bit greedy · O(32n)

static class Node {
    Node[] next = new Node[2];                 // children: bit 0 / bit 1
}

public static int findMaximumXOR(int[] nums) {
    Node root = new Node();
    for (int x : nums) {                       // insert: 32 bits, MSB first
        Node n = root;
        for (int b = 31; b >= 0; b--) {
            int bit = (x >>> b) & 1;
            if (n.next[bit] == null) n.next[bit] = new Node();
            n = n.next[bit];
        }
    }
    int best = 0;
    for (int x : nums) {                       // query: chase opposite bits
        Node n = root;
        int xor = 0;
        for (int b = 31; b >= 0; b--) {
            int bit = (x >>> b) & 1;
            int want = 1 - bit;                // opposite bit -> XOR bit = 1
            if (n.next[want] != null) {
                xor |= 1 << b;
                n = n.next[want];
            } else {
                n = n.next[bit];               // forced to settle for a match
            }
        }
        best = Math.max(best, xor);
    }
    return best;
}

Takeaway. Tries aren't just for letters — any fixed-alphabet sequence works, and a 32-bit number IS a 32-character string over {0,1}. The greedy is sound because bits are positional: winning bit 31 outweighs losing all bits below, so prefer-the-opposite-bit per level is optimal. This bitwise-trie tool solves a family of XOR-maximization problems (LC 1707, 1938). Mirror (brute-force cross-checked): code/13-tries/MaximumXOR.java.

hard Word Search II — LeetCode 212

The problem. Given a letter grid and a list of words, return every word that can be traced through adjacent cells (each cell used once per word). Doing Word Search (page 10) once per word is too slow when there are 10⁴ words.

Example. board [[o,a,a,n],[e,t,a,e],[i,h,k,r],[i,f,l,v]], words ["oath","pea","eat","rain"] → ["eat","oath"].

One DFS, all words at once: walk the board and the trie together · O(m·n·4^L)

public class WordSearchII {

    private static class Node {
        Node[] children = new Node[26];
        String word;                        // non-null = a complete word ends here
    }

    public static List<String> findWords(char[][] board, String[] words) {
        // Build ONE trie of all target words — the shared prefix structure
        // is what lets a single DFS search for every word simultaneously.
        Node root = new Node();
        for (String w : words) {
            Node cur = root;
            for (char ch : w.toCharArray()) {
                int c = ch - 'a';
                if (cur.children[c] == null) cur.children[c] = new Node();
                cur = cur.children[c];
            }
            cur.word = w;                   // store the word AT its end node
        }

        List<String> res = new ArrayList<>();
        for (int r = 0; r < board.length; r++)
            for (int c = 0; c < board[0].length; c++)
                dfs(board, r, c, root, res);
        return res;
    }

    private static void dfs(char[][] b, int r, int c, Node node, List<String> res) {
        if (r < 0 || r >= b.length || c < 0 || c >= b[0].length) return;
        char ch = b[r][c];
        if (ch == '#') return;              // visited on this path
        Node next = node.children[ch - 'a'];
        if (next == null) return;           // NO word continues this way — prune the
                                            // whole subtree of paths right here
        if (next.word != null) {
            res.add(next.word);             // found a complete word
            next.word = null;               // de-duplicate: report each word once
        }
        b[r][c] = '#';                      // mark visited (page 10's trick)
        dfs(b, r + 1, c, next, res);
        dfs(b, r - 1, c, next, res);
        dfs(b, r, c + 1, next, res);
        dfs(b, r, c - 1, next, res);
        b[r][c] = ch;                       // restore on backtrack
    }
}

Takeaway. The trie replaces "loop over words" with "carry the trie node alongside the DFS": one board walk explores every word simultaneously, and a null child prunes all words sharing that dead prefix at once. Nulling next.word after reporting is the tidy de-duplication trick. This is the standard hard-level demonstration of why tries exist. Mirror: code/13-tries/WordSearchII.java.

hard Stream of Characters — LeetCode 1032

The problem. query(letter) appends a character to a growing stream and asks: does some dictionary word equal a suffix of the stream right now?

Example. words ["cd","f","kl"], stream "abcdefghijkl…" — true at 'd' (suffix "cd"), at 'f' ("f"), at 'l' ("kl"); false everywhere else.

Trie of REVERSED words; walk the stream tail backwards · query O(maxLen)

public StreamChecker(String[] words) {
    for (String w : words) {
        maxLen = Math.max(maxLen, w.length());
        Node n = root;
        for (int i = w.length() - 1; i >= 0; i--) {   // insert REVERSED
            int c = w.charAt(i) - 'a';
            if (n.next[c] == null) n.next[c] = new Node();
            n = n.next[c];
        }
        n.isWord = true;
    }
}

public boolean query(char letter) {
    stream.append(letter);
    Node n = root;
    // Newest character first: a suffix of the stream read backwards
    // is a prefix of some reversed word. Cap the walk at maxLen.
    for (int i = stream.length() - 1;
         i >= 0 && i >= stream.length() - maxLen; i--) {
        n = n.next[stream.charAt(i) - 'a'];
        if (n == null) return false;          // no word has this tail
        if (n.isWord) return true;            // a full reversed word matched
    }
    return false;
}

Takeaway. The reversal is the entire idea: suffix problems become prefix problems when you flip the reading direction, and prefixes are what tries are built for. The maxLen cap bounds each query regardless of stream length. This is the toy version of multi-pattern stream matching — production engines (Aho–Corasick, used in intrusion detection and spam filters) add failure links so the walk never restarts. Mirror: code/13-tries/StreamChecker.java.

Pitfalls

  • Forgetting the word-end flag — the #1 trie bug. Without isWord, search("ca") returns true because "cat" passes through. Test exactly this case out loud.
  • search vs startsWith differ by ONE condition. Write the shared walk helper first; both queries become one-liners and the interviewer sees clean design.
  • Index arithmetic: ch - 'a' assumes lowercase a–z. Confirm the alphabet with the interviewer before writing new Node[26]; digits or uppercase silently corrupt it with negative/out-of-range indexes.
  • Don't claim O(1) search. Trie operations are O(L) in the key length — so is hashing, actually (computing hashCode reads every char), which is a sharp observation to offer.
  • Own the memory story. If asked "why not always use a trie?", the answer is node overhead — 26 references per node — and the fix names: radix/Patricia tree (collapse single-child chains), DAWG (share suffixes).
  • In design questions, decorate nodes. Frequencies, top-K caches, or values at nodes turn the trie into typeahead, prefix-count, or MapSum — say "I'll hang extra data on the nodes" and you've generalized the tool.

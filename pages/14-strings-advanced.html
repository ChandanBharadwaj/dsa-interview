String Algorithms: Palindromes, Hashing & KMP

The three ideas that turn quadratic string problems linear: expand from centers, hash windows in O(1), and never re-read a character you've already matched.

Phase 5 · Week 8 Palindromes appear constantly — KMP is a senior-level signal

Before you start

Why this topic exists. Strings are just char arrays, so every earlier array pattern applies — yet string questions have their own recurring shapes: "longest palindrome", "find this pattern inside that text", "is it a repetition of itself?". Naive answers are O(n²) or O(n·m); this page gives you the three standard escapes. Palindrome expansion you'll use monthly; Rabin-Karp and KMP you may never type in production — but explaining why they're linear is exactly the kind of question that separates senior candidates.

Prerequisites

  • 02 · Two Pointers — palindrome checks are converging pointers.
  • 03 · Sliding Window — a rolling hash is a sliding window whose state is a number.
  • 01 · Arrays & Hashing — count arrays and hashing intuition.

Interview relevance: palindromes appear constantly; KMP and rolling hashes are the advanced-string differentiators.

How this builds on the previous topic

Tries (page 13) organized many words for prefix lookup; this page searches for patterns inside one long text. The two meet later: Aho-Corasick — a trie whose nodes carry KMP-style failure links — scans a text for thousands of patterns simultaneously. Master single-pattern KMP here and that whole family opens up.

Core concepts

1. Recap: immutability and StringBuilder

Java Strings are immutable: every "modification" allocates a new object. s += c in a loop copies the whole string built so far — O(n²) total, the classic trap from page 00. Inside algorithms use StringBuilder (amortized O(1) append, plus reverse() and deleteCharAt), or work on char[] / indexes and materialize a String once at the end. Also remember substring(i, j) copies O(j−i) characters — prefer passing indexes around.

2. Palindromes: expand around the center

Checking one palindrome is two pointers converging (page 02). Finding the longest palindromic substring flips the direction: instead of testing substrings from outside in, grow them from the center out — because every palindrome is symmetric around some center.

Key counting insight: a string of length n has exactly 2n−1 centers — n single characters (odd-length palindromes like aba) and n−1 gaps between characters (even-length ones like abba). Expand from each center while the ends match: O(n) centers × O(n) max expansion = O(n²) time, O(1) space — and unlike the brute force, each expansion step does O(1) work because it extends a known palindrome by one ring. (An O(n) algorithm exists — Manacher's — worth name-dropping, never expected in interviews.)

Since Manacher's is the "how would you make it linear?" follow-up, here is its one idea — the mirror argument — in interview-sized form. Maintain the palindrome with the rightmost reach seen so far, spanning [l, r] with center c. When you process a new center i inside that span, its mirror i' = l + (r − i) sits symmetrically on the other side of c — and because everything inside [l, r] is a mirror image around c, whatever palindrome radius i' achieved, i inherits — up to the edge: radius[i] ≥ min(radius[i'], r − i). So instead of expanding from scratch, you start from that inherited radius and only pay for expansion steps that push past r. That's the whole complexity argument: r never moves left, every paid expansion step moves r right, and r can cross at most n positions ⇒ all expansions together cost O(n), amortized exactly like the sliding-window proof on page 03. (The standard implementation first interleaves # separators — "abba" → "#a#b#b#a#" — so even-length palindromes become odd-length and one center array covers both.) Being able to state "inherit the mirror's radius, clipped at the border; only border-pushing steps cost" is worth more in an interview than reproducing the code.

3. Rolling hashes: compare windows as numbers

Treat a string as a number in base B: hash("cab") = c·B² + a·B + b (mod a large prime M). The power move is rolling: sliding the window one character right is O(1) — 

// slide window [i .. i+m-1] -> [i+1 .. i+m]
hash = (hash - text[i] * B^(m-1)) * B + text[i+m]     // all mod M

So after O(m) setup you can compare the pattern's hash against every text window in O(1) each — that's Rabin-Karp, expected O(n+m). The honesty clause: equal hashes do not prove equal strings (pigeonhole: 10⁹ hash values, far more possible strings), so verify char-by-char on a hash hit. Where hashing truly shines: many patterns of the same length (hash all into a HashSet, one scan finds them all) and 2D pattern matching — per-comparison, KMP can't do either.

How likely is a collision, really? Two facts calibrate your paranoia. First, the birthday bound: among k random hash values in a space of size M, expect the first collision around k ≈ √M. With M ≈ 10⁹ that's ~3×10⁴ — so an algorithm that implicitly compares all pairs of substrings (grouping by hash, counting distinct substrings) starts colliding at inputs of a few tens of thousands, even though a single pattern-vs-text comparison (n+m hash equalities, probability ~n/M of any false hit) is quite safe. Second, hashing with a fixed, published base and modulus is breakable on purpose: an adversary can construct inputs whose windows collide — competitive-programming judges do exactly this to fixed B=31, M=1e9+7 submissions, and it's the same threat model as the hash-DoS attacks on page 01. The defenses, in increasing strength: pick the base randomly at runtime, and/or run double hashing — two independent (base, modulus) pairs, comparing both — which squares the collision probability down to ~1/M² ≈ 10⁻¹⁸, effectively eliminating verification in practice. (One trap: using long overflow as an implicit mod-2⁶⁴ hash is known-broken — Thue–Morse-style inputs defeat it regardless of base.)

Why a random base works — the polynomial view. The hash of a window is a polynomial in B: h(B) = c₀·B^(m−1) + … + c_(m−1), coefficients = characters. Two different windows give two different polynomials, and their difference is a nonzero polynomial of degree < m — which has at most m−1 roots mod a prime M. A collision happens only if your randomly chosen B lands on one of those roots: probability ≤ (m−1)/M, no matter how adversarial the strings are. Rabin-Karp is "evaluate both strings at a random point and compare" — randomness in the algorithm replacing assumptions about the input, the same move as randomized quicksort pivots on page 06.

4. KMP: never re-read the text

Naive matching restarts after every mismatch: align pattern at i, compare, on failure try i+1 — worst case O(n·m) (think "aaaa…" vs "aaab"). KMP's observation: when a mismatch happens after matching k characters, you already know those k text characters — so the table can tell you, in advance, the next alignment worth trying. That table is the LPS array: for each pattern prefix, the length of the longest proper prefix that is also a suffix of it.

Hand-build it for our animation's pattern ABABC — do this once slowly and LPS stops being scary:

iprefix so farlongest proper prefix = suffixlps[i]

1ABnone (A ≠ B)0

3ABABAB2
4ABABCnone0

During the scan, a mismatch with j characters matched doesn't reset j to 0 — it falls back to lps[j-1], the longest chunk of the match that's still usable, and the text pointer i never moves backwards. Each character of text is consumed once; j can only fall as much as it previously rose ⇒ O(n+m) total. Watch it dodge a restart:

"j can only fall as much as it previously rose" deserves to be said as a real amortized proof, because interviewers push on it: why isn't the fall-back while loop O(m) per character, O(n·m) overall? Take j itself as the potential. Each of the n scan steps increments j at most once (the extend-on-match line), so across the whole run j gains at most n total. Every iteration of the fall-back loop strictly decreases j (to lps[j−1] < j), and j never goes negative — so the total number of fall-back iterations, summed over the entire scan, is bounded by the total gain: ≤ n. Any one character can trigger a long chain of fall-backs, but the chain spends savings that earlier characters deposited one at a time. That's the same bank-account argument as the monotonic stack on page 09 — and the LPS construction is the identical proof on m instead of n, giving O(n+m) overall, worst case, no randomness.

The deeper mental model — and the one that generalizes — is the automaton view. The states are "how many pattern characters am I currently matching" (0..m); each text character moves you between states; the LPS table is the failure function that says where to land when the desired transition doesn't exist. KMP is lazily simulating a DFA. You can make it eager: precompute the full transition table next[state][char] (O(m·Σ) space) and the scan becomes one branchless table lookup per character — O(1) unconditionally, no inner loop, which is why the automaton form is the streaming/network-appliance version. Two payoffs from this view: it explains why KMP suits streams you can't rewind (the state is a single integer — feed it chunks forever), and it is exactly what Aho-Corasick (page 13) builds when it grafts failure links onto a trie of many patterns: same failure-function idea, states = trie nodes instead of pattern prefixes.

Why the JDK doesn't bother with KMP. String.indexOf in current JDKs is a naive scan — no LPS table, no preprocessing — but it's a JIT-intrinsified naive scan: HotSpot replaces it with hand-tuned vectorized machine code that compares many characters per instruction. On real-world inputs (short patterns, text that mismatches within a character or two), expected cost is ~O(n) with a tiny constant, and O(n·m) requires adversarial periodic inputs like "aaaa…" vs "aaab" that production strings essentially never exhibit. KMP's guaranteed bound costs O(m) setup and pointer-chasing that beats SIMD only on those pathological inputs. The quotable summary: the library optimizes the constant on the common case; you reach for KMP when an adversary — or a guarantee — is in the room.

6. The Z-algorithm: KMP's friendlier twin

Z[i] = the length of the longest substring starting at position i that matches a prefix of the string. Where LPS answers "how much of my match survives a failure?", the Z-array answers the more direct question "how prefix-like is every position?" — and many people find it easier to reason about.

The linear-time construction reuses the exact window argument you just saw in Manacher's: maintain the rightmost match window [l, r) — the prefix-occurrence extending furthest right. For a new i inside the window, position i − l of the string has already computed the same local situation, so i inherits Z[i] ≥ min(Z[i − l], r − i), clipped at the border; only expansion steps pushing past r are paid, r only moves right ⇒ O(n) total. Same amortization, third appearance on this page — at this point it should feel like one idea wearing different costumes.

Z-array in ~10 lines — inherit, clip, extend

int[] zArray(String s) {
    int n = s.length();
    int[] z = new int[n];               // z[0] conventionally left 0 (or = n)
    for (int i = 1, l = 0, r = 0; i < n; i++) {
        if (i < r)                      // inside the window: inherit, clipped
            z[i] = Math.min(z[i - l], r - i);
        while (i + z[i] < n && s.charAt(z[i]) == s.charAt(i + z[i]))
            z[i]++;                     // paid steps: each pushes r rightward
        if (i + z[i] > r) { l = i; r = i + z[i]; }   // new rightmost window
    }
    return z;
}

Pattern matching becomes a one-liner of thought: build the Z-array of P + "#" + T (the sentinel # must appear in neither string, so no Z-value can spill across the boundary); every position with Z[i] == m is a match of P in T. O(n+m) time, and the Z-array and prefix function are information-equivalent — each is derivable from the other in O(n) — so anything KMP proves (periods, borders, Repeated Substring Pattern) has a Z-formulation. Interview calibration: KMP is the canonical name to invoke; the Z-algorithm is often the faster one to derive live without memorization.

7. Suffix arrays: every substring, one sorted index

Tries taught that a suffix trie holds every substring but costs O(n²) nodes (page 13). The practical fix is the suffix array: just the n suffixes' starting indexes, sorted lexicographically — O(n) integers, nothing else. The classic construction is doubling: sort suffixes by their first character, then by their first 2, 4, 8… — the trick being that a suffix's first 2k characters compare as the pair (rank of its first k, rank of the suffix k later), both already known from the previous round. Each of the log n rounds is a sort of n pairs ⇒ O(n log n) with radix sort per round, O(n log² n) with Arrays.sort — either is interview-respectable. (Linear-time constructions exist — SA-IS — strictly a name to know.)

The companion structure is the LCP array — the longest common prefix of each adjacent pair in the sorted order — built in O(n) by Kasai's algorithm, whose one insight is another amortization: walking suffixes in position order, dropping the first character of a suffix costs its LCP at most 1, so the running LCP value bounces within a total budget of O(n). Together they buy: find any pattern by binary-searching the sorted suffixes in O(m log n); longest repeated substring = max LCP entry; count of distinct substrings = n(n+1)/2 − ΣLCP. That last pair of one-liners is why suffix arrays appear in dedup, plagiarism detection, and bioinformatics indexes far more often than suffix trees, which win asymptotically but lose on memory constants. Beyond both sits the suffix automaton — the minimal DFA of all suffixes, O(n) states, buildable online — the strongest "name to know" in this family: say it exists, don't offer to build it.

Pattern templates (Java)

Expand-around-center — the palindrome workhorse

// For EACH of the 2n-1 centers, grow while the ends match.
// Odd-length centers: (i, i). Even-length centers: (i, i+1).
int expand(String s, int l, int r) {
    while (l >= 0 && r < s.length() && s.charAt(l) == s.charAt(r)) {
        l--;                       // extend one ring outward
        r++;
    }
    return r - l - 1;              // length of the palindrome found
}
// Caller pattern:
// for (int i = 0; i < n; i++) { use expand(s,i,i) and expand(s,i,i+1); }

LPS build + KMP scan — the shape to remember

// lps[i] = length of longest proper prefix of p[0..i] that is also its suffix
int[] lps = new int[m];
for (int i = 1, len = 0; i < m; i++) {
    while (len > 0 && p.charAt(i) != p.charAt(len))
        len = lps[len - 1];        // fall back to the next-best prefix
    if (p.charAt(i) == p.charAt(len)) len++;
    lps[i] = len;
}
// Scan: i walks the text ONCE; j tracks matched pattern chars
for (int i = 0, j = 0; i < n; i++) {
    while (j > 0 && text.charAt(i) != p.charAt(j))
        j = lps[j - 1];            // reuse the longest still-valid prefix
    if (text.charAt(i) == p.charAt(j)) j++;
    if (j == m) { /* match at i-m+1 */ j = lps[j - 1]; /* keep scanning */ }
}
Notice the twin loops. The LPS builder is KMP matching the pattern against itself — same fall-back while, same extend-on-match. Learn one loop, get the other free. And in a real interview it's perfectly acceptable to start with "naively this is O(n·m); if we need linear I'd use KMP — here's the idea…"

Decision cues:

  • "Longest/count palindromic …" → expand around center first; DP only if the interviewer pushes toward subsequences (19).
  • "Find pattern in text" and n·m fits the budget → say indexOf, then offer KMP/Rabin-Karp as the linear upgrades.
  • "Compare many substrings for equality" → rolling hash turns each comparison into O(1).
  • "Order doesn't matter" (anagram/permutation wording) → count arrays + window, not matching (03).
  • "Repeats / periods / shifts" → LPS: a string of length n with lps[n-1] = k repeats with period n−k. (That's Repeated Substring Pattern in one line.)

Where this shows up in real systems

grep, ripgrep & editors

Literal-string search in grep-family tools builds on linear-time matchers (Boyer-Moore variants — KMP's cousin that skips forward even faster) plus SIMD scanning; regex engines compile down to automata whose "failure transitions" are exactly the LPS idea generalized. Your IDE's find-in-files lives on this.

rsync & dedup pipelines

rsync finds unchanged blocks between two file versions by sliding a rolling checksum over the file — Rabin-Karp's exact trick — then confirming candidates with a strong hash. Content-defined chunking in backup/dedup systems (and plagiarism detectors like MOSS) fingerprint documents the same way.

Bioinformatics

DNA is a 4-letter alphabet text of billions of characters; finding gene subsequences uses Rabin-Karp-style k-mer hashing (BLAST seeds) and suffix structures. LeetCode's "Repeated DNA Sequences" is a toy version: hash every 10-mer, report duplicates.

WAFs & intrusion detection

Web application firewalls and IDS engines (Snort, Suricata) must match every packet against thousands of attack signatures at line rate — Aho-Corasick automata (trie + KMP failure links, page 13) make it one pass per packet regardless of rule count.

easy Repeated Substring Pattern — LeetCode 459

The problem. Can the string be built by repeating some substring of it two or more times?

Example. "abab" → true ("ab" × 2). "aba" → false. "abcabcabc" → true ("abc" × 3).

Variation 1 — try every divisor length · O(n · d(n))

public static boolean repeatedSubstringPatternBrute(String s) {
    int n = s.length();
    for (int len = 1; len <= n / 2; len++) {
        if (n % len != 0) continue;              // must tile the string exactly
        String unit = s.substring(0, len);
        boolean ok = true;
        for (int i = len; i < n && ok; i += len)
            ok = s.regionMatches(i, unit, 0, len);   // compare each tile
        if (ok) return true;
    }
    return false;
}

Variation 2 — the doubling trick · O(n) with a linear strStr

public static boolean repeatedSubstringPattern(String s) {
    // Slick fact: s is periodic  <=>  s appears inside (s+s) with both
    // endpoints strictly inside — i.e. at an index other than 0 and n.
    // Chopping the first and last character removes those trivial hits.
    String doubled = (s + s).substring(1, 2 * s.length() - 1);
    return doubled.contains(s);   // use KMP-strStr for a guaranteed O(n)
}

Takeaway. Why the trick works: if s = u repeated k≥2 times, then s+s contains s starting at |u| (an interior index). Conversely an interior occurrence forces a period that divides n. A third route: LPS from KMP — s is periodic iff n % (n − lps[n−1]) == 0 with lps[n−1] > 0; the LPS table literally encodes the period. Mirror: code/14-strings-advanced/RepeatedSubstringPattern.java.

easy Roman to Integer — LeetCode 13

The problem. Convert a Roman numeral (valid, 1–3999) to an integer, handling the six subtractive pairs (IV, IX, XL, XC, CD, CM).

Example. "LVIII" → 58. "MCMXCIV" → 1994.

Smaller-before-bigger counts negative · O(n)

public static int romanToInt(String s) {
    Map<Character, Integer> val = Map.of(
            'I', 1, 'V', 5, 'X', 10, 'L', 50,
            'C', 100, 'D', 500, 'M', 1000);
    int total = 0;
    for (int i = 0; i < s.length(); i++) {
        int v = val.get(s.charAt(i));
        // The one rule that unifies IV, IX, XL, XC, CD, CM:
        if (i + 1 < s.length() && v < val.get(s.charAt(i + 1)))
            total -= v;                // smaller before bigger: subtract
        else
            total += v;
    }
    return total;
}

Takeaway. Instead of six special-cased pairs, one generalization — "a symbol smaller than its right neighbor is subtractive" — makes the code trivial and unforgettable. Finding the rule that collapses the special cases is a transferable interview move; the reverse problem (LC 12) inverts it with a greedy over value-symbol pairs including the subtractive ones. Mirror: code/14-strings-advanced/RomanToInteger.java.

medium String Compression — LeetCode 443

The problem. Compress runs in place: ['a','a','b','b','c','c','c'] becomes ['a','2','b','2','c','3']; runs of length 1 get no count; counts ≥ 10 are written digit by digit. Return the new length; use O(1) extra space.

Example. [a,a,b,b,c,c,c] → 6, array starts a2b2c3. [a] → 1. [a,b,b,b,b,b,b,b,b,b,b,b,b] → 4 (a,b,1,2).

Read/write pointers over runs · O(n) time, O(1) space

public static int compress(char[] chars) {
    int write = 0;                            // where the next output char goes
    int read = 0;                             // start of the current run
    while (read < chars.length) {
        char c = chars[read];
        int runEnd = read;
        while (runEnd < chars.length && chars[runEnd] == c)
            runEnd++;                         // find the end of this run
        int runLen = runEnd - read;

        chars[write++] = c;                   // every run writes its character
        if (runLen > 1) {                     // runs of 1 write NO count
            // Counts >= 10 are written digit by digit, in order
            for (char d : String.valueOf(runLen).toCharArray())
                chars[write++] = d;
        }
        read = runEnd;                        // jump to the next run
    }
    return write;                             // compressed length
}

Takeaway. Run-length encoding with page 02's read/write pointers. It's safe to write into the array being read because the compressed form of a run is never longer than the run itself (a 1-char run writes 1 char; longer runs shrink or break even). This tiny algorithm is real: RLE is inside PNG filtering, fax encoding, and columnar database compression. Mirror: code/14-strings-advanced/StringCompression.java.

medium Longest Palindromic Substring — LeetCode 5

The problem. Return the longest contiguous palindrome inside s (n up to 1000; any one answer if several tie).

Example. "babad" → "bab" (or "aba"). "cbbd" → "bb".

Variation 1 — check every substring · O(n³), the baseline to beat

// For each of the O(n^2) substrings, an O(n) palindrome check.
// Fine for explaining WHAT we optimize: the checks repeat massive work —
// "bab" being a palindrome already tells you a lot about "xbabx".

Variation 2 — expand around every center (the expected answer) · O(n²), O(1) space

public static String longestPalindrome(String s) {
    if (s == null || s.isEmpty()) return "";
    int bestStart = 0, bestLen = 1; // any single char is a palindrome
    for (int center = 0; center < s.length(); center++) {
        // Odd-length palindromes: center is the single char at `center`.
        int odd = expand(s, center, center);
        // Even-length: center is the GAP between center and center+1.
        int even = expand(s, center, center + 1);
        int len = Math.max(odd, even);
        if (len > bestLen) {
            bestLen = len;
            // Recover the start index from the center and the length.
            bestStart = center - (len - 1) / 2;
        }
    }
    return s.substring(bestStart, bestStart + bestLen);
}

// Grow outward while the two ends match; return the palindrome length.
private static int expand(String s, int l, int r) {
    while (l >= 0 && r < s.length() && s.charAt(l) == s.charAt(r)) {
        l--; // step the left end outward
        r++; // step the right end outward
    }
    // The loop overshot by one on each side: length = (r-1)-(l+1)+1.
    return r - l - 1;
}

Takeaway. Every palindrome has a center — one character (odd length) or a gap (even length) — so 2n−1 expansions cover everything. Forgetting the even centers (finding "aba" but never "abba") is the classic bug. Manacher's algorithm does O(n) but is rarely expected — name it, don't derive it. Mirror: code/14-strings-advanced/LongestPalindromicSubstring.java.

medium Palindromic Substrings — LeetCode 647

The problem. Count all palindromic substrings (occurrences count separately even with equal text).

Example. "abc" → 3 ("a","b","c"). "aaa" → 6 ("a"×3, "aa"×2, "aaa").

Same expansion, counting instead of maximizing · O(n²), O(1) space

public static int countSubstrings(String s) {
    int count = 0;
    for (int center = 0; center < s.length(); center++) {
        count += countFrom(s, center, center);     // odd-length palindromes
        count += countFrom(s, center, center + 1); // even-length palindromes
    }
    return count;
}

// Each successful expansion step IS one more palindrome — count them all.
private static int countFrom(String s, int l, int r) {
    int count = 0;
    while (l >= 0 && r < s.length() && s.charAt(l) == s.charAt(r)) {
        count++;                                   // s[l..r] is a palindrome
        l--;
        r++;
    }
    return count;
}

Takeaway. Identical engine to LC 5, different bookkeeping: instead of tracking the longest, every step outward registers one palindrome. When two problems share an engine, learn the engine — the pairing here (maximize vs count) is a pattern you'll see again with DP tables. Mirror: code/14-strings-advanced/PalindromicSubstrings.java.

medium Find the Index of the First Occurrence (strStr) — LeetCode 28

The problem. Return the index of the first occurrence of needle in haystack, or −1. This is indexOf, implemented three ways.

Example. haystack="sadbutsad", needle="sad" → 0. haystack="leetcode", needle="leeto" → -1.

Variation 1 — naive sliding compare · O(n·m)

public static int strStrNaive(String haystack, String needle) {
    int n = haystack.length(), m = needle.length();
    for (int start = 0; start + m <= n; start++)
        if (haystack.regionMatches(start, needle, 0, m))  // O(m) compare
            return start;
    return -1;
}

The waste: after matching "abcab…" and failing, the naive scan restarts from scratch one position over, re-reading characters it has already seen. Both fixes below avoid re-reading the text.

Variation 2 — KMP: reuse matched prefixes · O(n+m) guaranteed

public static int strStr(String haystack, String needle) {
    int n = haystack.length(), m = needle.length();
    if (m == 0) return 0;
    int[] lps = buildLps(needle);
    int j = 0; // how many pattern chars are currently matched
    for (int i = 0; i < n; i++) {           // i NEVER decreases
        // On mismatch, fall back through the LPS chain: keep the
        // longest matched prefix that is still consistent.
        while (j > 0 && haystack.charAt(i) != needle.charAt(j))
            j = lps[j - 1];
        if (haystack.charAt(i) == needle.charAt(j)) j++; // extend match
        if (j == m) return i - m + 1;       // full match; return its start
    }
    return -1; // pattern never fully matched
}

// lps[i] = length of the longest proper prefix of needle[0..i] that
// is ALSO a suffix of it. Built with the same fall-back trick.
static int[] buildLps(String p) {
    int[] lps = new int[p.length()];
    int len = 0;                       // current matched prefix length
    for (int i = 1; i < p.length(); i++) {
        while (len > 0 && p.charAt(i) != p.charAt(len))
            len = lps[len - 1];        // shrink to the next-best prefix
        if (p.charAt(i) == p.charAt(len)) len++; // prefix grows by one
        lps[i] = len;
    }
    return lps;
}

Variation 3 — Rabin-Karp: rolling hash · expected O(n+m)

public static int strStrRabinKarp(String haystack, String needle) {
    int n = haystack.length(), m = needle.length();
    if (m == 0) return 0;
    if (m > n) return -1;

    final long MOD = 1_000_000_007L; // large prime keeps collisions rare
    final long BASE = 31;            // treat chars as digits in base 31

    // Highest power of BASE inside a window: BASE^(m-1) mod MOD.
    long pow = 1;
    for (int i = 0; i < m - 1; i++) pow = pow * BASE % MOD;

    // Hash of the needle and of the first text window.
    long target = 0, window = 0;
    for (int i = 0; i < m; i++) {
        target = (target * BASE + needle.charAt(i)) % MOD;
        window = (window * BASE + haystack.charAt(i)) % MOD;
    }

    for (int start = 0; ; start++) {
        // Hashes equal? Verify for real — collisions are possible.
        if (window == target
                && haystack.regionMatches(start, needle, 0, m))
            return start;
        if (start + m >= n) return -1; // no window left to slide into
        // Roll: remove the leftmost char, shift, append the next char.
        long out = haystack.charAt(start) * pow % MOD;
        window = (window - out + MOD) % MOD;  // +MOD avoids going negative
        window = (window * BASE + haystack.charAt(start + m)) % MOD;
    }
}

Takeaway. Three philosophies: naive re-reads the text; KMP remembers structure about the pattern (the LPS table) so the text pointer never backs up; Rabin-Karp summarizes windows as numbers so comparisons are O(1) until a hash hit demands verification. KMP wins on guaranteed bounds; rolling hashes win on flexibility (multiple patterns at once, 2D grids, chunking in rsync). Mirrors: code/14-strings-advanced/StrStrKMP.java and StrStrRabinKarp.java.

medium String to Integer (atoi) — LeetCode 8

The problem. Parse an int the way atoi does: skip leading spaces, read one optional sign, read digits until a non-digit, clamp to [Integer.MIN_VALUE, Integer.MAX_VALUE]. Everything else yields 0.

Example. " -042" → −42. "1337c0d3" → 1337. "words and 987" → 0. "-91283472332" → −2147483648 (clamped).

State-by-state parse with early clamping · O(n)

public static int myAtoi(String s) {
    int i = 0, n = s.length();
    while (i < n && s.charAt(i) == ' ') i++;        // 1. leading spaces

    int sign = 1;
    if (i < n && (s.charAt(i) == '+' || s.charAt(i) == '-')) {
        sign = s.charAt(i) == '-' ? -1 : 1;         // 2. at most ONE sign
        i++;
    }

    long value = 0;                                  // long: safe headroom
    while (i < n && s.charAt(i) >= '0' && s.charAt(i) <= '9') {
        value = value * 10 + (s.charAt(i) - '0');
        // 3. clamp EARLY — a 50-digit input would overflow even long
        if (sign == 1 && value > Integer.MAX_VALUE) return Integer.MAX_VALUE;
        if (sign == -1 && -value < Integer.MIN_VALUE) return Integer.MIN_VALUE;
        i++;
    }
    return (int) (sign * value);
}

Takeaway. Zero algorithms, pure specification discipline — which is exactly why it's asked: the signal is whether you enumerate the edge cases ("+-12"? spaces after the sign? digits then letters?) before coding. Clamping inside the loop matters; waiting until the end lets a long-enough input overflow the long. This is a miniature of writing any protocol parser to spec. Mirror: code/14-strings-advanced/MyAtoi.java.

medium Count and Say — LeetCode 38

The problem. The sequence starts "1"; each term describes the previous aloud: "1" → "one 1" → "11" → "two 1s" → "21" → "one 2, one 1" → "1211"… Return the nth term.

Example. n=4 → "1211". n=6 → "312211".

Run-length encode the previous term, n−1 times · O(output)

public static String countAndSay(int n) {
    String curr = "1";
    for (int round = 2; round <= n; round++) {
        StringBuilder next = new StringBuilder();
        int i = 0;
        while (i < curr.length()) {
            char digit = curr.charAt(i);
            int runLen = 0;
            while (i < curr.length() && curr.charAt(i) == digit) {
                runLen++;                    // measure the run of equal digits
                i++;
            }
            next.append(runLen).append(digit);   // "say" it: count, then digit
        }
        curr = next.toString();
    }
    return curr;
}

Takeaway. The inner loop is run-length encoding — the same run-scanning two-pointer as String Compression (above), applied iteratively. RLE itself is real: fax machines, bitmap image formats, and column-store databases all compress runs this way. The run-scan (anchor, advance while equal, emit) is worth having as a reflex; it appears in dozens of "group consecutive" problems. Mirror: code/14-strings-advanced/CountAndSay.java.

medium Reverse Words in a String — LeetCode 151

The problem. Reverse the word order; collapse all extra whitespace (leading, trailing, between words) to single spaces.

Example. " hello world " → "world hello". "a good   example" → "example good a".

Carve words from the end · O(n), no split()

public static String reverseWords(String s) {
    StringBuilder sb = new StringBuilder();
    int i = s.length() - 1;
    while (i >= 0) {
        while (i >= 0 && s.charAt(i) == ' ') i--;   // skip space runs
        if (i < 0) break;
        int end = i;                                // word ends here
        while (i >= 0 && s.charAt(i) != ' ') i--;   // find its start
        if (sb.length() > 0) sb.append(' ');        // exactly one separator
        sb.append(s, i + 1, end + 1);
    }
    return sb.toString();
}

Takeaway. split("\\s+") + reverse + join is the one-liner — say it, then show the manual version, because the follow-up ("char array, in place, O(1) extra") has a beautiful answer: reverse the whole string, then reverse each word — the same double-reversal trick as Rotate Array. Scanning backwards makes the whitespace rules enforce themselves. Mirror: code/14-strings-advanced/ReverseWords.java.

hard Text Justification — LeetCode 68

The problem. Lay words into lines of exactly maxWidth: pack greedily, pad gaps with spaces as evenly as possible (extra spaces to the left gaps); the last line and lines with one word are left-justified and right-padded.

Example. ["This","is","an","example","of","text","justification."], width 16 → "This    is    an", "example  of text", "justification.  ".

Greedy fill, then distribute spaces div/mod · O(total chars)

public static List<String> fullJustify(String[] words, int maxWidth) {
    List<String> lines = new ArrayList<>();
    int i = 0;
    while (i < words.length) {
        // Phase 1: greedily count words that fit (each needs ≥ 1 space between)
        int j = i, lineChars = 0;
        while (j < words.length
                && lineChars + words[j].length() + (j - i) <= maxWidth) {
            lineChars += words[j].length();
            j++;
        }
        int wordCount = j - i;
        int spaces = maxWidth - lineChars;          // total spaces to place
        StringBuilder line = new StringBuilder();

        if (j == words.length || wordCount == 1) {
            // Last line / lone word: left-justify, pad the right edge.
            for (int k = i; k < j; k++) {
                line.append(words[k]);
                if (k < j - 1) line.append(' ');
            }
            while (line.length() < maxWidth) line.append(' ');
        } else {
            // Even distribution: base spaces per gap, remainder to the left.
            int gaps = wordCount - 1;
            int base = spaces / gaps, extra = spaces % gaps;
            for (int k = i; k < j; k++) {
                line.append(words[k]);
                if (k < j - 1) {
                    int pad = base + (k - i < extra ? 1 : 0);
                    for (int p = 0; p < pad; p++) line.append(' ');
                }
            }
        }
        lines.add(line.toString());
        i = j;
    }
    return lines;
}

Takeaway. Hard by bookkeeping, not by algorithm — the div/mod split (base = spaces/gaps, first spaces%gaps gaps get one extra) plus two special cases are the problem. It's a beloved "real work" screen because it mirrors writing a layout engine to spec. Fun fact: this greedy is what most terminals do; TeX's prettier paragraphs use DP over break points to minimize raggedness. Mirror: code/14-strings-advanced/TextJustification.java.

Pitfalls

  • Missing even-length centers — expanding only from (i, i) finds aba but never abba. Always run both (i, i) and (i, i+1).
  • "Proper" prefix means strictly shorter. lps of "aaaa" is [0,1,2,3], never 4 — the whole string doesn't count as its own prefix. Get this wrong and KMP loops forever.
  • Negative mods: in Java (a - b) % MOD can be negative — write ((a - b) % MOD + MOD) % MOD. This bug is invisible until a hash silently mismatches.
  • Hash arithmetic overflows int fast: B·hash exceeds 2³¹ after a couple of characters. Use long throughout and reduce mod M at every step, not once at the end.
  • Palindromic substring ≠ subsequence. Substring is contiguous (this page); subsequence allows gaps and is a DP problem (page 19). Check which one the problem means.
  • Don't oversell hashing: say "expected O(n+m), and I verify on hash equality since collisions exist."

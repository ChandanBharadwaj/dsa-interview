Graph Basics: BFS & DFS

Model anything as nodes and edges, then master the only two traversals you'll ever need — and learn why grids are secretly graphs.

Phase 6 · Weeks 9–10 Nearly every senior interview loop includes one graph question

Before you start

Why this topic exists. Graphs are the most general data model in the book: social networks, road maps, dependency trees, microservices, grids in games — all nodes and edges. Interviewers love them because a graph question tests modeling ("can you SEE the graph hiding in this story?") plus algorithms (BFS/DFS) plus bookkeeping (the visited set) in one problem. The good news: two traversals — BFS and DFS — solve the vast majority of graph interview questions.

Prerequisites

  • 10 · Recursion & Backtracking — DFS is recursion on a graph
  • 11 · Trees & BSTs — level-order traversal is BFS; a tree is just a graph with no cycles
  • 09 · Stacks & Queues — BFS runs on a queue, iterative DFS on a stack

What the interviewer is actually testing

  • Can you build the graph from the problem's raw input (edge list, grid, word list) before traversing it?
  • Do you place the visited check at the right moment (the #1 graph bug — details below)?
  • Do you know BFS gives shortest paths in unweighted graphs, and why DFS doesn't?
  • Can you state O(V + E) and explain it (each node and each edge touched once)?

How this builds on the previous topic

Trees gave you traversal instincts in a world without cycles. Graphs add exactly one complication — you can reach the same node twice — and exactly one tool fixes it: the visited set. Tree BFS/DFS + visited set = graph BFS/DFS. That's genuinely the whole leap.

Core concepts

1. Representing a graph (step 1 of every solution)

Problems hand you edge lists, grids, or prerequisite pairs — almost never a ready-made graph. Your first lines of code build an adjacency list: for each node, the list of its neighbors.

Building an adjacency list from an edge list — memorize this idiom

// n nodes (0..n-1), undirected edges like [[0,1],[1,2],[0,3]]
List<List<Integer>> adj = new ArrayList<>();
for (int i = 0; i < n; i++) adj.add(new ArrayList<>());
for (int[] e : edges) {
    adj.get(e[0]).add(e[1]);   // edge u -> v
    adj.get(e[1]).add(e[0]);   // undirected: also v -> u (drop this line if directed)
}

2. BFS — explore in expanding waves

Breadth-first search uses a queue: visit the start, then everything 1 edge away, then 2 edges away… Because it exhausts distance-k nodes before touching distance-k+1, the first time BFS reaches a node is via a shortest path (fewest edges). That single property answers every "minimum number of steps" question.

3. DFS — explore one path to the end, then backtrack

Depth-first search dives along one path as far as possible, then backtracks — naturally recursive (the call stack is the "stack"). It doesn't find shortest paths, but it's the cleanest tool for "does a path exist", connected components, and flood fill, and the backbone of cycle detection and topological sort (next page).

4. Grids are graphs

A 2D grid is an implicit graph: each cell is a node; its up/down/left/right neighbors are its edges. No adjacency list needed — compute neighbors with a direction array. "Number of Islands" is literally "count connected components"; "Rotting Oranges" is literally "multi-source BFS". Interviewers dress graphs up as grids constantly — undressing them is the skill.

5. Multi-source BFS

When spreading starts from many places at once (all rotten oranges, all ocean-border cells), seed the queue with every source before starting. BFS then computes distance-to-nearest-source for every node in one pass — no need to run BFS per source.

Pattern templates (Java)

BFS — queue + visited-before-enqueue (memorize)

public void bfs(List<List<Integer>> adj, int start) {
    boolean[] visited = new boolean[adj.size()];
    Queue<Integer> queue = new ArrayDeque<>();
    visited[start] = true;              // mark WHEN ENQUEUING, not when polling
    queue.offer(start);
    while (!queue.isEmpty()) {
        int layer = queue.size();       // freeze the layer ONLY if you need distances
        for (int k = 0; k < layer; k++) {
            int cur = queue.poll();
            for (int nb : adj.get(cur)) {
                if (visited[nb]) continue;   // already discovered — skip
                visited[nb] = true;          // claim it now…
                queue.offer(nb);             // …so no one enqueues it twice
            }
        }
        // one layer processed == one unit of distance from start
    }
}

DFS — recursive (visited set instead of a tree's implicit no-cycles)

public void dfs(List<List<Integer>> adj, int cur, boolean[] visited) {
    visited[cur] = true;                 // mark on ARRIVAL
    for (int nb : adj.get(cur))
        if (!visited[nb])
            dfs(adj, nb, visited);       // dive deeper; backtracking is the return
}
The #1 graph bug: visited timing in BFS. If you mark nodes visited when you poll them (instead of when you offer them), the same node can sit in the queue multiple times — on dense graphs that's exponential blowup and wrong distance counts. Rule: the moment a node enters the queue, it is claimed.

Where this shows up in real systems

Social networks

"People you may know" is BFS 2–3 hops out from you, ranked by mutual-friend counts. Degrees of separation, friend clusters, influence spread — all BFS/DFS on the friendship graph (capstone maps the full pipeline).

Web crawlers

Googlebot is a giant BFS: seed URLs in a queue, fetch, extract links, enqueue unseen ones — the visited set (already-crawled URLs) is exactly your interview visited set, at the scale of the entire web.

Microservices & infrastructure

Blast-radius analysis ("if this service dies, what breaks?") is reachability — DFS over the dependency graph. Network topology discovery, Kubernetes resource graphs, Terraform plans: all traversals.

Games & robotics

Enemy pathfinding on uniform-cost maps is grid BFS; fog-of-war reveal and paint-bucket tools are flood fill. Weighted terrain upgrades it to Dijkstra/A* — the exact ladder this runbook climbs next.

Worked solution: Number of Islands (DFS flood fill) — Medium

Problem. Grid of '1' (land) and '0' (water) — count islands (4-directionally connected land).

Idea. The grid IS a graph; islands are connected components. Scan every cell; each time you find un-visited land, that's a new island — flood-fill (DFS) sinks the whole component so it's never recounted.

public static int numIslands(char[][] grid) {
    int rows = grid.length, cols = grid[0].length;
    int islands = 0;
    for (int r = 0; r < rows; r++) {
        for (int c = 0; c < cols; c++) {
            if (grid[r][c] == '1') { // fresh land -> a brand-new component
                islands++;
                sink(grid, r, c);    // erase the whole island
            }
        }
    }
    return islands;
}

// DFS flood fill: mark this cell as water, then recurse in 4 directions.
// Mutating the grid doubles as our "visited" set (no extra structure).
private static void sink(char[][] grid, int r, int c) {
    if (r < 0 || r >= grid.length || c < 0 || c >= grid[0].length
            || grid[r][c] != '1') // off the map, water, or already sunk
        return;
    grid[r][c] = '0';             // mark visited BEFORE recursing
    sink(grid, r + 1, c);         // down
    sink(grid, r - 1, c);         // up
    sink(grid, r, c + 1);         // right
    sink(grid, r, c - 1);         // left
}

Complexity. Time O(rows·cols) — each cell visited a constant number of times. Space O(rows·cols) worst case for the recursion (one giant snake-shaped island). Mention that mutating the input is a choice — offer a separate boolean[][] visited if the interviewer objects.

Worked solution: Rotting Oranges (multi-source BFS) — Medium

Problem. Fresh (1) and rotten (2) oranges; each minute rot spreads to adjacent fresh oranges. Minutes until nothing fresh remains, or −1 if impossible.

Idea. Seed the queue with ALL initially-rotten oranges (they all rot outward simultaneously), then expand one BFS layer per minute — BFS layers ARE elapsed time.

public static int orangesRotting(int[][] grid) {
    int rows = grid.length, cols = grid[0].length;
    Queue<int[]> queue = new ArrayDeque<>();
    int fresh = 0;

    // Seed: every rotten orange enters the queue simultaneously.
    for (int r = 0; r < rows; r++)
        for (int c = 0; c < cols; c++) {
            if (grid[r][c] == 2) queue.offer(new int[]{r, c});
            else if (grid[r][c] == 1) fresh++;
        }
    if (fresh == 0) return 0; // nothing left to rot -> 0 minutes

    int[][] dirs = {{1,0},{-1,0},{0,1},{0,-1}};
    int minutes = 0;
    // One full LAYER of the queue = one minute of spreading.
    while (!queue.isEmpty() && fresh > 0) {
        int layerSize = queue.size(); // freeze this minute's frontier
        for (int k = 0; k < layerSize; k++) {
            int[] cell = queue.poll();
            for (int[] d : dirs) {
                int nr = cell[0] + d[0], nc = cell[1] + d[1];
                if (nr < 0 || nr >= rows || nc < 0 || nc >= cols
                        || grid[nr][nc] != 1) continue; // only fresh oranges rot
                grid[nr][nc] = 2;               // mark rotten NOW ("visited")
                fresh--;                        // one fewer fresh orange
                queue.offer(new int[]{nr, nc}); // it spreads next minute
            }
        }
        minutes++; // a whole wavefront finished = one minute passed
    }
    return fresh == 0 ? minutes : -1; // stranded fresh orange -> impossible
}

Complexity. Time O(rows·cols), space O(rows·cols) for the queue. The fresh > 0 loop guard avoids counting a final minute where nothing spreads.

Worked solution: Clone Graph (BFS + HashMap) — Medium

Problem. Deep-copy an undirected graph given one node reference.

Idea. The trap is cycles — blind recursion clones forever. Keep a HashMap original → clone that doubles as the visited set: clone a node the first time you see it, and always wire edges between clones.

static class Node {
    int val;
    List<Node> neighbors = new ArrayList<>();
    Node(int val) { this.val = val; }
}

public static Node cloneGraph(Node node) {
    if (node == null) return null;
    // Maps each original node -> its clone. Also our "visited" set.
    Map<Node, Node> clones = new HashMap<>();
    clones.put(node, new Node(node.val)); // clone the start node up front
    Queue<Node> queue = new ArrayDeque<>();
    queue.offer(node);

    while (!queue.isEmpty()) {
        Node cur = queue.poll();
        for (Node nb : cur.neighbors) {
            if (!clones.containsKey(nb)) {        // first time seeing nb
                clones.put(nb, new Node(nb.val)); // clone BEFORE enqueue
                queue.offer(nb);                  // explore it later
            }
            // Wire the CLONE edge: clone(cur) -> clone(nb).
            clones.get(cur).neighbors.add(clones.get(nb));
        }
    }
    return clones.get(node);
}

Complexity. Time O(V + E), space O(V) for the map + queue. The same map-as-visited trick powers "Copy List with Random Pointer" (page 08) — say so for pattern credit.

Pitfalls & interview tips

  • Visited on enqueue, not on poll (BFS). Repeat it like a mantra — it's the most common graph bug in interviews.
  • Build the graph first. Don't traverse edge lists directly; spend the O(E) to build an adjacency list and the rest of the code writes itself.
  • Deep recursion can crash. A 300×300 grid can recurse ~90,000 deep — beyond Java's default stack. Mention the iterative (explicit stack) rewrite; writing it is rarely required, knowing it is.
  • Directions array beats eight ifs. int[][] dirs = {{1,0},{-1,0},{0,1},{0,-1}} — less code, fewer typos, easy 8-direction upgrade.
  • Weighted edges ⇒ BFS is wrong. The moment steps have different costs, switch to Dijkstra — that's the opening insight of page 17.
  • Say the complexity as a formula: "O(V + E) — every node dequeued once, every edge relaxed once (twice if undirected)."

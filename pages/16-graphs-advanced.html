Graphs Advanced: Topological Sort & Union-Find

Ordering dependency graphs (Kahn's and DFS post-order), merging groups incrementally with union-find, and 2-coloring for bipartite checks.

Phase 6 · Weeks 9–10 Course Schedule & Accounts Merge are interview staples

Before you start

Why this topic exists. BFS and DFS (page 15) answer "can I reach X?" and "how far is X?". Real graph questions are often subtler: in what order can I process nodes when some depend on others (build systems, course prerequisites)? And which nodes belong to the same group when connections arrive one at a time (merging user accounts, detecting the edge that creates a cycle)? Topological sort answers the first; union-find answers the second — and union-find does it without ever traversing the graph, which is why it feels like a magic trick the first time.

Prerequisites

  • 15 · Graph Basics — adjacency lists, BFS with a queue, DFS, connected components.
  • 09 · Stacks & Queues — ArrayDeque as the queue behind Kahn's algorithm.
  • 10 · Recursion & Backtracking — the DFS variant of topo sort is a post-order recursion.

Interview relevance: Course Schedule and Accounts Merge are staples; topological sort and union-find are the two tools under test.

How this builds on the previous topic

Page 15 gave you the two traversal engines and the habit of building an adjacency list as step one. This page adds structure on top: Kahn's algorithm is literally BFS where the queue admission rule is "indegree hit zero" instead of "not yet visited", and union-find replaces whole traversals with two tiny array operations. Everything here feeds page 17 — Kruskal's minimum spanning tree is just sorted edges + the union-find you learn now.

Core concepts

1. Topological sort: lining up a dependency graph

A topological order of a directed graph lists the nodes so that every edge u→v has u appearing before v. It's "a valid order to take the courses" or "a valid build order for modules".

One condition makes or breaks it: the graph must be a DAG (directed acyclic graph). If A depends on B and B depends on A, no order can satisfy both — a topological order exists if and only if there is no directed cycle. This is why every topo-sort algorithm doubles as a cycle detector, and why interviewers love Course Schedule: one problem tests both skills.

Kahn's algorithm (BFS flavor) — the one to write in interviews

Think of how you'd actually plan courses: first take everything with no prerequisites, cross them off, see what that unblocks, repeat. Kahn's algorithm is exactly that, with an indegree array (how many unmet prerequisites each node has) and a queue of currently-takeable nodes. When it finishes, either every node was processed (valid order) or some nodes never reached indegree 0 — those sit on a cycle.

DFS post-order (the other flavor)

Alternative: run DFS and record each node after all its descendants are done (post-order); the reverse of that record is a topological order — a node is only stamped once everything it points to is already stamped. Cycle detection needs the three-color trick: seeing a gray node (still on the recursion stack) again means you followed a back edge into your own ancestry — a directed cycle. Both variants are O(V+E); Kahn's is usually easier to get right under pressure and hands you the order for free.

The two flavors are mirror images, and saying so cleanly is a senior signal: Kahn's peels nodes with no remaining incoming constraints off the front of the order, while DFS stamps a node only once everything downstream of it is finished — it discovers the order back-to-front, which is exactly why you reverse the finish sequence. Same theorem underneath, approached from opposite ends: a node may be emitted only when it is a source of the not-yet-emitted subgraph. Two upgrades interviewers actually ask for:

  • Lexicographically smallest valid order: swap Kahn's ArrayDeque for a PriorityQueue so the smallest currently-unblocked node is always emitted next. Cost rises to O(E + V log V) — the standard follow-up to Course Schedule II, and the same move Reconstruct Itinerary (worked below) applies to edges.
  • Return the cycle, not just "false": in the DFS variant, carry an explicit path stack; when an edge hits a gray node v, the cycle is the stack's suffix starting at v — pop and collect. In Kahn's, use the fact that after the queue drains, every leftover node still has an unmet edge from another leftover node: start at any leftover, repeatedly step to such a predecessor, and within V steps a node repeats — the walk between the two visits is your cycle.

2. Union-find (disjoint set union): groups without traversal

Union-find maintains a collection of disjoint sets under two operations: find(x) — which set is x in? — and union(a,b) — merge two sets. The whole structure is one int[] parent array: each node points at a parent, roots point at themselves, and a set is identified by its root. find walks up the chain; union links one root under the other.

Two small optimizations make it fast enough to feel like O(1):

  • Path compression: while walking up in find, re-point every visited node closer to the root, flattening the tree for all future calls.
  • Union by size (or rank): always hang the smaller tree under the bigger root, so trees stay shallow.

Together they make each operation amortized O(α(n)) — the inverse Ackermann function, which is ≤ 4 for any input that fits in this universe. Say "effectively constant, formally inverse Ackermann" and move on.

Those two optimizations deserve their actual arguments — this is precisely the depth a senior loop probes for. First, why union by size caps tree height at log₂ n: a node's depth grows only when its root is hung under another root, and the rule guarantees the winning tree is at least as large — so every time a node gets one level deeper, the size of the tree containing it at least doubles. Starting from size 1 and capped at n, it can double at most log₂ n times, so no node ever sits deeper than log₂ n. That alone makes find a worst-case O(log n) operation, before compression does anything.

Path compression attacks history rather than height: each find flattens the exact chain it just paid to walk, so an expensive chain can never be charged twice. On its own it yields amortized O(log n). The celebrated bounds are for the combination: Hopcroft and Ullman proved O(log* n) per operation — the iterated logarithm, "how many times must you take log₂ to get down to 1", which is 5 even for 265536 — and Tarjan sharpened it to Θ(α(n)) amortized and proved that's optimal for this structure. The interview move is to know the ladder — log n → log* n → α(n) — and quote it; nobody expects the potential-function proof live.

Look closely at the find you'll memorize in Template 3 below: parent[x] = parent[parent[x]] is not full compression — it's path halving, re-pointing every other node on the walk to its grandparent in a single pass. Full compression re-points every visited node all the way to the root, which needs recursion or a second pass. Path halving and its sibling path splitting (re-point every node, not every other one) both preserve the α(n) bound with one loop and no recursion — which is why practical implementations prefer them.

3. What union-find unlocks

  • Cycle detection in undirected graphs: process edges one by one; if union(a,b) reports the endpoints were already in the same set, this edge closes a cycle. (That is the entire solution to Redundant Connection, worked below.)
  • Counting components without traversal: start count = n, decrement on every successful union. No BFS, no visited set.
  • The accounts-merge grouping pattern: when items share an attribute, union them. For Accounts Merge: give each account an index, keep a Map<String,Integer> from email → first account index seen; on a repeat email, union the two account indices. Afterwards, group all emails by find(accountIndex) and sort each group. The same "union by shared key, then group by root" shape powers identity resolution and fraud-ring detection at real companies.

4. Bipartite check: 2-coloring with BFS

A graph is bipartite if its nodes split into two camps with every edge crossing between camps (think: match interns to teams, or "possible bipartition" problems phrased as "people who dislike each other"). Test it by painting: color a start node 1, all its neighbors −1, their neighbors 1 again… If you ever find an edge whose endpoints got the same color, the graph contains an odd-length cycle and is not bipartite.

Bipartite check — BFS 2-coloring, O(V+E)

public static boolean isBipartite(int[][] graph) {
    int n = graph.length;
    int[] color = new int[n];                    // 0 = uncolored, 1 / -1 = the two camps
    for (int start = 0; start < n; start++) {    // graph may be disconnected: try every start
        if (color[start] != 0) continue;         // already colored by an earlier BFS
        color[start] = 1;
        Deque<Integer> queue = new ArrayDeque<>();
        queue.offer(start);
        while (!queue.isEmpty()) {
            int u = queue.poll();
            for (int v : graph[u]) {
                if (color[v] == 0) {             // uncolored neighbor: paint the opposite camp
                    color[v] = -color[u];
                    queue.offer(v);
                } else if (color[v] == color[u]) {
                    return false;                // same camp across an edge -> odd cycle
                }
            }
        }
    }
    return true;
}

The theorem behind the painting — bipartite ⇔ no odd cycle — deserves both directions, because "why does this work?" is the natural follow-up. (⇒) If the graph is bipartite, every edge crosses camps, so walking around any cycle alternates camp, camp, camp… and needs an even number of steps to land back where it started: odd cycles are impossible. (⇐) If no odd cycle exists, color each node by the parity of its BFS distance from the start. A violating edge would join two nodes whose distances share a parity — but then their two BFS-tree paths plus that edge close a walk of odd total length, and any odd closed walk contains a genuine odd cycle (splice out repeated vertices; each removed loop has some parity, and an odd total can't be built from even pieces alone). Contradiction. So the BFS 2-coloring isn't a heuristic that happens to work — it is the constructive proof, and the edge it fails on is your witness to an odd cycle.

Edges arriving over time? Bipartite meets union-find. BFS needs the whole graph up front; the DSU formulation doesn't. Allocate 2n slots — node u and a shadow u+n meaning "the camp opposite u". For each edge (u,v): if find(u) == find(v) the two endpoints are provably same-camp — an odd cycle just closed, not bipartite; otherwise union(u, v+n) and union(v, u+n). Each edge costs O(α(n)), and the check works mid-stream — the same incremental superpower union-find brings everywhere else. (The space-tighter variant, DSU with parity, stores each node's camp-flip relative to its root and updates it during path compression — name it, implement the 2n version.)

5. Bridges & articulation points (concept only)

A bridge is an edge whose removal disconnects the graph; an articulation point is such a node. Tarjan's low-link DFS finds them in O(V+E) by tracking, for each node, the earliest-discovered ancestor reachable from its subtree. They appear in interviews rarely (LeetCode's Critical Connections in a Network) — know the vocabulary and that "one DFS with discovery times and low-links" solves it; implementing it live is almost never required outside specialist loops.

"One DFS with discovery times and low-links" deserves to be unpacked, because the mechanism is only three ideas. Stamp disc[u] — the time DFS first reaches u — and define low[u] as the smallest discovery time reachable from u's subtree using any number of tree edges plus at most one back edge. Computing it is two mins during the DFS: absorb low[child] after each recursive call, and absorb disc[v] whenever an edge lands on an already-visited non-parent v (a back edge — an escape route toward the ancestors). Then the conditions read off directly: tree edge u–v is a bridge iff low[v] > disc[u] — v's entire subtree has no escape route reaching u or above, so cutting the edge strands it. Non-root u is an articulation point iff some child v has low[v] >= disc[u] (note ≥, not >: escaping only to u itself still leaves the subtree dependent on u); the DFS root is special — it's an articulation point iff it has two or more DFS children.

Tarjan bridges (LC 1192 Critical Connections in a Network) · O(V+E)

int timer = 0;
int[] disc, low;                        // disc[u] == 0 means "not yet visited"

void dfs(int u, int parent, List<List<Integer>> adj, List<List<Integer>> bridges) {
    disc[u] = low[u] = ++timer;         // low starts as "my own discovery time"
    for (int v : adj.get(u)) {
        if (v == parent) continue;      // don't ride the tree edge straight back up
        if (disc[v] == 0) {             // tree edge: recurse, then absorb the child's low
            dfs(v, u, adj, bridges);
            low[u] = Math.min(low[u], low[v]);
            if (low[v] > disc[u])                // subtree of v has NO escape past u
                bridges.add(List.of(u, v));      // -> removing u-v disconnects it
        } else {
            low[u] = Math.min(low[u], disc[v]);  // back edge: an escape route upward
        }
    }
}

One caveat for graphs with parallel edges: skipping "the parent" by node id wrongly ignores a second u–v edge (which makes u–v un-cuttable). Track the incoming edge index instead of the parent node.

The same low-link idea aimed at directed graphs finds strongly connected components — maximal sets in which every node reaches every other. Tarjan's SCC keeps the current DFS path's nodes on an explicit stack; when a node finishes with low[u] == disc[u], u is the root of a component, and everything above it on the stack pops off as one SCC. Kosaraju's is the easier one to memorize: DFS once recording finish times, reverse every edge, then DFS again in decreasing finish order — each second-pass tree is exactly one SCC. Both are O(V+E); Kosaraju costs two passes, Tarjan one.

The condensation trick. Contract each SCC to a single super-node and the resulting graph — the condensation — is always a DAG (a cycle between two super-nodes would merge them into one SCC). That means everything from section 1 applies to any directed graph after one O(V+E) preprocessing pass: "condense, then topo-sort, then DP over the DAG" is the standard shape for hard directed-graph problems (2-SAT is the famous production example). Find Eventual Safe States, worked below, is secretly this: safe nodes are the ones whose condensation super-node reaches only terminal super-nodes.

Pattern templates (Java)

Template 1 — Kahn's topological sort (indegree + queue) with cycle detection

// Returns a topological order, or an empty list if the graph has a cycle.
public static List<Integer> topoSort(int n, int[][] edges) {   // edges: [from, to]
    // Step 1: adjacency list + indegree (how many unmet dependencies each node has)
    List<List<Integer>> adj = new ArrayList<>();
    for (int i = 0; i < n; i++) adj.add(new ArrayList<>());
    int[] indegree = new int[n];
    for (int[] e : edges) {
        adj.get(e[0]).add(e[1]);
        indegree[e[1]]++;
    }
    // Step 2: seed the queue with every node that depends on nothing
    Deque<Integer> queue = new ArrayDeque<>();
    for (int i = 0; i < n; i++)
        if (indegree[i] == 0) queue.offer(i);
    // Step 3: repeatedly emit a ready node and release its dependents
    List<Integer> order = new ArrayList<>();
    while (!queue.isEmpty()) {
        int u = queue.poll();
        order.add(u);
        for (int v : adj.get(u))
            if (--indegree[v] == 0) queue.offer(v);   // last dependency just cleared
    }
    // Step 4: nodes still > 0 indegree never became ready -> they sit on a cycle
    return order.size() == n ? order : List.of();
}

Template 2 — DFS post-order topo sort (three colors; gray-hit = cycle)

// state: 0 = white (untouched), 1 = gray (on the current recursion stack), 2 = black (finished)
// Returns false if a cycle is found. Popping `order` afterwards yields a topological order.
static boolean dfs(int u, List<List<Integer>> adj, int[] state, Deque<Integer> order) {
    state[u] = 1;                                  // entering: mark gray
    for (int v : adj.get(u)) {
        if (state[v] == 1) return false;           // edge into a gray node = back edge = CYCLE
        if (state[v] == 0 && !dfs(v, adj, state, order)) return false;
    }
    state[u] = 2;                                  // every descendant finished
    order.push(u);                                 // post-order push; the stack reverses it
    return true;
}

Template 3 — Union-find with path compression + union by size (memorize this)

class UnionFind {
    int[] parent;   // parent[x] = x's parent; a root points to itself
    int[] size;     // size[root] = number of nodes in that tree (for union by size)
    int count;      // how many disjoint sets remain

    UnionFind(int n) {
        parent = new int[n];
        size = new int[n];
        count = n;
        for (int i = 0; i < n; i++) { parent[i] = i; size[i] = 1; }
    }

    int find(int x) {
        while (parent[x] != x) {
            parent[x] = parent[parent[x]];  // path compression: skip to grandparent
            x = parent[x];
        }
        return x;
    }

    boolean union(int a, int b) {
        int ra = find(a), rb = find(b);
        if (ra == rb) return false;         // already together (this edge would close a cycle)
        if (size[ra] < size[rb]) { int t = ra; ra = rb; rb = t; }  // ra = bigger tree
        parent[rb] = ra;                    // smaller root hangs under bigger root
        size[ra] += size[rb];
        count--;                            // two sets became one
        return true;
    }

    boolean connected(int a, int b) { return find(a) == find(b); }
}
Make union return boolean. Returning "did a merge actually happen?" gives you cycle detection (false = redundant edge), component counting (count), and Kruskal's accept/skip decision (page 17) — all from the same five lines.

Decision cues — when the problem says…

  • "prerequisites", "build order", "compile before", "alien dictionary of word order" → topological sort.
  • "is it possible to finish…" on a directed dependency graph → cycle detection (Kahn count or gray-node DFS).
  • "merge accounts / groups / provinces", "number of connected components", "which edge creates the cycle" → union-find.
  • "split into two groups", "everyone dislikes someone in the other group" → bipartite 2-coloring.

Where this shows up in real systems

Build systems & package managers

Maven, Gradle, make, and npm install all topologically sort a dependency DAG to decide build/install order — and the dreaded "circular dependency detected" error is exactly Kahn's processed < n check firing. Gradle's task graph is a literal DAG you can print.

Workflow orchestration & spreadsheets

Airflow calls its unit of work a DAG and schedules tasks in topological order, releasing a task when its last upstream finishes — the indegree array, distributed. Excel does the same for formula cells, and Kubernetes operators resolve resource dependency order the same way.

Network partitions & image segmentation

Union-find answers "did the cluster split?" as links fail — merge on every healthy link, count roots. In vision, classic segmentation merges adjacent similar pixels with UF, growing regions in near-linear time. It is also the engine inside Kruskal's MST (page 17).

Identity resolution & fraud rings

Payment companies union accounts that share a card, device, or address; a fraud ring surfaces as one suspiciously large root. This is Accounts Merge at production scale — see the capstone (page 25) for the full pipeline.

medium Number of Provinces — LeetCode 547

The problem. An n×n symmetric matrix where isConnected[i][j] == 1 means cities i and j are directly connected. Count the provinces — groups of directly or transitively connected cities.

Example. [[1,1,0],[1,1,0],[0,0,1]] → 2 (cities 0+1, city 2). [[1,0,0],[0,1,0],[0,0,1]] → 3.

Variation 1 — DFS over components (page 15's method) · O(n²)

public static int findCircleNumDFS(int[][] isConnected) {
    int n = isConnected.length;
    boolean[] seen = new boolean[n];
    int provinces = 0;
    for (int i = 0; i < n; i++)
        if (!seen[i]) {                       // a new, uncounted province
            provinces++;
            dfs(isConnected, seen, i);        // absorb everything reachable
        }
    return provinces;
}
private static void dfs(int[][] m, boolean[] seen, int i) {
    seen[i] = true;
    for (int j = 0; j < m.length; j++)
        if (m[i][j] == 1 && !seen[j]) dfs(m, seen, j);
}

Variation 2 — union-find (no traversal at all) · O(n² · α(n))

public static int findCircleNum(int[][] isConnected) {
    int n = isConnected.length;
    UnionFind uf = new UnionFind(n);         // n cities start as n separate provinces
    for (int i = 0; i < n; i++)
        for (int j = i + 1; j < n; j++)      // upper triangle only: matrix is symmetric
            if (isConnected[i][j] == 1)
                uf.union(i, j);              // direct link -> same province; count-- inside
    return uf.count;                         // survivors = number of disjoint sets
}

Takeaway. Both are fine here; the decision rule generalizes: one-shot static graph → DFS is simplest; edges arriving over time with connectivity queries in between → union-find, because it absorbs each edge in near-O(1) without ever re-traversing. Mirror: code/16-graphs-advanced/NumberOfProvinces.java.

medium Course Schedule — LeetCode 207

The problem. numCourses courses; each prerequisite pair [a, b] means you must take b before a. Can every course be finished?

Example. 2 courses, [[1,0]] → true (0 then 1). 2 courses, [[1,0],[0,1]] → false (each requires the other — a cycle).

Kahn's algorithm: peel zero-indegree nodes · O(V+E)

public static boolean canFinish(int numCourses, int[][] prerequisites) {
    // Step 1: adjacency list + indegree. Pair [a, b] means edge b -> a (b unlocks a).
    List<List<Integer>> adj = new ArrayList<>();
    for (int i = 0; i < numCourses; i++) adj.add(new ArrayList<>());
    int[] indegree = new int[numCourses];
    for (int[] p : prerequisites) {
        adj.get(p[1]).add(p[0]);       // b -> a
        indegree[p[0]]++;              // a has one more unmet prerequisite
    }

    // Step 2: every course with zero prerequisites is takeable right now.
    Deque<Integer> queue = new ArrayDeque<>();
    for (int i = 0; i < numCourses; i++)
        if (indegree[i] == 0) queue.offer(i);

    // Step 3: take a ready course; that satisfies one prereq of each dependent.
    int taken = 0;
    while (!queue.isEmpty()) {
        int course = queue.poll();     // safe: all its prerequisites are done
        taken++;
        for (int next : adj.get(course))
            if (--indegree[next] == 0) // that was next's LAST unmet prerequisite
                queue.offer(next);     // now takeable
    }

    // Step 4: a never-taken course means its indegree never hit 0 -> it sits on
    // (or behind) a directed cycle, so no valid order exists.
    return taken == numCourses;
}

Takeaway. Topological sort's existence test and cycle detection are the same thing: everything gets taken ⇔ no cycle. Kahn's is iterative (no stack-depth worries) and the count check makes cycle detection free. The DFS variant (white/gray/black coloring) proves the same result post-order. Mirror: code/16-graphs-advanced/CourseSchedule.java.

medium Course Schedule II — LeetCode 210

The problem. Same setup — but return an actual valid ordering of the courses (any one), or an empty array if impossible.

Example. 4 courses, [[1,0],[2,0],[3,1],[3,2]] → [0,1,2,3] or [0,2,1,3].

Kahn's, recording the take-order · O(V+E)

public static int[] findOrder(int numCourses, int[][] prerequisites) {
    List<List<Integer>> adj = new ArrayList<>();
    for (int i = 0; i < numCourses; i++) adj.add(new ArrayList<>());
    int[] indegree = new int[numCourses];
    for (int[] p : prerequisites) {
        adj.get(p[1]).add(p[0]);
        indegree[p[0]]++;
    }

    Deque<Integer> queue = new ArrayDeque<>();
    for (int i = 0; i < numCourses; i++)
        if (indegree[i] == 0) queue.offer(i);

    int[] order = new int[numCourses];
    int taken = 0;
    while (!queue.isEmpty()) {
        int course = queue.poll();
        order[taken++] = course;           // THE only change from LC 207:
        for (int next : adj.get(course))   // record the order courses are taken
            if (--indegree[next] == 0)
                queue.offer(next);
    }
    return taken == numCourses ? order : new int[0];   // cycle -> no valid order
}

Takeaway. One added line turns the yes/no answer into the schedule itself — the take-order is a topological order. This is exactly how build systems (Maven, Gradle, make) and dependency installers decide execution order. Mirror: code/16-graphs-advanced/CourseScheduleII.java.

medium Redundant Connection — LeetCode 684

The problem. A tree on nodes 1..n plus exactly one extra edge. Return the edge whose removal restores a tree; if several qualify, the one latest in the input.

Example. [[1,2],[1,3],[2,3]] → [2,3]. [[1,2],[2,3],[3,4],[1,4],[1,5]] → [1,4].

Union-find: the first edge that closes a cycle · O(n·α(n))

public static int[] findRedundantConnection(int[][] edges) {
    UnionFind uf = new UnionFind(edges.length + 1);  // nodes are labeled 1..n; index 0 unused
    for (int[] e : edges) {
        // union returns false when both endpoints already share a root —
        // adding this edge would create the cycle, so it's the redundant one.
        if (!uf.union(e[0], e[1])) return e;
    }
    return new int[0];                               // unreachable for valid input
}

Takeaway. Union-find's boolean union IS an undirected cycle detector: "endpoints already connected" means this edge is extra. Processing in input order automatically satisfies the return-the-latest rule. Re-running DFS per edge would be O(n²); this is the "incremental edges → union-find" cue in its clearest form. Mirror: code/16-graphs-advanced/RedundantConnection.java.

medium Accounts Merge — LeetCode 721

The problem. Each account is [name, email, email, …]. Two accounts belong to the same person if they share any email (names can repeat across different people). Merge each person's accounts; output [name, sorted unique emails…] per person.

Example. John{a,b}, John{c}, John{b,d} → merged John{a,b,d} + separate John{c} (the shared email b links accounts 0 and 2).

Union accounts through shared emails, then group · O(total emails · α)

public static List<List<String>> accountsMerge(List<List<String>> accounts) {
    int n = accounts.size();
    UnionFind uf = new UnionFind(n);            // one set per ACCOUNT index
    Map<String, Integer> emailOwner = new HashMap<>();  // email -> first account seen

    // Pass 1: an email seen in two accounts unions those accounts.
    for (int i = 0; i < n; i++)
        for (int j = 1; j < accounts.get(i).size(); j++) {
            String email = accounts.get(i).get(j);
            Integer owner = emailOwner.putIfAbsent(email, i);
            if (owner != null) uf.union(owner, i);   // same email, same person
        }

    // Pass 2: bucket every email under its account's ROOT.
    Map<Integer, TreeSet<String>> groups = new HashMap<>();  // TreeSet: sorted + deduped
    for (int i = 0; i < n; i++) {
        int root = uf.find(i);
        groups.computeIfAbsent(root, k -> new TreeSet<>())
              .addAll(accounts.get(i).subList(1, accounts.get(i).size()));
    }

    // Pass 3: emit [name, emails...] per group (name from any member account).
    List<List<String>> res = new ArrayList<>();
    for (Map.Entry<Integer, TreeSet<String>> e : groups.entrySet()) {
        List<String> row = new ArrayList<>();
        row.add(accounts.get(e.getKey()).get(0));   // the person's name
        row.addAll(e.getValue());
        res.add(row);
    }
    return res;
}

Takeaway. The identity-resolution pattern: entities (accounts) union through shared attributes (emails), then a group-by-root pass collects each set. This is literally how fraud systems link accounts sharing a device or card, and how CRMs deduplicate customers (capstone). Mirror: code/16-graphs-advanced/AccountsMerge.java.

medium Is Graph Bipartite? — LeetCode 785

The problem. Can the graph's nodes be split into two groups so every edge crosses between the groups (no edge inside a group)?

Example. [[1,3],[0,2],[1,3],[0,2]] (a 4-cycle) → true (groups {0,2} and {1,3}). [[1,2,3],[0,2],[0,1,3],[0,2]] (contains a triangle) → false.

2-coloring by BFS · O(V+E)

public static boolean isBipartite(int[][] graph) {
    int n = graph.length;
    int[] color = new int[n];               // 0 = uncolored, 1 / -1 = the two groups
    Deque<Integer> queue = new ArrayDeque<>();

    for (int start = 0; start < n; start++) {   // graph may be disconnected
        if (color[start] != 0) continue;
        color[start] = 1;                   // arbitrary first color
        queue.offer(start);
        while (!queue.isEmpty()) {
            int u = queue.poll();
            for (int v : graph[u]) {
                if (color[v] == 0) {
                    color[v] = -color[u];   // neighbor takes the OPPOSITE color
                    queue.offer(v);
                } else if (color[v] == color[u]) {
                    return false;           // same-colored neighbors: odd cycle
                }
            }
        }
    }
    return true;
}

Takeaway. Bipartite ⇔ 2-colorable ⇔ no odd-length cycle — the BFS proves it constructively by propagating alternating colors and failing exactly when an edge joins same-colored nodes. Watch the disconnected-graph loop: single-component code passes examples and fails hidden tests. Real use: two-sided matching problems (jobs↔candidates) start with a bipartite check. Mirror: code/16-graphs-advanced/IsBipartite.java.

medium Find Eventual Safe States — LeetCode 802

The problem. In a directed graph, a node is safe if every possible walk from it ends at a terminal node (no outgoing edges) — equivalently, if it can never reach a cycle. Return all safe nodes, sorted.

Example. graph [[1,2],[2,3],[5],[0],[5],[],[]] → [2,4,5,6] (0, 1, 3 sit on or feed the 0→1→3→0 cycle).

Three-color DFS · O(V + E)

private static final int WHITE = 0, GRAY = 1, BLACK = 2;   // black = proven safe

private static boolean dfs(int[][] g, int u, int[] color) {
    if (color[u] != WHITE)
        return color[u] == BLACK;       // GRAY again = we're in a cycle
    color[u] = GRAY;                    // u joins the current path
    for (int v : g[u])
        if (!dfs(g, v, color))
            return false;               // some route reaches a cycle: unsafe
    color[u] = BLACK;                   // every exit proved safe — memoized
    return true;
}

Takeaway. The three-color scheme from the topological-sort section, promoted to the star of the show: gray = "on the current path" detects cycles, black doubles as a memoization mark so each node is fully processed once. Equivalent alternative worth naming: reverse all edges and run Kahn's algorithm from the terminal nodes — safe nodes are the ones that ever reach in-degree 0. Mirror: code/16-graphs-advanced/EventualSafeStates.java.

medium Minimum Height Trees — LeetCode 310

The problem. A tree with n nodes: which roots minimize the tree's height? (There are at most two.)

Example. n=6, edges [[3,0],[3,1],[3,2],[3,4],[5,4]] → roots 3 and 4.

Peel leaves inward until ≤ 2 remain · O(n)

List<Integer> leaves = new ArrayList<>();
for (int i = 0; i < n; i++)
    if (adj.get(i).size() == 1) leaves.add(i);

int remaining = n;
while (remaining > 2) {                  // the center is at most 2 nodes
    remaining -= leaves.size();
    List<Integer> next = new ArrayList<>();
    for (int leaf : leaves) {
        int nb = adj.get(leaf).iterator().next();   // a leaf's only neighbor
        adj.get(nb).remove(leaf);        // peel the leaf off
        if (adj.get(nb).size() == 1)     // neighbor just became a leaf
            next.add(nb);
    }
    leaves = next;
}
return leaves;                           // the tree's center(s)

Takeaway. The min-height root is the tree's center — the node(s) minimizing the farthest distance — and peeling leaves layer-by-layer converges on it like burning a rope from all ends at once. Structurally this is Kahn's algorithm with "degree 1" in place of "in-degree 0". Why at most two survivors: each peel round shortens every longest path by one from both ends, so the middle is 1 node (odd diameter) or 2 (even). Mirror: code/16-graphs-advanced/MinHeightTrees.java.

medium Number of Operations to Make Network Connected — LeetCode 1319

The problem. n computers, given cables between pairs. One operation moves an existing cable elsewhere. Minimum operations to connect all computers, or −1.

Example. n=4, connections [[0,1],[0,2],[1,2]] → 1 (move the redundant 1–2 cable to computer 3). n=6 with 4 cables → −1 (5 needed).

Union-find: components and spare cables · O(E α(n))

public static int makeConnected(int n, int[][] connections) {
    if (connections.length < n - 1) return -1;   // a spanning tree needs n−1

    parent = new int[n];
    for (int i = 0; i < n; i++) parent[i] = i;
    int components = n;
    for (int[] c : connections)
        if (union(c[0], c[1]))
            components--;                        // useful cable: merged two
        // else: redundant cable — exactly the kind we can move
    return components - 1;                       // one move per extra component
}

Takeaway. Two counting facts close the problem before any cleverness: k components need exactly k−1 connecting moves, and total cables ≥ n−1 guarantees enough redundant ones exist (any component with a cycle has spares). Union-find's "did this edge merge or was it redundant?" return value is precisely the classifier needed — the same signal Redundant Connection (above) built on. Mirror: code/16-graphs-advanced/NetworkConnected.java.

hard Alien Dictionary — LeetCode 269

The problem. Words sorted by an unknown alphabet. Recover a valid letter ordering, or "" if the input is contradictory. (Premium on LeetCode; a canonical topological-sort hard.)

Example. ["wrt","wrf","er","ett","rftt"] → "wertf": wrt<wrf gives t<f; wrf<er gives w<e; er<ett gives r<t; ett<rftt gives e<r. ["abc","ab"] → "" (a word before its own prefix is impossible).

Extract pairwise constraints, then topo-sort the letters · O(total chars)

public static String alienOrder(String[] words) {
    // Graph over the letters that actually appear.
    Map<Character, Set<Character>> adj = new HashMap<>();
    Map<Character, Integer> indegree = new HashMap<>();
    for (String w : words)
        for (char c : w.toCharArray()) {
            adj.putIfAbsent(c, new HashSet<>());
            indegree.putIfAbsent(c, 0);
        }

    // Each ADJACENT word pair contributes at most ONE edge: the first
    // position where they differ says thisChar < thatChar.
    for (int i = 0; i + 1 < words.length; i++) {
        String a = words[i], b = words[i + 1];
        // Contradiction: "abc" before "ab" can't happen in any alphabet.
        if (a.length() > b.length() && a.startsWith(b)) return "";
        for (int k = 0; k < Math.min(a.length(), b.length()); k++) {
            char x = a.charAt(k), y = b.charAt(k);
            if (x != y) {
                if (adj.get(x).add(y))          // dedupe repeated constraints
                    indegree.merge(y, 1, Integer::sum);
                break;                          // ONLY the first difference matters
            }
        }
    }

    // Standard Kahn's over the letter graph.
    Deque<Character> queue = new ArrayDeque<>();
    for (var e : indegree.entrySet())
        if (e.getValue() == 0) queue.offer(e.getKey());
    StringBuilder order = new StringBuilder();
    while (!queue.isEmpty()) {
        char c = queue.poll();
        order.append(c);
        for (char next : adj.get(c))
            if (indegree.merge(next, -1, Integer::sum) == 0)
                queue.offer(next);
    }
    // Letters left un-taken sit on a cycle: contradictory input.
    return order.length() == indegree.size() ? order.toString() : "";
}

Takeaway. The graph isn't given — it's extracted: each adjacent word pair yields one "x before y" edge at the first differing position, then Kahn's does the rest. The two traps are both in extraction: the prefix contradiction check, and breaking after the first difference (later positions say nothing). Modeling-then-standard-algorithm is the anatomy of most graph hards. Mirror: code/16-graphs-advanced/AlienDictionary.java.

hard Reconstruct Itinerary — LeetCode 332

The problem. Use every flight ticket exactly once starting from "JFK"; among valid itineraries return the lexicographically smallest. A valid itinerary always exists.

Example. tickets JFK→SFO, JFK→ATL, SFO→ATL, ATL→JFK, ATL→SFO → JFK, ATL, JFK, SFO, ATL, SFO (plain greedy-smallest would strand itself in SFO first).

Hierholzer's algorithm: post-order Eulerian path · O(E log E)

public static List<String> findItinerary(List<List<String>> tickets) {
    Map<String, PriorityQueue<String>> graph = new HashMap<>();
    for (List<String> t : tickets)
        graph.computeIfAbsent(t.get(0), k -> new PriorityQueue<>())
             .add(t.get(1));               // heap = smallest destination first

    LinkedList<String> route = new LinkedList<>();
    dfs("JFK", graph, route);
    return route;
}

private static void dfs(String airport, Map<String, PriorityQueue<String>> graph,
                        LinkedList<String> route) {
    PriorityQueue<String> dests = graph.get(airport);
    while (dests != null && !dests.isEmpty())
        dfs(dests.poll(), graph, route);   // poll() consumes the ticket forever
    // POST-order: an airport is added only when it has no unused tickets
    // left — dead ends sink to the route's end, detours splice themselves in.
    route.addFirst(airport);
}

Takeaway. "Use every edge exactly once" is an Eulerian path — different beast from visiting every node. The magic is the post-order append: greedy DFS may strand itself early, but the stuck airport just gets logged last, and the unexplored side-loops naturally insert before it. The heap gives lexicographic order at no extra logic. Same algorithm reassembles DNA fragments (de Bruijn graphs) in bioinformatics. Mirror: code/16-graphs-advanced/ReconstructItinerary.java.

Pitfalls

  • Read the edge direction in Course Schedule. [a, b] means b comes first (edge b→a). Building the adjacency list backwards passes the easy tests and fails the rest — fix the direction before coding.
  • Don't use a plain visited-set for directed cycle detection. A node can be safely revisited via a different path (a "diamond" A→B→D, A→C→D has no cycle). You need the gray/black distinction — or just use Kahn's count.
  • Union roots, not the nodes themselves. parent[b] = a instead of parent[find(b)] = find(a) corrupts the structure. Always find first.
  • Watch 1-indexed nodes. LeetCode graph problems often label nodes 1..n — size the parent array n+1 (as in Redundant Connection) or subtract 1 everywhere, consistently.
  • Path compression alone is enough in interviews. Compression alone is near-O(log n) amortized; adding union by size gets α(n).
  • Recursive DFS depth: 10⁵ nodes in a chain overflows the Java call stack. Mention the iterative alternative (or that Kahn's is iterative already).

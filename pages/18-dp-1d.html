Dynamic Programming 1D: Recursion + Remembering

DP has a scary reputation it doesn't deserve. It is exactly the recursion you already know from page 10, plus a cache — nothing more. This page builds that idea from Fibonacci up to Coin Change and LIS, one gentle rung at a time.

Phase 7 · Weeks 11–12 The most feared topic — and a fixture of every senior loop

Before you start

Why this topic exists. Many optimization problems ("minimum cost", "number of ways", "longest such-and-such") have brute-force solutions that re-solve the same smaller problem millions of times. DP is the observation that if you write each subproblem's answer down once, exponential blow-ups collapse to linear or quadratic work. You've done this professionally for years — it's memoization, the same instinct as caching an expensive service call. Interviews love DP because it tests whether you can define the subproblem, not just code a loop.

Prerequisites

  • 10 · Recursion & Backtracking — DP is recursion with a cache; the recursion-tree picture is essential.
  • 00 · Foundations — you'll be comparing O(2ⁿ) vs O(n) constantly.

Interview relevance: the most feared topic — the 4-step recipe (state in words, recurrence, base cases, order) is what actually gets graded.

How this builds on the previous topic

Page 17 ended with DP on a DAG: min path sum works with a plain loop because restricted movement means no cycles — each cell's answer depends only on already-solved cells. That is the whole of DP: solve subproblems in an order where dependencies are always ready. Backtracking (page 10) explored a tree of choices and kept every path; DP notices that many branches of that tree are the same subproblem and refuses to solve any of them twice.

Core concepts

1. The moment DP clicks: Fibonacci's exploding tree

Plain recursive fib(n) = fib(n-1) + fib(n-2) is correct and hopeless: the call tree doubles at every level, O(2ⁿ). Look at the tree for fib(5) — count how often fib(2) is recomputed:

Static view: fib(5)'s call tree. Every amber node is a subproblem already solved somewhere to its left. Cache the answers and the tree collapses to the n+1 blue nodes — O(2ⁿ) becomes O(n).

There are only n+1 distinct subproblems (fib(0)…fib(n)), yet the naive tree performs ~2ⁿ calls. DP = keep the tree's logic, delete the duplicate work. The two properties that make this legal are worth naming in an interview: overlapping subproblems (the same call recurs) and optimal substructure (the best answer for n is built from best answers for smaller inputs).

2. Two directions, one idea: memoization vs tabulation

Take Climbing Stairs (LeetCode 70): you climb 1 or 2 steps at a time; how many distinct ways to reach step n? The last move was a 1-step from n−1 or a 2-step from n−2, so ways(n) = ways(n−1) + ways(n−2) — Fibonacci in a costume. Here is the same recurrence solved both ways:

Top-down — write the recursion you believe in, then cache it

public static int climbMemo(int n) {
    int[] memo = new int[n + 1];          // memo[i] = ways to reach step i (0 = not computed yet)
    return waysTo(n, memo);
}

private static int waysTo(int i, int[] memo) {
    if (i <= 2) return i;                 // base cases: 1 way to step 1, 2 ways to step 2
    if (memo[i] != 0) return memo[i];     // already solved this subproblem? reuse the answer
    // Last move was either a 1-step from i-1 or a 2-step from i-2 — add both worlds
    memo[i] = waysTo(i - 1, memo) + waysTo(i - 2, memo);
    return memo[i];
}

Bottom-up — same recurrence, but fill an array smallest-first with a loop

public static int climbTab(int n) {
    if (n <= 2) return n;
    int[] dp = new int[n + 1];            // dp[i] = number of distinct ways to reach step i
    dp[1] = 1;                            // base case: one way to stand on step 1
    dp[2] = 2;                            // base case: 1+1 or 2
    for (int i = 3; i <= n; i++) {
        dp[i] = dp[i - 1] + dp[i - 2];    // arrive via a 1-step or via a 2-step
    }
    return dp[n];
}
They are the same algorithm. Memoization solves subproblems in the order recursion discovers them (lazily, top-down); tabulation solves them in an order you choose (eagerly, bottom-up). Same states, same recurrence, same O(n). In an interview, derive top-down first — it mirrors how you think — then convert to a loop if the interviewer wants it or if recursion depth (n ≈ 10⁵ would overflow Java's stack) is a risk.

There is a picture worth keeping for the rest of your DP career: draw every distinct subproblem as a node and an edge from each state to the states its recurrence reads — you get a DAG of subproblems (acyclic by necessity: a cycle would mean a state depends on itself, and the recursion would never bottom out). In that picture, memoization is a depth-first search of the DAG starting at the state you asked about, caching answers in post-order; tabulation is a topological order of the same DAG, written as a loop. For 1D DPs the topological order is usually just "left to right", which is why the loop looks so innocent. Page 17's closing observation — min path sum works with a plain sweep because the grid is a DAG — was this same fact, met from the other direction.

The DAG view also settles "which direction should I code?" beyond taste. Top-down wins when the reachable states are sparse: the DFS visits only states actually reachable from the start, while a bottom-up loop grinds through every state whether or not anything needs it — think Coin Change with coins {100, 500} and a large amount, where most amounts are unreachable. Bottom-up wins when the state space is dense anyway, because it costs no call stack — memoized recursion over ~10⁵ states risks StackOverflowError on the JVM's default thread stack (page 9 has the frame arithmetic) — and because a predictable fill order is what unlocks the space optimizations of §4.

3. The 4-step recipe (use it on every DP problem, forever)

  1. State, in words. "dp[i] = <the answer to the question> restricted to the first i elements / amount i / prefix of length i." If you can't say it in one plain sentence, the design isn't done.
  2. Recurrence. Stand at state i and list the last choices that could have led here; combine their subanswers with max/min/sum.
  3. Base cases. The smallest states where the answer is known without any choice (empty prefix, amount 0). Get these wrong and everything downstream is wrong.
  4. Order & answer. Fill states so dependencies are ready before they're needed (usually left→right for 1D); say which cell holds the final answer.

Step 1 hides a fork that decides everything downstream, and interviewers probe it deliberately: is dp[i] the answer for the prefix 0..i (element i merely considered, not necessarily used), or the best answer ending exactly at i (element i mandatory)? House Robber uses the prefix form — dp[i] may or may not rob house i, and the final answer is the last cell. Kadane's maximum subarray and LIS are forced into the ending-at form: a subarray or an increasing run is only extendable if you know where it ends, so "best in the first i elements" doesn't decompose — the best subarray of a prefix tells you nothing about whether nums[i+1] can extend anything. The tell: if the object being optimized is contiguous or must physically end somewhere, define the state as "ending exactly at i" and take the max over all cells at the end; if elements can be freely skipped, the prefix state works and the answer sits in the last cell. The two forms have different base cases and different final answers — mixing them is the most common whiteboard DP bug, which is why the pitfalls below open with it.

4. Space optimization: look at what you actually read

Climbing Stairs reads only dp[i-1] and dp[i-2] — so the array is a luxury. Keep two variables and slide them. This "shrink the table to what the recurrence touches" move applies to most 1D DPs and to 2D tables row-by-row (page 19):

O(1) space — the array was never needed

public static int climb(int n) {
    if (n <= 2) return n;
    int prev2 = 1, prev1 = 2;             // ways to reach steps 1 and 2
    for (int i = 3; i <= n; i++) {
        int cur = prev1 + prev2;          // the recurrence, without the array
        prev2 = prev1;                    // slide the two-cell window forward
        prev1 = cur;
    }
    return prev1;
}

6. Why greedy fails — the coins that break it

With coins [1, 3, 4] and amount 6, the greedy habit ("always take the biggest coin") gives 4 + 1 + 1 = 3 coins. The optimum is 3 + 3 = 2 coins. Greedy commits to the 4 and can never un-commit; DP keeps every amount's best answer alive, so the 3+3 world survives. This tiny counterexample is the fastest way to justify DP to an interviewer — memorize it. (US-style coin systems happen to be greedy-safe, which is exactly why the habit feels right; page 21 covers when greedy is provably fine.)

7. Why the tails array works — the proof behind O(n log n) LIS

The Longest Increasing Subsequence walkthrough below (LeetCode 300) gives both solutions; this section adds the part a senior loop actually probes — why the tails trick is correct. Don't re-derive the code; internalize the two invariants.

Invariant 1: tails[k] is the smallest possible tail value of any increasing subsequence of length k+1 among the elements processed so far. The exchange argument for why "smallest tail" is the right thing to track: of two increasing subsequences of equal length, the one with the smaller tail can be extended by everything that extends the other, and possibly more — so remembering the smaller tail is never worse. Any strategy holding a larger tail can be exchanged, slot by slot, into this one without ever losing a unit of length. That is the same "my choice dominates yours" template page 21 uses to certify greedy algorithms — LIS-in-O(n log n) is a greedy wearing DP's clothes.

Invariant 2: tails is strictly increasing — which is what legalizes the binary search. Proof by contradiction: suppose tails[k] >= tails[k+1]. Take the length-(k+2) increasing subsequence whose tail is tails[k+1] and drop its last element: what remains is a length-(k+1) increasing subsequence whose tail is strictly less than tails[k+1] ≤ tails[k] — contradicting invariant 1's claim that tails[k] was already minimal. Both invariants survive every step: replacing the first tail ≥ x with x lowers one slot without disturbing the order, and appending happens only when x exceeds every tail (a genuinely longer subsequence now exists).

Two sentences worth saying unprompted: the array's length is the LIS length, but its contents are generally not an actual LIS (each slot's value may come from a different subsequence — reconstructing a real LIS needs extra bookkeeping); and lower-bound (first tail ≥ x) versus upper-bound (first tail > x) is exactly the strict-vs-non-decreasing switch. O(n log n) time, O(n) space; the search itself is page 7's lower-bound template verbatim.

Counting DPs: take the modulus at every add, not at the end. "Number of ways" answers grow exponentially, which is why constraints say "return the answer modulo 10⁹+7". The discipline: apply % MOD at every addition — dp[i] = (dp[i-1] + dp[i-2]) % MOD — never once at the end, because the overflow already happened in the intermediate sums. And when a counting recurrence subtracts (inclusion–exclusion), write (a - b + MOD) % MOD: Java's % keeps the sign of the dividend, so a plain (a - b) % MOD can hand you a negative "count". Page 23 covers the full modular toolbox.

Pattern templates (Java)

Template 1 — top-down memoization (derive this first in interviews)

// 1) STATE: solve(i) answers the question for input restricted to "i" (prefix, amount, index...)
// 2) memo: one slot per distinct state; a sentinel (-1) marks "not computed"
int[] memo;                                    // size = number of states, filled with -1

int solve(int i) {
    if (i is a base case) return baseAnswer;   // 3) BASE: known without any choice
    if (memo[i] != -1) return memo[i];         // cache hit: this subtree vanishes
    int best = WORST;                          // identity for max/min/sum
    for (each choice c available at state i) {
        // 4) RECURRENCE: last choice c reduces state i to a smaller state
        best = combine(best, cost(c) + solve(smaller(i, c)));
    }
    return memo[i] = best;                     // store before returning — the whole point
}

Template 2 — bottom-up tabulation (mechanical conversion of template 1)

int[] dp = new int[n + 1];                     // dp[i] = same meaning as solve(i) above
dp[base states] = base answers;                // base cases go in FIRST
for (int i = smallest non-base; i <= n; i++) { // ORDER: dependencies (smaller states) already filled
    for (each choice c at state i) {
        dp[i] = combine(dp[i], cost(c) + dp[smaller(i, c)]);
    }
}
return dp[n];                                  // ANSWER: state matching the original question
// Space bonus: if dp[i] only reads dp[i-1], dp[i-2] — keep 2 variables, drop the array.

Where this shows up in real systems

TeX / LaTeX line breaking

Knuth-Plass paragraph justification is a 1D DP: dp[i] = minimum "badness" of typesetting the first i words, choosing where the last line break goes. Greedy (break as late as possible — what most word processors do) produces visibly worse paragraphs; TeX's DP output is why math papers look good.

Retry & workflow planning

Choosing checkpoint/retry points in batch pipelines (Spark stage recomputation, workflow engines) is a min-cost DP over the step sequence: each state is "completed through step i", each choice is where to place the next checkpoint, balancing snapshot cost vs recompute-on-failure cost.

Reinforcement learning

Value iteration — the algorithm behind classic RL and MDP solvers — is literally the Bellman equation: V(s) = max over actions of reward + V(next state). Richard Bellman coined "dynamic programming" in the 1950s; the fancy name was chosen partly to sound impressive to funders.

Diff & spell-check tooling

git diff, spell correction and fuzzy search are the two-string cousins of this page's DPs (LCS / edit distance — page 19). The 1D skills — state definition, recurrence, base cases — transfer directly; the table just grows a second dimension.

easy Climbing Stairs — LeetCode 70 (all three versions)

Problem: climbing 1 or 2 steps at a time, count the distinct ways to reach step n.

Example. n=2 → 2 (1+1, 2). n=3 → 3 (1+1+1, 1+2, 2+1). n=5 → 8 — it is Fibonacci in disguise.

Idea: partition all ways by the last move: it was a 1-step (from n−1) or a 2-step (from n−2), and these sets don't overlap, so add them. That single sentence is the recurrence; the three versions in §2 and §4 above (memo → tab → two variables) are the full solution ladder. See code/18-dp-1d/ClimbingStairs.java for all three together with tests.

Complexity: time O(n); space O(n) memo/tab, O(1) optimized.

easy Min Cost Climbing Stairs — LeetCode 746

Problem: each step has a fee paid when you leave it; start at step 0 or 1; reach the top (one past the last step) as cheaply as possible.

Idea: same two-predecessor shape as Climbing Stairs, but combining with min instead of +. Define dp[i] = cheapest cost to stand on step i (folding the fee into arrival); the top is reachable from either of the last two steps. Only two previous values are read → two variables suffice.

MinCostClimbingStairs.java

// dp[i] = cheapest total cost to STAND ON step i (you pay cost[i] when you leave it,
// so we fold the payment into arriving: dp[i] = cost[i] + min(dp[i-1], dp[i-2])).
public static int minCostClimbingStairs(int[] cost) {
    int n = cost.length;
    // Only the last two dp values matter — keep two variables instead of an array.
    int prev2 = cost[0];                       // cheapest way to be on step 0
    int prev1 = cost[1];                       // cheapest way to be on step 1
    for (int i = 2; i < n; i++) {
        int cur = cost[i] + Math.min(prev1, prev2);  // arrive from i-1 or i-2, whichever is cheaper
        prev2 = prev1;
        prev1 = cur;
    }
    // The "top" is one past the last step; we can jump there from either of the last two steps.
    return Math.min(prev1, prev2);
}

Complexity: time O(n), space O(1).

medium House Robber — LeetCode 198

Problem: maximize loot from a row of houses where robbing two adjacent houses trips the alarm.

Example. [1,2,3,1] → 4 (houses 0 and 2). [2,7,9,3,1] → 12 (2+9+1).

Idea: dp[i] = best loot after considering houses 0..i (considered, not necessarily robbed — precision here prevents bugs). Rob house i → add nums[i] to dp[i−2]; skip → keep dp[i−1]. Take the max. The animation in §5 walks this exact array.

HouseRobber.java

// dp[i] = the most money you can have after CONSIDERING houses 0..i
// At each house you face one binary choice:
//   rob it  -> you must have skipped i-1, so you get dp[i-2] + nums[i]
//   skip it -> you keep whatever was best through i-1, i.e. dp[i-1]
public static int rob(int[] nums) {
    int n = nums.length;
    if (n == 1) return nums[0];
    int prev2 = nums[0];                        // best through house 0
    int prev1 = Math.max(nums[0], nums[1]);     // best through house 1
    for (int i = 2; i < n; i++) {
        int robIt  = prev2 + nums[i];           // take this house + best two houses back
        int skipIt = prev1;                     // pass, carry the previous best forward
        int cur = Math.max(robIt, skipIt);      // the DP choice
        prev2 = prev1;                          // slide the 2-value window
        prev1 = cur;
    }
    return prev1;                               // best after considering every house
}

Complexity: time O(n), space O(1). Note skipping two houses in a row can be optimal (try [2,1,1,2]) — the recurrence handles it automatically because dp[i−1] already contains that world.

medium Coin Change — LeetCode 322

Problem: fewest coins (each denomination reusable) summing to amount, or −1 if impossible.

Idea: dp[a] = fewest coins making amount a. The last coin used was some c, leaving the subproblem a−c: dp[a] = 1 + min over c of dp[a−c]. Fill amounts small→large so every dp[a−c] is ready. An unreachable amount keeps a sentinel value larger than any real answer.

CoinChange.java

// dp[a] = fewest coins that sum to amount a (unbounded: each coin reusable).
// Recurrence: dp[a] = 1 + min over coins c of dp[a - c].
public static int coinChange(int[] coins, int amount) {
    int[] dp = new int[amount + 1];
    Arrays.fill(dp, amount + 1);            // "infinity" sentinel: no way found yet
    dp[0] = 0;                              // base case: zero coins make amount 0
    for (int a = 1; a <= amount; a++) {     // solve every amount from small to large
        for (int c : coins) {
            if (c <= a && dp[a - c] + 1 < dp[a]) {
                // Use one coin c on top of the best solution for the remainder.
                dp[a] = dp[a - c] + 1;
            }
        }
    }
    // If dp[amount] still holds the sentinel, the amount is unreachable.
    return dp[amount] > amount ? -1 : dp[amount];
}

Complexity: time O(amount × coins), space O(amount). This is "unbounded knapsack" — coins may repeat because dp[a−c] can itself already use coin c. Compare the 0/1 version on page 19 where the loop direction flips.

medium Word Break — LeetCode 139

Problem: can string s be split entirely into words from a dictionary (words reusable)?

Idea: dp over prefixes: dp[i] = "the first i characters can be segmented". For prefix i, guess the last word: it spans s[j..i) for some cut j where dp[j] is already true. Bounding j by the longest dictionary word keeps the inner scan short.

WordBreak.java

// dp[i] = "can the first i characters s[0..i) be segmented into dictionary words?"
// dp[i] is true if SOME earlier true cut j exists where s[j..i) is a word.
public static boolean wordBreak(String s, List<String> wordDict) {
    Set<String> dict = new HashSet<>(wordDict);   // O(1) word lookups
    int maxLen = 0;                               // longest dictionary word — bounds the inner scan
    for (String w : dict) maxLen = Math.max(maxLen, w.length());

    int n = s.length();
    boolean[] dp = new boolean[n + 1];
    dp[0] = true;                                 // base case: the empty prefix is always "segmented"
    for (int i = 1; i <= n; i++) {                // for every prefix length i...
        // try every possible LAST word s[j..i); no word is longer than maxLen
        for (int j = Math.max(0, i - maxLen); j < i; j++) {
            if (dp[j] && dict.contains(s.substring(j, i))) {
                dp[i] = true;                     // prefix j was breakable + one more word reaches i
                break;                            // one witness is enough — stop scanning
            }
        }
    }
    return dp[n];                                 // can we segment the WHOLE string?
}

Complexity: time O(n × maxLen) substring checks (each up to O(maxLen) to hash) — commonly quoted O(n²) without the maxLen bound; space O(n) plus the dictionary set.

medium Longest Increasing Subsequence — LeetCode 300 (O(n²) and O(n log n))

Problem: length of the longest strictly increasing subsequence (not necessarily contiguous).

Idea 1 (O(n²)): dp[i] = LIS length ending exactly at index i. To extend, look at every earlier j with nums[j] < nums[i] and take the best dp[j] + 1. The answer is the max over all i — an LIS can end anywhere.

Idea 2 (O(n log n)): keep tails[k] = the smallest possible tail value of any increasing subsequence of length k+1. Read that twice — it's the subtle part. Smaller tails are strictly better (easier to extend later), and tails is always sorted, so each new element binary-searches the first tail ≥ it and improves that slot; if it beats every tail, the LIS just got longer. The array's length is the answer (its contents are generally NOT an actual LIS).

LongestIncreasingSubsequence.java

// ---------- O(n^2) DP: dp[i] = length of the longest increasing subsequence ENDING at i ----------
public static int lengthOfLIS(int[] nums) {
    int n = nums.length;
    int[] dp = new int[n];
    Arrays.fill(dp, 1);                       // every element alone is an LIS of length 1
    int best = 1;
    for (int i = 1; i < n; i++) {
        for (int j = 0; j < i; j++) {         // try appending nums[i] to every earlier LIS...
            if (nums[j] < nums[i]) {          // ...that it can legally extend
                dp[i] = Math.max(dp[i], dp[j] + 1);
            }
        }
        best = Math.max(best, dp[i]);         // the answer can end anywhere, so track the max
    }
    return best;
}

// ---------- O(n log n) "patience" version ----------
// tails[k] = the SMALLEST possible tail value of any increasing subsequence of length k+1.
// tails is always sorted, so each element binary-searches its landing spot.
public static int lengthOfLISFast(int[] nums) {
    int[] tails = new int[nums.length];
    int size = 0;                             // how many pile tails are in use
    for (int x : nums) {
        // find the FIRST tail >= x (lower bound) — that pile's tail can be improved to x
        int lo = 0, hi = size;
        while (lo < hi) {
            int mid = lo + (hi - lo) / 2;
            if (tails[mid] < x) lo = mid + 1;
            else hi = mid;
        }
        tails[lo] = x;                        // replace (improve) that tail, or...
        if (lo == size) size++;               // ...x beats every tail: a longer subsequence exists
    }
    return size;                              // number of piles = LIS length
}

Complexity: O(n²)/O(n) for the DP; O(n log n)/O(n) for tails. Using lower-bound (first ≥ x) makes the subsequence strictly increasing — equal elements replace rather than extend.

medium Maximum Product Subarray — LeetCode 152

The problem. Find the contiguous subarray with the largest product. Negatives make this tricky: two of them multiply into a positive, so the "worst" running value can suddenly become the best.

Example. [2,3,-2,4] → 6 ([2,3]). [-2,0,-1] → 0. [-2,3,-4] → 24 (the whole array — the two negatives cancel).

Track BOTH the max and min product ending here · O(n) time, O(1) space

public static int maxProduct(int[] nums) {
    // maxHere / minHere = the largest and smallest product of a subarray
    // ENDING at the current index. We need the min too: multiplied by a
    // negative, the smallest product becomes the largest.
    int maxHere = nums[0], minHere = nums[0], best = nums[0];
    for (int i = 1; i < nums.length; i++) {
        int x = nums[i];
        if (x < 0) {                      // a negative flips the roles:
            int t = maxHere;              // the old min is about to become
            maxHere = minHere;            // the new max, and vice versa
            minHere = t;
        }
        // Either extend the running product, or start fresh at x
        maxHere = Math.max(x, maxHere * x);
        minHere = Math.min(x, minHere * x);
        best = Math.max(best, maxHere);
    }
    return best;
}

Takeaway. Kadane's maximum-subarray idea (carry "best ending here") needs a twin here: because multiplication by a negative swaps extremes, you must carry the minimum as well. Carrying two running states instead of one is a mini state-machine — the training wheels for page 20. Zeros reset both states naturally via the "start fresh at x" branch. Mirror: code/18-dp-1d/MaxProductSubarray.java.

medium House Robber II — LeetCode 213

The problem. Same as House Robber, but the houses form a circle — the first and last house are adjacent, so robbing both trips the alarm.

Example. [2,3,2] → 3 (can't take both 2s — they're neighbors on the circle). [1,2,3,1] → 4. [5] → 5.

Break the circle into two lines · O(n) time, O(1) space

public static int rob(int[] nums) {
    int n = nums.length;
    if (n == 1) return nums[0];
    // The circle only adds ONE constraint: house 0 and house n-1 can't
    // both be robbed. So the answer is the better of two LINE problems:
    //   case A: house 0 allowed, house n-1 excluded  -> rob nums[0..n-2]
    //   case B: house 0 excluded, house n-1 allowed  -> rob nums[1..n-1]
    return Math.max(robLine(nums, 0, n - 2),
                    robLine(nums, 1, n - 1));
}

// Plain House Robber (LC 198) on nums[lo..hi] with two rolling variables.
private static int robLine(int[] nums, int lo, int hi) {
    int skip = 0, rob = 0;                 // best if we skip / rob the previous house
    for (int i = lo; i <= hi; i++) {
        int newRob = skip + nums[i];       // rob i: previous must be skipped
        skip = Math.max(skip, rob);        // skip i: carry the better past
        rob = newRob;
    }
    return Math.max(skip, rob);
}

Takeaway. Don't invent circular DP — reduce to the linear problem you already solved: any valid plan omits house 0 or house n−1, so solving both restricted lines and taking the max covers every possibility. Case analysis that removes a constraint is a general trick (the same move solves circular versions of other line-DP problems). Mirror: code/18-dp-1d/HouseRobberII.java.

medium Maximum Subarray — LeetCode 53

The problem. The contiguous subarray with the largest sum (elements may be negative; the subarray must be non-empty).

Example. [-2,1,-3,4,-1,2,1,-5,4] → 6, from [4,-1,2,1]. All-negative [-3,-1,-2] → −1 (the least-bad single element).

Variation 1 — brute force · O(n²)

public static int maxSubArrayBrute(int[] nums) {
    int best = Integer.MIN_VALUE;
    for (int i = 0; i < nums.length; i++) {
        int sum = 0;
        for (int j = i; j < nums.length; j++) {
            sum += nums[j];                  // extend the (i..j) subarray
            best = Math.max(best, sum);
        }
    }
    return best;
}

Variation 2 — Kadane's algorithm · O(n) time, O(1) space

public static int maxSubArray(int[] nums) {
    // DP in disguise: endingHere = best sum of a subarray ENDING at i.
    int endingHere = nums[0];
    int best = nums[0];
    for (int i = 1; i < nums.length; i++) {
        // Extend the previous run, or abandon it and restart at nums[i] —
        // a negative running sum can only drag the future down.
        endingHere = Math.max(nums[i], endingHere + nums[i]);
        best = Math.max(best, endingHere);
    }
    return best;
}

Takeaway. The most famous 1D DP there is, and the state definition is the whole lesson: not "best so far" but "best ending here" — the same ending-here trick as Maximum Product Subarray (above), which extends it to track a min too. Two things to say unprompted: it handles all-negative arrays (no "empty subarray" cheat), and it generalizes to circular arrays (LC 918) via max(normal Kadane, total − min-Kadane). See it face off against sliding window and prefix sums in playbook drill 7. Mirror (cross-checked against brute force): code/18-dp-1d/MaximumSubarray.java.

medium Decode Ways — LeetCode 91

The problem. Letters map A=1..Z=26; count the ways to decode a digit string. Zeros are only valid inside "10" or "20".

Example. "12" → 2 (AB, L). "226" → 3. "06" → 0. "100" → 0.

Climbing stairs with validity gates · O(n) time, O(1) space

public static int numDecodings(String s) {
    int n = s.length();
    if (n == 0 || s.charAt(0) == '0') return 0;   // leading zero: dead
    int prev2 = 1;                  // dp[0]: empty prefix decodes one way
    int prev1 = 1;                  // dp[1]: the first (non-zero) char
    for (int i = 2; i <= n; i++) {
        int curr = 0;
        if (s.charAt(i - 1) != '0')               // 1-digit step "X"
            curr += prev1;
        int two = (s.charAt(i - 2) - '0') * 10 + (s.charAt(i - 1) - '0');
        if (two >= 10 && two <= 26)               // 2-digit step "XY"
            curr += prev2;
        if (curr == 0) return 0;                  // stuck (e.g. "30", "100")
        prev2 = prev1;
        prev1 = curr;
    }
    return prev1;
}

Takeaway. Structurally Climbing Stairs — each position closes a 1-step or a 2-step — but the steps carry validity conditions, and the zeros are where every wrong submission dies: '0' can't stand alone, "30" traps it with no partner, and the two-digit range check must be ≥ 10 (else "06" sneaks in). Run "10", "100", "27" by hand before declaring victory. Mirror: code/18-dp-1d/DecodeWays.java.

medium Perfect Squares — LeetCode 279

The problem. The fewest perfect squares (1, 4, 9, 16, …) summing to n.

Example. 12 → 3 (4+4+4, not 9+1+1+1). 13 → 2 (4+9).

Coin Change with the squares as coins · O(n√n)

public static int numSquares(int n) {
    int[] dp = new int[n + 1];              // dp[x] = fewest squares for x
    for (int x = 1; x <= n; x++) {
        dp[x] = Integer.MAX_VALUE;
        for (int root = 1; root * root <= x; root++)
            dp[x] = Math.min(dp[x], dp[x - root * root] + 1);
    }
    return dp[n];                            // dp[0] = 0 seeds everything
}

Takeaway. Spotting the reduction — "the coin set is {1,4,9,…}" — turns a number-theory-looking problem into Coin Change (above) verbatim; no new DP needed. Since squares include 1, every amount is reachable, so the unreachable-sentinel handling Coin Change needs disappears. Bonus fact worth dropping: Lagrange's four-square theorem caps every answer at 4 (the mirror asserts it), and a BFS-on-amounts view finds the answer level by level. Mirror: code/18-dp-1d/PerfectSquares.java.

medium Delete and Earn — LeetCode 740

The problem. Taking a copy of value v earns v points but deletes every element equal to v−1 or v+1. Maximize total points.

Example. [3,4,2] → 6 (take 4 — deletes the 3 — then take 2). [2,2,3,3,3,4] → 9 (all three 3s; the 2s and 4 die).

Reduce to House Robber on the value axis · O(n + maxVal)

public static int deleteAndEarn(int[] nums) {
    int maxVal = 0;
    for (int x : nums) maxVal = Math.max(maxVal, x);
    // Bucket first: taking one v earns ALL copies of v (they don't
    // delete each other) — so per value it's all-or-nothing.
    long[] gain = new long[maxVal + 1];      // gain[v] = v * count(v)
    for (int x : nums) gain[x] += x;

    // Adjacent values conflict = adjacent houses. Rob the value line:
    long skip = 0, take = gain[0];
    for (int v = 1; v <= maxVal; v++) {
        long newTake = skip + gain[v];       // taking v forces skipping v−1
        skip = Math.max(skip, take);         // skipping v keeps the better of v−1
        take = newTake;
    }
    return (int) Math.max(skip, take);
}

Takeaway. A reduction in two moves: bucket duplicates (all copies of v live or die together → gain[v] = v·count), then notice "taking v kills v±1" is exactly House Robber's adjacency constraint — just indexed by value instead of position. Interviews love this problem precisely because it tests whether you can see an old DP under a new costume. Mirror: code/18-dp-1d/DeleteAndEarn.java.

Pitfalls

  • Say the state sentence before writing code. Most DP failures are a fuzzy state ("dp[i] is... the answer... at i?"). If dp[i] is "best ending AT i" (LIS) the final answer is a max over all cells; if it's "best considering 0..i" (robber) the answer is the last cell. Know which yours is.
  • Base cases off-by-one: dp[0] usually means the empty prefix/amount, not the first element. Coin Change's dp[0]=0 and Word Break's dp[0]=true both encode "the empty problem is trivially solved".
  • Memoize before returning. Computing the value and forgetting memo[i] = best keeps the code correct but exponential — the classic "my DP times out" bug.
  • Java recursion depth: top-down on n = 10⁵ states can overflow the default JVM stack. Convert to tabulation, or mention -Xss — the interviewer will nod either way.
  • Use a safe "infinity": Integer.MAX_VALUE overflows the moment you add 1 to it. Use a domain bound (Coin Change's amount + 1) or MAX_VALUE / 2.
  • Counting problems overflow int: "number of ways" answers grow like Fibonacci — climbing stairs overflows int at n = 46. Ask about constraints; switch to long or a modulus when counting.

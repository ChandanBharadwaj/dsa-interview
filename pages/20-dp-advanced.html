DP Advanced: State Machines, Intervals, Bitmasks & Trees

Four DP families beyond arrays and grids — how to recognize each from the problem statement, and the one worked classic for every family.

Phase 7 · Weeks 11–12 Senior/staff differentiator — where "knows DP" becomes "owns DP"

Before you start

Why this topic exists. Pages 18–19 taught DP where the state is "position in an array" or "cell in a table". Real interview hards go further: the state can be a mode you're in (holding a stock or not), a range (balloons between index l and r), a set (which cities are already visited, packed into an int), or a subtree. The DP recipe never changes — define state, write the recurrence, order the computation — what changes is the shape of the state. This page trains you to recognize the four shapes.

Interview relevance: a senior differentiator — recognizing WHICH advanced family (state machine, interval, bitmask, tree) fits the input is the whole game.

How this builds on the previous topic

In 2D grid DP the state was where you are. Here the state becomes what situation you're in — a strictly more powerful idea. Once you can say "my state is (day, holding-or-not)" or "(range l..r)", the recurrences practically write themselves using the exact skills from pages 18–19.

Core concepts

1. State-machine DP — when the problem has modes

Stock problems are the canonical family. Each day you're in one of a few states (holding a share / just sold / free to buy), and each day you take one transition (buy, sell, rest). Draw the machine, and each state's recurrence is just "the best way to arrive here":

Why sold exists at all: after selling there's a one-day cooldown, so "not holding" splits into two genuinely different situations — just sold (can't buy tomorrow) and rest (free to buy). When one flag changes what moves are legal, that flag belongs in the state. That sentence generalizes to every state-machine problem (with-fee, at-most-k-transactions, etc.).

2. Interval DP — when the state is a range

Signature: "do operations on a sequence until it's consumed; order changes the score" — burst balloons, matrix-chain multiplication, merging stones. State: dp[l][r] = best result for the sub-range. Solve short ranges first, then longer ones built from them.

The key inversion: pick the LAST operation, not the first. In Burst Balloons, if you pick the first balloon to pop, the two sides still interact (their neighbors change later) — subproblems aren't independent. Pick the balloon that pops last in range (l, r): at that moment its neighbors are exactly the walls l and r, so the left and right sides are fully independent. Almost every interval-DP problem has this same "condition on the last move" trick.

3. Bitmask DP — when the state is a subset (n ≤ ~20)

An int is 32 flags (page 22): bit i set = "item i already used". So dp[mask] means "best result having used exactly this subset" — 2ⁿ states you can iterate with a plain for-loop. The giveaway is the constraint: n ≤ 20 with "visit all / assign all / partition all".

The bitmask vocabulary (used in TSP, partition-to-k-subsets, job assignment)

int full = (1 << n) - 1;                 // all n items used
for (int mask = 0; mask <= full; mask++) // enumerate every subset
    for (int i = 0; i < n; i++) {
        if ((mask & (1 << i)) != 0) continue;  // item i already in the subset
        int next = mask | (1 << i);            // subset with item i added
        // dp[next] = best(dp[next], dp[mask] + cost of adding i)
    }

4. Tree DP — when the input is a tree

Run the bottom-up pattern from page 11: post-order DFS where each node returns a small array of answers for its subtree — one per "state of this node" (robbed / not robbed). The parent combines children's answers in O(1). It's House Robber's rob/skip choice, walked over a tree instead of a row.

Pattern templates (Java)

State-machine rolling update — read yesterday, write today

// one variable per state; temporaries enforce "all reads see yesterday"
int newHold = Math.max(hold, rest - price);   // stay, or buy from rest
int newSold = hold + price;                   // sell what we held
int newRest = Math.max(rest, sold);           // stay, or cooldown just ended
hold = newHold; sold = newSold; rest = newRest;

Interval DP skeleton — lengths outward, split point inside

for (int len = 2; len <= n; len++)            // small ranges first
    for (int l = 0; l + len <= n; l++) {
        int r = l + len;
        for (int k = l + 1; k < r; k++)       // k = the LAST operation's position
            dp[l][r] = Math.max(dp[l][r], dp[l][k] + dp[k][r] + score(l, k, r));
    }

Tree DP skeleton — post-order, return one answer per node-state

int[] dfs(TreeNode node) {                    // returns {bestIfUsed, bestIfSkipped}
    if (node == null) return new int[]{0, 0};
    int[] L = dfs(node.left), R = dfs(node.right);   // children first
    int use  = node.val + L[1] + R[1];        // using node forbids using children
    int skip = Math.max(L[0], L[1]) + Math.max(R[0], R[1]);
    return new int[]{use, skip};
}

Decision cues

  • "…with cooldown / with fee / at most k transactions" → add a mode to the state, don't invent a new algorithm.
  • Score of an operation depends on what's still around it → interval DP, condition on the last move.
  • n ≤ 20 is a flashing sign: the intended solution is exponential-in-subsets — bitmask.
  • Tree + "can't pick adjacent" or "best path/coverage" → post-order returning per-state answers.

Where this shows up in real systems

Database query optimizers

Choosing a join order for k tables is interval/bitmask DP over table subsets — PostgreSQL's planner literally runs DP up to a table-count threshold (then falls back to heuristics). "dp[subset of tables] = cheapest plan producing that subset."

Trading & inventory state machines

Position management (flat/long/short with transaction costs and holding constraints) is the stock-DP state machine at industrial scale — backtesting engines evaluate exactly these transition graphs over price series.

Routing & last-mile delivery

Exact TSP solvers for small stop-counts (a driver's 15-stop run) use Held-Karp bitmask DP — dp[visited-set][current-stop]. Beyond ~20 stops, heuristics take over, but the DP is the correctness baseline.

Compilers & bioinformatics

Optimal expression evaluation and matrix-chain ordering are interval DP; RNA secondary-structure prediction (Nussinov algorithm) is dp[l][r] over a sequence — the same skeleton as Burst Balloons.

medium Best Time to Buy and Sell Stock II — LeetCode 122 (two-state machine)

Problem. Unlimited buy/sell (one share at a time) — maximize profit.

Idea. Two states after each day: hold (own a share) and free (don't). Each day, each state keeps its best value over "stay" and "transition in".

public static int maxProfit(int[] prices) {
    int hold = -prices[0];                       // day 0: bought at prices[0]
    int free = 0;                                // day 0: did nothing
    for (int d = 1; d < prices.length; d++) {
        int p = prices[d];
        // Compute both new states from YESTERDAY's values (read before write).
        int newHold = Math.max(hold, free - p);  // keep holding, or buy today
        int newFree = Math.max(free, hold + p);  // keep resting, or sell today
        hold = newHold;
        free = newFree;
    }
    return free;                                 // end without a share — holding one is wasted money
}

Complexity. O(n) time, O(1) space. (A greedy "sum every up-tick" also solves this one — but the state-machine framing is the one that generalizes to cooldown/fee/k-transactions, which greedy cannot.)

medium Best Time ... with Cooldown — LeetCode 309 (three-state machine)

Problem. Same as Stock II, but after selling you must skip a day before buying again.

Idea. The cooldown makes "not holding" ambiguous — split it: sold (sold today, tomorrow is a forced cooldown) vs rest (idle, allowed to buy). Then read the transitions off the machine in the animation above.

public static int maxProfit(int[] prices) {
    int hold = -prices[0];                          // day 0: bought
    int sold = Integer.MIN_VALUE / 2;               // day 0: can't have sold yet ("-infinity", halved to avoid overflow)
    int rest = 0;                                   // day 0: did nothing
    for (int d = 1; d < prices.length; d++) {
        int p = prices[d];
        // All three reads use YESTERDAY's values, so compute into temporaries first.
        int newHold = Math.max(hold, rest - p);     // keep holding, or buy (only from rest — not right after a sale!)
        int newSold = hold + p;                     // sell the share we were holding
        int newRest = Math.max(rest, sold);         // idle, or yesterday's cooldown just ended
        hold = newHold; sold = newSold; rest = newRest;
    }
    return Math.max(sold, rest);                    // end not holding (ending in `hold` wastes a purchase)
}

Complexity. O(n) time, O(1) space. Two details interviewers probe: why buy reads rest (not sold) — that IS the cooldown; and why −∞ is halved — a raw MIN_VALUE would overflow when added to a price.

medium House Robber III — LeetCode 337 (tree DP)

Problem. Houses form a binary tree; robbing a node forbids robbing its direct children. Maximize the loot.

Idea. Post-order DFS where every subtree reports two numbers: best-if-robbed and best-if-skipped. The parent combines them in O(1) — the bottom-up pattern from page 11, now carrying DP state.

// res[0] = best loot in this subtree if we ROB this node
// res[1] = best loot in this subtree if we SKIP this node
public static int rob(TreeNode root) {
    int[] res = dfs(root);
    return Math.max(res[0], res[1]);                 // free to rob or skip the root
}

private static int[] dfs(TreeNode node) {
    if (node == null) return new int[]{0, 0};        // empty subtree: nothing either way
    int[] left = dfs(node.left);                     // solve children first (post-order)
    int[] right = dfs(node.right);
    // Rob this node -> both children MUST be skipped (adjacency constraint along edges).
    int robThis = node.val + left[1] + right[1];
    // Skip this node -> each child is independently free: take its better option.
    int skipThis = Math.max(left[0], left[1]) + Math.max(right[0], right[1]);
    return new int[]{robThis, skipThis};
}

Complexity. O(n) time (each node once), O(h) space for recursion. The common wrong answer — "skip node ⇒ rob both children" — misses that skipping can also beat robbing at a child; taking each child's max fixes it.

medium Last Stone Weight II — LeetCode 1049

The problem. Smash any two stones; the difference survives. Minimize the final remaining weight.

Example. [2,7,4,1,8,1] → 1. [31,26,33,21,40] → 5.

Reduce to closest-to-half subset sum · O(n · total/2)

public static int lastStoneWeightII(int[] stones) {
    // Every smash sequence ends at |sum(A) − sum(B)| for SOME partition
    // (each stone's weight ends up + or −). Minimize = get sum(A) as
    // close to total/2 as possible from below.
    int total = 0;
    for (int s : stones) total += s;
    int half = total / 2;

    boolean[] dp = new boolean[half + 1];   // dp[s]: some subset sums to s
    dp[0] = true;
    for (int stone : stones)
        for (int s = half; s >= stone; s--) // backwards: one use per stone
            dp[s] |= dp[s - stone];

    for (int s = half; s >= 0; s--)         // best achievable ≤ half
        if (dp[s])
            return total - 2 * s;           // |(total−s) − s|
    return total;
}

Takeaway. The smash game is a decoy — proving every outcome is a signed sum (and every partition is achievable) collapses it to Partition Equal Subset Sum's optimization twin. Compare the trio: LC 416 asks can we hit total/2, Target Sum counts hits of a target, this minimizes the miss — one boolean knapsack, three question types. Mirror: code/20-dp-advanced/LastStoneWeightII.java.

hard Burst Balloons — LeetCode 312 (interval DP)

Problem. Bursting balloon i earns nums[left]·nums[i]·nums[right] using its current neighbors; balloons then close ranks. Maximize total coins.

Idea. Condition on the balloon that pops last between walls l and r: at that moment its neighbors are exactly l and r, so the two sides are independent subproblems. Pad both ends with virtual 1-balloons so edge products are uniform.

public static int maxCoins(int[] nums) {
    int n = nums.length;
    int[] b = new int[n + 2];                       // padded array
    b[0] = 1; b[n + 1] = 1;                         // virtual boundary balloons
    for (int i = 0; i < n; i++) b[i + 1] = nums[i];

    int[][] dp = new int[n + 2][n + 2];             // defaults to 0: empty ranges score 0
    for (int len = 2; len <= n + 1; len++) {        // solve small gaps first (range length = r-l)
        for (int l = 0; l + len <= n + 1; l++) {
            int r = l + len;
            for (int k = l + 1; k < r; k++) {       // choose which balloon pops LAST in (l,r)
                int coins = dp[l][k]                // first clear everything left of k...
                          + dp[k][r]                // ...then everything right of k...
                          + b[l] * b[k] * b[r];     // ...then k pops between walls l and r
                dp[l][r] = Math.max(dp[l][r], coins);
            }
        }
    }
    return dp[0][n + 1];                            // burst everything between the two virtual walls
}

Complexity. O(n³) time (ranges × split points), O(n²) space. If you remember one thing: "first balloon" couples the sides; "last balloon" decouples them.

medium Stone Game — LeetCode 877

The problem. Piles of stones in a row; Alice and Bob alternately take a whole pile from either end (Alice first). Total stones are odd, so no ties. Both play optimally — does Alice win?

Example. [5,3,4,5] → true: Alice can always engineer taking the larger of the two "colors" of positions.

Interval DP on the score difference · O(n²)

public static boolean stoneGame(int[] piles) {
    int n = piles.length;
    // dp[l][r] = best score DIFFERENCE (current player minus opponent)
    // achievable from piles l..r. The "current player" flips implicitly:
    // whatever the opponent gains from the smaller range counts NEGATIVE.
    int[][] dp = new int[n][n];
    for (int l = n - 1; l >= 0; l--) {
        dp[l][l] = piles[l];                 // one pile: take it, lead by its value
        for (int r = l + 1; r < n; r++) {
            // Take left end: gain piles[l], then opponent plays [l+1..r]
            int takeLeft = piles[l] - dp[l + 1][r];
            // Take right end: gain piles[r], then opponent plays [l..r-1]
            int takeRight = piles[r] - dp[l][r - 1];
            dp[l][r] = Math.max(takeLeft, takeRight);
        }
    }
    return dp[0][n - 1] > 0;                 // Alice leads on the full row
}

Takeaway. Two ideas in one: interval state (a shrinking range, like Burst Balloons) and the game-theory trick of tracking the difference so one table serves both players — the subtraction is the turn flip. (Fun fact: with even piles Alice can always win by parity, so return true passes — but the DP is what generalizes to Stone Game II/III and odd pile-counts.) Mirror: code/20-dp-advanced/StoneGame.java.

medium Partition to K Equal Sum Subsets — LeetCode 698

The problem. Can the array be split into exactly k non-empty subsets with equal sums?

Example. nums=[4,3,2,3,5,2,1], k=4 → true: (5), (1,4), (2,3), (2,3) — each sums to 5. nums=[1,2,3,4], k=3 → false (10 isn't divisible by 3).

Fill buckets one at a time with a used-mask · exponential, heavily pruned

public static boolean canPartitionKSubsets(int[] nums, int k) {
    int total = 0;
    for (int x : nums) total += x;
    if (total % k != 0) return false;        // must divide evenly
    int target = total / k;
    Arrays.sort(nums);                       // ascending; we'll scan from the top
    if (nums[nums.length - 1] > target) return false;  // one number too big

    // used = bitmask of consumed elements (n <= 16 in the constraints).
    return fill(nums, new boolean[nums.length], k, 0, 0, target);
}

// Complete `k` buckets; `sum` is the current bucket's fill, `start` the
// scan position within the current bucket (avoids permuting choices).
private static boolean fill(int[] nums, boolean[] used, int k,
                            int start, int sum, int target) {
    if (k == 1) return true;                 // last bucket gets the exact remainder
    if (sum == target)                       // bucket complete — start the next one
        return fill(nums, used, k - 1, 0, 0, target);
    for (int i = start; i < nums.length; i++) {
        if (used[i] || sum + nums[i] > target) continue;
        // Prune duplicates: an unused equal neighbor already failed here
        if (i > start && nums[i] == nums[i - 1] && !used[i - 1]) continue;
        used[i] = true;                      // choose
        if (fill(nums, used, k, i + 1, sum + nums[i], target)) return true;
        used[i] = false;                     // un-choose
        if (sum == 0) return false;          // key prune: if this element can't
                                             // START a bucket, no one can — fail fast
    }
    return false;
}

Takeaway. Backtracking (page 10) wearing DP's constraint hat: the n ≤ 16 limit signals the exponential-in-subsets family. The two prunes do all the work — duplicate-skip, and "if a bucket can't even be started with this element, abort". A pure bitmask-DP formulation (dp over 2ⁿ masks) exists too and is worth mentioning; the pruned backtracking is faster in practice. Mirror: code/20-dp-advanced/PartitionKSubsets.java.

hard Best Time to Buy and Sell Stock IV — LeetCode 188

The problem. Same stock rules, but at most k complete transactions (buy+sell). Maximize profit.

Example. k=2, prices=[3,2,6,5,0,3] → 7 (buy 2, sell 6; buy 0, sell 3). k=2, prices=[2,4,1] → 2.

The machine generalized: hold/free per transaction count · O(n·k)

public static int maxProfit(int k, int[] prices) {
    int n = prices.length;
    if (n == 0 || k == 0) return 0;
    // k >= n/2 means "effectively unlimited" — collapse to Stock II's greedy
    // (a transaction needs 2 days, so more than n/2 can never all be used).
    if (k >= n / 2) {
        int profit = 0;
        for (int i = 1; i < n; i++)
            if (prices[i] > prices[i - 1]) profit += prices[i] - prices[i - 1];
        return profit;
    }

    // hold[t] = best cash while holding, having STARTED t transactions.
    // free[t] = best cash while not holding, having COMPLETED t transactions.
    int[] hold = new int[k + 1];
    int[] free = new int[k + 1];
    Arrays.fill(hold, Integer.MIN_VALUE / 2);   // can't hold before buying
    for (int p : prices) {
        for (int t = k; t >= 1; t--) {          // backward: each day adds ≤1 action per t
            free[t] = Math.max(free[t], hold[t] + p);     // sell: completes transaction t
            hold[t] = Math.max(hold[t], free[t - 1] - p); // buy: starts transaction t
        }
    }
    return free[k];                              // best with ≤ k completed transactions
}

Takeaway. Stock II had states {hold, free}; adding "how many transactions so far" multiplies the state space by k+1 — the machine doesn't change, it gets indexed. The k ≥ n/2 collapse to the unlimited case is the expected optimization (without it, huge k wastes O(n·k)). Cooldown, fee, and k-limit are all the same lesson: every rule that changes what's legal becomes a state dimension. Mirror: code/20-dp-advanced/StockIV.java.

hard Longest Increasing Path in a Matrix — LeetCode 329

The problem. The longest strictly increasing path in a grid, moving in the four cardinal directions.

Example. [[9,9,4],[6,6,8],[2,1,1]] → 4 (1→2→6→9).

Memoized DFS — DP on the implicit DAG · O(mn)

private static int dfs(int[][] m, int r, int c, int[][] memo) {
    if (memo[r][c] != 0) return memo[r][c];   // solved before — reuse
    int best = 1;                             // the cell alone
    for (int[] d : DIRS) {
        int nr = r + d[0], nc = c + d[1];
        if (nr >= 0 && nr < m.length && nc >= 0 && nc < m[0].length
                && m[nr][nc] > m[r][c])       // strictly uphill only
            best = Math.max(best, 1 + dfs(m, nr, nc, memo));
    }
    memo[r][c] = best;
    return best;
}
// answer = max of dfs over every start cell

Takeaway. Remember why grid DP was "forbidden" for free-movement grids (page 17): cycles. Here strict increase makes cycles impossible — the grid becomes a DAG, and memoized DFS is DP running in topological order without ever computing that order. The contrast is the interview payload: movement rules decide whether a grid is a DAG, and DAG-ness decides whether DP applies. Visited-marking is unnecessary (the uphill rule can't revisit) — adding it anyway is the common self-inflicted bug. Mirror: code/20-dp-advanced/LongestIncreasingPath.java.

hard Regular Expression Matching — LeetCode 10

The problem. Full-string matching where '.' matches any character and '*' means zero or more of the preceding element.

Example. "aa" vs "a*" → true. "ab" vs ".*" → true. "mississippi" vs "mis*is*p*." → false. "aab" vs "c*a*b" → true (c* matches zero c's).

2D boolean table; '*' branches zero-or-more · O(mn)

public static boolean isMatch(String s, String p) {
    int m = s.length(), n = p.length();
    boolean[][] dp = new boolean[m + 1][n + 1];   // dp[i][j]: s[0..i) vs p[0..j)
    dp[0][0] = true;                        // empty matches empty
    // Empty s can still match a*, a*b*, …: each '*' erases its element.
    for (int j = 2; j <= n; j++)
        if (p.charAt(j - 1) == '*')
            dp[0][j] = dp[0][j - 2];

    for (int i = 1; i <= m; i++)
        for (int j = 1; j <= n; j++) {
            char pc = p.charAt(j - 1);
            if (pc == '*') {
                char prev = p.charAt(j - 2);
                dp[i][j] = dp[i][j - 2]     // ZERO copies: drop "x*" entirely
                    || ((prev == '.' || prev == s.charAt(i - 1))
                        && dp[i - 1][j]);   // ONE MORE copy: consume s's char,
            } else {                        //   keep the same "x*" active
                dp[i][j] = (pc == '.' || pc == s.charAt(i - 1))
                    && dp[i - 1][j - 1];    // plain/dot char: both advance
            }
        }
    return dp[m][n];
}

Takeaway. The key reframe: '*' isn't a character, it's a modifier — the unit is the two-character block "x*", which the DP either deletes (dp[i][j−2]) or expands one copy at a time (dp[i−1][j] with the same j — the star stays active). The empty-string row seeding is where most attempts die. This table is a tabular NFA simulation, the same construction real regex engines compile to. Mirror: code/20-dp-advanced/RegexMatching.java.

Pitfalls

  • Rolling-state bug: updating hold before free reads it silently uses today's value. Always compute all new states into temporaries, then assign.
  • −∞ sentinels overflow. Integer.MIN_VALUE + price wraps around; use MIN_VALUE / 2 or a long.
  • Interval DP loop order matters: iterate by range length, not by l then r — dp[l][r] needs all shorter sub-ranges already solved.
  • Bitmask off-by-ones: (1 << n) - 1 is "all used"; test membership with (mask >> i & 1) == 1; never shift by ≥ 32 on an int.
  • Don't force DP where a mode-less greedy works (Stock II alone is greedy-able) — but SAY the state-machine version too, since it's what generalizes when the follow-up adds a cooldown or fee.
  • Name states out loud before coding. "hold means best cash while owning a share".

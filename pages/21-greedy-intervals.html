Greedy & Intervals

When a locally best choice is provably globally safe — and the interval toolkit (merge, schedule, count rooms) that appears in almost every interview pool.

Phase 7 · Weeks 11–12 Intervals show up in every phone-screen pool

Before you start

Why this topic exists. You just spent three pages on dynamic programming, which carefully considers every choice. Greedy is the opposite bet: make the obviously-best local choice, never look back, and finish in one pass. Sometimes that's wrong (DP exists for a reason) — but for a well-defined family of problems it is provably right, and interviews love asking you to tell the two apart. Interval problems are the most common playground for that judgment call, and they're also the questions your calendar app solves every day.

Prerequisites

  • 06 · Sorting — every interval problem starts with a sort and a comparator.
  • 12 · Heaps — Meeting Rooms II uses a min-heap of end times.
  • 18 · DP 1D — you'll contrast greedy with DP (remember coin change with coins [1,3,4]).

Interview relevance: intervals appear in every phone-screen pool; the tested skill is the sort-by-start-or-end decision plus a correctness argument for the greedy choice.

How this builds on the previous topic

Page 20 ended with DP variants that enumerate whole families of choices. Greedy is what you try first: if a local rule can be proven safe, you replace an O(n·states) DP with a single O(n log n) sort-and-sweep. The decision test "can I prove the greedy choice, or do I need DP?" is exactly the skill this page adds on top of the DP instinct you already have.

Core concepts

1. Greedy = local choice + a safety proof

A greedy algorithm makes the best-looking choice right now and never reconsiders. That's only an algorithm — not a correct one — until you can argue the choice is safe.

The standard argument is the exchange argument, and told simply it goes: "take any optimal solution; I can rewrite it to start with my greedy choice without making it worse." If that rewrite is always possible, greedy's first pick is safe; apply the same logic to the rest and the whole greedy answer is optimal. You rarely write this proof formally in an interview — but saying one honest sentence of it ("earliest end time leaves the most room, so swapping it into any optimal schedule can't hurt") is what separates "I memorized this" from "I understand this".

When greedy fails. Coin change with coins [1,3,4] and target 6: greedy grabs 4, then 1, then 1 — three coins. Optimal is 3+3 — two coins. The local choice (biggest coin) is not safe, so this problem needs DP (page 18). The practical interview test: try 2–3 small counterexamples first. If you can't break the greedy rule quickly, start sketching why it's safe; if you can, it's DP.

2. The canonical greedy moves (non-interval)

  • Furthest reach (Jump Game): walk the array keeping one number — the furthest index reachable so far. Every jump either extends it or doesn't; you never need to know which jumps got you there.
  • Reset point (Gas Station): if you run out of gas somewhere between station i and j, no station in (i..j] can be the answer either — so restart from j+1. One pass, O(n).
  • Last occurrence (Partition Labels): a cut is legal only when no letter inside the current chunk appears again later — so precompute each letter's last index and extend the chunk to cover them.
  • Pair extremes (task assignment): sorting and matching smallest-with-largest (or biggest-job-to-freest-worker) is a greedy that shows up in load-balancing style questions.

3. Intervals: the "sort by start or sort by end?" question

Almost every interval problem is: sort, then one sweep. The entire game is choosing the sort key — and the two choices answer two different questions:

START"What does the merged timeline look like?"Once sorted by start, an interval can only overlap the block you're currently building — after the first gap, nothing later can reach back.Merge Intervals, Insert Interval, Meeting Rooms I

4. Counting rooms: heap vs sweep line

Meeting Rooms II ("minimum rooms so no two meetings share one") has two classic solutions, and strong candidates can sketch both:

  • Min-heap of end times (page 12): process meetings in start order; the heap holds one end-time per occupied room. If the earliest-freeing room (heap top) is free before your meeting starts, reuse it (poll), otherwise you need a new room. Final heap size = answer.
  • Sweep line (+1/−1): forget which meeting is which — a room count only changes at events. Sort all starts and all ends separately; walk time forward adding 1 per start, subtracting 1 per passed end; the peak of that counter is the answer. This "event counting" trick generalizes to any "max concurrent X" question (connections, calls, car pooling — see the difference array on page 04).

Pattern templates (Java)

Template 1 — merge / sweep intervals (sort by START)

// Works for: Merge Intervals, Insert Interval, Meeting Rooms I, "any overlap?" checks
Arrays.sort(intervals, (a, b) -> Integer.compare(a[0], b[0])); // NEVER a[0]-b[0]: overflow risk
List<int[]> out = new ArrayList<>();
int[] cur = intervals[0];                 // block currently being grown
for (int i = 1; i < intervals.length; i++) {
    if (intervals[i][0] <= cur[1]) {      // overlap (touching counts) -> absorb
        cur[1] = Math.max(cur[1], intervals[i][1]);   // max() handles nested intervals
    } else {                              // gap: starts only increase -> cur is final
        out.add(cur);
        cur = intervals[i];
    }
}
out.add(cur);                             // the last block is still open — commit it

Template 2 — activity selection (sort by END, keep the compatible ones)

// Works for: Non-overlapping Intervals, max meetings in one room, Min Arrows
Arrays.sort(intervals, (a, b) -> Integer.compare(a[1], b[1])); // earliest END first
int kept = 0;
int end = Integer.MIN_VALUE;              // end of the last interval we kept
for (int[] iv : intervals) {
    if (iv[0] >= end) {                   // starts after the last kept one ends -> compatible
        kept++;                           // greedy: earliest end leaves maximum room
        end = iv[1];
    }
    // else: skip it — it clashes with a provably better pick
}
// kept = max compatible set;  intervals.length - kept = min removals
One decision to make out loud. Does your problem treat touching intervals ([1,4] and [4,5]) as overlapping? Merge Intervals says yes (<=); Non-overlapping Intervals says no (>= is fine). Confirm with the interviewer — it's a one-character bug either way.

Where this shows up in real systems

Calendar & meeting systems

Google Calendar / Outlook conflict detection is Meeting Rooms I; "find a free room" across a building is Meeting Rooms II with real room identities (the heap version, since you must know which room frees up). Free/busy views are literally Merge Intervals over attendees' schedules.

CPU & task schedulers

EDF (Earliest Deadline First) — the classic real-time scheduling policy in RTOSes and Linux's SCHED_DEADLINE — is activity selection's cousin: always run the task with the earliest deadline, provably optimal for schedulable single-core task sets.

Register allocation in compilers

A variable is "live" over an interval of instructions. Linear-scan register allocation (used in JIT compilers like HotSpot's C1) sorts live intervals by start and greedily assigns registers, spilling when the active set exceeds the register count — Meeting Rooms II where rooms are CPU registers.

Booking, ads & Huffman

Hotel/seat booking engines count overlap per resource (sweep line); ad servers pack non-overlapping slots into breaks (activity selection). And Huffman coding — the compression step inside zip/JPEG — is a famous greedy: repeatedly merge the two rarest symbols (a heap, page 12).

medium Jump Game — LeetCode 55 (greedy reach)

Problem in one line: from index 0, each cell tells you your max jump length — can you reach the last index?

Idea: you never need to know the actual path. Keep one number — the furthest index reachable using everything seen so far. Each cell you can stand on may extend it. The exchange argument is trivial here: any successful path only ever stands on reachable cells, and "furthest" dominates every other measure of progress.

code/21-greedy-intervals/JumpGame.java

public static boolean canJump(int[] nums) {
    int furthest = 0;                                 // furthest index we can reach with jumps seen so far
    for (int i = 0; i < nums.length; i++) {
        if (i > furthest) return false;               // we can't even stand on i -> stuck before the end
        furthest = Math.max(furthest, i + nums[i]);   // jumping from i could extend our reach to i + nums[i]
        if (furthest >= nums.length - 1) return true; // last index is already within reach -> done early
    }
    return true;                                      // loop finished => every index (incl. last) was reachable
}

Complexity: time O(n) — one pass; space O(1).

medium Partition Labels — LeetCode 763 (last occurrence)

Problem in one line: split a string into as many chunks as possible so each letter appears in at most one chunk; return chunk sizes.

Idea: a cut after position i is legal only if no letter in s[start..i] occurs again after i. So first record every letter's last occurrence. Sweeping left to right, the chunk's required end is the max last-occurrence of the letters seen so far; the moment i catches up to that end, nothing inside points past it — cut greedily (cutting as early as possible clearly maximizes the number of chunks).

code/21-greedy-intervals/PartitionLabels.java

public static List<Integer> partitionLabels(String s) {
    int[] last = new int[26];                       // last[c] = final index where letter c appears
    for (int i = 0; i < s.length(); i++) {
        last[s.charAt(i) - 'a'] = i;                // later occurrences overwrite earlier ones
    }
    List<Integer> sizes = new ArrayList<>();
    int start = 0, end = 0;                         // current partition is s[start..end]
    for (int i = 0; i < s.length(); i++) {
        // every letter we meet drags the partition's end out to its last occurrence
        end = Math.max(end, last[s.charAt(i) - 'a']);
        if (i == end) {                             // nothing inside points past i -> safe to cut here
            sizes.add(end - start + 1);             // record this partition's length
            start = i + 1;                          // next partition begins after the cut
        }
    }
    return sizes;
}

Complexity: time O(n) — two passes; space O(1) (the 26-slot array is constant).

medium Merge Intervals — LeetCode 56 (sort by start)

Problem in one line: given intervals in any order, return the merged, non-overlapping set.

Idea: exactly the animation above. Sort by start; grow a current block; a new interval either overlaps it (starts at or before its end — extend with max, because a nested interval must not shrink the end) or opens a gap (commit the block; since starts only increase, nothing later can reach back into it).

code/21-greedy-intervals/MergeIntervals.java

public static int[][] merge(int[][] intervals) {
    // Sort by start. Integer.compare avoids the subtraction-overflow bug of (a[0] - b[0]).
    Arrays.sort(intervals, (a, b) -> Integer.compare(a[0], b[0]));
    List<int[]> out = new ArrayList<>();
    int[] cur = intervals[0];                       // the interval we are currently growing
    for (int i = 1; i < intervals.length; i++) {
        int[] next = intervals[i];
        if (next[0] <= cur[1]) {                    // starts before cur ends -> overlap (touching counts)
            cur[1] = Math.max(cur[1], next[1]);     // extend; max() because next may be nested inside cur
        } else {                                    // gap -> cur can never grow again (starts only increase)
            out.add(cur);                           // commit the finished merged interval
            cur = next;                             // start growing a fresh one
        }
    }
    out.add(cur);                                   // don't forget the last open interval
    return out.toArray(new int[0][]);
}

Complexity: time O(n log n) for the sort, O(n) sweep; space O(n) for the output (O(log n) sort stack).

medium Non-overlapping Intervals — LeetCode 435 (sort by end)

Problem in one line: remove the minimum number of intervals so the rest don't overlap.

Idea: flip it: keep the maximum non-overlapping set (activity selection), then removals = n − kept. Sorting by end time makes greedy safe: the earliest-ending interval leaves the most room for everything after it, and any optimal solution can be rewritten to start with it (exchange argument) — so keep it, then repeat among intervals that start after it ends.

code/21-greedy-intervals/NonOverlappingIntervals.java

public static int eraseOverlapIntervals(int[][] intervals) {
    // Sort by END time: the interval finishing earliest leaves the most room
    // for everything after it (exchange argument: any optimal schedule can
    // swap its first pick for the earliest-ending one without losing a slot).
    Arrays.sort(intervals, (a, b) -> Integer.compare(a[1], b[1]));
    int kept = 0;                                   // size of the max non-overlapping set
    int end = Integer.MIN_VALUE;                    // end of the last interval we kept
    for (int[] iv : intervals) {
        if (iv[0] >= end) {                         // starts at/after last end -> no overlap, keep it
            kept++;
            end = iv[1];                            // future picks must start at/after this
        }                                           // else: overlaps the kept set -> it's a removal
    }
    return intervals.length - kept;                 // minimum removals = everything we couldn't keep
}

Complexity: time O(n log n); space O(1) beyond the sort.

medium Meeting Rooms II — LeetCode 253 (heap AND sweep line)

Problem in one line: given meeting [start,end) intervals, find the minimum number of rooms needed.

Idea A (heap): take meetings in start order; the min-heap holds the end time of every occupied room, earliest on top. If the earliest-freeing room is free before your meeting starts, reuse it; otherwise open a new room. Idea B (sweep): the room count only changes at events — +1 at every start, −1 at every end. Sort starts and ends separately and walk them in time order; the peak count is the answer. Note the sweep never pairs an end with "its" start — it doesn't need to, because counts are anonymous.

code/21-greedy-intervals/MeetingRoomsII.java

// Solution A — min-heap of end times: the heap IS the set of occupied rooms.
public static int minMeetingRoomsHeap(int[][] intervals) {
    Arrays.sort(intervals, (a, b) -> Integer.compare(a[0], b[0])); // process meetings in start order
    PriorityQueue<Integer> endTimes = new PriorityQueue<>();       // min-heap: earliest-freeing room on top
    for (int[] m : intervals) {
        // If the room that frees earliest is free before we start, reuse it
        if (!endTimes.isEmpty() && endTimes.peek() <= m[0]) {
            endTimes.poll();                       // that meeting ended -> room released
        }
        endTimes.offer(m[1]);                      // our meeting now occupies a room until m[1]
    }
    return endTimes.size();                        // rooms still tracked = peak simultaneous meetings
}

// Solution B — sweep line: +1 at each start, -1 at each end, track the peak.
public static int minMeetingRoomsSweep(int[][] intervals) {
    int n = intervals.length;
    int[] starts = new int[n], ends = new int[n];
    for (int i = 0; i < n; i++) {
        starts[i] = intervals[i][0];
        ends[i] = intervals[i][1];
    }
    Arrays.sort(starts);                           // walk both timelines in time order
    Arrays.sort(ends);
    int rooms = 0, best = 0, e = 0;                // e = index of next meeting to finish
    for (int s = 0; s < n; s++) {
        // release every meeting that finished before (or exactly when) this one starts
        while (e < n && ends[e] <= starts[s]) {
            rooms--;
            e++;
        }
        rooms++;                                   // this meeting starts -> needs a room
        best = Math.max(best, rooms);              // record the peak concurrency
    }
    return best;                                   // peak = minimum rooms required
}

Complexity: both O(n log n) time. Heap: O(n) space, and it can tell you which room frees up (useful in the real-system version). Sweep: O(n) space for the two sorted arrays, simpler logic.

medium Jump Game II — LeetCode 45

The problem. Same setup as Jump Game (each value = max jump length), but now reaching the last index is guaranteed — return the minimum number of jumps.

Example. [2,3,1,1,4] → 2 (index 0 → 1 → 4).

Implicit BFS layers over indices · O(n)

public static int jump(int[] nums) {
    int jumps = 0;
    int right = 0;              // end of the current BFS layer
    int furthest = 0;           // furthest reach from within this layer
    for (int i = 0; i < nums.length - 1; i++) {   // last index excluded!
        furthest = Math.max(furthest, i + nums[i]);
        if (i == right) {       // layer exhausted — forced to jump again
            jumps++;
            right = furthest;
        }
    }
    return jumps;
}

Takeaway. This greedy is BFS wearing O(1) space: "everything reachable in k jumps" is a contiguous index range, so the queue collapses into two integers (layer end, next layer end). Seeing minimum-steps = BFS-layers explains both why it works and why the loop excludes the last index (arriving ends the walk — no jump departs from it). Companion insight to Jump Game's max-reach scan (above). Mirror: code/21-greedy-intervals/JumpGameII.java.

medium Valid Parenthesis String — LeetCode 678

The problem. A string of '(', ')' and '*', where each '*' may act as '(', ')' or nothing. Can it be a valid parenthesis string?

Example. "(*)" → true. "(*))" → true. ")(" → false. "*(" → false.

Track the RANGE of possible open counts · O(n), O(1)

public static boolean checkValidString(String s) {
    int lo = 0;                 // fewest possible unmatched '(' so far
    int hi = 0;                 // most possible unmatched '(' so far
    for (char c : s.toCharArray()) {
        if (c == '(') {
            lo++;
            hi++;
        } else if (c == ')') {
            lo--;
            hi--;
        } else {                // '*': could be ')' (lo--), '(' (hi++), or ""
            lo--;
            hi++;
        }
        if (hi < 0) return false;   // beyond repair: ')' exceeds every option
        lo = Math.max(lo, 0);       // a real count can't be negative
    }
    return lo == 0;             // 0 lies inside [lo, hi] -> some choice works
}

Takeaway. Instead of branching on every '*' (3ⁿ) or running two-pass heuristics, carry an interval of feasible states — the wildcard just widens it. The range stays contiguous, so two ints represent every possible world at once; feasibility at the end is "does the range contain 0". This interval-of-possibilities compression is a mini technique that resurfaces in scheduling and simulation problems. Mirror: code/21-greedy-intervals/ValidParenthesisString.java.

medium Hand of Straights — LeetCode 846

The problem. Can the cards be partitioned into groups of exactly groupSize consecutive values?

Example. [1,2,3,6,2,3,4,7,8], size 3 → true ([1,2,3], [2,3,4], [6,7,8]). [1,1,2,2,3,3], size 2 → false.

Smallest card must lead a group · O(n log n)

public static boolean isNStraightHand(int[] hand, int groupSize) {
    if (hand.length % groupSize != 0) return false;
    TreeMap<Integer, Integer> counts = new TreeMap<>();
    for (int card : hand) counts.merge(card, 1, Integer::sum);

    while (!counts.isEmpty()) {
        int start = counts.firstKey();          // FORCED: nothing smaller left,
        for (int v = start; v < start + groupSize; v++) {  // so it must lead
            Integer c = counts.get(v);
            if (c == null) return false;        // gap — the straight breaks
            if (c == 1) counts.remove(v);
            else counts.put(v, c - 1);
        }
    }
    return true;
}

Takeaway. The greedy is airtight because the smallest remaining card has no choice: no smaller card exists to precede it, so it must start a group — a forced move, the strongest kind of greedy argument. TreeMap supplies "smallest remaining" in O(log n) (page 24's ordered-map pattern). Identical problem in different clothes: LC 1296 "Divide Array in Sets of K Consecutive Numbers". Mirror: code/21-greedy-intervals/HandOfStraights.java.

medium Gas Station — LeetCode 134

The problem. Circular route of stations: at station i you gain gas[i] and spend cost[i] driving to the next. Return the unique starting station from which you can complete a full lap (tank starts empty), or −1.

Example. gas=[1,2,3,4,5], cost=[3,4,5,1,2] → 3 (start at station 3 with net +3, survive the early deficits). gas=[2,3,4], cost=[3,4,3] → -1 (total gas < total cost).

Variation 1 — simulate from every start · O(n²)

public static int canCompleteCircuitBrute(int[] gas, int[] cost) {
    int n = gas.length;
    for (int start = 0; start < n; start++) {
        int tank = 0;
        int i = 0;
        for (; i < n; i++) {                       // drive one full lap
            int station = (start + i) % n;
            tank += gas[station] - cost[station];
            if (tank < 0) break;                   // stranded — this start fails
        }
        if (i == n) return start;
    }
    return -1;
}

Variation 2 — greedy reset (optimal) · O(n)

public static int canCompleteCircuit(int[] gas, int[] cost) {
    int total = 0;      // net gas over the whole circle
    int tank = 0;       // gas since the current candidate start
    int start = 0;      // current candidate starting station
    for (int i = 0; i < gas.length; i++) {
        int net = gas[i] - cost[i];
        total += net;
        tank += net;
        if (tank < 0) {
            // KEY FACT: if we ran dry reaching station i+1, then no station
            // between the old start and i can work either — each of them
            // would arrive here with even LESS gas (they skip the early
            // surplus that we had). So jump the candidate past i wholesale.
            start = i + 1;
            tank = 0;
        }
    }
    // Sufficiency: if total net >= 0, the final candidate completes the lap.
    return total >= 0 ? start : -1;
}

Takeaway. Two greedy facts carry it: (1) failure at i eliminates every start up to i in one stroke — that's what collapses O(n²) to O(n); (2) total net ≥ 0 guarantees some start works, so the surviving candidate is it. This eliminate-a-whole-range reasoning is the same shape as the two-pointer discard proofs on page 02. Mirror: code/21-greedy-intervals/GasStation.java.

medium Insert Interval — LeetCode 57

The problem. A sorted, non-overlapping interval list plus one new interval: insert it, merging where necessary, and return the (still sorted, non-overlapping) result.

Example. intervals [[1,3],[6,9]], new [2,5] → [[1,5],[6,9]]. intervals [[1,2],[3,5],[6,7],[8,10],[12,16]], new [4,8] → [[1,2],[3,10],[12,16]].

Three phases: before / absorb / after · O(n)

public static int[][] insert(int[][] intervals, int[] newInterval) {
    List<int[]> res = new ArrayList<>();
    int i = 0, n = intervals.length;

    // Phase 1: intervals entirely BEFORE the new one (end < newStart)
    while (i < n && intervals[i][1] < newInterval[0])
        res.add(intervals[i++]);

    // Phase 2: everything that overlaps the new interval gets absorbed.
    // Overlap test: interval starts before (or at) the new one's end.
    while (i < n && intervals[i][0] <= newInterval[1]) {
        newInterval[0] = Math.min(newInterval[0], intervals[i][0]);  // stretch left
        newInterval[1] = Math.max(newInterval[1], intervals[i][1]);  // stretch right
        i++;
    }
    res.add(newInterval);                    // the merged blob goes in once

    // Phase 3: the untouched tail
    while (i < n)
        res.add(intervals[i++]);

    return res.toArray(new int[0][]);
}

Takeaway. Because the input is already sorted and disjoint, the overlap zone is one contiguous run — so a single left-to-right pass with three phases does it, no sorting needed (contrast Merge Intervals, which must sort first). The two boundary comparisons (end < newStart, start ≤ newEnd) are the entire correctness surface — check them against touching intervals like [1,2]+[2,3]. Mirror: code/21-greedy-intervals/InsertInterval.java.

medium Minimum Number of Arrows to Burst Balloons — LeetCode 452

The problem. Balloons are horizontal intervals [start, end]; a vertical arrow shot at x pops every balloon whose interval contains x (ends inclusive). Minimize arrows to pop all balloons.

Example. [[10,16],[2,8],[1,6],[7,12]] → 2 (x=6 pops [2,8]+[1,6]; x=11 pops [10,16]+[7,12]). [[1,2],[3,4],[4,5]] → 2.

Sort by end; shoot at each first-unpopped balloon's end · O(n log n)

public static int findMinArrowShots(int[][] points) {
    if (points.length == 0) return 0;
    // Sort by END (long-safe compare — coordinates hit ±2^31).
    Arrays.sort(points, (a, b) -> Integer.compare(a[1], b[1]));

    int arrows = 1;
    long arrowX = points[0][1];        // greedy: shoot at the earliest end
    for (int[] p : points) {
        if (p[0] > arrowX) {           // this balloon starts after the arrow -> missed
            arrows++;                  // need a fresh arrow...
            arrowX = p[1];             // ...again at the new balloon's end
        }
        // else: p straddles arrowX (its end >= arrowX by sort order) — already popped
    }
    return arrows;
}

Takeaway. This is Non-overlapping Intervals inverted: an arrow at the earliest unpopped balloon's end catches every balloon overlapping it — the exchange argument says no other x pops a superset. Arrows = number of "overlap groups" = n − (max intervals you could keep non-overlapping). Same sort, same greedy, opposite bookkeeping. Note the inclusive ends: [1,2] and [2,3] share x=2 and need one arrow — which is why the miss test is strict >. Mirror: code/21-greedy-intervals/MinArrows.java.

hard Candy — LeetCode 135

The problem. Children in a line with ratings; each gets ≥ 1 candy, and anyone rated higher than an adjacent child must get more candy than that neighbor. Minimize total candies.

Example. ratings [1,0,2] → 5 (candies 2,1,2). [1,2,2] → 4 (candies 1,2,1 — equal ratings owe nothing).

Two sweeps, one per constraint direction · O(n)

public static int candy(int[] ratings) {
    int n = ratings.length;
    int[] candies = new int[n];
    java.util.Arrays.fill(candies, 1);          // floor: one candy each

    for (int i = 1; i < n; i++)                 // pass 1: rising from the left
        if (ratings[i] > ratings[i - 1])
            candies[i] = candies[i - 1] + 1;

    for (int i = n - 2; i >= 0; i--)            // pass 2: rising from the right
        if (ratings[i] > ratings[i + 1])
            candies[i] = Math.max(candies[i],   // max preserves pass 1's rule
                    candies[i + 1] + 1);

    int total = 0;
    for (int c : candies) total += c;
    return total;
}

Takeaway. The constraint "greater than BOTH neighbors when higher" splits into two one-directional constraints, each trivially solvable by a sweep — and Math.max merges the solutions without breaking either (raising a value never violates a greater-than rule). Decomposing a bidirectional constraint into two independent passes is the reusable idea; Trapping Rain Water's maxLeft/maxRight (page 02) is the same shape. Pitfalls

  • Comparator overflow: (a, b) -> a[0] - b[0] breaks when values straddle ±2·10⁹ (subtraction wraps). Always Integer.compare(a[0], b[0]) or Comparator.comparingInt(a -> a[0]).
  • Forgetting to sort (or sorting by the wrong key) is the #1 interval bug. Say the sort key and why before writing any loop code.
  • Touching intervals: decide up front whether [1,4] and [4,5] overlap. Merge Intervals: yes. Non-overlapping / Min Arrows: check the problem statement — Arrows counts touching as shared (<= keeps them together).
  • Commit the last block. Sweep loops end with an open current interval — the trailing out.add(cur) after the loop is forgotten constantly.
  • Greedy without a sentence of proof is a red flag. One line — "earliest end leaves the most room" / "furthest reach dominates any path" — plus one tested counterexample attempt is exactly the bar.
  • Meeting Rooms heap direction: it's a min-heap of end times and you poll at most one room per meeting (you only need to free one to reuse it). Java's PriorityQueue is already a min-heap — no comparator needed.

Bit Manipulation

Two's complement, Java's operators (including >>>), and the small bag of tricks — XOR cancellation, x & (x−1), masks — that turns "clever" screener questions into 5-minute wins.

Before you start

Why this topic exists. Bit questions aren't deep — they're a vocabulary test. Interviewers use them as fast screeners because each trick is either known (2 minutes) or unknown (unsolvable on the spot; nobody derives XOR cancellation under pressure). Core concepts

1. Two's complement in ninety seconds

A Java int is 32 bits. Non-negative numbers are plain binary; a negative number is stored as two's complement: -x == ~x + 1 (flip all bits, add one). Consequences worth knowing cold: the top bit is the sign bit; the range is asymmetric (−2³¹ … 2³¹−1, so -Integer.MIN_VALUE overflows back to itself!); and arithmetic "just works" without special cases — which is exactly why the x−1 and -x tricks below behave so nicely.

The >> vs >>> trap. Looping while (n != 0) { …; n >>= 1; } never terminates for negative n — the sign bit keeps refilling the top (−1 stays −1 forever). Use >>> when treating an int as 32 raw bits (Reverse Bits, Number of 1 Bits). C/C++ don't have >>>; it exists in Java precisely because Java lacks unsigned ints.

The single most useful trick deserves a demo. Kernighan's popcount — count 1-bits by deleting one per loop:

Pattern templates (Java)

The mask toolbox — memorize as a unit

// --- single-bit operations on position i (0 = least significant) ---
int setBit    = x |  (1 << i);        // turn bit i on
int clearBit  = x & ~(1 << i);        // turn bit i off  (~ makes the "all except i" mask)
int toggleBit = x ^  (1 << i);        // flip bit i
boolean isSet = ((x >> i) & 1) == 1;  // read bit i

// --- whole-number tricks ---
boolean isOdd  = (x & 1) == 1;
boolean isPow2 = x > 0 && (x & (x - 1)) == 0;  // exactly one set bit
int lowestBit  = x & (-x);                     // e.g. 10110100 -> 00000100

// --- int as a set (the bitmask-DP engine, page 20) ---
int add      = mask | (1 << i);       // set.add(i)
int remove   = mask & ~(1 << i);      // set.remove(i)
boolean has  = (mask & (1 << i)) != 0;// set.contains(i)  — note != 0, NOT == 1!
int size     = Integer.bitCount(mask);// the JDK's intrinsic popcount (use in real code)
// iterate all subsets of n items: for (int m = 0; m < (1 << n); m++) { ... }
In production, reach for the JDK. Integer.bitCount, Integer.lowestOneBit, Integer.reverse, Long.numberOfTrailingZeros — all compile to single CPU instructions (POPCNT etc.). Interviews want the manual versions once to prove you know why they work; say "in real code I'd call Integer.bitCount" for bonus credit.

Decision cue: if the problem says "at most 20 items" and asks about subsets → int-as-bitset (and probably bitmask DP, page 20). If it says "appears twice except one" or "missing from 0..n" → XOR. If it's genuinely arbitrary data → hashing (page 01); bit tricks are a memory/speed optimization, not a general replacement for sets.

Where this shows up in real systems

Permission & feature flags

Unix mode bits (rwxr-xr-- = 754) are three octal digits of masks — chmod is OR/AND-NOT. The same pattern powers feature-flag bitfields, protocol header flags (TCP SYN/ACK), and int modifiers in java.lang.reflect.Modifier — isPublic() is just (mod & PUBLIC) != 0.

Bloom filters & bitmap indexes

A Bloom filter (page 24) is k hash functions setting bits in one array — Cassandra/HBase consult it before touching disk. Analytics DBs (Druid, ClickHouse, Lucene) use Roaring bitmaps — compressed bitsets — to intersect "users matching filter A AND B" as bitwise ANDs over millions of rows per microsecond.

Networking: CIDR masks

10.0.0.0/16 means "the top 16 bits identify the network": a router tests (ip & mask) == network for every packet, billions of times a second. Subnetting, VPC route tables, and firewall rules are all AND-with-a-mask — the same x & ~(1 << i) algebra from the template above.

Game engines & crypto

Entity-component systems store each entity's components as a bitmask, so "all entities with Physics AND Render" is one AND per entity; collision layers work the same way. And every round of AES/SHA is built from XOR, shifts, and rotates — XOR's self-inverse property (c ^ key = m) is literally why stream ciphers decrypt.

Problems, fully worked (easy → hard)

easy Single Number — LeetCode 136 (XOR cancellation)

Problem in one line: every element appears exactly twice except one — find it in O(n) time, O(1) space.

Idea: the O(1)-space constraint rules out a HashSet, which is the interviewer's way of pointing at XOR. Because XOR is commutative and self-canceling (a^a=0, a^0=a), folding the whole array into one accumulator annihilates every pair regardless of order, leaving exactly the unpaired value.

code/22-bit-manipulation/SingleNumber.java

public static int singleNumber(int[] nums) {
    int acc = 0;                     // XOR identity: a ^ 0 = a
    for (int x : nums) {
        acc ^= x;                    // pairs cancel (a ^ a = 0); the loner survives
    }
    return acc;                      // all duplicates vanished — only the single remains
}

Complexity: time O(n), space O(1). Works for negatives too — XOR doesn't care what the bits mean.

easy Counting Bits — LeetCode 338 (the i >> 1 DP)

Problem in one line: for every i in 0..n, output how many 1-bits i has — in O(n) total, not O(n log n).

Idea: i >> 1 is i with its last bit chopped off — a strictly smaller number whose answer is already in the table. Gluing the chopped bit back on adds i & 1. That's a complete DP recurrence in one line, a tiny sibling of the 1-D DPs from page 18.

code/22-bit-manipulation/CountingBits.java

public static int[] countBits(int n) {
    int[] dp = new int[n + 1];               // dp[0] = 0: zero has no set bits (base case)
    for (int i = 1; i <= n; i++) {
        // i >> 1 < i, so dp[i >> 1] is already computed when we need it
        dp[i] = dp[i >> 1] + (i & 1);        // popcount of the prefix + the dropped last bit
    }
    return dp;
}

Complexity: time O(n) — constant work per entry; space O(n) for the answer itself.

easy Missing Number — LeetCode 268 (XOR version)

Problem in one line: nums contains n distinct numbers from the range 0..n; return the one that's missing.

Idea: XOR every index 0..n together with every array value. Each present number equals some index, so they pair up and cancel; the missing number is the only range member with no partner. Seeding the accumulator with n covers the index that has no array slot. Same effect as the Gauss sum n(n+1)/2 − sum, but with zero overflow concerns.

code/22-bit-manipulation/MissingNumber.java

public static int missingNumber(int[] nums) {
    int acc = nums.length;               // seed with n: the index that has no array slot
    for (int i = 0; i < nums.length; i++) {
        acc ^= i;                        // fold in the index 0..n-1
        acc ^= nums[i];                  // fold in the value; value==index pairs cancel out
    }
    return acc;                          // every number in 0..n appeared twice except the missing one
}

Complexity: time O(n), space O(1).

easy Number of 1 Bits — LeetCode 191

The problem. Count the set bits (1s) in a 32-bit integer's binary representation (the "population count" / Hamming weight).

Example. 11 (1011) → 3. 128 (10000000) → 1. −3 (two's complement 111...101) → 31.

Variation 1 — check all 32 positions · always 32 steps

public static int hammingWeightScan(int n) {
    int count = 0;
    for (int i = 0; i < 32; i++)
        count += (n >>> i) & 1;      // >>> (unsigned shift): works for negatives too
    return count;
}

Variation 2 — Kernighan: clear the lowest set bit (the animation above) · one step per 1-bit

public static int hammingWeight(int n) {
    int count = 0;
    while (n != 0) {
        n &= n - 1;    // n-1 flips the lowest 1 and everything under it;
        count++;       // AND-ing erases exactly that lowest 1
    }
    return count;      // loop ran once per set bit — not per bit position
}

Takeaway. n & (n−1) is the single most useful bit trick — it powers the power-of-two test (n & (n−1) == 0) and Counting Bits below. Guard the loop with != 0, not > 0: negative inputs are valid bit patterns. (Production code calls Integer.bitCount, which compiles to a hardware POPCNT.) Mirror: code/22-bit-manipulation/NumberOf1Bits.java.

easy Reverse Bits — LeetCode 190

The problem. Reverse the 32 bits of an integer: bit 0 swaps with bit 31, bit 1 with bit 30, and so on.

Example. 00000010100101000001111010011100 → 00111001011110000010100101000000.

Peel from one end, push onto the other · 32 steps

public static int reverseBits(int n) {
    int result = 0;
    for (int i = 0; i < 32; i++) {
        result <<= 1;              // make room at the bottom of the output
        result |= n & 1;           // move n's lowest bit into it
        n >>>= 1;                  // consume that bit (unsigned shift!)
    }
    return result;
}

Takeaway. The same two-pointer reversal idea as strings, played on bits: take from one end, append to the other. The >>> matters — a signed shift would smear the sign bit across negative inputs. Follow-up worth knowing: for many calls, reverse a byte at a time via a 256-entry lookup table. Mirror: code/22-bit-manipulation/ReverseBits.java.

medium Sum of Two Integers — LeetCode 371

The problem. Add two integers without ever using + or -.

Example. a=1, b=2 → 3. a=−2, b=3 → 1.

The half-adder loop: XOR = sum, AND<<1 = carry · ≤32 iterations

public static int getSum(int a, int b) {
    while (b != 0) {
        int carry = (a & b) << 1;  // positions where BOTH have 1 generate a
                                   // carry into the next column
        a = a ^ b;                 // XOR adds each column, ignoring carries
        b = carry;                 // now add the carries in — repeat until none
    }
    return a;
}

Takeaway. This is literally how a CPU's adder circuit works: XOR is a half-adder's sum line, AND its carry line, and the loop ripples carries until they vanish (at most 32 rounds). Two's complement makes negatives work for free — no special cases. A neat consequence: subtraction is getSum(a, getSum(~b, 1)), since −b = ~b + 1. Mirror: code/22-bit-manipulation/SumTwoIntegers.java.

medium Single Number II — LeetCode 137

The problem. Every element appears exactly three times except one, which appears once. Find it in O(n) time and O(1) space (plain XOR no longer works — triples don't cancel).

Example. [2,2,3,2] → 3. [0,1,0,1,0,1,99] → 99.

Variation 1 — count each bit position mod 3 · 32 passes of logic

public static int singleNumberBitCount(int[] nums) {
    int result = 0;
    for (int bit = 0; bit < 32; bit++) {
        int count = 0;
        for (int x : nums)
            count += (x >>> bit) & 1;      // how many numbers have this bit set?
        // Triples contribute multiples of 3; a remainder means the
        // single number owns this bit.
        if (count % 3 != 0)
            result |= 1 << bit;
    }
    return result;                          // works for negatives: bit 31 included
}

Variation 2 — two-variable state machine (elegant) · one pass

public static int singleNumber(int[] nums) {
    // Per bit position, (twos, ones) counts appearances mod 3:
    // 00 -> 01 -> 10 -> 00. The update formulas implement that cycle
    // for all 32 positions in parallel.
    int ones = 0, twos = 0;
    for (int x : nums) {
        ones = (ones ^ x) & ~twos;   // enter `ones` unless already seen twice
        twos = (twos ^ x) & ~ones;   // enter `twos` when leaving `ones`
    }
    return ones;                     // bits seen 3k+1 times = the single number
}

Takeaway. The generalization of XOR-cancellation: XOR counts bits mod 2; this counts them mod 3 (and mod-k variants follow the same recipe). The bit-count version is the one to derive under pressure; the state machine is the same arithmetic compressed into two registers. Mirror: code/22-bit-manipulation/SingleNumberII.java.

Pitfalls

  • Precedence bites: & binds looser than ==, so x & 1 == 1 parses as x & (1 == 1) — a compile error at best. Always parenthesize: (x & 1) == 1, (mask & bit) != 0.
  • Test with != 0, not == 1: mask & (1 << 5) yields 32, not 1. Comparing to 1 silently fails for every bit except bit 0.
  • >> on negatives never reaches 0 — use >>> for "loop over all 32 bits" logic, or Kernighan's loop which sidesteps shifting entirely.
  • 1 << 32 is 1, not 0: Java masks shift amounts to 5 bits for int (6 for long), so shifts wrap around. Building a long mask needs 1L << i — 1 << 40 is really 1 << 8.
  • Integer.MIN_VALUE edge: -x and Math.abs(x) both overflow for it (they return MIN_VALUE again). Power-of-two checks must exclude non-positives first: x > 0 && ….
  • Name the trick out loud ("pairs cancel under XOR", "this clears the lowest set bit") — bit solutions look like magic; one sentence of why turns magic into engineering, which is what's being graded.

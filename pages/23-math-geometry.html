Core concepts

1. GCD — Euclid's two-liner

Why it works: any number dividing both a and b also divides a − b, hence also a mod b. So gcd(a, b) = gcd(b, a mod b) — the pair shrinks fast (the argument at least halves every two steps ⇒ O(log min(a,b))). LCM comes free: lcm = a / gcd(a,b) * b — divide first to dodge overflow. 

static long gcd(long a, long b) {
    return b == 0 ? a : gcd(b, a % b);   // when b hits 0, a holds the answer
}

2. Primes — trial division vs the sieve

One number? Trial-divide by 2..√n — O(√n), because a factor above √n implies a partner below it. Many numbers (or "count primes below n")? The sieve of Eratosthenes: repeatedly take the next unmarked number (a prime) and cross out its multiples. Total work n/2 + n/3 + n/5 + … = O(n log log n) — effectively linear.

3. Modular arithmetic — surviving "answer mod 10⁹+7"

Counting problems overflow long almost immediately, so answers are requested modulo a large prime. The rules: take mod after every add and every multiply; multiply in long (two ints near 10⁹ overflow int on multiplication); and for subtraction use ((a - b) % M + M) % M because Java's % can return negatives. 

static final int MOD = 1_000_000_007;
long sum = (a + b) % MOD;               // a, b already reduced
long prod = (a * b) % MOD;              // a, b are longs — the product fits
long diff = ((a - b) % MOD + MOD) % MOD; // Java % keeps sign of the dividend!

4. Fast exponentiation — xⁿ in O(log n)

x¹³ isn't 13 multiplications: 13 = 1101₂, so x¹³ = x⁸·x⁴·x¹. Square-and-multiply: walk the exponent's bits (page 22!), squaring the base each step, multiplying it in when the bit is 1. This is also how RSA computes mod-exponents on 2048-bit numbers — the same 15-line loop.

5. Randomized algorithms — the two that get asked

Fisher-Yates shuffle: walk i from n−1 down to 1, swap a[i] with a random index in [0..i]. Every permutation gets probability 1/n! — provably uniform. The tempting "swap each i with any random j in [0..n−1]" is biased (it generates nⁿ equally likely outcomes, which n! doesn't divide evenly). Reservoir sampling: pick a uniform random item from a stream of unknown length in O(1) space — keep the k-th item with probability 1/k. Proof sketch: item i survives with (1/i)·(i/(i+1))·…·((n−1)/n) = 1/n. ∎

6. Geometry-lite — intervals per axis

Interview geometry is rarely trigonometry. Two axis-aligned rectangles overlap ⇔ their x-intervals overlap AND their y-intervals overlap — reuse page 21's interval logic per axis (a.left < b.right && b.left < a.right). Circle vs point: compare squared distances (dx*dx + dy*dy <= r*r) — no sqrt, no floating-point drama.

Pattern templates (Java)

Fast power with modulus — the exact loop to memorize

static long powMod(long base, long exp, long mod) {
    long result = 1;
    base %= mod;
    while (exp > 0) {
        if ((exp & 1) == 1)            // current bit of the exponent is 1
            result = result * base % mod;   // multiply this power of x in
        base = base * base % mod;      // x -> x^2 -> x^4 -> x^8 ...
        exp >>= 1;                     // next bit
    }
    return result;
}

nCk via Pascal's rule — safe small-scale combinatorics (it's just DP)

// C(n,k) = C(n-1,k-1) + C(n-1,k): "use item n or don't" — House Robber logic in disguise
long[][] c = new long[n + 1][n + 1];
for (int i = 0; i <= n; i++) {
    c[i][0] = 1;                                   // one way to choose nothing
    for (int j = 1; j <= i; j++)
        c[i][j] = (c[i-1][j-1] + c[i-1][j]) % MOD; // mod every add — factorials explode fast
}

Java randomness — which class when

Random r = new Random();          // interviews & reproducible tests (seedable)
ThreadLocalRandom.current();      // production hot paths — no contention between threads
new SecureRandom();               // tokens/crypto ONLY — slower, unpredictable
r.nextInt(bound);                 // uniform 0..bound-1 — never abuse Math.abs(nextInt() % bound): biased AND breaks on MIN_VALUE

Where this shows up in real systems

Cryptography runs on powMod

RSA encryption/decryption is modular exponentiation on 2048-bit numbers; Diffie-Hellman key exchange is g^secret mod p. The square-and-multiply loop above — plus BigInteger — is the beating heart of TLS handshakes securing every HTTPS request.

Sampling in observability & ML

Metrics agents keep a fair sample of latencies from an unbounded stream via reservoir sampling (it's in Dropwizard/Micrometer histograms); ML training pipelines Fisher-Yates-shuffle datasets every epoch so batches stay i.i.d.

Sharding & hashing

shard = hash(key) mod N is the first-cut partitioner in every distributed store (and why prime table sizes and good hash mixing matter); GCD/LCM logic sizes ring buffers and aligns rate windows (capstone).

Games, graphics & geo

Collision detection is per-axis interval overlap (AABB checks) exactly as above; map apps pre-filter "restaurants within 2 km" with squared-distance bounding boxes before any precise math; card games ship Fisher-Yates verbatim.

Worked solution: Count Primes (sieve of Eratosthenes) — Medium

Problem. Count primes strictly less than n (n up to 5·10⁶ — trial division per number is too slow).

Idea. One shared boolean array; every prime crosses out its multiples starting at p·p (smaller multiples were already crossed out by smaller primes). What survives is prime.

public static int countPrimes(int n) {
    if (n < 3) return 0;                       // no primes below 2
    boolean[] composite = new boolean[n];      // false = presumed prime
    int count = 0;
    for (int i = 2; i < n; i++) {
        if (composite[i]) continue;            // already crossed out — skip
        count++;                               // i survived every smaller prime -> prime
        // Cross out multiples starting at i*i: anything below that, like 2*i or 3*i,
        // was already handled by the smaller prime factor (2, 3, ...).
        for (long m = (long) i * i; m < n; m += i)  // long: i*i overflows int for i ~ 50k
            composite[(int) m] = true;
    }
    return count;
}

Complexity. Time O(n log log n), space O(n). Two crossable details interviewers look for: starting at i*i, and the long cast that prevents i*i overflowing int.

Worked solution: Pow(x, n) (fast exponentiation) — Medium

Problem. Implement xⁿ for integer n (which may be negative — or Integer.MIN_VALUE).

Idea. Square-and-multiply over n's bits: O(log n) multiplications. Negative n means 1/x^|n| — but |MIN_VALUE| doesn't fit in an int, so widen to long first.

public static double myPow(double x, int n) {
    long e = n;                       // widen BEFORE negating: -(MIN_VALUE) overflows int
    if (e < 0) {
        x = 1 / x;                    // x^-e = (1/x)^e
        e = -e;
    }
    double result = 1;
    while (e > 0) {
        if ((e & 1) == 1)             // this bit of the exponent is set
            result *= x;              // fold the current power of x into the answer
        x *= x;                       // x -> x^2 -> x^4 -> x^8 ...
        e >>= 1;                      // move to the next bit
    }
    return result;
}

Complexity. O(log n) time, O(1) space. Mentioning the MIN_VALUE widening unprompted is an instant credibility point — it's the entire reason this "easy" problem is rated Medium.

Worked solution: Shuffle an Array (Fisher-Yates) — Medium

Problem. Design a class with shuffle() returning a uniformly random permutation and reset() returning the original array.

Idea. Fisher-Yates: for i from the end down, swap a[i] with a random index in [0..i]. Each of the n! permutations is produced by exactly one random-choice sequence ⇒ uniform.

private final int[] original;          // pristine copy for reset()
private int[] current;
private final Random rand = new Random();

public ShuffleArray(int[] nums) {
    original = nums.clone();           // defensive copies — don't alias caller's array
    current = nums.clone();
}

public int[] reset() {
    current = original.clone();        // hand back (a copy of) the original order
    return current;
}

public int[] shuffle() {
    for (int i = current.length - 1; i > 0; i--) {
        int j = rand.nextInt(i + 1);   // uniform in [0..i] — the i+1 bound IS the algorithm
        int t = current[i]; current[i] = current[j]; current[j] = t;
    }
    return current;
}

Complexity. O(n) per shuffle, O(n) space for the pristine copy. If asked "why not rand.nextInt(n) every time?": that yields nⁿ equally likely runs, and n! doesn't divide nⁿ — some permutations must come up more often. The [0..i] bound is what makes it exactly n! outcomes.

Pitfalls & interview tips

  • Java's % can be negative: -7 % 3 == -1. Normalize with ((a % m) + m) % m whenever a subtraction is involved.
  • Multiply before you mod — in a long. Two ints ≈ 10⁹ multiply to ≈ 10¹⁸: fits long, destroys int. Declare accumulators long by reflex.
  • Math.pow is for doubles. Integer exponentiation via Math.pow silently loses precision around 2⁵³ — always the square-and-multiply loop for exact/modular work.
  • Math.abs can return a negative: Math.abs(Integer.MIN_VALUE) is still MIN_VALUE. This poisons "abs then mod" hashing tricks — mask instead.
  • Compare squared distances, never sqrt results — faster and immune to floating-point equality traps.
  • Say the sieve's complexity right: it's O(n log log n) — knowing that odd little bound (and starting the crossing at p·p) is the whole signal.

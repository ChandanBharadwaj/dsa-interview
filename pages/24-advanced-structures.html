Core concepts

1. Range queries on data that changes: segment tree

Prefix sums (page 04) answer "sum of a[i..j]" in O(1) — but one update forces an O(n) rebuild. When both query and update must be fast, a segment tree stores the answer for every "segment": the root covers the whole array, each node's children cover its two halves, leaves are single elements. Any range [i..j] can be assembled from O(log n) nodes, and an update only touches the O(log n) nodes on the leaf-to-root path.

Store it in a plain array with children at 2i+1 / 2i+2 — the same trick heaps use (page 12), sized 4n to be safe. Swap + for Math.min and the identity 0 for Integer.MAX_VALUE and you have a range-minimum tree — the skeleton is identical for any associative merge (sum, min, max, gcd).

2. Fenwick / Binary Indexed Tree: the lightweight cousin

For sum-like operations that are invertible (you can subtract), the Fenwick tree does the same job in ~10 lines and n+1 ints. It is 1-based; slot i stores the sum of the last i & -i elements ending at i (the lowest-set-bit trick from page 22). Prefix query: keep stripping the lowest set bit. Update: keep adding it. Both walk O(log n) slots. Range sum is then just prefix(r) − prefix(l−1), exactly like page 04. Reach for a segment tree only when the operation can't be inverted (min/max) or you need fancier queries.

3. TreeMap patterns: nearest-neighbor lookups

TreeMap (a red-black tree, page 11) is the interview answer whenever you need order + lookup: "the closest key ≤ x" is floorEntry(x), "the next key ≥ x" is ceilingEntry(x), all O(log n). The killer application is the interval map: store bookings as start → end, and a new interval conflicts only with its floor neighbor (which might still be running) or its ceiling neighbor (which might start too soon) — two lookups replace a scan of every interval. That's the whole "My Calendar" pattern, worked below.

4. The LRU cache — the flagship

Spec: get(key) and put(key, value) in O(1), evicting the least-recently-used entry when capacity is exceeded. A HashMap alone gives O(1) lookup but has no idea what "least recent" is; a list alone knows the order but takes O(n) to find a key. The classic composition: a HashMap that points at nodes of a hand-rolled doubly linked list. The map finds the node in O(1); the list moves it to the front or evicts from the tail in O(1) — a doubly linked list is the only structure that can unlink a middle node in O(1), if you already hold a reference to it, which is precisely what the map provides.

The "cheat" version — LinkedHashMap does all of this for you

// LinkedHashMap = hash map + doubly linked list ALREADY WIRED TOGETHER.
// Constructor flag accessOrder=true makes iteration order = recency order,
// and removeEldestEntry() is a built-in eviction hook. Mention this version
// to show JDK fluency — then build the manual one, because that is the test.
class LruViaLinkedHashMap extends LinkedHashMap<Integer, Integer> {
    private final int capacity;
    LruViaLinkedHashMap(int capacity) {
        super(16, 0.75f, true);          // true = order by ACCESS, not insertion
        this.capacity = capacity;
    }
    @Override
    protected boolean removeEldestEntry(Map.Entry<Integer, Integer> eldest) {
        return size() > capacity;        // auto-evict after each put
    }
}

5. Quick tour of the rest of the arsenal

  • LFU cache (concept): evict the least-frequently-used instead. O(1) design: a map key→node, a map frequency→doubly-linked-list of nodes with that frequency, plus a running minFreq pointer. On access, a node migrates from list f to list f+1. Same building blocks as LRU, one more layer.
  • O(1) randomized set: getRandom() needs dense indexing (array), remove() needs O(1) location (map value→index). Deleting from the middle would shift — so swap the victim with the last element and pop. Worked below.
  • Rate limiters: a token bucket refills tokens at rate r up to a burst cap — O(1) memory, allows bursts; a sliding window log keeps a queue of recent timestamps and counts them (page 03's window, literally) — exact but O(requests) memory. Interviews love asking which and why.
  • Bloom filter: a bit array + k hash functions. add sets k bits; mightContain checks them. It can say "definitely not present" or "probably present" — false positives only, never false negatives, and you tune the size/k for a target error rate (~10 bits per key ≈ 1% at k≈7). Perfect as a cheap gate in front of an expensive lookup.
  • Skip list (concept): a sorted linked list with express lanes — each node is promoted to the next level with probability ½, giving O(log n) search/insert without rotation logic. This is how Redis implements sorted sets.
Design-question reflex. Read the required operations and their target costs FIRST, then pick one structure per requirement and wire them together: "O(1) lookup" → hash map; "ordered / nearest" → TreeMap; "cheapest next" → heap; "O(1) reorder or evict" → doubly linked list; "random element" → array. Naming the contract before coding is exactly what a senior hire does.

Pattern templates (Java)

Segment tree — range sum + point update, the full skeleton

public class SegmentTree {

    // Array-backed segment tree for range SUM. Node i covers a segment [lo..hi];
    // its children 2i+1 / 2i+2 cover the two halves (same trick as heaps, page 12).
    // 4n slots is always enough, even for non-power-of-two n.
    private final int[] tree;
    private final int n;

    public SegmentTree(int[] nums) {
        n = nums.length;
        tree = new int[4 * n];
        build(0, 0, n - 1, nums);
    }

    private void build(int node, int lo, int hi, int[] nums) {
        if (lo == hi) { tree[node] = nums[lo]; return; }        // leaf = one element
        int mid = lo + (hi - lo) / 2;
        build(2 * node + 1, lo, mid, nums);                     // left child = left half
        build(2 * node + 2, mid + 1, hi, nums);                 // right child = right half
        tree[node] = tree[2 * node + 1] + tree[2 * node + 2];   // parent = merge of children
    }

    public void update(int index, int val) { update(0, 0, n - 1, index, val); }

    private void update(int node, int lo, int hi, int index, int val) {
        if (lo == hi) { tree[node] = val; return; }             // reached the leaf
        int mid = lo + (hi - lo) / 2;
        if (index <= mid) update(2 * node + 1, lo, mid, index, val);
        else              update(2 * node + 2, mid + 1, hi, index, val);
        tree[node] = tree[2 * node + 1] + tree[2 * node + 2];   // re-merge on the way up
    }

    public int query(int left, int right) { return query(0, 0, n - 1, left, right); }

    private int query(int node, int lo, int hi, int left, int right) {
        if (right < lo || hi < left) return 0;                  // no overlap: sum identity
        if (left <= lo && hi <= right) return tree[node];       // fully covered: cached sum
        int mid = lo + (hi - lo) / 2;                           // partial overlap: recurse
        return query(2 * node + 1, lo, mid, left, right)
             + query(2 * node + 2, mid + 1, hi, left, right);
    }
}

TreeMap nearest-neighbor idioms

TreeMap<Integer, Integer> m = new TreeMap<>();
m.floorKey(x);     // greatest key <= x, or null      — "the booking before mine"
m.ceilingKey(x);   // smallest key >= x, or null      — "the booking after mine"
m.firstKey();      // global minimum (m.lastKey() = maximum)
m.floorEntry(x);   // same but returns the (key, value) pair
// Null checks are mandatory: every one of these returns null at the edges.

Token-bucket rate limiter — the O(1)-memory design

class TokenBucket {
    private final double ratePerSec, burst;   // refill speed and bucket cap
    private double tokens; private long last; // current tokens, last refill time
    TokenBucket(double ratePerSec, double burst) {
        this.ratePerSec = ratePerSec; this.burst = burst;
        this.tokens = burst; this.last = System.nanoTime();
    }
    synchronized boolean allow() {
        long now = System.nanoTime();
        // Lazy refill: add tokens for the elapsed time, capped at burst.
        tokens = Math.min(burst, tokens + (now - last) / 1e9 * ratePerSec);
        last = now;
        if (tokens < 1) return false;         // bucket empty -> reject/throttle
        tokens -= 1;                          // spend one token for this request
        return true;
    }
}

Where this shows up in real systems

Redis

Sorted sets are skip lists; eviction policies are literally allkeys-lru and allkeys-lfu (an approximated LRU/LFU over sampled keys). The structure you build by hand in this page's flagship problem is a production eviction engine.

Caches at every layer

CDN edges, browser caches, database buffer pools and your service's in-process cache all pair an O(1) index with a recency/frequency list. InnoDB's buffer pool is a tuned LRU with a midpoint insertion hack to survive scans.

Bloom filters in storage engines

Cassandra, HBase and RocksDB attach a bloom filter to each SSTable so a read can skip files that definitely lack the key; Chrome historically shipped a bloom filter of malicious URLs for Safe Browsing. False positives cost a lookup; false negatives never happen.

Order books & API gateways

Exchange matching engines keep price levels in tree maps — best bid/ask is firstKey(), and "match against the closest price" is a floor/ceiling query. In front of it all, API gateways (Kong, Envoy, cloud rate limiting) run token buckets per client key.

easy Design HashMap — LeetCode 706

The problem. Implement put/get/remove for integer keys without using built-in hash tables.

Example. put(1,1), put(2,2), get(1) → 1, put(2,1) overwrites, remove(2), get(2) → −1.

Separate chaining: bucket array + linked chains · O(1) average

private static final int BUCKETS = 1024;
private final Node[] table = new Node[BUCKETS];

private int index(int key) {
    return Integer.hashCode(key) & (BUCKETS - 1);   // power-of-two mask
}

public void put(int key, int value) {
    int i = index(key);
    for (Node n = table[i]; n != null; n = n.next)
        if (n.key == key) {         // key exists: overwrite in place
            n.value = value;
            return;
        }
    table[i] = new Node(key, value, table[i]);      // else prepend a node
}

public void remove(int key) {
    int i = index(key);
    Node prev = null;
    for (Node n = table[i]; n != null; prev = n, n = n.next)
        if (n.key == key) {
            if (prev == null) table[i] = n.next;    // chain head
            else prev.next = n.next;                // splice out mid-chain
            return;
        }
}

Takeaway. This is the data structure the whole runbook leans on, built from parts: hash → bucket → chain walk, with collisions handled by chaining (keys 1024 apart in the mirror share a bucket). What production adds: resizing at a load factor (~0.75), treeifying long chains (Java 8+), and better hash mixing. Being able to sketch this in five minutes is the difference between using HashMap and understanding it. Mirror: code/24-advanced-structures/DesignHashMap.java.

medium LRU Cache — LeetCode 146 (the flagship)

Problem in one line: build a fixed-capacity key→value cache where get and put run in O(1) and the least-recently-used entry is evicted on overflow.

Idea: a HashMap maps each key to a node in a doubly linked list ordered by recency. Every access unlinks the node and relinks it behind the HEAD sentinel; eviction takes the node before the TAIL sentinel. The two sentinels mean insert/remove never branch on nulls, and nodes carry their key so an evicted node can delete its own map entry.

public class LRUCache {

    // Doubly-linked-list node. It stores the KEY as well as the value:
    // when we evict from the tail we must also delete the map entry,
    // and the key is the only way to find it.
    private static class Node {
        int key, value;
        Node prev, next;
        Node(int key, int value) { this.key = key; this.value = value; }
    }

    private final int capacity;
    private final Map<Integer, Node> map = new HashMap<>();  // key -> its node, O(1) locate
    // Sentinel head/tail: real nodes live strictly between them, so unlink/link
    // NEVER needs a null check. This one trick removes 90% of the edge-case bugs.
    private final Node head = new Node(0, 0);   // head.next  = most-recently-used
    private final Node tail = new Node(0, 0);   // tail.prev  = least-recently-used

    public LRUCache(int capacity) {
        this.capacity = capacity;
        head.next = tail;                        // empty list: sentinels point at each other
        tail.prev = head;
    }

    public int get(int key) {
        Node n = map.get(key);
        if (n == null) return -1;                // miss
        moveToFront(n);                          // this access makes it most-recent
        return n.value;
    }

    public void put(int key, int value) {
        Node n = map.get(key);
        if (n != null) {                         // key exists: update value, refresh recency
            n.value = value;
            moveToFront(n);
            return;
        }
        if (map.size() == capacity) {            // full: evict the least-recently-used
            Node lru = tail.prev;                // the real node just before the tail sentinel
            unlink(lru);                         // O(1) removal from the list...
            map.remove(lru.key);                 // ...and from the map — why nodes store keys
        }
        Node fresh = new Node(key, value);
        map.put(key, fresh);
        linkFront(fresh);                        // new entries are most-recent by definition
    }

    // Splice a node out of the list. Sentinels guarantee prev/next are non-null.
    private void unlink(Node n) {
        n.prev.next = n.next;
        n.next.prev = n.prev;
    }

    // Insert right after the head sentinel — the most-recent position.
    private void linkFront(Node n) {
        n.next = head.next;
        n.prev = head;
        head.next.prev = n;
        head.next = n;
    }

    private void moveToFront(Node n) {
        unlink(n);
        linkFront(n);
    }
}

Complexity: time O(1) for both operations (map lookup + a constant number of pointer writes); space O(capacity) for the map and list.

medium Insert Delete GetRandom O(1) — LeetCode 380

Problem in one line: a set supporting insert, remove and getRandom (uniform) all in average O(1).

Idea: uniform randomness needs dense integer indexing → ArrayList; O(1) removal needs to know where a value lives → HashMap value→index. Removing from the middle of an array is O(n), so instead overwrite the victim's slot with the last element and pop the tail — order doesn't matter in a set, so nobody notices the swap.

public class RandomizedSet {

    // The trick: an ArrayList gives O(1) access at a random index, and a
    // HashMap (value -> its index in the list) gives O(1) location for delete.
    // Delete = swap-with-last + remove last, so nothing ever shifts.
    private final List<Integer> values = new ArrayList<>();
    private final Map<Integer, Integer> indexOf = new HashMap<>();
    private final Random rnd = new Random();

    public boolean insert(int val) {
        if (indexOf.containsKey(val)) return false;   // already present
        indexOf.put(val, values.size());              // it will sit at the end
        values.add(val);
        return true;
    }

    public boolean remove(int val) {
        Integer i = indexOf.get(val);
        if (i == null) return false;                  // not present
        int last = values.get(values.size() - 1);
        values.set(i, last);                          // fill the hole with the LAST element
        indexOf.put(last, i);                         // ...and record its new position
        values.remove(values.size() - 1);             // removing the tail is O(1)
        indexOf.remove(val);                          // (works even when val WAS the last)
        return true;
    }

    public int getRandom() {
        // Uniform because the list is dense: every slot holds a live value.
        return values.get(rnd.nextInt(values.size()));
    }
}

Complexity: time average O(1) per operation; space O(n) for the list and map.

medium Range Sum Query — Mutable — LeetCode 307 (Fenwick tree)

Problem in one line: support update(index, val) and sumRange(left, right) on an array, both fast.

Idea: a Fenwick tree keeps partial sums so that both a prefix query and a point update touch only O(log n) slots, navigated by adding or stripping the lowest set bit (i & -i, page 22). A range sum is the difference of two prefix sums, exactly like page 04 — but now updates are cheap too.

public class NumArray {

    // Fenwick tree / Binary Indexed Tree, 1-based. tree[i] stores the sum of
    // the (i & -i) elements ending at position i — that lowest-set-bit trick
    // from page 22 is the whole data structure.
    private final int[] tree;
    private final int[] nums;   // current values, kept so update() can compute a delta
    private final int n;

    public NumArray(int[] nums) {
        this.n = nums.length;
        this.nums = new int[n];
        this.tree = new int[n + 1];               // index 0 unused: 0 has no lowest set bit
        for (int i = 0; i < n; i++) update(i, nums[i]);   // n * O(log n) build
    }

    public void update(int index, int val) {
        int delta = val - nums[index];            // BIT stores sums, so apply the CHANGE
        nums[index] = val;
        for (int i = index + 1; i <= n; i += i & (-i))    // hop to every covering range
            tree[i] += delta;
    }

    public int sumRange(int left, int right) {
        return prefix(right + 1) - prefix(left);  // classic prefix-sum subtraction (page 04)
    }

    // Sum of the first i elements: strip the lowest set bit until 0.
    private int prefix(int i) {
        int s = 0;
        for (; i > 0; i -= i & (-i))
            s += tree[i];
        return s;
    }
}

Complexity: time O(log n) per update and per query (build O(n log n)); space O(n).

medium My Calendar I — LeetCode 729 (TreeMap neighbors)

Problem in one line: accept a booking [start, end) only if it overlaps no previously accepted booking.

Idea: keep bookings in a TreeMap keyed by start. A candidate can only collide with two neighbors: the latest booking starting at or before it (must have ended by our start) and the earliest booking starting at or after it (must start at or after our end). Two O(log n) lookups replace an O(n) scan — the archetypal interval-map pattern.

public class MyCalendar {

    // start -> end of every booked interval [start, end), kept sorted by start.
    // The red-black tree behind TreeMap makes "nearest interval" an O(log n) question.
    private final TreeMap<Integer, Integer> booked = new TreeMap<>();

    public boolean book(int start, int end) {
        // Nearest booking starting AT OR BEFORE us: it must end before we begin.
        Map.Entry<Integer, Integer> before = booked.floorEntry(start);
        if (before != null && before.getValue() > start) return false;
        // Nearest booking starting AT OR AFTER us: it must start after we end.
        Map.Entry<Integer, Integer> after = booked.ceilingEntry(start);
        if (after != null && after.getKey() < end) return false;
        booked.put(start, end);        // no conflict on either side -> safe to book
        return true;
    }
}

Complexity: time O(log n) per booking; space O(n) for the map.

medium Design Hit Counter — LeetCode 362

The problem. Count hits in the past 5 minutes (300 seconds): hit(t) records a hit at timestamp t, getHits(t) returns hits in (t−300, t]. Timestamps arrive in non-decreasing order.

Example. hit(1), hit(2), hit(3) → getHits(4) = 3; hit(300) → getHits(300) = 4; getHits(301) = 3 (the hit at t=1 expired).

Variation 1 — queue of timestamps (sliding-window log) · O(1) amortized, memory ∝ hits

public class HitCounterQueue {
    private final Deque<Integer> hits = new ArrayDeque<>();

    public void hit(int timestamp) { hits.offer(timestamp); }

    public int getHits(int timestamp) {
        // Evict everything older than the window (page 03's sliding log)
        while (!hits.isEmpty() && hits.peek() <= timestamp - 300)
            hits.poll();
        return hits.size();
    }
}

Variation 2 — 300 circular buckets · O(1), memory O(300) regardless of traffic

public class HitCounter {
    // One bucket per second of the window. times[i] remembers WHICH second
    // the bucket currently represents, so stale buckets self-invalidate.
    private final int[] times = new int[300];
    private final int[] counts = new int[300];

    public void hit(int timestamp) {
        int i = timestamp % 300;             // this second's bucket
        if (times[i] != timestamp) {         // bucket still holds an OLD second
            times[i] = timestamp;            // claim it for the current second
            counts[i] = 0;
        }
        counts[i]++;
    }

    public int getHits(int timestamp) {
        int total = 0;
        for (int i = 0; i < 300; i++)
            if (timestamp - times[i] < 300)  // bucket's second is inside the window
                total += counts[i];
        return total;
    }
}

Takeaway. The queue is exact but grows with traffic; the circular buckets cap memory at the window size by trading per-hit storage for per-second aggregation — precisely the sliding-window-log vs sliding-window-counter trade in production rate limiters (Redis, Envoy). "Millions of hits per second?" → buckets. Mirror: code/24-advanced-structures/HitCounter.java.

medium Time Based Key-Value Store — LeetCode 981

The problem. set(key, value, timestamp) stores a value version; get(key, timestamp) returns the value with the largest timestamp ≤ the query ("" if none). Set timestamps are strictly increasing per key.

Example. set("foo","bar",1); get("foo",1) → "bar"; get("foo",3) → "bar"; set("foo","bar2",4); get("foo",4) → "bar2"; get("foo",3) → "bar" still.

Per-key version list + floor binary search · set O(1), get O(log n)

public class TimeMap {
    // key -> its versions, appended in timestamp order (so each list is
    // ALREADY SORTED — that's what makes binary search legal).
    private final Map<String, List<int[]>> times = new HashMap<>();   // {timestamp}
    private final Map<String, List<String>> vals = new HashMap<>();

    public void set(String key, String value, int timestamp) {
        times.computeIfAbsent(key, k -> new ArrayList<>()).add(new int[]{timestamp});
        vals.computeIfAbsent(key, k -> new ArrayList<>()).add(value);
    }

    public String get(String key, int timestamp) {
        List<int[]> ts = times.get(key);
        if (ts == null) return "";
        // Floor search: LAST index with ts[i] <= timestamp.
        int lo = 0, hi = ts.size() - 1, ans = -1;
        while (lo <= hi) {
            int mid = lo + (hi - lo) / 2;
            if (ts.get(mid)[0] <= timestamp) {
                ans = mid;                   // candidate; maybe a later one fits too
                lo = mid + 1;
            } else {
                hi = mid - 1;
            }
        }
        return ans == -1 ? "" : vals.get(key).get(ans);
    }
}

Takeaway. "Latest version at or before time T" is a floor query — binary search on the append-only (hence sorted) version list, or equivalently TreeMap.floorEntry per key. This is MVCC in miniature: versioned KV stores (etcd, DynamoDB point-in-time reads) answer exactly this query shape. Mirror: code/24-advanced-structures/TimeMap.java.

medium Design Circular Queue — LeetCode 622

The problem. A fixed-capacity FIFO queue in an array where the tail wraps around to reuse freed front slots: enQueue, deQueue, Front, Rear, isEmpty, isFull.

Example. capacity 3: enQueue 1,2,3; enQueue(4) → false; deQueue; enQueue(4) → true (wrapped into slot 0).

head + size, indices mod capacity · all ops O(1)

private final int[] data;
private int head = 0;               // index of the front element
private int size = 0;               // storing SIZE avoids the full/empty ambiguity

public boolean enQueue(int value) {
    if (size == data.length) return false;          // full
    data[(head + size) % data.length] = value;      // one past the tail
    size++;
    return true;
}

public boolean deQueue() {
    if (size == 0) return false;
    head = (head + 1) % data.length;                // front slides right
    size--;
    return true;
}

public int Rear() {
    return size == 0 ? -1 : data[(head + size - 1) % data.length];
}

Takeaway. The design decision worth voicing: with only head and tail pointers, "head == tail" is ambiguous (empty or full?) — classic fixes are storing the size (done here) or sacrificing one slot. Ring buffers are everywhere real systems need bounded queues without allocation: ArrayDeque's internals, Kafka partitions, audio pipelines, and kernel I/O rings. Mirror: code/24-advanced-structures/CircularQueue.java.

medium Design Underground System — LeetCode 1396

The problem. checkIn(id, station, t), checkOut(id, station, t), and getAverageTime(start, end) — the average travel time over all completed trips on that exact route.

Example. Trips Leyton→Waterloo of 12, 10, 14 minutes → getAverageTime("Leyton","Waterloo") = 12.0.

Open journeys map + per-route running aggregates · all ops O(1)

private final Map<Integer, String> inStation = new HashMap<>();  // id -> start
private final Map<Integer, Integer> inTime = new HashMap<>();    // id -> t
private final Map<String, long[]> routes = new HashMap<>();      // "a->b" -> {sum, count}

public void checkOut(int id, String stationName, int t) {
    String route = inStation.remove(id) + "->" + stationName;
    int started = inTime.remove(id);
    long[] agg = routes.computeIfAbsent(route, k -> new long[2]);
    agg[0] += t - started;          // running total — never store all trips
    agg[1]++;                       // trip count
}

public double getAverageTime(String startStation, String endStation) {
    long[] agg = routes.get(startStation + "->" + endStation);
    return (double) agg[0] / agg[1];
}

Takeaway. Two small decisions carry the design: keep running aggregates (sum, count) instead of trip lists — averages stay O(1) and memory stays flat — and remove() on checkout both reads and cleans in one call. This open-session + aggregate shape is the skeleton of every metrics pipeline ("request started / request finished / p50 by route"). A favorite "can you write clean code fast?" screen. Mirror: code/24-advanced-structures/UndergroundSystem.java.

hard LFU Cache — LeetCode 460

The problem. Like LRU, but evict the least frequently used key; ties break by least-recent use. get and put must both be O(1).

Example. capacity 2: put(1,1), put(2,2), get(1) → 1 makes key 1's count 2; put(3,3) evicts key 2 (count 1 < 2); get(2) → −1.

Frequency buckets of LinkedHashSets + a minFreq pointer · all ops O(1)

public class LFUCache {
    private final int capacity;
    private final Map<Integer, Integer> values = new HashMap<>();
    private final Map<Integer, Integer> freq = new HashMap<>();   // key -> use count
    // freq -> keys with that count, in LRU order (LinkedHashSet keeps
    // insertion order, giving us the recency tie-break for free).
    private final Map<Integer, LinkedHashSet<Integer>> buckets = new HashMap<>();
    private int minFreq = 0;              // smallest frequency present in the cache

    public LFUCache(int capacity) { this.capacity = capacity; }

    public int get(int key) {
        if (!values.containsKey(key)) return -1;
        touch(key);                       // bump its frequency
        return values.get(key);
    }

    public void put(int key, int value) {
        if (capacity == 0) return;
        if (values.containsKey(key)) {    // update existing: new value + bump
            values.put(key, value);
            touch(key);
            return;
        }
        if (values.size() == capacity) {
            // Evict: the LRU member of the minimum-frequency bucket.
            int evict = buckets.get(minFreq).iterator().next();  // oldest inserted
            buckets.get(minFreq).remove(evict);
            values.remove(evict);
            freq.remove(evict);
        }
        values.put(key, value);
        freq.put(key, 1);
        buckets.computeIfAbsent(1, k -> new LinkedHashSet<>()).add(key);
        minFreq = 1;                      // a brand-new key resets the floor
    }

    // Move key from bucket f to bucket f+1, maintaining minFreq.
    private void touch(int key) {
        int f = freq.get(key);
        buckets.get(f).remove(key);
        // If that emptied the minimum bucket, the floor rises by exactly 1
        if (f == minFreq && buckets.get(f).isEmpty()) minFreq++;
        freq.put(key, f + 1);
        buckets.computeIfAbsent(f + 1, k -> new LinkedHashSet<>()).add(key);
    }
}

Takeaway. LFU = LRU's structure one level up: instead of one recency list, keep one per frequency, plus a minFreq floor. The subtle invariant: minFreq only ever rises by 1 on a bump (the touched key came from the min bucket) and resets to 1 on inserts — that's why no scan for the new minimum is needed. Redis offers exactly this policy as allkeys-lfu. Mirror: code/24-advanced-structures/LFUCache.java.

hard Range Module — LeetCode 715

The problem. Track half-open ranges of numbers: addRange(left, right), queryRange (is every point of [left, right) tracked?), removeRange. All must stay fast under heavy mixing.

Example. add [10,20), remove [14,16): query [10,14) → true, [13,15) → false, [16,17) → true.

TreeMap of disjoint intervals; touch only neighbors · O(log n) amortized

private final TreeMap<Integer, Integer> map = new TreeMap<>();  // start -> end

public void addRange(int left, int right) {
    Map.Entry<Integer, Integer> e = map.floorEntry(left);
    if (e != null && e.getValue() >= left) {        // left neighbor reaches us
        left = Math.min(left, e.getKey());          // absorb it
        right = Math.max(right, e.getValue());
        map.remove(e.getKey());
    }
    while ((e = map.ceilingEntry(left)) != null && e.getKey() <= right) {
        right = Math.max(right, e.getValue());      // swallow overlappers
        map.remove(e.getKey());
    }
    map.put(left, right);                           // insert the merged blob
}

public boolean queryRange(int left, int right) {
    Map.Entry<Integer, Integer> e = map.floorEntry(left);
    return e != null && e.getValue() >= right;      // ONE interval must cover it
}

public void removeRange(int left, int right) {
    Map.Entry<Integer, Integer> e = map.floorEntry(left);
    if (e != null && e.getValue() > left) {
        int end = e.getValue();
        map.put(e.getKey(), left);                  // keep the left stub
        if (end > right) map.put(right, end);       // and a right stub if any
    }
    while ((e = map.ceilingEntry(left)) != null && e.getKey() < right) {
        if (e.getValue() > right)
            map.put(right, e.getValue());           // right stub survives
        map.remove(e.getKey());
    }
}

Takeaway. The culmination of the TreeMap-interval pattern (My Calendar's grown-up sibling): keep tracked intervals disjoint, and every operation reduces to floor/ceiling neighbor surgery — merge on add, stub-trim on remove, single-cover check on query. Because the invariant holds, query is one floorEntry. Interval maps like this manage allocated memory ranges, file locks, and calendar availability in real systems. The mirror randomizes 500 mixed ops against a boolean-array oracle. Mirror: code/24-advanced-structures/RangeModule.java.

Pitfalls

  • Store the key inside the LRU node. The #1 LRU bug: you evict tail.prev and then can't remove its map entry because the node only held a value.
  • Use sentinel head/tail nodes. Without them, linking at an empty/one-element list needs four special cases; with them, unlink/linkFront are branch-free.
  • Remember that get mutates. In LRU, a read reorders the list; forgetting moveToFront in get passes the sample test and fails the hidden ones.
  • Fenwick trees are 1-based. Convert once at the API boundary (index + 1) and never think about it again; mixing conventions mid-function is the classic off-by-one.
  • TreeMap navigation calls return null at the edges — floorEntry/ceilingEntry need null checks before dereferencing, or your calendar crashes on its first booking.
  • Say the contract out loud: "get must be O(1), so a TreeMap's O(log n) is already disqualified" — narrating cost targets is how design rounds are scored.

Real-World Systems: Where All of It Lives

Seven systems you use every day, decomposed into the exact structures and algorithms from pages 00–24. This is the retention page: every pattern gets a face.

Phase 9 · Week 14 The payoff page — retention through stories

You now know every tool; here's how real systems compose them. Nothing on this page is new — that's the point. A recommendation feed is hashing + graphs + heaps. Navigation is Dijkstra with a heuristic. Git is a hash-addressed DAG plus LCS. When a system-design interviewer asks "how would you build X?", they are asking which pages 01–24 you'd bolt together and why. Each section below gives the architecture, the exact DSA inventory (with page links), and honest notes on what production adds beyond the interview version.

1. Recommendation systems (flagship deep-dive)

Netflix's "Because you watched…", YouTube's Up Next, Amazon's "Customers also bought" — all are the same four-stage funnel. You cannot score 10 million items per user per request, so the pipeline's whole job is shrinking the candidate set cheaply before spending money on precision.

Stage by stage, in your vocabulary

Candidate generation starts from the user-item bipartite graph: users on one side, items on the other, an edge per watch (page 15). "Users who watched X also watched Y" is a two-hop neighbor traversal — from X to its watchers, from them to everything else they watched — with a HashMap counting how often each Y appears (frequency counting, page 01). Those co-visitation counts feed item-item collaborative filtering: two items are similar when their watcher sets overlap a lot, measured with Jaccard (|A∩B| / |A∪B|) or cosine similarity — both computed from the same hash-set machinery. Other candidate sources run in parallel: sliding-window trending counts over the last N hours (page 03), and the search box's trie-powered typeahead with per-node cached top-K completions (page 13).

Filtering removes what must never surface: already-watched titles (a bloom filter per user makes this check nearly free — false positives just hide a rewatch, an acceptable error; page 24), region blocks, age ratings. Ranking scores the surviving ~1000 candidates with an ML model, then a size-K min-heap keeps the best K in O(m log k) (page 12). Re-ranking applies business rules — diversity, freshness, "no three sequels in a row" — small heap- and rule-based shuffles over ~100 items.

Scale honesty: counting exact co-visitation over all pairs is quadratic and infeasible at catalog scale. Production embeds items as vectors and uses approximate nearest-neighbor search; MinHash and locality-sensitive hashing make "similar sets collide in the same bucket" — literally a hash function designed to collide (the honest extension of page 01), and modern ANN indexes (HNSW) are graphs navigated greedily (page 15).

A working miniature

The whole skeleton — co-visitation counting, seen-filtering, heap top-K — fits in one small class. This runs (see code/25-real-world-systems/MiniRecommender.java) and recommends sensibly on a toy watch log: 

public class MiniRecommender {

    // Co-visitation matrix: item -> (other item -> number of users who watched both).
    // This IS the "users who watched X also watched Y" signal, stored as nested hashing (page 01).
    private final Map<String, Map<String, Integer>> coVis = new HashMap<>();
    // Watch history: user -> items already seen. Doubles as our "don't re-recommend"
    // filter — production uses a bloom filter (page 24) for exactly this role.
    private final Map<String, Set<String>> watched = new HashMap<>();

    public void recordWatch(String user, String item) {
        Set<String> items = watched.computeIfAbsent(user, u -> new LinkedHashSet<>());
        if (!items.add(item)) return;             // duplicate event, nothing new to learn
        for (String other : items) {              // pair the new item with the user's history:
            if (other.equals(item)) continue;     // one hop through the user-item bipartite
            bump(item, other);                    // graph (page 15), counted with a HashMap
            bump(other, item);                    // co-visitation is symmetric
        }
    }

    private void bump(String a, String b) {
        coVis.computeIfAbsent(a, k -> new HashMap<>()).merge(b, 1, Integer::sum);
    }

    public List<String> recommend(String user, int k) {
        Set<String> seen = watched.getOrDefault(user, Set.of());
        // Candidate generation + scoring: aggregate co-visitation counts over
        // everything the user has watched (frequency counting, page 01).
        Map<String, Integer> score = new HashMap<>();
        for (String item : seen)
            for (Map.Entry<String, Integer> e : coVis.getOrDefault(item, Map.of()).entrySet())
                if (!seen.contains(e.getKey()))   // filtering stage: drop already-watched
                    score.merge(e.getKey(), e.getValue(), Integer::sum);
        // Ranking stage: top-K with a size-K MIN-heap (page 12). The worst of the
        // current best K sits on top and is evicted when something better arrives:
        // O(m log k) over m candidates instead of sorting everything at O(m log m).
        PriorityQueue<Map.Entry<String, Integer>> heap = new PriorityQueue<>(
            (a, b) -> a.getValue().equals(b.getValue())
                ? b.getKey().compareTo(a.getKey())            // tie-break so results are stable
                : Integer.compare(a.getValue(), b.getValue()));
        for (Map.Entry<String, Integer> e : score.entrySet()) {
            heap.offer(e);
            if (heap.size() > k) heap.poll();     // drop the current worst candidate
        }
        LinkedList<String> out = new LinkedList<>();
        while (!heap.isEmpty()) out.addFirst(heap.poll().getKey());  // reverse: best first
        return out;
    }
}
What production adds: learned embeddings instead of raw counts, ANN indexes instead of exact scoring, feature stores and A/B-tested ranking models, and freshness/decay so last year's binge doesn't dominate. But the request path is still recognizably: graph hop → hash count → filter → heap.

2. Google Maps & navigation

Roads become a weighted directed graph: intersections are nodes, road segments are edges, and the weight is expected travel time, continuously re-estimated from live traffic — ETA quality is a weight-estimation problem sitting on top of a shortest-path problem (page 17). Plain Dijkstra explores in every direction; A* adds an admissible heuristic (straight-line time to the destination) so the search leans toward the goal. Continent-scale routing precomputes shortcuts — contraction hierarchies — so a query touches thousands of nodes, not millions: the classic precompute-vs-query trade.

3. Git

Git is a content-addressed object store: every blob, tree and commit is named by the hash of its bytes (page 01 — the ultimate "group by canonical key": identical content dedupes itself). Commits form a DAG — each commit points at its parent(s) (pages 15/16); git log walks it in reverse topological order. git diff is the LCS problem from page 19: keep the longest common subsequence of lines, print the rest as +/−. A merge finds the two branches' nearest common ancestor commit — the LCA idea from page 11 lifted onto a DAG — and three-way-merges both sides' changes against that base.

4. Databases & search engines

A B-tree index is a fat, shallow BST tuned for disk pages: O(log n) point lookups and ordered range scans (pages 07/11), while a hash index is O(1) but can only answer equality — the HashMap-vs-TreeMap trade at terabyte scale (page 24). Write-heavy stores (Cassandra, RocksDB, LevelDB) use LSM trees instead: writes land in a sorted in-memory memtable, get flushed to immutable sorted files, and background compaction k-way-merges those files exactly like merge sort's merge step driven by a heap (pages 06/12). Each SSTable carries a bloom filter so reads skip files that definitely lack the key (page 24). Search engines invert the data: term → sorted posting list of document ids; a multi-term AND intersects sorted lists with two pointers (page 02). The query planner even chooses join orders with DP over subsets (page 20).

5. Web-scale infrastructure

Sharding asks "which server owns key k?" — naive hash(k) % n reshuffles nearly everything when n changes. Consistent hashing places servers and keys on a ring; a key belongs to the first server clockwise from it. In Java, that lookup is literally ring.ceilingEntry(hash) on a TreeMap, wrapping to firstEntry() — the page-24 pattern verbatim. Adding a node moves only its slice. In front: API gateways run token-bucket rate limiters per client (pages 03/24); load balancers pick the least-loaded backend from a heap (page 12); failed calls retry with exponential backoff — doubling, the page-07 halving intuition run backwards.

6. Social networks

The friend graph is an adjacency list keyed by user id. "People you may know" is a BFS bounded to 2–3 hops: friends-of-friends who aren't already friends, ranked by mutual-friend count — a HashMap tally over the frontier (pages 15/01). Degrees of separation is plain BFS depth. Community detection and fraud-ring discovery lean on union-find: accounts sharing devices, cards or addresses union into components, and a suspiciously large component is a signal (page 16). The feed itself is ranking: merge candidate posts and keep top-K by score with a heap (page 12), while "Trending" is a sliding window of hashtag counts over the last hours (page 03).

Production adds: the graph is sharded (TAO-style caches at Meta), suggestion candidates are precomputed offline, and ranking is a learned model — but its candidates still arrive via BFS-shaped jobs.

7. Schedulers & streaming

Everything that decides "what runs next" is a priority queue: OS run queues, Kubernetes pod scheduling (priority classes), thread-pool task queues (page 12). Timer systems — cron daemons, Netty's timeouts — keep the nearest deadline on top of a min-heap or in a timer wheel (a ring of buckets, page 09's circular queue with a clock hand). Kafka is at heart a set of giant per-partition FIFO queues with consumer offsets (page 09); keys are hashed to partitions (page 01), preserving order only within a partition. Stream processors (Flink/Spark) compute rolling aggregates with sliding windows (page 03) and cumulative metrics with prefix-sum-style running totals (page 04).

Exercises: map the system yourself

No LeetCode here — the capstone skill is decomposition. Sketch your answer before opening each solution.

sketch 1. Which structures power Uber-style driver matching?
sketch 2. Design the data path for Twitter/X "Trending in your country".
sketch 3. Map Spotify's Discover Weekly onto the recommendation funnel.
sketch 4. What makes Dropbox sync only the changed parts of a big file?
sketch 5. Which structures run a Ticketmaster-style seat-booking flow?
Answer sketches
  1. Driver matching: hash the map into geo-cells (grid = implicit graph, pages 01/05); candidate drivers from the rider's cell and neighbors; rank by ETA with a min-heap (12), where ETA is a shortest-path estimate on the road graph (17); surge pricing reads sliding-window demand counters (03).
  2. Trending: stream of hashtag events, hashed and counted per sliding window (01/03 — count-min sketches when exact maps don't fit); a size-K min-heap per region keeps the top K (12); re-rank with rules exactly like a recommender's last stage.
  3. Discover Weekly: candidate generation from co-listening similarity (01/15) plus audio-embedding ANN neighbors (LSH, extended 01); filter out tracks you already play with a seen-filter (24); rank with a model and take top ~30 via heap (12) — the funnel from section 1 verbatim.
  4. Dropbox sync: rsync-style content-defined chunking with a rolling hash (14) finds chunk boundaries that survive insertions; chunks are content-addressed by hash so unchanged chunks dedupe (01, same trick as Git); the folder tree is walked like any tree (11) and change notifications queue up FIFO (09).
  5. Seat booking: seat holds are intervals with expiry — a TreeMap per event for free/held ranges (24/21); expiring holds sit in a min-heap by deadline (12); the purchase queue under load is a FIFO with backpressure (09); per-user request caps are token buckets (24).
The closing move. Any system-design question you'll ever face maps onto these seven skeletons: a funnel (recommendations), a weighted graph search (maps), a hash-addressed DAG (git), an indexed store (databases), a hash ring with guards (infra), a social graph (BFS + union-find), or a scheduler (heaps + queues). Name the skeleton, then name the structure per stage — that's the whole game. Keep the master cheat-sheet open while you practice mocks.

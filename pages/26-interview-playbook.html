The Interview Playbook: One Problem, Many Algorithms

How to run the 45 minutes — and seven drills where a single problem is attacked with every reasonable algorithm, compared head-to-head, so choosing the right one becomes a reflex.

Phase 9 · Week 14 The synthesis page — read after the topics, revisit before every interview

Before you start

Why this page exists. The topic pages taught you the tools one at a time. Real problems don't arrive labeled — the actual interview skill is standing in front of an unlabeled problem, listing the two or three algorithms that could work, comparing their costs out loud, and picking one for a stated reason. This page trains exactly that: first the process for the 45 minutes, then seven "decision gym" drills where one problem is run through multiple algorithms with the pros, cons, and the factor that flips the verdict.

Prerequisites

Everything — this is the synthesis. Each drill links back to the pages where the individual solutions are fully worked.

Part 1 — Running the 45 minutes

Interviews are lost on process as often as on knowledge: diving into code before understanding the problem, optimizing silently, declaring "done" without testing. Use the same six phases every time, with rough timeboxes for a 45-minute slot:

PhaseTimeWhat you doWhat it sounds like
1. Clarify0–4 minRestate the problem; ask about sizes, ranges, duplicates, sortedness, output format. Write one worked example yourself."Can values be negative? How large is n? So for [3,1,2], k=2, the answer would be … — correct?"
2. Brute force4–8 minState the obvious solution and its cost. Don't code it — it's the baseline that proves you understand the problem."Brute force checks every pair: O(n²). With n up to 10⁵ that's 10¹⁰ operations — too slow, so let's improve."
3. Compare candidates8–14 minThe phase this page trains. Name 2–3 applicable algorithms, give each one's cost and drawback, pick one for a stated reason."A HashMap gives O(n) with O(n) memory; sorting plus two pointers gives O(n log n) with O(1) memory but scrambles indexes. We need indexes, so HashMap."
4. Code14–32 minCode the chosen approach top-to-bottom, narrating decisions. Name the invariant of any loop that isn't obvious."The stack holds indexes of days still waiting for a warmer day, temperatures decreasing."
5. Test32–39 minTrace your example through the actual code, line by line. Then probe edges: empty, size 1, duplicates, extremes."For [2,7], target 9: i=0, map empty, insert 2… i=1, need=2, found — returns [0,1]. Empty array? Loop doesn't run, we return the fallback."
6. Wrap up39–45 minState final time and space with one sentence of justification; volunteer the natural follow-up."O(n) time, O(n) space. If memory were tight and the array were sorted, two pointers would drop it to O(1)."

The failure modes that end interviews

FailureWhat it looks likeThe fix
Coding at minute 2Typing before the example is even settled; discovering the misread at minute 25Phase 1 is mandatory. A wrong problem solved perfectly scores zero.
Skipping the brute forceHunting the clever answer cold, freezing when it doesn't comeSay the brute force. It banks partial credit and its bottleneck usually points at the fix.
Silent thinking60 seconds of quiet — the interviewer can't grade a blankNarrate dead ends too: "sorting kills the index requirement, so that's out."
Optimizing the wrong axisMicro-tuning constants while the algorithm is still O(n²)Fix the complexity class first; constants never matter in interviews.
"Done" without a traceHanding over untested code with an off-by-onePhase 5 catches most bugs in two minutes. Never skip it.
Fighting the hintInterviewer nudges toward heaps; candidate keeps digging their holeHints are steering, not judgment. Take them immediately and visibly.
When you're stuck, say one of these — each restarts progress honestly: "Let me re-examine the constraints — n ≤ 20 suggests something exponential is intended." · "The brute force is slow because it recomputes X; what structure remembers X?" · "Let me try a smaller example and look for the pattern." · "I see two candidate approaches; let me cost them both quickly."

Part 2 — The decision gym: one problem, many algorithms

Each drill below is a real interview problem attacked with every reasonable algorithm. Read the problem, list your own candidates first, then open the comparison. The verdict line matters less than the flip factors — the input changes that would switch the winner. Full code for every approach lives on the linked pages.

drill 1 "Find a pair summing to target" — 3 algorithms

The problem. Given an array and a target, find two elements that sum to it (Two Sum, LC 1 / LC 167).

AlgorithmTimeSpaceProsCons
Brute-force pairsO(n²)O(1)trivial to write; fine for n ≤ ~10³too slow beyond that
HashMap complement (01)O(n)O(n)one pass; unsorted input; keeps original indexesO(n) memory; average-case bound
Sort + two pointers (02)O(n log n)O(1)no extra memory; generalizes to 3Sum/k-sumsorting destroys indexes; slower asymptotically

Verdict. Unsorted + indexes needed → HashMap. Already sorted, or O(1) memory demanded → two pointers.

Flip factors. Input arrives sorted → two pointers wins outright (LC 167 exists precisely to test this). Asked for all pairs / triples → sort first anyway (dedup needs adjacency), so two pointers. Streaming data → HashMap is the only one that works online.

drill 2 "Top-K elements" — 4 algorithms

The problem. Return the k largest (or most frequent) elements (LC 215 / LC 347 / LC 973).

AlgorithmTimeSpaceProsCons
Full sort (06)O(n log n)O(1)–O(n)two lines; output comes ordereddoes far more work than needed for small k
Size-k min-heap (12)O(n log k)O(k)works on streams; memory scales with k, not nlog k per element; result not fully sorted
Quickselect (06)O(n) avgO(1)fastest one-shot; in placemutates input; O(n²) worst case; needs all data in memory
Bucket by count (01)O(n)O(n)true linear time for frequency problemsonly works when the key (a count) is bounded by n

Verdict. One-shot on an array you may mutate → quickselect. Streaming or huge-n, small-k → heap. Frequency flavor → bucket. "I'll just sort" is acceptable only with the caveat said aloud.

Flip factors. k ≈ n → sorting is fine (the heap's advantage vanishes). Data doesn't fit in memory → heap is the only option. Need the k results in order → sort the k survivors afterwards (+O(k log k), usually negligible).

drill 3 "Count the connected groups" — 3 algorithms

The problem. Count connected components — islands in a grid, provinces in a matrix, linked accounts (LC 200 / LC 547 / LC 721).

AlgorithmTimeSpaceProsCons
DFS flood fill (15)O(V+E)O(V) stack worst caseleast code; natural on gridsrecursion depth can blow the stack on huge/snaky regions
BFS (15)O(V+E)O(V) queueno recursion risk; gives distances for freeslightly more code than DFS
Union-find (16)O(E·α)O(V)handles incremental edges; components merge in O(1); no traversalmore setup; overkill for a one-shot static graph

Verdict. Static one-shot graph → DFS (or BFS if depth worries you). Edges arriving over time with connectivity queries in between → union-find, no contest.

Flip factors. The follow-up "now islands appear one cell at a time — count after each" (LC 305) flips the verdict to union-find instantly: re-running DFS per addition is O(V) each, union-find is ~O(1). Grouping by shared attributes (Accounts Merge) → union-find, because "merge these two sets" is its native operation.

drill 4 "Cheapest path through a grid" — 4 algorithms (the screenshot problem)

The problem. Travel from the top-left to the bottom-right of a grid at minimum cost (LC 64 / LC 1091 / LC 1631). The movement rule and the cost model decide everything.

AlgorithmWhen it's the answerTimeFails when…
Grid DP (19)movement restricted (right/down only) — the grid is a DAGO(mn)free movement creates cycles: no valid fill order exists
BFS (17)free movement, every step costs the sameO(mn)weighted steps: layer count ≠ cost
Dijkstra (17)free movement, non-negative weightsO(mn log mn)negative weights (→ Bellman-Ford)
Minimize-the-max Dijkstra (17)path cost = worst single step (effort/elevation)O(mn log mn)— (binary-search+BFS and union-find also work here)

Verdict. Ask two questions in order: Which moves are legal? (restricted → DP; free → graph search) and What does a step cost? (uniform → BFS; weighted → Dijkstra; max-of-steps → the Dijkstra variant). Two questions classify every grid-path problem you'll ever see.

Flip factors. This drill IS the flip-factor lesson: LC 64 and LC 1091 look nearly identical and share zero algorithms. The movement rule — one line of the problem statement — is the entire decision. This was the confusion in the post that started this runbook; if this table reads as obvious now, that goal is met.

drill 5 "How many meeting rooms?" — 4 algorithms

The problem. Given meeting intervals, find the minimum number of rooms — i.e. the maximum overlap (LC 253, and Car Pooling LC 1094 is the same question).

AlgorithmTimeSpaceProsCons
Min-heap of end times (21)O(n log n)O(n)simulates room reuse directly — easy to extend to "which room"heap machinery for what's really just counting
Sweep line: sort +1/−1 events (21)O(n log n)O(n)minimal code; generalizes to any max-overlap questionloses track of which meeting is in which room
Difference array (04)O(n + R)O(R)O(1) per interval; beats sorting when coordinates are smallneeds a bounded coordinate range R (times of day: yes; arbitrary longs: no)
TreeMap of boundary deltas (24)O(n log n)O(n)handles unbounded coordinates AND online insertion (My Calendar III)most machinery of the four

Verdict. One-shot with arbitrary times → sweep line (least code) or heap (if a follow-up about room assignment is likely). Bounded integer coordinates → difference array. Bookings arriving online → TreeMap.

Flip factors. "Which room does each meeting get?" → heap (it names rooms; the counters don't). "Positions are 0..1000" → difference array turns the sort into an array walk. "Bookings stream in, report the max after each" → TreeMap deltas.

drill 6 "Sum of a range, many times" — 4 structures

The problem. Answer many sum(i..j) queries over an array (LC 303 / LC 307).

StructureQueryUpdateBuildUse when
Recompute per queryO(n)O(1)—a handful of queries; don't over-engineer
Prefix array (04)O(1)O(n) ✗O(n)array never changes — the immutable sweet spot
Fenwick / BIT (24)O(log n)O(log n)O(n log n)point updates + prefix-style (invertible) aggregates
Segment tree (24)O(log n)O(log n)O(n)non-invertible aggregates (min/max/gcd) or range updates

Verdict. The mutability question decides everything: immutable → prefix array; mutable with sums → Fenwick (half the code of a segment tree); mutable with min/max → segment tree, because you can't subtract a minimum back out.

Flip factors. "Now the array can change" is the exact flip LC 303 → LC 307 tests. "Range minimum instead of sum" kills Fenwick's prefix trick (min isn't invertible) → segment tree. "Only 10 queries total" → just loop; simplicity is also an engineering answer.

drill 7 "Best contiguous stretch" — window vs prefix-map vs DP

The problem. Find the best subarray/substring satisfying a condition — the family covering LC 3, LC 76, LC 560, LC 53.

AlgorithmTimeWorks whenBreaks when
Sliding window (03)O(n)validity is monotonic: growing the window only pushes one way (all-positive sums, distinct-char counts)negatives (or any non-monotonic measure) — shrinking might help OR hurt, so the greedy window logic is simply wrong
Prefix sum + HashMap (04)O(n)the condition is an equation on prefix values (sum == k, balance repeats)the condition is an inequality/optimization over unsorted prefixes
Kadane-style running DP (18)O(n)optimizing a value where "best ending here" has a recurrence (max sum, max product)counting all occurrences — DP carries one best, not a tally

Verdict. All three are O(n) — the choice is about correctness, not speed, which makes this the subtlest drill. Ask: is window-validity monotonic? Is the condition a prefix equation? Am I optimizing or counting?

Flip factors. "Subarray sum equals k, all positives" → window works. Add one negative number → window silently wrong, prefix-map required (LC 560's whole point). "Maximum sum" instead of "sum equals k" → Kadane. And if "substring" becomes "subsequence", the contiguity is gone — none of these apply; you're in DP-table territory (19).

The seven drills, compressed into the sentence you should say when each situation appears:

SituationThe sentence to say

Top-K"One-shot mutable → quickselect O(n); streaming or small k → size-k heap O(n log k); bounded counts → bucket O(n)."

How to practice this page. Don't reread it — drill it. Pick any problem from the earlier pages, and before opening the solution, write down: every algorithm that could plausibly apply, each one's time/space, one con each, your pick, and the input change that would flip your pick. Two minutes per problem. After twenty problems, phase 3 of the interview becomes the part you look forward to.

Pitfalls

  • Presenting one algorithm as the only option. Even when the best choice is obvious, naming the runner-up and why it loses is what demonstrates judgment rather than memorization.
  • Comparing on time alone. Half the verdicts above hinge on memory, mutation rights, streaming, or index preservation — read the constraints for all of them.
  • Ignoring the flip factor mid-interview. Follow-ups ("now the data streams in…", "now the array can change…") are usually a deliberate flip of your chosen algorithm. Expect the flip; it's the second half of the question.
  • Over-engineering small inputs. n ≤ 100 makes O(n²) instantly fine — saying "brute force is actually sufficient here, and simpler" is a strong answer, not a cop-out.
  • Letting the comparison eat the clock. Phase 3 is five minutes, spoken, no code. Compare, commit, and move — a finished good solution beats an unfinished perfect one.
